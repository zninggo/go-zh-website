介绍

数据竞争是并发系统中最常见且最难调试的错误类型之一。 当两个 goroutine（协程）并发地访问同一个变量，且至少有一个访问是写操作时，就会发生数据竞争。 详情请参阅 Go 内存模型。

下面是一个可能导致程序崩溃和内存损坏的数据竞争示例： 

func main() {
	c := make(chan bool)
	m := make(map[string]string)
	go func() {
		m["1"] = "a" // 第一次冲突访问。
		c <- true
	}()
	m["2"] = "b" // 第二次冲突访问。
	<-c
	for k, v := range m {
		fmt.Println(k, v)
	}
}

使用方法

为了帮助诊断此类错误，Go 语言内置了一个数据竞争检测器。 要使用它，只需在 go 命令中添加 -race 标志： 

$ go test -race mypkg    // 测试包
$ go run -race mysrc.go  // 运行源文件
$ go build -race mycmd   // 构建命令
$ go install -race mypkg // 安装包

选项

环境变量 GORACE 用于设置竞争检测器的选项。 格式如下： 

GORACE="option1=val1 option2=val2"

可用选项如下：

  • log_path（默认值 stderr）：竞争检测器会将其报告写入名为 log_path.pid 的文件中。 特殊名称 stdout 和 stderr 分别表示将报告写入标准输出和标准错误。
  • exitcode（默认值 66）：检测到竞争后程序退出时使用的退出状态码。
  • strip_path_prefix（默认值 ""）：从所有报告的文件路径中剥离此前缀，以使报告更简洁。
  • history_size（默认值 1）：每个 goroutine（协程）的内存访问历史大小为 32K * 2**history_size 个元素。 增加此值可以避免报告中出现“failed to restore the stack”错误，但会增加内存使用量。
  • halt_on_error（默认值 0）：控制程序在报告第一个数据竞争后是否退出。
  • atexit_sleep_ms（默认值 1000）：在主 goroutine（协程）退出前休眠的毫秒数。

示例： 

$ GORACE="log_path=/tmp/race/report strip_path_prefix=/my/go/sources/" go test -race

排除测试

当你使用 -race 标志进行构建时，go 命令会定义一个额外的 构建标签 race。 你可以使用此标签在运行竞争检测器时排除某些代码和测试。 一些示例： 

// +build !race

package foo

// 此测试包含数据竞争。参见 issue 123。
func TestFoo(t *testing.T) {
	// ...
}

// 此测试在竞争检测器下由于超时而失败。
func TestBar(t *testing.T) {
	// ...
}

// 此测试在竞争检测器下运行时间过长。
func TestBaz(t *testing.T) {
	// ...
}

如何使用

首先，使用竞争检测器运行你的测试（go test -race）。 竞争检测器只能发现运行时发生的数据竞争，因此无法发现未执行的代码路径中的竞争。 如果你的测试覆盖不完整， 你可以通过在实际工作负载下运行使用 -race 构建的二进制文件来发现更多数据竞争。

典型数据竞争

以下是一些典型的数据竞争示例。所有这些都可以被竞争检测器检测到。

循环计数器上的竞争

func main() {
	var wg sync.WaitGroup
	wg.Add(5)
	var i int
	for i = 0; i < 5; i++ {
		go func() {
			fmt.Println(i) // 这里的 'i' 可能不是你期望的那个。
			wg.Done()
		}()
	}
	wg.Wait()
}

函数字面量中的变量 i 与循环使用的变量是同一个， 因此 goroutine（协程）中的读取操作与循环的递增操作存在竞争。 （此程序通常会打印 55555，而不是 01234。） 可以通过复制变量来修复该程序： 

func main() {
	var wg sync.WaitGroup
	wg.Add(5)
	var i int
	for i = 0; i < 5; i++ {
		go func(j int) {
			fmt.Println(j) // 很好。读取循环计数器的本地副本。
			wg.Done()
		}(i)
	}
	wg.Wait()
}

意外共享的变量

// ParallelWrite 将数据写入 file1 和 file2，返回错误。
func ParallelWrite(data []byte) chan error {
	res := make(chan error, 2)
	f1, err := os.Create("file1")
	if err != nil {
		res <- err
	} else {
		go func() {
			// 这个 err 与主 goroutine 共享，
			// 因此这里的写入与下面的写入存在竞争。
			_, err = f1.Write(data)
			res <- err
			f1.Close()
		}()
	}
	f2, err := os.Create("file2") // 第二个冲突的对 err 的写入。
	if err != nil {
		res <- err
	} else {
		go func() {
			_, err = f2.Write(data)
			res <- err
			f2.Close()
		}()
	}
	return res
}

修复方法是在 goroutine 中引入新变量（注意使用 :=）： 

			...
			_, err := f1.Write(data)
			...
			_, err := f2.Write(data)
			...

未受保护的全局变量

如果以下代码从多个 goroutine 调用，会导致对 service map 的竞争。 对同一 map 的并发读写是不安全的： 

var service map[string]net.Addr

func RegisterService(name string, addr net.Addr) {
	service[name] = addr
}

func LookupService(name string) net.Addr {
	return service[name]
}

为使代码安全，应使用互斥锁保护访问： 

var (
	service   map[string]net.Addr
	serviceMu sync.Mutex
)

func RegisterService(name string, addr net.Addr) {
	serviceMu.Lock()
	defer serviceMu.Unlock()
	service[name] = addr
}

func LookupService(name string) net.Addr {
	serviceMu.Lock()
	defer serviceMu.Unlock()
	return service[name]
}

未受保护的基本类型变量

数据竞争也可能发生在基本类型变量上（如 bool、int、int64 等），如下例所示： 

type Watchdog struct{ last int64 }

func (w *Watchdog) KeepAlive() {
	w.last = time.Now().UnixNano() // 第一个冲突的访问。
}

func (w *Watchdog) Start() {
	go func() {
		for {
			time.Sleep(time.Second)
			// 第二个冲突的访问。
			if w.last < time.Now().Add(-10*time.Second).UnixNano() {
				fmt.Println("No keepalives for 10 seconds. Dying.")
				os.Exit(1)
			}
		}
	}()
}

即使是这种"无害"的数据竞争，也可能由于内存访问的非原子性、编译器优化的干扰，或访问处理器内存时的重排序问题而导致难以调试的问题。

此竞争的典型修复方法是使用 channel 或互斥锁。 为了保持无锁行为，也可以使用 sync/atomic 包。 

type Watchdog struct{ last int64 }

func (w *Watchdog) KeepAlive() {
	atomic.StoreInt64(&w.last, time.Now().UnixNano())
}

func (w *Watchdog) Start() {
	go func() {
		for {
			time.Sleep(time.Second)
			if atomic.LoadInt64(&w.last) < time.Now().Add(-10*time.Second).UnixNano() {
				fmt.Println("No keepalives for 10 seconds. Dying.")
				os.Exit(1)
			}
		}
	}()
}

未同步的发送和关闭操作

如本例所示，对同一 channel 的未同步发送和关闭操作也可能构成竞态条件： 

c := make(chan struct{}) // 或带缓冲的 channel

// 竞态检测器无法推断以下发送和关闭操作之间的 happens-before 关系。
// 这两个操作未同步且并发发生。
go func() { c <- struct{}{} }()
close(c)

根据 Go 内存模型，对 channel 的发送操作在该 channel 上相应的接收操作完成之前发生。 要同步发送和关闭操作，请使用保证在关闭前完成发送的接收操作： 

c := make(chan struct{}) // 或带缓冲的 channel

go func() { c <- struct{}{} }()
<-c
close(c)

要求

竞态检测器需要启用 cgo，并且在非 Darwin 系统上需要安装 C 编译器。 竞态检测器支持 linux/amd64、linux/ppc64le、 linux/arm64、linux/s390x、 linux/loong64、freebsd/amd64、 netbsd/amd64、darwin/amd64、 darwin/arm64 和 windows/amd64。

在 Windows 上，竞态检测器运行时对已安装的 C 编译器版本敏感； 从 Go 1.21 开始，使用 -race 构建程序需要一个包含了 mingw-w64 运行时库版本 8 或更高版本的 C 编译器。您可以通过调用 --print-file-name libsynchronization.a 参数来测试您的 C 编译器。 较新的兼容 C 编译器将打印该库的完整路径，而较旧的 C 编译器只会回显该参数。

运行时开销

竞态检测的成本因程序而异，但对于典型程序，内存使用量可能增加 5-10 倍，执行时间可能增加 2-20 倍。

竞态检测器目前为每个 defer 和 recover 语句额外分配 8 字节。 这些额外的分配 直到 goroutine 退出才被回收。 这意味着，如果您有一个长期运行的 goroutine 周期性地发出 defer 和 recover 调用， 程序的内存使用量可能会无限增长。这些内存分配不会出现在 runtime.ReadMemStats 或 runtime/pprof 的输出中。
