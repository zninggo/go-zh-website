Go 发行包中包含一个名为 "go" 的命令， 该命令能够自动完成 Go 包（packages）和命令（commands）的下载、构建、安装和测试。 本文将介绍我们编写这个新命令的原因、它的功能边界以及使用方法。

设计初衷

您可能看过早期 Go 技术讲座中 Rob Pike 的玩笑话：Go 语言的灵感诞生于等待一个大型 Google 服务器编译完成的漫长过程。 这确实是 Go 诞生的真实动机：创建一门能够良好支撑 Google 编写和运行的大型软件开发的语言。 从一开始我们就明确，这样的语言必须提供清晰表达代码库间依赖关系的方式，因此引入了包（package）分组和显式的导入（import）代码块。 同样从一开始就清楚，您可能需要任意语法来描述导入的代码；这就是为什么导入路径是字符串字面量（string literal）。

Go 从诞生之初就有一个明确目标：仅依靠源代码本身包含的信息来构建 Go 程序，无需开发者编写 makefile 或任何 makefile 的现代替代品。 如果 Go 需要配置文件来说明如何构建程序，那么它就已经失败了。

最初，并没有 Go 编译器，早期开发的重点是先构建编译器，再为其构建库。 为了快速推进，我们暂时使用 make 和编写 makefile 的方式来自动化构建 Go 代码的过程。 当编译单个包需要多次调用 Go 编译器时，我们甚至用程序来自动生成 makefile。 如果您查阅代码库的历史记录，应该还能找到这个程序。

新 go 命令的目的正是回归这一理想：Go 程序的编译过程不应依赖任何配置，除了编写必要的导入（import）语句外，开发者无需额外交付。

配置与约定

实现无需配置系统的简洁性，在于建立并遵循约定。 系统的有效性完全依赖于这些约定的遵守。 Go 首次发布时，许多人发布的包必须安装在特定位置、使用特定名称、通过特定构建工具才能使用。 这可以理解：这是大多数其他语言的工作方式。 过去几年，我们不断提醒用户关于 goinstall 命令 （现已由 go get 取代） 及其约定：第一，导入路径（import path）以已知方式从源代码的 URL 派生； 第二，在本地文件系统中存储源代码的位置以已知方式从导入路径派生； 第三，源代码树中的每个目录对应一个包（package）； 第四，包的构建仅依赖源代码中的信息。 如今，绝大多数包都遵循了这些约定。 这使得 Go 生态系统更简单、更强大。

我们收到过许多请求，希望允许在包目录中放置 makefile 来提供超出源代码的少量额外配置。 但这会引入新的规则。 由于我们没有应允此类请求，才得以编写 go 命令，并彻底摆脱了对 make 或其他构建系统的依赖。

需要理解的是，go 命令不是一个通用的构建工具。 它无法被配置，也只尝试构建 Go 包（packages）。 这些是重要的简化假设：它们不仅简化了实现，更重要的是简化了工具本身的使用。

Go 的约定

go 命令要求代码遵循一些关键且成熟的约定。

首先，导入路径（import path）以已知方式从源代码的 URL 派生。 对于 Bitbucket、GitHub、Google Code 和 Launchpad，代码仓库的根目录由仓库的主 URL 标识（不含 https:// 前缀）。 子目录则通过在该路径后添加名称来命名。 例如，Google 日志包 glog 的源代码通过运行以下命令获取：

git clone https://github.com/golang/glog
因此， glog 包的导入路径为 "github.com/golang/glog"。

这些路径虽然较长，但换来了一个自动管理的导入路径命名空间，以及像 go 命令这样的工具能够通过未知导入路径推断出源代码获取位置的能力。

其次，源代码在本地文件系统中的存储位置以已知方式从导入路径派生，具体为 $GOPATH/src/<import-path>。 如果未设置 $GOPATH，则默认为用户主目录下的 go 子目录。 如果 $GOPATH 设置为路径列表，go 命令会尝试列表中每个目录的 <dir>/src/<import-path>。

根据约定，每个这样的目录树都包含一个名为 "bin" 的顶级目录（用于存放编译后的可执行文件）、一个名为 "pkg" 的顶级目录（用于存放可导入的已编译包），以及 "src" 目录（用于存放包源文件）。 这种结构使得每个目录树都是自包含的：编译产物和源代码总是在相邻位置。

这些命名约定也支持反向操作：从目录名推导出其导入路径。 这种映射对于 go 命令的许多子命令至关重要，下文将会看到。

第三，源代码树中的每个目录都对应一个单独的包（package）。通过将目录限制为单一包，我们无需创建混合导入路径——这种路径需要先指定目录再指定该目录内的包。此外，大多数文件管理工具和用户界面都以目录作为基本操作单元。将 Go 的核心单元——包——与文件系统结构绑定，意味着文件系统工具也能成为 Go 包管理工具。复制、移动或删除一个包，等同于复制、移动或删除一个目录。

第四，每个包仅使用源文件中包含的信息进行构建。这使得工具更有可能适应不断变化的构建环境和条件。例如，如果我们允许额外的配置（如编译器标志或命令行指令），那么每当构建工具更新时，这些配置也需要同步更新；并且这种配置本质上与特定工具链的使用绑定。

go 命令入门

最后，简要介绍如何使用 go 命令。 如前所述，Unix 系统上默认的 $GOPATH 路径是 $HOME/go。 我们将在此存放程序。 如需使用其他位置，可设置 $GOPATH 环境变量；详情请参阅《如何编写 Go 代码》。

首先添加一些源代码。假设我们需要使用 codesearch 项目中的索引库以及一个左倾红黑树。可以通过 "go get" 子命令同时安装两者：

$ go get github.com/google/codesearch/index
$ go get github.com/petar/GoLLRB/llrb
$

这两个项目现已下载并安装到 $HOME/go 目录下，其中包含 src/github.com/google/codesearch/index/ 和 src/github.com/petar/GoLLRB/llrb/ 两个目录， 以及这些库及其依赖项的编译包（位于 pkg/ 目录）。

由于我们使用版本控制系统（Mercurial 和 Git）检出源码，源代码树还包含对应仓库中的其他文件（如相关包）。"go list" 子命令可列出其参数对应的导入路径，其中模式 "./..." 表示从当前目录（"./"）开始查找其下的所有包（"..."）：

$ cd $HOME/go/src
$ go list ./...
github.com/google/codesearch/cmd/cgrep
github.com/google/codesearch/cmd/cindex
github.com/google/codesearch/cmd/csearch
github.com/google/codesearch/index
github.com/google/codesearch/regexp
github.com/google/codesearch/sparse
github.com/petar/GoLLRB/example
github.com/petar/GoLLRB/llrb
$

我们还可以测试这些包：

$ go test ./...
?   	github.com/google/codesearch/cmd/cgrep	[无测试文件]
?   	github.com/google/codesearch/cmd/cindex	[无测试文件]
?   	github.com/google/codesearch/cmd/csearch	[无测试文件]
ok  	github.com/google/codesearch/index	0.203s
ok  	github.com/google/codesearch/regexp	0.017s
?   	github.com/google/codesearch/sparse	[无测试文件]
?       github.com/petar/GoLLRB/example          [无测试文件]
ok      github.com/petar/GoLLRB/llrb             0.231s
$

如果 go 子命令在未指定路径的情况下调用，它将在当前目录下执行操作：

$ cd github.com/google/codesearch/regexp
$ go list
github.com/google/codesearch/regexp
$ go test -v
=== RUN   TestNstateEnc
--- PASS: TestNstateEnc (0.00s)
=== RUN   TestMatch
--- PASS: TestMatch (0.00s)
=== RUN   TestGrep
--- PASS: TestGrep (0.00s)
PASS
ok  	github.com/google/codesearch/regexp	0.018s
$ go install
$

"go install" 子命令会将包的最新版本安装到 pkg 目录中。由于 go 命令能分析依赖图，"go install" 还会递归安装此包导入但已过期的所有依赖包。

请注意，"go install" 能够根据目录命名约定自动确定当前目录中包的导入路径名称。虽然如果能自定义存放源代码的目录名称会更方便（我们可能不会选择这么长的名称），但这需要工具额外的配置和复杂性。多输入一两个目录名，换来的是更简洁强大的特性，这点代价是值得的。

局限性

如前所述，go 命令并非通用构建工具。 特别是它不具备在构建过程中生成 Go 源文件的功能，尽管它提供了 go generate 工具，可以在构建前自动创建 Go 文件。 对于更高级的构建需求，你可能需要编写 makefile（或自选构建工具的配置文件）来运行生成 Go 文件的工具，然后将这些生成的源文件提交到代码仓库。这对包作者而言虽然增加了工作量，但对用户而言却大大简化了流程——他们只需使用 "go get" 即可，无需获取和构建额外工具。
