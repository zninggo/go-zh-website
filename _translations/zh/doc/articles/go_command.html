Go 语言发行版中包含一个名为 "go" 的命令，它能够自动完成 Go 包和命令的下载、构建、安装和测试。本文将探讨我们为何编写这个新命令、它是什么、不是什么，以及如何使用它。

开发动机

你可能看过早期的 Go 语言演讲，其中 Rob Pike 开玩笑说，Go 的构想是在等待一个大型 Google 服务器编译时产生的。这确实就是 Go 语言的开发动机：构建一种能良好支持 Google 编写和运行的大型软件的语言。从一开始就明确，这样的语言必须提供一种清晰表达代码库间依赖关系的方式，因此引入了包（package）的分组和显式的导入（import）块。同样从一开始就明确，你可能希望用任意语法描述被导入的代码；这就是为什么导入路径是字符串字面量（string literal）。

Go 语言最初的一个明确目标是，仅利用源码本身包含的信息就能构建 Go 程序，无需编写 makefile 或众多现代 makefile 替代工具。如果 Go 语言需要一个配置文件来解释如何构建你的程序，那么 Go 就失败了。

起初，并没有 Go 编译器，最初的开发工作集中在构建一个编译器，然后为其构建库。为了方便起见，我们推迟了构建 Go 代码自动化的实现，转而使用 make 并编写 makefile。当编译一个单独的包需要多次调用 Go 编译器时，我们甚至使用了一个程序来为我们生成 makefile。你可以在仓库历史记录中找到它。

新 go 命令的目的是回归这个理想：Go 程序应该无需配置或开发人员付出额外努力（除了编写必要的导入语句）即可编译。

配置与约定

实现无需配置的系统简洁性的方法是建立约定。该系统只有在遵循这些约定的情况下才能工作。当我们首次推出 Go 时，许多人发布的包必须安装在特定位置、使用特定名称、使用特定的构建工具才能使用。这是可以理解的：在大多数其他语言中就是这样工作的。在过去的几年里，我们一直提醒人们关于 goinstall 命令（现已被 go get 取代）及其约定：首先，导入路径以一种已知的方式从源代码的 URL 派生；其次，源代码在本地文件系统中的存储位置以一种已知的方式从导入路径派生；第三，源代码树中的每个目录对应一个单独的包；第四，包仅使用源代码中的信息构建。如今，绝大多数包都遵循这些约定。Go 生态系统因此变得更简单、更强大。

我们收到了许多请求，希望允许在包目录中使用 makefile 来提供源代码之外的一点点额外配置。但这会引入新的规则。因为我们没有同意这些请求，所以我们能够编写 go 命令并消除对 make 或任何其他构建系统的使用。

需要理解的是，go 命令不是一个通用的构建工具。它不可配置，并且除了构建 Go 包之外，它不尝试构建任何其他东西。这些都是重要的简化假设：它们不仅简化了实现，更重要的是，也简化了工具本身的使用。

Go 的约定

go 命令要求代码遵守一些关键的、广为接受的约定。

首先，导入路径以一种已知的方式从源代码的 URL 派生。对于 Bitbucket、GitHub、Google Code 和 Launchpad，仓库的根目录由其主 URL 标识，不带 https:// 前缀。子目录通过向该路径追加来命名。例如，Google 日志包 glog 的源代码通过运行以下命令获取

git clone https://github.com/golang/glog
因此，glog 包的导入路径是 "github.com/golang/glog"。

这些路径比较长，但作为交换，我们获得了一个自动管理的导入路径命名空间，以及像 go 命令这样的工具能够查看不熟悉的导入路径并推断出从何处获取源代码的能力。

其次，源代码在本地文件系统中的存储位置以一种已知的方式从导入路径派生，具体为 $GOPATH/src/<import-path>。如果未设置，$GOPATH 默认为用户主目录中名为 go 的子目录。如果 $GOPATH 设置为一个路径列表，go 命令会依次尝试每个目录下的 <dir>/src/<import-path>。

根据约定，这些目录树中的每一个都包含一个名为 "bin" 的顶级目录（用于存放编译后的可执行文件）、一个名为 "pkg" 的顶级目录（用于存放可导入的已编译包）以及 "src" 目录（用于存放包源文件）。强制这种结构使我们能够保持每个目录树自包含：编译形式和源文件总是在一起。

这些命名约定也让我们能够反向工作，从目录名得到其导入路径。这种映射对于 go 命令的许多子命令很重要，我们将在下面看到。

第三，源码树中的每个目录对应一个单独的包。通过限制每个目录仅对应一个包，我们无需创建混合导入路径——那种既指定目录又指定该目录内特定包的路径。此外，大多数文件管理工具和用户界面都以目录作为基本操作单元。将Go语言的基本单位——包——与文件系统结构绑定，意味着文件系统工具即可成为Go包管理工具。复制、移动或删除一个包，就等同于复制、移动或删除一个目录。

第四，每个包的构建仅依赖其源文件中存在的信息。这使得工具更有可能适应不断变化的构建环境和条件。例如，如果我们允许额外的配置（如编译器标志或命令行指令），那么每次构建工具变更时都需要更新该配置；并且这种配置天生就与特定工具链的使用绑定在一起。

开始使用go命令

最后，快速介绍如何使用go命令。如上所述，在Unix系统上默认的 $GOPATH 是 $HOME/go。我们将把程序存储在那里。如需使用其他位置，你可以设置 $GOPATH；详情请参阅《如何编写Go代码》。

首先我们添加一些源代码。假设我们要使用codesearch项目中的索引库以及一个左倾红黑树。我们可以通过 "go get" 子命令同时安装两者：

$ go get github.com/google/codesearch/index
$ go get github.com/petar/GoLLRB/llrb
$

这两个项目现在已被下载并安装到 $HOME/go 中，该目录包含 src/github.com/google/codesearch/index/ 和 src/github.com/petar/GoLLRB/llrb/ 这两个子目录，以及这些库及其依赖项的已编译包（位于 pkg/ 目录中）。

由于我们使用了版本控制系统（Mercurial和Git）来检出源码，源码树中也包含了相应仓库中的其他文件，例如相关的包。"go list" 子命令会列出其参数对应的导入路径，而模式 "./..." 表示从当前目录（"./"）开始，查找该目录下的所有包（"..."）：

$ cd $HOME/go/src
$ go list ./...
github.com/google/codesearch/cmd/cgrep
github.com/google/codesearch/cmd/cindex
github.com/google/codesearch/cmd/csearch
github.com/google/codesearch/index
github.com/google/codesearch/regexp
github.com/google/codesearch/sparse
github.com/petar/GoLLRB/example
github.com/petar/GoLLRB/llrb
$

我们也可以测试这些包：

$ go test ./...
?   	github.com/google/codesearch/cmd/cgrep	[no test files]
?   	github.com/google/codesearch/cmd/cindex	[no test files]
?   	github.com/google/codesearch/cmd/csearch	[no test files]
ok  	github.com/google/codesearch/index	0.203s
ok  	github.com/google/codesearch/regexp	0.017s
?   	github.com/google/codesearch/sparse	[no test files]
?       github.com/petar/GoLLRB/example          [no test files]
ok      github.com/petar/GoLLRB/llrb             0.231s
$

如果调用go子命令时未列出路径，则它会在当前目录上运行：

$ cd github.com/google/codesearch/regexp
$ go list
github.com/google/codesearch/regexp
$ go test -v
=== RUN   TestNstateEnc
--- PASS: TestNstateEnc (0.00s)
=== RUN   TestMatch
--- PASS: TestMatch (0.00s)
=== RUN   TestGrep
--- PASS: TestGrep (0.00s)
PASS
ok  	github.com/google/codesearch/regexp	0.018s
$ go install
$

这个 "go install" 子命令会将包的最新副本安装到pkg目录中。由于go命令能够分析依赖关系图，"go install" 还会递归地安装该包导入但已过时的任何依赖包。

请注意，"go install" 能够根据目录命名的约定确定当前目录中包的导入路径名称。如果我们能自由选择存放源代码的目录名称会更方便些，而且我们可能也不会选择这么长的名字，但这种功能将需要工具进行额外的配置并增加复杂性。多输入一两个目录名称，是为换取更高的简洁性和强大功能所付出的微小代价。

局限性

如前所述，go命令并非通用构建工具。特别是，它没有任何在构建过程中生成Go源文件的功能，尽管它确实提供了go generate，该命令可以在构建之前自动创建Go文件。对于更高级的构建设置，你可能需要编写Makefile（或你所选构建工具的配置文件）来运行创建Go文件的工具，然后将这些生成的源文件签入你的代码仓库。这对你——包作者——来说工作量更大，但对你的用户来说则轻松很多，他们可以使用 "go get" 而无需获取并构建任何额外工具。
