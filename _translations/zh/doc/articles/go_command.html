Go 语言发行版中包含一个名为 "go" 的命令，它可以自动完成 Go 包和命令的下载、构建、安装与测试。本文将探讨我们为何编写这个新命令，它的功能定位、非设计目标，以及如何使用它。

设计初衷

你可能在早期的 Go 技术演讲中看到过 Rob Pike 的玩笑，说 Go 的灵感源于等待大型谷歌服务器编译时的漫长等待。这确实是 Go 诞生的核心动机：构建一门能高效构建谷歌所编写和运行的大型软件的语言。从一开始就很明确，这样的语言必须能清晰表达代码库之间的依赖关系，因此有了包的分组和显式的导入代码块。同样明确的是，你可能需要任意语法来描述被导入的代码，这就是为什么导入路径是字符串字面量。

Go 语言从诞生之初就有一个明确目标：仅凭源代码本身的信息就能构建 Go 代码，无需编写 Makefile 或任何其他现代构建系统的配置文件。如果 Go 需要配置文件来说明如何构建程序，那么 Go 就算失败了。

最初，Go 编译器尚未出现，早期开发集中在构建编译器和相应的库。为了便捷起见，我们曾使用 make 并编写 Makefile 来推迟构建过程的自动化。当编译单个包需要多次调用 Go 编译器时，我们甚至用程序自动生成 Makefile——你可以在仓库历史记录中找到它。

新 go 命令的诞生，正是我们回归这一理想的体现：Go 程序的编译不应依赖额外的配置，开发者只需编写必要的导入语句即可。

配置与约定

实现免配置系统的关键在于建立约定。系统的有效性完全取决于这些约定是否被遵守。Go 语言刚发布时，许多发布的包要求必须安装在特定路径下、使用特定名称、依赖特定构建工具才能使用。这可以理解：大多数其他语言就是这样运作的。过去几年里，我们一直提醒用户使用 goinstall 命令（现已被 go get 取代）并遵循其约定：首先，导入路径需通过已知规则从源代码 URL 推导；其次，本地文件系统中的存储路径需通过已知规则从导入路径推导；再次，源码树中的每个目录对应一个包；最后，包的构建仅依赖源代码中的信息。如今，绝大多数包都遵循这些约定，Go 生态因此变得更简洁强大。

我们曾收到许多请求，希望允许包目录中的 Makefile 提供少量额外配置。但这会引入新的规则。正因为我们没有接受这类请求，才得以编写出 go 命令，彻底摆脱了对 make 或其他构建系统的依赖。

需要理解的是，go 命令并非通用构建工具。它不可配置，也仅用于构建 Go 包。这是重要的简化假设：不仅简化了实现，更重要的是简化了工具本身的使用。

Go 的约定规范

go 命令要求代码遵循一些关键且成熟的约定。

第一，导入路径需通过已知规则从源代码 URL 推导。对于 Bitbucket、GitHub、Google Code 和 Launchpad 平台，仓库根目录通过主 URL 标识（不包含 https:// 前缀），子目录则通过追加路径命名。例如，Google 日志包 glog 的源代码可通过以下命令获取：

git clone https://github.com/golang/glog

因此，glog 包的导入路径为 "github.com/golang/glog"。

这些路径虽然较长，但换来了自动管理的导入路径命名空间，以及像 go 命令这样的工具能够通过陌生导入路径推断源代码获取位置的能力。

第二，本地文件系统中的源代码存储路径需通过已知规则从导入路径推导，具体为 $GOPATH/src/<import-path>。若未设置，$GOPATH 默认为用户主目录下的 go 子目录。若 $GOPATH 设置为路径列表，go 命令会依次尝试 <dir>/src/<import-path> 路径。

按照约定，每个目录树包含顶层目录 "bin"（存放编译后的可执行文件）、顶层目录 "pkg"（存放可导入的编译包）以及 "src" 目录（存放包源代码）。这种结构使每个目录树自成体系：编译产物与源代码始终保持相邻。

这些命名约定还支持反向推导：从目录名称推导出导入路径。这种映射关系对 go 命令的许多子命令至关重要，下文将具体说明。

第三，源代码树中的每个目录对应一个单独的包。通过将目录限制为单个包，我们无需创建混合导入路径——先指定目录再指定目录内的包。此外，大多数文件管理工具和用户界面都以目录作为基本工作单位。将 Go 语言的基本单元——包——与文件系统结构绑定，意味着文件系统工具可以直接作为 Go 包管理工具使用。复制、移动或删除一个包，就等同于复制、移动或删除一个目录。

第四，每个包仅使用源代码文件中包含的信息进行构建。这使得工具更有可能适应不断变化的构建环境和条件。例如，如果我们允许使用额外的配置（如编译器标志或命令行指令），那么每次构建工具更新时都需要同步更新这些配置；同时，这也会使得配置与特定工具链的使用产生固有绑定。

go 命令入门指南

最后，简要介绍如何使用 go 命令。如前所述，Unix 系统上默认的 $GOPATH 是 $HOME/go，我们将把程序存储在此处。若要使用其他路径，可以设置 $GOPATH；详情请参阅《如何编写 Go 代码》。

我们首先添加一些源代码。假设我们要使用 codesearch 项目中的索引库以及一个左倾红黑树。可以通过 "go get" 子命令同时安装这两个库：

$ go get github.com/google/codesearch/index
$ go get github.com/petar/GoLLRB/llrb
$

这两个项目现在已下载并安装到 $HOME/go 目录中，该目录包含 src/github.com/google/codesearch/index/ 和 src/github.com/petar/GoLLRB/llrb/ 两个子目录，以及这些库及其依赖项对应的已编译包（位于 pkg/ 目录）。

由于我们使用了版本控制系统（Mercurial 和 Git）来检出源代码，源代码树中还包含相应仓库中的其他文件，例如相关包。"go list" 子命令会列出其参数对应的导入路径，而模式 "./..." 表示从当前目录（"./"）开始查找该目录下的所有包（"..."）：

$ cd $HOME/go/src
$ go list ./...
github.com/google/codesearch/cmd/cgrep
github.com/google/codesearch/cmd/cindex
github.com/google/codesearch/cmd/csearch
github.com/google/codesearch/index
github.com/google/codesearch/regexp
github.com/google/codesearch/sparse
github.com/petar/GoLLRB/example
github.com/petar/GoLLRB/llrb
$

我们也可以测试这些包：

$ go test ./...
?   	github.com/google/codesearch/cmd/cgrep	[无测试文件]
?   	github.com/google/codesearch/cmd/cindex	[无测试文件]
?   	github.com/google/codesearch/cmd/csearch	[无测试文件]
ok  	github.com/google/codesearch/index	0.203秒
ok  	github.com/google/codesearch/regexp	0.017秒
?   	github.com/google/codesearch/sparse	[无测试文件]
?       github.com/petar/GoLLRB/example          [无测试文件]
ok      github.com/petar/GoLLRB/llrb             0.231秒
$

如果执行 go 子命令时未列出任何路径，它将作用于当前目录：

$ cd github.com/google/codesearch/regexp
$ go list
github.com/google/codesearch/regexp
$ go test -v
=== RUN   TestNstateEnc
--- PASS: TestNstateEnc (0.00秒)
=== RUN   TestMatch
--- PASS: TestMatch (0.00秒)
=== RUN   TestGrep
--- PASS: TestGrep (0.00秒)
PASS
ok  	github.com/google/codesearch/regexp	0.018秒
$ go install
$

"go install" 子命令会将包的最新副本安装到 pkg 目录中。由于 go 命令能够分析依赖关系图，"go install" 还会递归地安装该包所导入的、但已过时的任何依赖包。

请注意，"go install" 能够通过目录命名约定确定当前目录中包的导入路径。如果我们能自由选择存放源代码的目录名称可能会更方便（我们可能也不会选择这么长的名字），但这种能力需要在工具中引入额外的配置和复杂性。多输入一两个目录名，换来的是工具更简洁强大，这个代价是值得的。

局限性

如前所述，go 命令并非通用构建工具。特别是，它不具备在构建过程中生成 Go 源代码文件的功能，尽管它提供了go generate命令，可以在构建之前自动化创建 Go 文件。对于更高级的构建设置，您可能需要编写 makefile（或您选择的构建工具的配置文件）来运行生成 Go 文件的工具，然后将生成的源文件提交到您的版本库。这会给包作者带来额外工作，但对您的用户而言则大大减轻了负担——他们只需使用 "go get" 即可，无需获取和构建任何额外工具。
