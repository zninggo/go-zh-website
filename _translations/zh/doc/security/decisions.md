## 概述 {#overview}

本文档包含 Go 安全团队就各类常见报告问题所做出的决定。它主要作为一份参考，列出了我们认为不属于漏洞的情况。

此列表并非详尽无遗。

## 漏洞 {#vulns}

### 远程代码执行 {#rce}

如果某个场景允许攻击者在不期望代码执行的情况下执行代码，这将属于**私有跟踪**漏洞。此决定优先于所有其他决定。

此决定不涵盖那些本应执行代码的函数。

**涵盖的场景**：

- 一个解析函数在处理恶意输入时执行了代码。
- 一个恶意请求导致 HTTP 服务器执行了代码。

**不涵盖的场景**：

- `go test` 命令会运行测试。在攻击者控制的测试上运行 `go test` 不在我们的威胁模型范围内。

### 恐慌 {#panics}

在处理攻击者控制的输入时发生恐慌，可能构成一个漏洞。

在应处理攻击者控制请求的服务器中（例如 `net/http` 服务器）发生恐慌，通常属于**私有跟踪**漏洞。

在客户端（例如 `net/http` 客户端）中发生恐慌，通常属于**公开跟踪**漏洞。

在处理可能恶意输入的包（例如 `archive/zip` 或 `image/png`）的解析函数中发生恐慌，通常属于**公开跟踪**漏洞。

由于输入本身的固有大小（例如一个非常大的图像）导致的恐慌，通常不被归类为漏洞。

对于那些设计目的并非明确为解析潜在恶意数据的函数，无效输入不在我们的威胁模型范围内，通常也不在安全漏洞的处理范围之内。例如，图像*解析器*应能防御无效输入，但图像*编码器*中的恐慌可能是一个缺陷，但不会作为漏洞来处理。

### 过度的资源消耗 {#quadratic}

我们通常将过度的 CPU 或内存消耗（例如，运行时间与输入大小呈 O(n²) 关系的函数）视为等同于恐慌。

### 构建恶意代码 {#malicious-build}

构建一个攻击者控制的程序应该是安全的，即使运行它并不安全。例如，我们期望构建不受信任的代码（然后在沙箱中运行）是安全的。我们强烈建议任何构建不受信任代码的人将此视为一项纵深防御措施，并且构建不受信任代码也应在非特权沙箱环境中进行。

数据泄露不在我们的威胁模型范围内。构建一个攻击者控制的程序可能会产生包含任意本地文件内容的输出。

*执行*恶意代码也不在我们的威胁模型范围内。运行恶意程序显然会允许它进行恶意行为。

**属于漏洞，通常是私有跟踪**：

- 一个恶意模块可能导致 `go build` 执行任意代码。

**不作为安全问题处理**：

- `go build` 产生了一个包含任意本地文件文本的错误。
- 一个编译后的可执行文件包含了任意本地文件的内容。
- 一个恶意程序可以损坏运行时状态并执行任意代码。
- 一个编译错误允许混淆恶意代码。

## 非漏洞 {#non-vuln}

### 攻击者控制的环境 {#attacker-control}

如果一次攻击依赖于攻击者能够控制程序运行的环境，那么它不是一个漏洞。

这包括但不限于攻击者有能力向 `$PATH` 添加程序或设置任意环境变量。

### image, x/image：大图片 {#large-image}

解析一个大图片可能会分配大量内存。例如，一个 65536x65536 像素的 32 位色深图像，存储未压缩数据需要 16MiB。

许多图像压缩格式可以将一个大的简单图像压缩成非常小的文件大小。解码这个小文件可能会分配大量内存。

解析不受信任图像的用户应在解析前验证图像大小，使用如 [image.DecodeConfig](https://pkg.go.dev/image#DecodeConfig) 这样的函数。

我们认为图像解析函数解码一个大的、压缩良好的图像并不构成漏洞。

### net/http：重定向 {#http-redirect}

`net/http` 包的 HTTP 客户端会处理重定向。它在重定向处理中实现了与安全相关的行为。例如，当跟随重定向到一个不是初始请求域的子域或精确匹配的域时，它会剥离 `Authorization` 头部。

剥离头部是一种纵深防御措施，以避免配置错误或被入侵的服务器无意中将包含敏感头部的客户端请求转发到不受信任的目的地。

未能在重定向时剥离头部本身并不允许攻击者获取通过头部传递的凭据。然而，它可能与其他漏洞（例如，一个存在开放重定向漏洞的服务器）结合使用来实现这一点。

改变 HTTP 客户端的行为存在很高的风险，会破坏依赖于当前行为的现有用户。例如，该客户端的同源策略目前允许子域（从 example.com 重定向到 www.example.com 将保留头部），而 WHATWG Fetch 标准则不允许。使客户端与标准保持一致可能是值得的，但在安全版本中这样做更可能给现有用户带来麻烦，而不是解决真正的漏洞。

由于重定向清理是一项纵深防御措施，并且对其进行更改存在风险，我们认为 HTTP 客户端重定向的所有方面都不在安全漏洞处理流程的范围内。
