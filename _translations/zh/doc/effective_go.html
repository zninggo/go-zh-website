引言

Go 是一种开源编程语言，专注于简洁、可靠和高效， 旨在简化大规模软件的构建过程。 尽管它借鉴了现有语言的思路， 但其独特特性使得高效的 Go 程序与其近亲语言编写的程序 在风格上截然不同。 将 C++ 或 Java 程序直接翻译为 Go 往往难以获得理想效果——Java 程序 应当用 Java 编写，而非 Go。 反之，若以 Go 的思维审视问题， 则可能产生一个成功但风格迥异的程序。 换言之， 要写好 Go，理解其特性与惯用法至关重要。 同时，了解 Go 编程的既定规范也必不可少， 例如命名、格式化、程序结构等， 这样才能使你编写的程序易于其他 Go 程序员理解。

本文档提供编写清晰、地道 Go 代码的建议。 它是 语言规范、 Go 语言之旅 和 如何编写 Go 代码 的补充， 建议您优先阅读这些材料。

格式

格式问题争议最大， 实则影响最小。 人们能够适应不同格式风格， 但若能统一则更佳； 若所有人都遵循相同风格， 就能减少在该议题上耗费的时间。 关键在于如何实现这一乌托邦，而无需冗长的 规范性风格指南。

在 Go 中我们采用独特方法： 让机器处理大部分格式问题。 gofmt 程序 （亦可通过 go fmt 调用，后者作用于包级别而非源文件级别） 读取 Go 程序后， 以标准缩进和垂直对齐格式输出源码， 保留注释并在必要时重新排版。 若需了解如何处理新布局情况，运行 gofmt 即可； 若结果不符预期，应调整程序（或向 gofmt 提交缺陷报告）， 而非变通处理。

例如，无需费时对齐结构体字段的注释。 gofmt 会自动完成。对于声明 

type T struct {
    name string // name of the object
    value int // its value
}

gofmt 将自动对齐列： 

type T struct {
    name    string // name of the object
    value   int    // its value
}

标准包中的所有 Go 代码均使用 gofmt 格式化。

部分格式细节仍需注意。简要说明如下：

缩进
使用制表符缩进，gofmt 默认输出制表符。 仅在必要时使用空格。
行长度
Go 无行长度限制。无需担心溢出穿孔卡片。 若某行过长，可换行并以额外制表符缩进。
括号
Go 比 C 和 Java 需要更少括号：控制结构（if、 for、switch）语法中不含括号。 且运算符优先级层次更简洁清晰，故 
x<<8 + y<<16
的含义如空格所示，与其他语言不同。

注释

Go 提供 C 风格 /* */ 块注释 和 C++ 风格 // 行注释。 行注释是常态； 块注释主要作为包注释使用， 但也可用于表达式内部或禁用大段代码。

位于顶层声明前、无空行分隔的注释， 被视为对声明本身的说明。 这些“文档注释”是 Go 包或命令的主要文档。 关于文档注释的更多信息，请参阅“Go 文档注释”。

命名

在 Go 中，命名与其他语言同等重要。 它们甚至影响语义： 名称在包外的可见性由其首字母是否大写决定。 因此值得花时间讨论 Go 程序中的命名规范。

包名

当导入包时，包名成为其内容的访问器。执行 

import "bytes"

这样，导入该包的代码就可以使用 bytes.Buffer 这样的表述方式。如果包的所有使用者都能用同一个名称来指代其内容，将会非常方便，这就要求包名本身应该足够好：简短、简洁、有力。按照惯例，包名应为小写单词；无需使用下划线或 mixedCaps（混合大小写）风格。 请尽量选用简短的名字，因为包的每个使用者都需要输入这个名称。 无需预先担心命名冲突问题。 包名只是导入时的默认名称；它不必在整个源代码范围内唯一，在极少数出现冲突的情况下，导入方可以在本地选择不同的名称来使用。无论如何，混淆情况很少发生，因为导入路径中的文件名已经明确指定了具体使用的是哪个包。

另一个惯例是，包名应是其源目录的基本名称； src/encoding/base64 目录下的包通过 "encoding/base64" 路径导入，但其包名是 base64， 而非 encoding_base64 或 encodingBase64。

包的导入方将使用包名来引用其内容，因此包内导出的名称可以利用这一点来避免重复。 （不要使用 import . 这种写法，它虽然能简化某些必须在包外运行的测试，但其他情况下应避免使用。） 例如，bufio 包中的缓冲读取器类型被命名为 Reader， 而非 BufReader，因为用户将其视为 bufio.Reader， 这是一个清晰而简洁的名称。 此外， 由于导入的实体总是通过其包名进行寻址，bufio.Reader 不会与 io.Reader 冲突。 同样，用于创建 ring.Ring 新实例的函数——在 Go 中即构造函数的定义—— 通常应命名为 NewRing，但由于 Ring 是该包导出的唯一类型，并且该包名为 ring， 因此它被直接命名为 New， 包的使用者看到的将是 ring.New。 善用包结构来帮助你选择好的名称。

另一个简短的例子是 once.Do； once.Do(setup) 读起来很通顺，若写成 once.DoOrWaitUntilDone(setup) 反而不会更好。 冗长的名字并不自动带来更好的可读性。 一份有益的文档注释往往比一个冗长的名字更有价值。

Getters（获取器）

Go 不提供自动的 getter 和 setter 支持。 自己提供 getter 和 setter 并无不妥，通常也适宜这样做，但将 Get 放入 getter 方法的名称中既非惯用法，也无必要。如果你有一个名为 owner（小写，未导出）的字段，其 getter 方法应命名为 Owner（大写，已导出），而非 GetOwner。 使用大写名称进行导出的惯例提供了区分 字段与方法的钩子。 如果需要 setter 函数，它可能会被命名为 SetOwner。 这两个名称在实践中读起来都很自然： 

owner := obj.Owner()
if owner != user {
    obj.SetOwner(user)
}

接口命名

按照惯例，单方法接口的命名方式是 方法名加上 -er 后缀或类似修饰，以构成一个动作名词：Reader、 Writer、Formatter、 CloseNotifier 等。

这类名称有很多，遵循它们及其所代表的功能是非常有益的。 Read、Write、Close、Flush、 String 等都有 规范的函数签名和含义。为避免混淆， 除非你的方法具有相同的签名和含义，否则不要使用这些名称。 反之，如果你的类型实现了一个与某个知名类型上的方法具有相同含义的方法， 请使用相同的名称和签名； 将你的字符串转换方法命名为 String 而非 ToString。

MixedCaps（混合大小写）

最后，Go 的惯例是使用 MixedCaps 或 mixedCaps 而非下划线来书写 多词名称。

分号

与 C 类似，Go 的形式语法使用分号来终止语句， 但与 C 不同的是，这些分号不会出现在源代码中。 相反，词法分析器在扫描时使用一条简单规则来自动插入分号， 因此输入文本中大多不含分号。

规则如下：如果换行符前的最后一个标记（token）是一个标识符 （包括像 int 和 float64 这样的词）、 一个基本字面量（如数字或字符串常量），或者是以下标记之一： 

break continue fallthrough return ++ -- ) }

则词法分析器总会在该标记后插入一个分号。 这可以概括为：“如果换行符出现在一个可能结束语句的标记之后，则插入一个分号”。

在右花括号之前也可以省略分号， 因此像下面这样的语句： 

    go func() { for { dst <- <-src } }()

无需分号。 惯用的 Go 程序仅在诸如 for 循环子句等地方使用分号，以分隔初始化、条件和后续元素。如果你在一行中书写多条语句，也需要用分号分隔。

分号插入规则的一个后果是， 你不能将控制结构（if、for、switch、 或 select）的开括号放在下一行。如果你这样做，会在括号前插入一个分号，这可能导致意外的结果。应像这样书写： 

if i < f() {
    g()
}

不要像这样写 

if i < f()  // 错误！
{           // 错误！
    g()
}

控制结构

Go 的控制结构与 C 相关，但在重要方面有所不同。 没有 do 或 while 循环，只有一个 略作泛化的 for； switch 更灵活； if 和 switch 像 for 一样接受一个可选的 初始化语句； break 和 continue 语句 接受一个可选的标签来标识要跳出或继续的目标； 并且有新的控制结构，包括类型开关和 多路通信复用器 select。 语法也略有不同： 没有括号， 并且主体必须始终用大括号分隔。

If

在 Go 中，一个简单的 if 看起来像这样： 

if x > 0 {
    return y
}

强制使用大括号鼓励将简单的 if 语句 写在多行上。无论如何，这样做都是好的风格， 特别是当主体包含控制语句（如 return 或 break）时。

由于 if 和 switch 接受初始化 语句，因此经常可以看到用它来设置局部变量。 

if err := file.Chmod(0664); err != nil {
    log.Print(err)
    return err
}

在 Go 库中，你会发现 当一个 if 语句不流向下一个语句——也就是说， 主体以 break、continue、 goto 或 return 结束——时，不必要的 else 被省略。 

f, err := os.Open(name)
if err != nil {
    return err
}
codeUsing(f)

这是一个常见情况的示例，代码必须防护一系列 错误条件。如果成功的控制流 沿页面向下流动，并在出现时消除错误情况，那么代码 读起来就很顺畅。由于错误情况往往以 return 语句结束，因此生成的代码不需要 else 语句。 

f, err := os.Open(name)
if err != nil {
    return err
}
d, err := f.Stat()
if err != nil {
    f.Close()
    return err
}
codeUsing(f, d)

重声明与重赋值

附带一提：上一节的最后一个示例展示了 := 短变量声明形式工作方式的一个细节。 调用 os.Open 的声明是， 

f, err := os.Open(name)

此语句声明了两个变量：f 和 err。 几行之后，对 f.Stat 的调用是， 

d, err := f.Stat()

看起来它声明了 d 和 err。 但是请注意，err 同时出现在两个语句中。 这种重复是合法的：err 由第一个语句声明， 但只在第二个语句中重新赋值。 这意味着对 f.Stat 的调用使用了上面 声明的现有 err 变量，并只是赋予它一个新值。

在 := 声明中，变量 v 即使 已经声明过，也可能出现，条件是：

  • 此声明与 v 的现有声明在同一作用域 （如果 v 已在外部作用域声明，该声明将创建一个新变量 §），
  • 初始化中对应的值可以赋值给 v，并且
  • 该声明至少创建了一个其他变量。

这个不寻常的特性是纯粹的务实主义， 使得例如在长的 if-else 链中轻松使用单个 err 值。 你会经常看到它的使用。

§ 值得注意的是，在 Go 中，函数参数和返回值的作用域 与函数体相同，即使它们在词法上出现在 包含函数体的大括号之外。

For

Go 的 for 循环与 C 的相似——但又不相同。 它统一了 for 和 while，并且没有 do-while。 有三种形式，只有一种有分号。 

// 类似 C 的 for
for init; condition; post { }

// 类似 C 的 while
for condition { }

// 类似 C 的 for(;;)
for { }

短变量声明使得在循环中直接声明索引变量变得容易。 

sum := 0
for i := 0; i < 10; i++ {
    sum += i
}

如果你在遍历数组、切片、字符串或映射， 或者从通道读取，可以使用 range 子句来 管理循环。 

for key, value := range oldMap {
    newMap[key] = value
}

如果你只需要 range 中的第一项（键或索引），可以省略第二项： 

for key := range m {
    if key.expired() {
        delete(m, key)
    }
}

如果你只需要 range 中的第二项（值），可以使用空白标识符（下划线）来丢弃第一项： 

sum := 0
for _, value := range array {
    sum += value
}

空白标识符有许多用途，如后面一节所述。

对于字符串，range 会为您完成更多工作，通过解析 UTF-8 来分解出各个 Unicode 码点。 错误的编码会消耗一个字节并产生替代符文 U+FFFD。 （名称（及其关联的内建类型）rune 是 Go 语言中用于表示单个 Unicode 码点的术语。 详情请参见语言规范。） 循环 

for pos, char := range "日本\x80語" { // \x80 是非法的 UTF-8 编码
    fmt.Printf("character %#U starts at byte position %d\n", char, pos)
}

会打印出 

character U+65E5 '日' starts at byte position 0
character U+672C '本' starts at byte position 3
character U+FFFD '�' starts at byte position 6
character U+8A9E '語' starts at byte position 7

最后，Go 语言没有逗号运算符，并且 ++ 和 -- 是语句而非表达式。 因此，如果你想在 for 循环中操作多个变量， 你应该使用并行赋值（尽管这样就不能使用 ++ 和 --）。 

// 反转切片 a
for i, j := 0, len(a)-1; i < j; i, j = i+1, j-1 {
    a[i], a[j] = a[j], a[i]
}

Switch

Go 的 switch 比 C 的更通用。 表达式不必是常量甚至不必是整数， 分支项会从上到下进行求值，直到找到匹配项， 并且如果 switch 没有表达式，它会以 true 为条件进行切换。 因此，可以（并且惯用）将一个 if-else-if-else 链写成 switch 形式。 

func unhex(c byte) byte {
    switch {
    case '0' <= c && c <= '9':
        return c - '0'
    case 'a' <= c && c <= 'f':
        return c - 'a' + 10
    case 'A' <= c && c <= 'F':
        return c - 'A' + 10
    }
    return 0
}

没有自动贯穿（fall through），但可以用逗号分隔的列表来组合多个分支条件。 

func shouldEscape(c byte) bool {
    switch c {
    case ' ', '?', '&', '=', '#', '+', '%':
        return true
    }
    return false
}

尽管在 Go 中 break 语句远不如在某些类 C 语言中常见， 但它们可用于提前终止一个 switch 语句。 不过，有时需要跳出的是外层的循环，而不是 switch 本身， 在 Go 中可以通过给循环加上标签并 "break" 到该标签来实现。 下面的例子展示了这两种用法。 

Loop:
    for n := 0; n < len(src); n += size {
        switch {
        case src[n] < sizeOne:
            if validateOnly {
                break
            }
            size = 1
            update(src[n])

        case src[n] < sizeTwo:
            if n+1 >= len(src) {
                err = errShortInput
                break Loop
            }
            if validateOnly {
                break
            }
            size = 2
            update(src[n] + src[n+1]<<shift)
        }
    }

当然，continue 语句也接受可选的标签， 但它只适用于循环。

作为本节的结束，这里有一个用于字节切片的比较例程，它使用了两个 switch 语句： 

// Compare 比较两个字节切片，返回一个整数指示结果。
// 如果 a == b 则结果为 0，如果 a < b 则为 -1，如果 a > b 则为 +1。
func Compare(a, b []byte) int {
    for i := 0; i < len(a) && i < len(b); i++ {
        switch {
        case a[i] > b[i]:
            return 1
        case a[i] < b[i]:
            return -1
        }
    }
    switch {
    case len(a) > len(b):
        return 1
    case len(a) < len(b):
        return -1
    }
    return 0
}

类型开关

switch 也可以用来发现一个接口变量的动态类型。这种 类型开关 使用类型断言的语法，括号内是关键字 type。 如果 switch 在表达式中声明了一个变量，那么该变量在每个分支中都会具有对应的类型。 在这种情况下，惯用做法是复用同一个名字，实际上相当于在每个 case 中声明了一个同名但类型不同的新变量。 

var t interface{}
t = functionOfSomeType()
switch t := t.(type) {
default:
    fmt.Printf("unexpected type %T\n", t)     // %T 会打印 t 的实际类型
case bool:
    fmt.Printf("boolean %t\n", t)             // t 的类型是 bool
case int:
    fmt.Printf("integer %d\n", t)             // t 的类型是 int
case *bool:
    fmt.Printf("pointer to boolean %t\n", *t) // t 的类型是 *bool
case *int:
    fmt.Printf("pointer to integer %d\n", *t) // t 的类型是 *int
}

函数

多重返回值

Go 语言的一个不寻常的特性是函数和方法 可以返回多个值。这种形式可以用来改进 C 程序中的几个笨拙的惯用法：例如带内错误返回（如用 -1 表示 EOF） 和修改通过地址传递的参数。

在 C 中，写入错误通过负的计数值发出信号，错误代码则隐藏在一个易变的位置。 在 Go 中，Write 可以返回一个计数值 和 一个错误信息："是的，你写入了一些字节，但不是全部，因为设备已满"。 包 os 中文件的 Write 方法的签名是： 

func (file *File) Write(b []byte) (n int, err error)

正如文档所说，当 n != len(b) 时， 它返回写入的字节数和一个非 nil 的 error。 这是一种常见的风格；更多示例请参见关于错误处理的章节。

类似的方法也无需传递指针来模拟引用参数，从而消除了向返回值传递指针的需要。下面是一个简单的函数示例，用于从字节切片中的某个位置抓取一个数字，并返回该数字以及下一个位置。 

func nextInt(b []byte, i int) (int, int) {
    for ; i < len(b) && !isDigit(b[i]); i++ {
    }
    x := 0
    for ; i < len(b) && isDigit(b[i]); i++ {
        x = x*10 + int(b[i]) - '0'
    }
    return x, i
}

你可以用它来扫描输入切片 b 中的数字，如下所示： 

    for i := 0; i < len(b); {
        x, i = nextInt(b, i)
        fmt.Println(x)
    }

命名结果参数

Go 函数的返回或结果“参数”可以被命名，并像传入参数一样作为常规变量使用。当它们被命名时，在函数开始执行时会被初始化为其类型的零值；如果函数执行了一个没有参数的 return 语句，则结果参数的当前值将作为返回值。

这些名称不是强制性的，但它们可以使代码更短更清晰：它们就是文档。如果我们给 nextInt 的结果命名，那么哪个返回的 int 代表哪个含义就变得显而易见。 

func nextInt(b []byte, pos int) (value, nextPos int) {

因为命名结果会被初始化并关联到一个简单的 return 语句，所以它们既能简化代码也能使意图更清晰。这是一个充分利用此特性的 io.ReadFull 版本： 

func ReadFull(r Reader, buf []byte) (n int, err error) {
    for len(buf) > 0 && err == nil {
        var nr int
        nr, err = r.Read(buf)
        n += nr
        buf = buf[nr:]
    }
    return
}

延迟执行

Go 的 defer 语句会安排一个函数调用（即 延迟 函数）在执行该 defer 的函数返回之前立即运行。这是一种不寻常但有效的处理方式，适用于诸如必须释放资源等场景，无论函数通过哪条路径返回。典型的例子是解锁互斥锁或关闭文件。 

// Contents 以字符串形式返回文件内容。
func Contents(filename string) (string, error) {
    f, err := os.Open(filename)
    if err != nil {
        return "", err
    }
    defer f.Close()  // f.Close 将在我们完成时运行。

    var result []byte
    buf := make([]byte, 100)
    for {
        n, err := f.Read(buf[0:])
        result = append(result, buf[0:n]...) // append 稍后讨论。
        if err != nil {
            if err == io.EOF {
                break
            }
            return "", err  // 如果在这里返回，f 将会被关闭。
        }
    }
    return string(result), nil // 如果在这里返回，f 将会被关闭。
}

延迟调用 Close 这样的函数有两个好处。首先，它保证你永远不会忘记关闭文件，这是在后续编辑函数以添加新的返回路径时容易犯的错误。其次，它意味着关闭操作紧挨着打开操作放置，这比将关闭放在函数末尾要清晰得多。

延迟函数的参数（如果该函数是方法，还包括接收器）在 defer 执行时被求值，而不是在 call 执行时。除了避免担心变量在函数执行过程中改变值外，这意味着一个单独的延迟调用点可以延迟多次函数执行。这里是一个简单的例子。 

for i := 0; i < 5; i++ {
    defer fmt.Printf("%d ", i)
}

延迟函数按照后进先出（LIFO）的顺序执行，因此当函数返回时，这段代码会导致打印出 4 3 2 1 0。一个更合理的例子是通过程序跟踪函数执行的简单方法。我们可以编写几个简单的跟踪例程，像这样： 

func trace(s string)   { fmt.Println("entering:", s) }
func untrace(s string) { fmt.Println("leaving:", s) }

// 像这样使用它们：
func a() {
    trace("a")
    defer untrace("a")
    // 做一些事情....
}

我们可以利用延迟函数的参数在 defer 执行时被求值这一事实来做得更好。跟踪例程可以设置用于取消跟踪例程的参数。这个例子： 

func trace(s string) string {
    fmt.Println("entering:", s)
    return s
}

func un(s string) {
    fmt.Println("leaving:", s)
}

func a() {
    defer un(trace("a"))
    fmt.Println("in a")
}

func b() {
    defer un(trace("b"))
    fmt.Println("in b")
    a()
}

func main() {
    b()
}

会输出： 

entering: b
in b
entering: a
in a
leaving: a
leaving: b

对于习惯了其他语言中块级资源管理的程序员来说，defer 可能显得有些特别，但它最有趣和最强大的应用恰恰源于它不是基于块而是基于函数的特性。在关于 panic 和 recover 的章节中，我们将看到它可能性的另一个例子。

使用 new 进行分配

Go 有两种分配原语，即内置函数 new 和 make。 它们的用途和适用类型不同，这可能会令人困惑，但规则其实很简单。 我们先讨论 new。 它是一个用于分配内存的内置函数，但与某些语言中的同名函数不同，它不会 初始化 内存，只会将其 清零。 也就是说，new(T) 会为一个类型为 T 的新变量分配清零的存储空间，并返回其地址，即一个类型为 *T 的值。 用 Go 的术语来说，它返回一个指向新分配的、类型为 T 的零值的指针。

从 Go 1.26 开始，new 也接受一个（值）表达式作为参数，该表达式指定了变量的初始值。 例如，new(int64(300)) 会分配一个类型为 int64 的新变量，将其初始化为 300，并返回其地址。

由于 new 返回的内存是清零的，因此在设计数据结构时，最好安排好使每种类型的零值无需进一步初始化即可使用。 这意味着数据结构的用户可以使用 new 创建一个实例并立即开始使用。 例如，bytes.Buffer 的文档指出，“Buffer 的零值是一个可立即使用的空缓冲区。” 同样，sync.Mutex 也没有显式的构造函数或 Init 方法。 相反，sync.Mutex 的零值被定义为一个未锁定的互斥锁。

零值即用的特性具有传递性。考虑以下类型声明。 

type SyncedBuffer struct {
    lock    sync.Mutex
    buffer  bytes.Buffer
}

SyncedBuffer 类型的值在分配或仅声明后也立即可以使用。 在接下来的代码片段中，p 和 v 都将在无需进一步安排的情况下正确工作。 

p := new(SyncedBuffer)  // 类型为 *SyncedBuffer
var v SyncedBuffer      // 类型为  SyncedBuffer

构造函数与复合字面量

有时零值不够好，需要初始化构造函数，如下例所示，该示例源自 os 包。 

func NewFile(fd int, name string) *File {
    if fd < 0 {
        return nil
    }
    f := new(File)
    f.fd = fd
    f.name = name
    f.dirinfo = nil
    f.nepipe = 0
    return f
}

这里有很多样板代码。我们可以使用 复合字面量 来简化它，复合字面量是一种表达式，每次求值时都会创建一个新的实例。 

func NewFile(fd int, name string) *File {
    if fd < 0 {
        return nil
    }
    f := File{fd, name, nil, 0}
    return &f
}

请注意，与 C 语言不同，返回一个局部变量的地址是完全没问题的；与该变量关联的存储在函数返回后仍然存在。 实际上，对复合字面量取地址每次求值时都会分配一个新实例，因此我们可以将最后两行合并。 

    return &File{fd, name, nil, 0}

复合字面量的字段按顺序排列，并且必须全部列出。 然而，通过将元素显式标记为 字段:值 对，初始化器可以以任何顺序出现，缺失的字段则保留为它们各自的零值。因此我们可以说 

    return &File{fd: fd, name: name}

作为一个极限情况，如果一个复合字面量根本不包含任何字段，它将创建该类型的零值。表达式 new(File) 和 &File{} 是等价的。

复合字面量也可以用于数组、切片和映射，字段标签分别是相应的索引或映射键。 在这些示例中，无论 Enone、Eio 和 Einval 的值如何，只要它们互不相同，初始化都能正常工作。 

a := [...]string   {Enone: "no error", Eio: "Eio", Einval: "invalid argument"}
s := []string      {Enone: "no error", Eio: "Eio", Einval: "invalid argument"}
m := map[int]string{Enone: "no error", Eio: "Eio", Einval: "invalid argument"}

使用 make 进行分配

回到分配的话题。 内置函数 make(T, args) 的用途与 new(T) 不同。 它仅用于创建切片、映射和通道，并且返回一个 已初始化（而非 清零）的类型为 T（而非 *T）的值。 区别的原因在于，这三种类型在底层表示的是数据结构的引用，这些数据结构在使用前必须初始化。 例如，一个切片是一个包含三项的描述符：一个指向数据（在数组内部）的指针、长度和容量，在这些项初始化之前，切片是 nil。 对于切片、映射和通道，make 会初始化内部数据结构并为值的使用做好准备。 例如， 

make([]int, 10, 100)

会分配一个包含 100 个 int 的数组，然后创建一个长度为 10、容量为 100 的切片结构，该切片指向该数组的前 10 个元素。 （创建切片时，容量可以省略；有关更多信息，请参阅切片部分。） 相比之下，new([]int) 返回一个指向新分配的、已清零的切片结构的指针，即一个指向 nil 切片值的指针。

这些示例说明了 new 和 make 之间的区别。 

var p *[]int = new([]int)       // 分配切片结构；*p == nil；很少使用
var v  []int = make([]int, 100) // 切片 v 现在引用一个包含 100 个 int 的新数组
// 不必要的复杂写法： var p *[]int = new([]int) *p = make([]int, 100, 100) // 符合惯例的写法： v := make([]int, 100)

请记住，make 只适用于映射（map）、切片（slice）和通道（channel）， 并且它不返回指针。 若需获得显式指针，请使用 new 分配，或显式地取变量的地址。

数组

数组在规划内存详细布局时非常有用，有时 可以避免内存分配，但最主要的是， 它们是构建切片的基础，而切片是下一节的主题。 为此主题奠定基础，这里先简要介绍数组。

Go 语言中的数组与 C 语言有重大区别。 在 Go 中：

  • 数组是值类型。将一个数组赋值给另一个数组会复制所有元素。
  • 特别地，如果将数组传递给函数，函数将接收到该数组的副本，而非指向它的指针。
  • 数组的大小是其类型的一部分。类型 [10]int 和 [20]int 是不同的。

值类型的特性很有用，但也可能带来开销；如果你想要类似 C 的行为和效率， 可以传递指向数组的指针。 

func Sum(a *[3]float64) (sum float64) {
    for _, v := range *a {
        sum += v
    }
    return
}

array := [...]float64{7.0, 8.5, 9.1}
x := Sum(&array)  // 注意显式取地址运算符

但这种风格并非惯用的 Go 写法。 应改用切片。

切片

切片是对数组的封装，为数据序列提供更通用、更强大、更便捷的 接口。除了像变换矩阵这样有显式维度的场景外，Go 中大多数 数组编程都是用切片而非简单数组完成的。

切片持有对底层数组的引用，如果将一个切片赋值给另一个， 两者将引用同一个数组。如果函数接受一个切片参数，它对切片元素所做的 修改将对调用者可见，类似于传递指向底层数组的指针。因此，Read 函数可以接受一个切片参数，而非指针和计数； 切片的长度设置了要读取数据量的上限。以下是 os 包中 File 类型的 Read 方法的签名： 

func (f *File) Read(buf []byte) (n int, err error)

该方法返回读取的字节数和一个错误值（如果有的话）。 要读取较大缓冲区 buf 的前 32 字节， 可以对该缓冲区进行切片（这里用作动词）。 

    n, err := f.Read(buf[0:32])

这种切片操作很常见且高效。实际上，暂且不论效率，以下代码片段 也能读取缓冲区的前 32 字节。 

    var n int
    var err error
    for i := 0; i < 32; i++ {
        nbytes, e := f.Read(buf[i:i+1])  // 每次读取一个字节。
        n += nbytes
        if nbytes == 0 || e != nil {
            err = e
            break
        }
    }

切片的长度可以更改，只要它仍在底层数组的限制范围内即可； 只需将其赋值给自身的一个切片。切片的容量（通过内置函数 cap 访问）报告了切片可能具有的最大长度。这里是一个向切片追加数据的函数。 如果数据超过容量，切片将被重新分配。 函数返回结果切片。该函数利用了一个事实： 对 nil 切片应用 len 和 cap 是合法的，且返回 0。 

func Append(slice, data []byte) []byte {
    l := len(slice)
    if l + len(data) > cap(slice) {  // 重新分配
        // 分配所需空间的两倍，为未来增长预留。
        newSlice := make([]byte, (l+len(data))*2)
        // copy 函数是预定义的，适用于任何切片类型。
        copy(newSlice, slice)
        slice = newSlice
    }
    slice = slice[0:l+len(data)]
    copy(slice[l:], data)
    return slice
}

我们必须返回该切片，因为虽然 Append 可以修改 slice 的元素，但切片本身（持有指针、长度和容量的运行时数据结构） 是按值传递的。

向切片追加元素这一想法非常实用，因此被内置函数 append 所封装。不过，要理解该函数的设计，我们需要更多信息， 因此我们将在后面再讨论它。

二维切片

Go 的数组和切片是一维的。 要创建二维数组或切片的等价物，必须定义数组的数组或切片的切片，例如： 

type Transform [3][3]float64  // 一个 3x3 数组，实质上是数组的数组。
type LinesOfText [][]byte     // 字节切片的切片。

由于切片是可变长度的，因此每个内部切片可以具有不同的长度。 这很常见，就像我们的 LinesOfText 示例一样：每行都有独立的长度。 

text := LinesOfText{
    []byte("Now is the time"),
    []byte("for all good gophers"),
    []byte("to bring some fun to the party."),
}

有时需要分配一个二维切片，例如在处理像素扫描线时就会出现这种情况。 有两种方法可以实现。 一种是独立分配每个切片；另一种 是分配一个单独的数组，并将各个切片指向其中。 使用哪种方法取决于你的应用。 如果切片可能会增长或收缩，则应独立分配它们， 以避免覆盖下一行；如果不会，则用单次分配来构造对象可能更高效。 供参考，以下是两种方法的简要示意。 第一种，一次一行分配： 

// 分配顶层切片。
picture := make([][]uint8, YSize) // 每行对应一个y单位。
// 遍历各行，为每行分配切片。
for i := range picture {
    picture[i] = make([]uint8, XSize)
}

现在采用单次分配，再分割为各行： 

// 分配顶层切片，与之前相同。
picture := make([][]uint8, YSize) // 每行对应一个y单位。
// 分配一个大切片来存储所有像素。
pixels := make([]uint8, XSize*YSize) // 类型为 []uint8，尽管 picture 是 [][]uint8。
// 遍历各行，从剩余像素切片的前端切分出每行。
for i := range picture {
    picture[i], pixels = pixels[:XSize], pixels[XSize:]
}

映射

映射是一种方便且强大的内置数据结构，它将一种类型的值（键）与另一种类型的值（元素或值）关联起来。 键可以是任何定义了等号运算符的类型， 例如整数、浮点数、复数、字符串、指针、接口（只要动态类型支持相等性）、结构体和数组。 切片不能用作映射键， 因为切片未定义相等性。 与切片类似，映射持有一个底层数据结构的引用。 如果你将映射传递给一个函数， 该函数更改了映射的内容，那么更改在调用者中是可见的。

映射可以使用通常的复合字面量语法构造， 键值对之间用冒号分隔， 因此在初始化期间很容易构建它们。 

var timeZone = map[string]int{
    "UTC":  0*60*60,
    "EST": -5*60*60,
    "CST": -6*60*60,
    "MST": -7*60*60,
    "PST": -8*60*60,
}

对映射的赋值和获取在语法上看起来与对数组和切片的操作相同， 只是索引不必是整数。 

offset := timeZone["EST"]

尝试使用一个在映射中不存在的键来获取映射值， 将返回该映射条目类型的零值。 例如，如果映射包含整数，查找一个不存在的键将返回 0。 集合可以实现为一个值类型为 bool 的映射。 将映射条目设置为 true 以将该值加入集合， 然后通过简单的索引来测试它。 

attended := map[string]bool{
    "Ann": true,
    "Joe": true,
    ...
}

if attended[person] { // 如果 person 不在映射中，结果为 false
    fmt.Println(person, "was at the meeting")
}

有时你需要区分是条目缺失还是返回了零值。 例如，对于 "UTC"，是有这个条目，还是因为它根本不在映射中而返回了 0？ 你可以使用一种多重赋值的形式来区分。 

var seconds int
var ok bool
seconds, ok = timeZone[tz]

出于显而易见的原因，这被称为“comma ok”惯用法。 在这个例子中，如果 tz 存在，seconds 将被设置为相应的值，并且 ok 为 true； 如果不存在，seconds 将被设置为零，ok 将 为 false。 这里有一个函数将它与良好的错误报告结合在一起： 

func offset(tz string) int {
    if seconds, ok := timeZone[tz]; ok {
        return seconds
    }
    log.Println("unknown time zone:", tz)
    return 0
}

要测试映射中是否存在某个键而不关心实际值，你可以使用空标识符（_） 来代替通常用于存储值的变量。 

_, present := timeZone[tz]

要删除一个映射条目，使用内置的 delete 函数，它的参数是映射和要删除的键。 即使该键在映射中已经不存在，这样做也是安全的。 

delete(timeZone, "PDT")  // 现在处于标准时间

打印

Go语言的格式化输出风格与C语言的 printf 函数族相似，但功能更丰富且更通用。这些函数位于 fmt 包中，且首字母大写：fmt.Printf、fmt.Fprintf、fmt.Sprintf 等。字符串相关函数（如 Sprintf 等）返回字符串，而非填充到提供的缓冲区。

你无需提供格式字符串。对于 Printf、Fprintf 和 Sprintf，每组都有另一对函数，例如 Print 和 Println。这些函数不接受格式字符串，而是为每个参数生成默认格式。Println 版本还会在参数之间插入空格，并在输出末尾添加换行符；而 Print 版本仅在相邻参数都不是字符串时才添加空格。在下面的例子中，每一行都产生相同的输出。 

fmt.Printf("Hello %d\n", 23)
fmt.Fprint(os.Stdout, "Hello ", 23, "\n")
fmt.Println("Hello", 23)
fmt.Println(fmt.Sprint("Hello ", 23))

格式化打印函数 fmt.Fprint 及其类似函数的第一个参数可以是任何实现了 io.Writer 接口的对象；变量 os.Stdout 和 os.Stderr 就是常见的实例。

至此，Go语言开始与C语言有所不同。首先，数字格式（如 %d）不接受表示符号性或大小的标志；打印例程使用参数的类型来决定这些属性。 

var x uint64 = 1<<64 - 1
fmt.Printf("%d %x; %d %x\n", x, x, int64(x), int64(x))

输出为 

18446744073709551615 ffffffffffffffff; -1 -1

如果你只需要默认转换（例如整数用十进制），可以使用通配格式 %v（代表 "值"）；其结果与 Print 和 Println 完全一致。此外，该格式可以打印任何值，甚至包括数组、切片、结构体和映射。下面是一个针对前文定义的时区映射的打印语句。 

fmt.Printf("%v\n", timeZone)  // 或者直接用 fmt.Println(timeZone)

输出如下： 

map[CST:-21600 EST:-18000 MST:-25200 PST:-28800 UTC:0]

对于映射，Printf 及其类似函数会按键的字典序排序输出。

当打印结构体时，修改后的格式 %+v 会为结构体的字段添加名称注解；而对于任何值，备用格式 %#v 会使用完整的Go语法打印该值。 

type T struct {
    a int
    b float64
    c string
}
t := &T{ 7, -2.35, "abc\tdef" }
fmt.Printf("%v\n", t)
fmt.Printf("%+v\n", t)
fmt.Printf("%#v\n", t)
fmt.Printf("%#v\n", timeZone)

输出为 

&{7 -2.35 abc   def}
&{a:7 b:-2.35 c:abc     def}
&main.T{a:7, b:-2.35, c:"abc\tdef"}
map[string]int{"CST":-21600, "EST":-18000, "MST":-25200, "PST":-28800, "UTC":0}

（请注意这些 & 符号。）那种带引号的字符串格式也可以通过 %q 应用于 string 或 []byte 类型的值。备用格式 %#q 会尽可能使用反引号代替。（%q 格式也适用于整数和 rune，产生单引号的 rune 常量。）此外，%x 不仅适用于整数，也适用于字符串、字节数组和字节切片，生成一长串十六进制字符串；使用带空格的格式（% x）会在字节之间添加空格。

另一个实用的格式是 %T，它打印值的类型。 

fmt.Printf("%T\n", timeZone)

输出为 

map[string]int

如果你想控制自定义类型的默认格式，只需在该类型上定义一个签名为 String() string 的方法即可。对于我们的简单类型 T，可能是这样。 

func (t *T) String() string {
    return fmt.Sprintf("%d/%g/%q", t.a, t.b, t.c)
}
fmt.Printf("%v\n", t)

输出格式为 

7/-2.35/"abc\tdef"

（如果你需要打印类型为 T 的值以及指向 T 的指针，那么 String 的接收者必须是值类型；本例使用了指针，因为对于结构体类型来说，这更高效且更符合习惯。更多信息请参阅下方关于 指针与值接收者 的章节。）

我们的 String 方法能够调用 Sprintf，因为打印例程是完全可重入的，可以这样包装。然而，关于这种做法有一个重要细节需要理解：不要通过以会导致 Sprintf 无限递归调用你的 String 方法的方式构造 String 方法。如果 Sprintf 调用试图将接收者直接作为字符串打印，就会再次调用该方法，从而发生这种情况。这是一个常见且容易犯的错误，如下例所示。 

type MyString string

func (m MyString) String() string {
    return fmt.Sprintf("MyString=%s", m) // 错误：会无限递归。
}

修复方法也很简单：将参数转换为基本字符串类型，该类型没有该方法。 

type MyString string
func (m MyString) String() string {
    return fmt.Sprintf("MyString=%s", string(m)) // 正确：注意类型转换。
}

在 初始化章节 中，我们将看到另一种避免这种递归的技术。

另一种打印技巧是将打印函数的参数直接传递给另一个打印函数。 Printf 的签名使用 ...interface{} 类型作为其最终参数，以指定在格式字符串之后可以出现任意数量（任意类型）的参数。 

func Printf(format string, v ...interface{}) (n int, err error) {

在 Printf 函数内部，v 的行为类似于 []interface{} 类型的变量，但如果将其传递给另一个可变参数函数，它就会像普通的参数列表一样。 下面是我们在上面用到的 log.Println 函数的实现。它将参数直接传递给 fmt.Sprintln 进行实际的格式化。 

// Println 的行为类似于 fmt.Println，但输出到标准日志记录器。
func Println(v ...interface{}) {
    std.Output(2, fmt.Sprintln(v...))  // Output 接收参数 (int, string)
}

我们在嵌套调用 Sprintln 时，在 v 后面写了 ...，这是为了告诉编译器将 v 视为一个参数列表；否则它只会把 v 作为一个单独的切片参数传递。

关于打印，这里涉及的内容比我们介绍的更多。详细信息请参阅 fmt 包的 godoc 文档。

顺便提一下，... 参数可以具有特定的类型，例如 ...int 可以用于一个选择整数列表中最小值的 min 函数： 

func Min(a ...int) int {
    min := int(^uint(0) >> 1)  // 最大 int 值
    for _, i := range a {
        if i < min {
            min = i
        }
    }
    return min
}

追加（Append）

现在我们有了缺失的那部分，可以用来解释内置函数 append 的设计了。append 的签名与我们上面自定义的 Append 函数不同。其示意如下： 

func append(slice []T, elements ...T) []T

其中 T 是任意给定类型的占位符。在 Go 中，你实际上无法编写一个类型 T 由调用者决定的函数。这就是 append 是内置的原因：它需要编译器的支持。

append 的作用是将元素追加到切片末尾并返回结果。之所以需要返回结果，是因为与我们手写的 Append 一样，底层数组可能会改变。这个简单的例子： 

x := []int{1,2,3}
x = append(x, 4, 5, 6)
fmt.Println(x)

会打印出 [1 2 3 4 5 6]。所以 append 的行为有点像 Printf，可以收集任意数量的参数。

但如果我们想像我们的 Append 那样，将一个切片追加到另一个切片呢？很简单：在调用点使用 ...，就像我们在上面调用 Output 时所做的那样。下面这段代码的输出与上面完全相同。 

x := []int{1,2,3}
y := []int{4,5,6}
x = append(x, y...)
fmt.Println(x)

如果没有那个 ...，代码将无法编译，因为类型会不匹配；y 的类型不是 int。

初始化（Initialization）

尽管表面上看起来与 C 或 C++ 的初始化差异不大，但 Go 中的初始化更为强大。可以在初始化过程中构建复杂的结构体，并且即使初始化对象来自不同的包，它们之间的顺序问题也能被正确处理。

方法（Methods）

指针与值（Pointers vs. Values）

正如我们在 ByteSize 中看到的，方法可以为任何命名类型（指针或接口除外）定义；接收器不必是结构体。

在前面关于切片的讨论中，我们编写了一个 Append 函数。我们可以将其定义为切片上的一个方法。为此，我们首先声明一个命名类型，以便将方法绑定到它，然后将该类型的一个值作为方法的接收器。 

type ByteSlice []byte

func (slice ByteSlice) Append(data []byte) []byte {
    // 函数体与上面定义的 Append 函数完全相同。
}

这仍然需要方法返回更新后的切片。我们可以通过重新定义方法，使其接受一个指向 ByteSlice 的 *指针* 作为接收器，从而消除这种笨拙性，这样方法就可以覆盖调用者的切片。 

func (p *ByteSlice) Append(data []byte) {
    slice := *p
    // 函数体同上，但没有返回语句。
    *p = slice
}

事实上，我们可以做得更好。如果我们将函数修改成看起来像标准的 Write 方法，就像这样， 

func (p *ByteSlice) Write(data []byte) (n int, err error) {
    slice := *p
    // 再次同上。
    *p = slice
    return len(data), nil
}

那么类型 *ByteSlice 就满足了标准接口 io.Writer，这非常方便。例如，我们可以向其写入内容。 

    var b ByteSlice
    fmt.Fprintf(&b, "This hour has %d days\n", 7)

我们传递了 ByteSlice 的地址，因为只有 *ByteSlice 满足 io.Writer。关于接收器使用指针与值的规则是：值方法可以在指针和值上调用，但指针方法只能在指针上调用。

这个规则的产生是因为指针方法可以修改接收器；如果在值上调用它们，方法将接收到该值的一个副本，因此任何修改都会被丢弃。语言因此禁止这种错误。不过，有一个便利的例外。当该值是可寻址时，语言会自动插入地址操作符来处理在值上调用指针方法的常见情况。在我们的例子中，变量 b 是可寻址的，因此我们可以直接使用 b.Write 来调用它的 Write 方法。编译器会为我们将其改写为 (&b).Write。

顺便说一下，在字节切片上使用 Write 的想法是 bytes.Buffer 实现的核心。

接口与其他类型（Interfaces and other types）

接口（Interfaces）

Go 语言中的接口提供了一种指定对象行为的方式：如果某个东西能做 *这* 件事，那么它就可以用在 *这里*。我们已经看过几个简单的例子了；自定义的打印函数可以通过 String 方法实现，而 Fprintf 可以向任何拥有 Write 方法的东西生成输出。只有一两个方法的接口在 Go 代码中很常见，通常以方法的名字来命名，例如将实现了 Write 的东西命名为 io.Writer。

一个类型可以实现多个接口。例如，如果一个集合实现了 sort.Interface（包含 Len()、Less(i, j int) bool 和 Swap(i, j int)），那么它就可以被 sort 包中的排序函数排序，同时它也可以拥有自定义的格式化器。在这个精心设计的例子中，Sequence 同时满足了两者。

类型转换（Conversions）

Sequence 的 String 方法重复实现了 Sprint 已经为切片完成的工作。 （该方法的复杂度为 O(N²)，效率较低。）如果在调用 Sprint 前将 Sequence 转换为普通的 []int，就能共享这一过程（同时提升速度）。 

func (s Sequence) String() string {
    s = s.Copy()
    sort.Sort(s)
    return fmt.Sprint([]int(s))
}

此方法是在 String 方法中安全调用 Sprintf 的又一个转换技巧示例。 由于忽略类型名称后这两种类型（Sequence 和 []int）本质相同，它们之间的转换是合法的。 这种转换不会创建新值，只是暂时将现有值视为新类型。 （其他合法的转换——如整数转浮点数——则会创建新值。）

在 Go 程序中，将表达式的类型转换以访问不同方法集是一种惯用法。例如，我们可以利用现有的 sort.IntSlice 类型，将整个示例简化为： 

type Sequence []int

// 打印方法 - 打印前对元素排序
func (s Sequence) String() string {
    s = s.Copy()
    sort.IntSlice(s).Sort()
    return fmt.Sprint([]int(s))
}

现在，我们不再要求 Sequence 实现多个接口（排序和打印），而是利用数据项可转换为多种类型（Sequence、sort.IntSlice 和 []int）的特性，每种类型各司其职。 这种方式在实践中较为少见，但效果显著。

接口转换与类型断言

类型开关是一种转换形式：它接收一个接口，并在每个 case 中将其转换为对应的类型。以下是 fmt.Printf 底层代码如何通过类型开关将值转为字符串的简化版本。 若值本身已是字符串，我们直接获取接口持有的实际字符串值；若值拥有 String 方法，则调用该方法获取结果。 

type Stringer interface {
    String() string
}

var value interface{} // 调用方提供的值。
switch str := value.(type) {
case string:
    return str
case Stringer:
    return str.String()
}

第一个 case 提取具体值；第二个 case 将接口转换为另一接口。这种混合类型的方式完全可行。

如果只关心单一类型呢？例如已知值包含 string 且只需提取它？单一 case 的类型开关可以实现，但 类型断言 同样适用。 类型断言接收一个接口值，并从中提取指定显式类型的值。其语法借鉴自类型开关的子句，但使用显式类型而非 type 关键字： 

value.(typeName)

结果是一个静态类型为 typeName 的新值。该类型必须是接口持有的具体类型，或是值可转换的另一接口类型。若要提取已知存在于值中的字符串，可以这样写： 

str := value.(string)

但若值实际不包含字符串，程序将在运行时崩溃。为防范此情况，可使用"逗号-ok"习惯用法安全地测试值是否为字符串： 

str, ok := value.(string)
if ok {
    fmt.Printf("字符串值为：%q\n", str)
} else {
    fmt.Printf("值不是字符串\n")
}

若类型断言失败，str 仍会存在且类型为 string，但其值将为零值（空字符串）。

作为能力展示，以下 if-else 语句等效于本节开头的类型开关： 

if str, ok := value.(string); ok {
    return str
} else if str, ok := value.(Stringer); ok {
    return str.String()
}

泛用性

若某类型仅为实现接口而存在，且除该接口定义外不再有导出方法，则无需导出该类型本身。 仅导出接口可明确表明：该值不具备接口描述之外的特殊行为。 这也避免了在每个通用方法的实例上重复编写文档。

在此类场景中，构造函数应返回接口值而非具体实现类型。例如，在哈希库中，crc32.NewIEEE 和 adler32.New 均返回接口类型 hash.Hash32。 在 Go 程序中将 CRC-32 算法替换为 Adler-32 时，只需修改构造函数调用；其余代码不受算法变更影响。

类似方法使得各 crypto 包中的流加密算法能与它们链式调用的分组密码分离。 crypto/cipher 包中的 Block 接口定义了分组密码的行为，提供对单数据块的加密能力。 随后，通过类比 bufio 包的设计，实现了该接口的密码包可用于构建流密码（以 Stream 接口表示），而无需了解分组加密的细节。

crypto/cipher 接口定义如下： 

type Block interface {
    BlockSize() int
    Encrypt(dst, src []byte)
    Decrypt(dst, src []byte)
}

type Stream interface {
    XORKeyStream(dst, src []byte)
}

这里是计数器模式（CTR）流的定义， 它能将分组密码转换为流密码；注意 分组密码的细节已被抽象隐藏： 

// NewCTR 返回一个 Stream，该 Stream 使用给定的 Block 以
// 计数器模式进行加密/解密。iv 的长度必须与 Block 的块大小相同。
func NewCTR(block Block, iv []byte) Stream

NewCTR 不仅适用于 特定的加密算法和数据源，也适用于任何 Block 接口的实现和任何 Stream。由于它们返回 接口值，因此将 CTR 加密替换为其他加密模式只需进行局部更改。构造函数 调用必须修改，但由于周边代码只需将结果视为 Stream，因此不会察觉其中的差异。

接口与方法

由于几乎任何类型都可以附带方法，因此几乎任何类型都可以 满足一个接口。一个直观的例子是 http 包，它定义了 Handler 接口。任何 实现了 Handler 的对象都可以处理 HTTP 请求。 

type Handler interface {
    ServeHTTP(ResponseWriter, *Request)
}

ResponseWriter 本身是一个接口，它提供了 将响应返回给客户端所需的方法。 这些方法包括标准的 Write 方法，因此 http.ResponseWriter 可以在任何可以使用 io.Writer 的地方使用。 Request 是一个结构体，它包含了 从客户端接收到的请求的已解析表示。

为了简洁起见，我们忽略 POST 请求，并假设 HTTP 请求总是 GET 请求；这一简化不会影响处理器的设置方式。 下面是一个简单的处理器实现，用于统计页面 被访问的次数。 

// 简单的计数器服务器。
type Counter struct {
    n int
}

func (ctr *Counter) ServeHTTP(w http.ResponseWriter, req *http.Request) {
    ctr.n++
    fmt.Fprintf(w, "counter = %d\n", ctr.n)
}

（延续我们的主题，请注意 Fprintf 如何向 http.ResponseWriter 输出内容。） 在真实的服务器中，对 ctr.n 的访问需要保护以防止 并发访问。 相关建议请参阅 sync 和 atomic 包。

作为参考，以下是将此类服务器附加到 URL 树节点的方式。 

import "net/http"
...
ctr := new(Counter)
http.Handle("/counter", ctr)

但为什么要把 Counter 做成结构体？一个整数就够了。 （接收器需要是指针，以便对调用者可见的递增操作。） 

// 更简单的计数器服务器。
type Counter int

func (ctr *Counter) ServeHTTP(w http.ResponseWriter, req *http.Request) {
    *ctr++
    fmt.Fprintf(w, "counter = %d\n", *ctr)
}

如果你的程序有一些内部状态需要在页面被访问时得到通知，该怎么办？可以将通道与网页绑定。 

// 每次访问时发送通知的通道。
// （可能希望该通道是有缓冲的。）
type Chan chan *http.Request

func (ch Chan) ServeHTTP(w http.ResponseWriter, req *http.Request) {
    ch <- req
    fmt.Fprint(w, "notification sent")
}

最后，假设我们想要在 /args 上显示 调用服务器二进制文件时使用的参数。 编写一个函数来打印参数很容易。 

func ArgServer() {
    fmt.Println(os.Args)
}

我们如何将其变成一个 HTTP 服务器？我们可以让 ArgServer 成为某个类型的方法，然后忽略该类型的值，但有一种更清晰的方式。 由于我们可以为除指针和接口外的任何类型定义方法， 我们可以为一个函数编写方法。 http 包中包含如下代码： 

// HandlerFunc 类型是一个适配器，允许将
// 普通函数用作 HTTP 处理器。如果 f 是一个具有
// 适当签名的函数，则 HandlerFunc(f) 是一个
// 调用 f 的 Handler 对象。
type HandlerFunc func(ResponseWriter, *Request)

// ServeHTTP 调用 f(w, req)。
func (f HandlerFunc) ServeHTTP(w ResponseWriter, req *Request) {
    f(w, req)
}

HandlerFunc 是一个拥有方法 ServeHTTP 的类型， 因此该类型的值可以处理 HTTP 请求。查看该方法的 实现：接收器是一个函数 f，而方法 调用了 f。这看起来可能有些奇怪，但它与，比如说， 接收器是一个通道而方法在该通道上发送消息并没有太大不同。

要使 ArgServer 成为一个 HTTP 服务器，我们首先修改它 以拥有正确的签名。 

// 参数服务器。
func ArgServer(w http.ResponseWriter, req *http.Request) {
    fmt.Fprintln(w, os.Args)
}

ArgServer 现在具有与 HandlerFunc 相同的签名， 因此可以将其转换为该类型以访问其方法， 就像我们将 Sequence 转换为 IntSlice 以访问 IntSlice.Sort 一样。 设置它的代码很简洁： 

http.Handle("/args", http.HandlerFunc(ArgServer))

当有人访问页面 /args 时， 在该页面上安装的处理器值为 ArgServer 类型为 HandlerFunc。 HTTP 服务器将调用该类型的 ServeHTTP 方法， 以 ArgServer 作为接收器，该方法接着将调用 ArgServer（通过 HandlerFunc.ServeHTTP 内部的调用 f(w, req)）。 然后参数就会被显示出来。

在本节中，我们用一个结构体、一个整数、 一个通道和一个函数构建了一个 HTTP 服务器，这一切都因为接口只是 方法的集合，而（几乎）任何类型都可以定义这些方法。

空白标识符

我们之前已经在 for range 循环 和映射的上下文中多次提到过空白标识符。 空白标识符可以被赋值或声明为任何类型的任意值，该值会被安全地丢弃。 这有点像向 Unix 的 /dev/null 文件写入： 它代表一个只写的值， 用作需要变量但实际值无关紧要时的占位符。 它的用途远不止我们已经见过的那些。

多重赋值中的空白标识符

在 for range 循环中使用空白标识符是 一种通用情况——多重赋值——的特例。

如果赋值语句左侧需要多个值， 但其中一个值不会被程序使用， 在赋值语句左侧使用空白标识符可以避免 创建伪变量，并清楚表明该值将被丢弃。 例如，当调用一个返回值和错误的函数， 但只有错误重要时，可以使用空白标识符丢弃无关的值。 

if _, err := os.Stat(path); os.IsNotExist(err) {
    fmt.Printf("%s does not exist\n", path)
}

偶尔你会看到为了忽略错误而丢弃错误值的代码；这是糟糕的做法。务必检查错误返回值；它们存在是有理由的。 

// 糟糕！如果路径不存在，此代码将崩溃。
fi, _ := os.Stat(path)
if fi.IsDir() {
    fmt.Printf("%s is a directory\n", path)
}

为副作用导入

像上一个例子中 fmt 或 io 这样未使用的导入 最终应该被使用或删除： 空白赋值标识代码是正在开发中的。 但有时，仅为了其副作用而导入一个包是有用的， 无需显式使用。 例如，在 net/http/pprof 包的 init 函数期间， 它注册了提供调试信息的 HTTP 处理器。它有一个导出的 API，但 大多数客户端只需要处理器注册并通过网页访问数据。 为了仅为其副作用导入包，请将包重命名为空白标识符： 

import _ "net/http/pprof"

接口检查

正如我们在前面讨论接口时所看到的， 一个类型无需显式声明它实现了某个接口。 相反，一个类型只需实现接口的方法就实现了该接口。 实际上，大多数接口转换是静态的，因此在编译时检查。 例如，将一个 *os.File 传递给一个 期望 io.Reader 的函数将无法编译，除非 *os.File 实现了 io.Reader 接口。

不过，有些接口检查确实发生在运行时。 一个例子是在 encoding/json 包中，它定义了一个 Marshaler 接口。当 JSON 编码器接收到一个实现了该接口的值时， 编码器会调用该值的编组方法将其转换为 JSON， 而不是执行标准转换。 编码器在运行时通过类型断言检查此属性，如下所示： 

m, ok := val.(json.Marshaler)

如果只需要询问一个类型是否实现了某个接口， 而不实际使用接口本身（也许作为错误检查的一部分）， 可以使用空白标识符忽略类型断言后的值： 

if _, ok := val.(json.Marshaler); ok {
    fmt.Printf("value %v of type %T implements json.Marshaler\n", val, val)
}

在包内实现类型时，有时需要确保该类型确实满足接口。例如，如果json.RawMessage类型需要自定义JSON表示形式，它就应该实现json.Marshaler接口，但这里没有静态转换能让编译器自动验证这一点。如果该类型不慎未能满足接口要求，JSON编码器仍会工作，但不会使用自定义实现。为确保实现正确性，可以在包中使用空标识符进行全局声明： 

var _ json.Marshaler = (*RawMessage)(nil)

在此声明中，将*RawMessage转换为Marshaler的赋值操作要求*RawMessage必须实现Marshaler接口，这一特性将在编译时进行检查。若json.Marshaler接口发生变更，此包将无法通过编译，从而提醒我们需要进行更新。

此结构中出现的空标识符表明该声明仅用于类型检查，而非创建变量。不过，无需对每个实现接口的类型都这样做。按照惯例，此类声明通常仅在代码中尚未存在静态转换时才会使用，这种情况相对少见。

嵌入

Go语言并未提供典型的、基于类型的子类化机制，但它具备通过将类型嵌入到结构体或接口中来"借用"部分实现的能力。

接口嵌入非常简单。我们之前已提及io.Reader和io.Writer接口；以下是它们的定义。 

type Reader interface {
    Read(p []byte) (n int, err error)
}
type Writer interface {
    Write(p []byte) (n int, err error)
}

io 包还导出了若干其他接口，用于定义可以实现多个此类方法的对象。 例如，存在一个 io.ReadWriter 接口，它同时包含 Read 和 Write 方法。 我们可以通过显式列出这两个方法来定义 io.ReadWriter，但更简单且更直观的方式是嵌入这两个接口来构成新接口，如下所示： 

// ReadWriter 是组合了 Reader 和 Writer 接口的接口。
type ReadWriter interface {
    Reader
    Writer
}

这段代码的含义正如其表面所示：一个 ReadWriter 既能做 Reader 能做的事，也能做 Writer 能做的事；它是所嵌入接口的联合体。 只有接口可以被嵌入到其他接口中。

同样的基本思想也适用于结构体，但其影响更为深远。bufio 包有两个结构体类型：bufio.Reader 和 bufio.Writer，它们各自当然都实现了 io 包中的相应接口。 而 bufio 也通过使用嵌入，将一个读取器和一个写入器组合到一个结构体中，从而实现了一个带缓冲的读取器/写入器：它在结构体内部列出了这些类型，但没有为它们指定字段名。 

// ReadWriter 存储了指向 Reader 和 Writer 的指针。
// 它实现了 io.ReadWriter。
type ReadWriter struct {
    *Reader  // *bufio.Reader
    *Writer  // *bufio.Writer
}

嵌入的元素是指向结构体的指针，因此在使用前必须初始化，使其指向有效的结构体。 这个 ReadWriter 结构体也可以写成： 

type ReadWriter struct {
    reader *Reader
    writer *Writer
}

但这样一来，为了提升这些字段的方法并满足 io 接口，我们还需要提供转发方法，如下所示： 

func (rw *ReadWriter) Read(p []byte) (n int, err error) {
    return rw.reader.Read(p)
}

通过直接嵌入结构体，我们避免了这种样板代码。 嵌入类型的方法会自动被继承，这意味着 bufio.ReadWriter 不仅拥有 bufio.Reader 和 bufio.Writer 的方法， 还同时满足以下三个接口： io.Reader、 io.Writer 和 io.ReadWriter。

嵌入与子类化有一个重要的区别。当我们嵌入一个类型时，该类型的方法会成为外部类型的方法， 但当这些方法被调用时，方法的接收者是内部类型，而不是外部类型。 在我们的示例中，当调用 bufio.ReadWriter 的 Read 方法时，其效果与上面写出的转发方法完全相同； 接收者是 ReadWriter 的 reader 字段，而不是 ReadWriter 本身。

嵌入也可以是一种简单的便利。 此示例展示了一个嵌入字段与一个常规命名字段并存的情况。 

type Job struct {
    Command string
    *log.Logger
}

Job 类型现在拥有了 *log.Logger 的 Print、Printf、Println 及其他方法。当然，我们也可以给 Logger 一个字段名，但这并非必须。现在，一旦初始化完成，我们就可以向 Job 记录日志了： 

job.Println("starting now...")

Logger 是 Job 结构体的一个常规字段，因此我们可以在 Job 的构造函数内部以通常的方式对其进行初始化，像这样： 

func NewJob(command string, logger *log.Logger) *Job {
    return &Job{command, logger}
}

或者使用复合字面量： 

job := &Job{command, log.New(os.Stderr, "Job: ", log.Ldate)}

如果我们需要直接引用嵌入字段，可以使用该字段的类型名（忽略包限定符）作为字段名，就像我们在 ReadWriter 结构体的 Read 方法中所做的那样。 在这里，如果我们需要访问 Job 变量 job 的 *log.Logger，我们会写作 job.Logger。 这在我们想要细化 Logger 的方法时会很有用。 

func (job *Job) Printf(format string, args ...interface{}) {
    job.Logger.Printf("%q: %s", job.Command, fmt.Sprintf(format, args...))
}

嵌入类型会引入名称冲突的问题，但其解决规则很简单。 首先，字段或方法 X 会隐藏类型中更深层嵌套部分的任何其他名为 X 的项。 如果 log.Logger 包含一个名为 Command 的字段或方法，那么 Job 的 Command 字段将覆盖它。

其次，如果相同的名称出现在相同的嵌套层级，通常是一个错误； 如果 Job 结构体包含另一个名为 Logger 的字段或方法，那么嵌入 log.Logger 就是错误的。 然而，如果这个重复的名称在类型定义之外的程序中从未被提及，那么它是允许的。 这一限制为防止外部嵌入类型被修改提供了某种保护；如果添加了一个字段，该字段与另一个子类型中的另一个字段冲突，但两个字段都从未被使用过，那就没有问题。

并发

通过通信来共享

并发编程是个大话题，这里仅能涉及一些Go语言特有的亮点。

在许多环境中，并发编程之所以困难，是因为要正确访问共享变量需要极其小心。Go鼓励采用不同的方式：让共享的值在通道上传递，并且实际上任何时刻都不会被多个执行线程主动共享。在任意给定时间，只有一个goroutine可以访问该值。从设计上就杜绝了数据竞争的可能性。为了推广这种思维方式，我们将其提炼为一句口号：

不要通过共享内存来通信； 而要通过通信来共享内存。

这种思路可能被过度使用。例如，用互斥锁包裹一个整型变量来实现引用计数，可能就是最佳选择。但作为一种高级方法，使用通道来控制访问，能让编写清晰、正确的程序变得更加容易。

思考这个模型的一种方式是：设想一个典型的单线程程序运行在单个CPU上。它无需任何同步原语。现在再运行一个这样的实例；它同样无需同步。接着让这两个实例通信；如果通信本身就是同步器，那么依然不需要其他同步机制。例如，Unix管道就完美契合这个模型。虽然Go的并发方案源于霍尔（Hoare）的通信顺序进程（CSP），但它也可以被看作Unix管道的一种类型安全的泛化。

Goroutines

它们被称为goroutine，是因为现有的术语——线程、协程、进程等——都带有不准确的隐含意义。goroutine的模型很简单：它是一个函数，与其他goroutine在同一个地址空间中并发执行。它非常轻量，其开销主要就是分配栈空间。并且栈初始很小，因此开销低，并按需通过分配（和释放）堆内存来增长。

Goroutine会被多路复用到多个操作系统线程上，因此如果其中一个阻塞了（例如在等待I/O），其他goroutine可以继续运行。这种设计隐藏了线程创建和管理的许多复杂性。

在函数或方法调用前加上go关键字，就可以在一个新的goroutine中运行该调用。调用完成后，该goroutine会静默退出。（其效果类似于Unix shell中用&符号在后台运行命令。） 

go list.Sort()  // 并发运行 list.Sort；不等待它完成。

函数字面量在goroutine调用中非常方便。 

func Announce(message string, delay time.Duration) {
    go func() {
        time.Sleep(delay)
        fmt.Println(message)
    }()  // 注意括号 - 必须调用该函数。
}

在Go中，函数字面量是闭包：其实现确保了函数所引用的变量在其活跃期间能够一直存在。

这些例子不太实用，因为函数没有方法来发出完成的信号。为此，我们需要通道。

Channels

与映射类似，通道使用make分配，其结果值充当底层数据结构的引用。如果提供了可选的整型参数，它将设置通道的缓冲区大小。默认值是零，即无缓冲或同步通道。 

ci := make(chan int)            // 无缓冲的整型通道
cj := make(chan int, 0)         // 无缓冲的整型通道
cs := make(chan *os.File, 100)  // 有缓冲的File指针通道

无缓冲通道将通信——即值的交换——与同步——保证两个计算（goroutine）处于已知状态——结合在一起。

使用通道有很多巧妙的惯用法。这里先介绍一个来入门。在上一节中，我们后台启动了一个排序任务。使用通道可以让启动排序的goroutine等待排序完成。 

c := make(chan int)  // 分配一个通道。
// 在一个goroutine中启动排序；完成后，通过通道发出信号。
go func() {
    list.Sort()
    c <- 1  // 发送一个信号；值是什么并不重要。
}()
doSomethingForAWhile()
<-c   // 等待排序完成；丢弃发送过来的值。

接收者总是会阻塞，直到有数据可以接收。如果通道是无缓冲的，发送者会阻塞，直到接收者已接收到值。如果通道有缓冲，发送者仅在值已复制到缓冲区时阻塞；如果缓冲区已满，则意味着需要等待直到某个接收者取走一个值。

缓冲通道可以用作信号量，例如用来限制吞吐量。在这个例子中，传入的请求被传递给handle，该函数向通道发送一个值，处理请求，然后从通道接收一个值，从而为下一个消费者准备好这个“信号量”。通道缓冲区的容量限制了对process的并发调用数量。 

var sem = make(chan int, MaxOutstanding)

func handle(r *Request) {
    sem <- 1    // 等待活跃队列排空。
    process(r)  // 可能耗时很长。
    <-sem       // 完成；允许下一个请求运行。
}

func Serve(queue chan *Request) {
    for {
        req := <-queue
        go handle(req)  // 不等待handle完成。
    }
}

一旦有MaxOutstanding个处理器正在执行process，任何更多的处理器在尝试向已满的通道缓冲区发送时都会阻塞，直到现有某个处理器完成并从缓冲区接收值。

然而，这种设计存在一个问题：Serve 函数会为每个传入请求创建新的 goroutine，即使在任何时刻只有 MaxOutstanding 个请求能实际运行。如果请求到来速度过快，程序可能会消耗无限的资源。我们可以通过修改 Serve 函数来限制 goroutine 的创建以解决这个缺陷： 

func Serve(queue chan *Request) {
    for req := range queue {
        sem <- 1
        go func() {
            process(req)
            <-sem
        }()
    }
}

（注意在 Go 1.22 之前的版本中，这段代码存在一个 bug：循环变量在所有 goroutine 间共享。详情请参阅 Go wiki。）

另一种有效管理资源的方法是启动固定数量的 handle goroutine，全部从请求 channel 中读取数据。goroutine 的数量限制了对 process 函数的并发调用次数。这个 Serve 函数还接受一个 channel，用于接收退出信号；在启动 goroutine 后，它会阻塞在该 channel 上等待退出指令。 

func handle(queue chan *Request) {
    for r := range queue {
        process(r)
    }
}

func Serve(clientRequests chan *Request, quit chan bool) {
    // 启动处理器
    for i := 0; i < MaxOutstanding; i++ {
        go handle(clientRequests)
    }
    <-quit  // 等待退出指令。
}

通道的通道

Go 语言最重要的特性之一是 channel 作为一等值，可以像其他任何值一样被分配和传递。这个特性的一个常见用途是实现安全的并行解复用。

在上一节的例子中，handle 是一个理想化的请求处理器，但我们没有定义它处理的类型。如果该类型包含一个用于回复的 channel，每个客户端就可以提供自己接收答案的路径。以下是 Request 类型的示意定义。 

type Request struct {
    args        []int
    f           func([]int) int
    resultChan  chan int
}

客户端提供函数及其参数，以及请求对象中用于接收答案的 channel。 

func sum(a []int) (s int) {
    for _, v := range a {
        s += v
    }
    return
}

request := &Request{[]int{3, 4, 5}, sum, make(chan int)}
// 发送请求
clientRequests <- request
// 等待响应。
fmt.Printf("answer: %d\n", <-request.resultChan)

在服务器端，只有处理函数需要更改。 

func handle(queue chan *Request) {
    for req := range queue {
        req.resultChan <- req.f(req.args)
    }
}

显然，要使其更实用还有很多工作要做，但这段代码为构建一个速率受限、并行、非阻塞的 RPC 系统提供了框架，而且完全没有使用互斥锁。

并行化

这些思想的另一个应用是在多个 CPU 核心上并行化计算。如果计算可以分解成可独立执行的独立部分，那么就可以进行并行化，并使用 channel 来信号通知每个部分何时完成。

假设我们要对一组项执行一个计算量大的操作，并且每个项的操作值是独立的，如这个理想化的示例所示。 

type Vector []float64

// 将操作应用于 v[i], v[i+1] ... 直到 v[n-1]。
func (v Vector) DoSome(i, n int, u Vector, c chan int) {
    for ; i < n; i++ {
        v[i] += u.Op(v[i])
    }
    c <- 1    // 信号通知此部分已完成
}

我们在循环中独立启动各部分，每个 CPU 一个。它们可以以任何顺序完成，但这无关紧要；我们只需在启动所有 goroutine 后，通过排空 channel 来统计完成信号。 

const numCPU = 4 // CPU 核心数

func (v Vector) DoAll(u Vector) {
    c := make(chan int, numCPU)  // 缓冲可选但合理。
    for i := 0; i < numCPU; i++ {
        go v.DoSome(i*len(v)/numCPU, (i+1)*len(v)/numCPU, u, c)
    }
    // 排空 channel。
    for i := 0; i < numCPU; i++ {
        <-c    // 等待一个任务完成
    }
    // 全部完成。
}

除了为 numCPU 创建常量值，我们还可以询问运行时合适的值。runtime.NumCPU 函数返回机器的硬件 CPU 核心数，因此我们可以这样写： 

var numCPU = runtime.NumCPU()

还有一个函数 runtime.GOMAXPROCS，它报告（或设置）Go 程序可以同时运行的用户指定核心数。默认值等于 runtime.NumCPU 的值，但可以通过设置同名的 shell 环境变量或使用正整数调用该函数来覆盖。使用零值调用则只查询当前值。因此，如果我们想遵循用户的资源请求，应该这样写： 

var numCPU = runtime.GOMAXPROCS(0)

请注意不要混淆并发（将程序构造成独立执行的组件）与并行（为在多 CPU 上提高效率而并行执行计算）的概念。虽然 Go 的并发特性可以使某些问题易于构造成并行计算，但 Go 是一种并发语言，而非并行语言，并非所有并行化问题都适合 Go 的模型。关于这一区别的讨论，请参阅 此博文 中引用的演讲。

一个有泄漏的缓冲区

并发编程的工具甚至可以让非并发的思想更易于表达。下面是一个从RPC包中抽象出来的例子。客户端goroutine循环从某个源（可能是网络）接收数据。为了避免反复分配和释放缓冲区，它维护了一个空闲列表，并使用一个有缓冲的channel来表示。如果channel为空，则分配一个新的缓冲区。一旦消息缓冲区准备就绪，它就被发送到服务器的 serverChan 上。 

var freeList = make(chan *Buffer, 100)
var serverChan = make(chan *Buffer)

func client() {
    for {
        var b *Buffer
        // 尝试获取一个缓冲区；如果没有可用的，则分配一个。
        select {
        case b = <-freeList:
            // 得到了一个；无需再做其他操作。
        default:
            // 没有空闲的，分配一个新的。
            b = new(Buffer)
        }
        load(b)              // 从网络读取下一条消息。
        serverChan <- b      // 发送给服务器。
    }
}

服务器循环接收来自客户端的每条消息，处理它，然后将缓冲区返回给空闲列表。 

func server() {
    for {
        b := <-serverChan    // 等待工作。
        process(b)
        // 如果有空间，则重用缓冲区。
        select {
        case freeList <- b:
            // 缓冲区已放回空闲列表；无需再做其他操作。
        default:
            // 空闲列表已满，继续执行。
        }
    }
}

客户端尝试从 freeList 获取缓冲区；如果没有可用的，它会分配一个新的。服务器向 freeList 发送时，会将 b 放回空闲列表，除非列表已满，在这种情况下缓冲区会被丢弃，由垃圾回收器回收。（select 语句中的 default 子句在其它case都未就绪时执行，这意味着这些 select 语句永远不会阻塞。）这个实现仅用几行代码就构建了一个泄漏桶式的空闲列表，它依赖有缓冲的channel和垃圾回收器来管理。

错误

库函数通常必须向调用者返回某种错误指示。 如前所述，Go的多返回值特性使得在返回正常值的同时返回详细的错误描述变得很容易。 利用这个特性来提供详细的错误信息是一种良好的风格。例如，正如我们将看到的，os.Open 在失败时不仅返回一个 nil 指针，还会返回一个描述错误原因的错误值。

按照惯例，错误的类型是 error，这是一个简单的内置接口。 

type error interface {
    Error() string
}

库作者可以在底层用更丰富的模型来实现这个接口，使得不仅能够看到错误，还能提供一些上下文信息。如前所述，除了通常的 *os.File 返回值外，os.Open 还返回一个错误值。如果文件打开成功，错误将为 nil，但当出现问题时，它会包含一个 os.PathError： 

// PathError 记录一个错误以及导致该错误的操作和文件路径。
type PathError struct {
    Op string    // "open", "unlink" 等。
    Path string  // 关联的文件。
    Err error    // 系统调用返回的错误。
}

func (e *PathError) Error() string {
    return e.Op + " " + e.Path + ": " + e.Err.Error()
}

PathError 的 Error 方法生成类似这样的字符串： 

open /etc/passwx: no such file or directory

这样一个包含了问题文件名、操作以及它触发的操作系统错误的错误信息，即使在远离导致它的调用的地方打印出来，也非常有用；它比单纯的 "no such file or directory" 信息量丰富得多。

在可行的情况下，错误字符串应标识其来源，例如通过添加一个前缀来命名产生错误的操作或包。例如，在 image 包中，由于未知格式导致的解码错误，其字符串表示为 "image: unknown format"。

关心精确错误细节的调用者可以使用类型开关或类型断言来查找特定错误并提取详细信息。对于 PathError，这可能包括检查内部的 Err 字段以判断是否是可恢复的失败。 

for try := 0; try < 2; try++ {
    file, err = os.Create(filename)
    if err == nil {
        return
    }
    if e, ok := err.(*os.PathError); ok && e.Err == syscall.ENOSPC {
        deleteTempFiles()  // 回收一些空间。
        continue
    }
    return
}

这里的第二个 if 语句是另一个类型断言。如果它失败，ok 将为 false，而 e 将为 nil。如果它成功，ok 将为 true，这意味着错误的类型是 *os.PathError，因此 e 也是，我们可以检查它以获取关于错误的更多信息。

恐慌

向调用者报告错误的常用方法是将一个 error 作为额外的返回值返回。典型的 Read 方法就是一个众所周知的例子；它返回一个字节数和一个 error。但如果错误是不可恢复的呢？有时程序根本无法继续执行。

为此，有一个内置函数 panic，它实际上创建一个运行时错误，该错误将停止程序（但请参阅下一节）。该函数接受一个任意类型的单一参数（通常是一个字符串），在程序终止时将其打印出来。这也是表示发生了不可能的情况的一种方式，例如退出无限循环。 

// 使用牛顿法实现的立方根玩具示例。
func CubeRoot(x float64) float64 {
    z := x/3   // 任意初始值
    for i := 0; i < 1e6; i++ {
        prevz := z
        z -= (z*z*z-x) / (3*z*z)
        if veryClose(z, prevz) {
            return z
        }
    }
    // 一百万次迭代仍未收敛；说明出了问题。
    panic(fmt.Sprintf("CubeRoot(%g) did not converge", x))
}

这只是一个例子，但真正的库函数应该避免使用 panic。如果问题可以被掩盖或绕过，让程序继续运行总是比终止整个程序要好。一个可能的例外是在初始化期间：如果库确实无法完成自身设置，那么使用恐慌，可以说，是合理的。 

var user = os.Getenv("USER")

func init() {
    if user == "" {
        panic("no value for $USER")
    }
}

Recover

当调用 panic 时（包括因运行时错误，如切片索引越界或类型断言失败而隐式调用），它会立即停止当前函数的执行，并开始展开（unwinding）该 goroutine 的栈，同时运行沿途的所有延迟函数。如果展开过程到达了该 goroutine 栈的顶部，程序就会终止。然而，可以使用内置函数 recover 来重新控制该 goroutine 并恢复正常执行。

对 recover 的调用会停止栈展开过程，并返回传递给 panic 的参数。因为在展开过程中唯一运行的代码是在延迟函数内部，所以 recover 只在延迟函数内部才有用。

recover 的一个应用场景是：在服务器内部关闭一个发生故障的 goroutine，而不影响其他正在执行的 goroutine。 

func server(workChan <-chan *Work) {
    for work := range workChan {
        go safelyDo(work)
    }
}

func safelyDo(work *Work) {
    defer func() {
        if err := recover(); err != nil {
            log.Println("work failed:", err)
        }
    }()
    do(work)
}

在这个例子中，如果 do(work) 发生恐慌，其结果会被记录下来，该 goroutine 会干净地退出，而不会干扰到其他 goroutine。在延迟闭包中不需要做其他任何事情；调用 recover 就能完全处理这种情况。

因为除非直接从延迟函数中调用，否则 recover 总是返回 nil，所以延迟代码可以调用那些自身也使用 panic 和 recover 的库函数而不会失败。例如，safelyDo 中的延迟函数可以在调用 recover 之前调用一个日志函数，而这段日志代码会不受恐慌状态的影响而正常运行。

有了我们的恢复模式，do 函数（以及它调用的任何代码）就可以通过调用 panic 来干净地摆脱任何糟糕的境地。我们可以利用这个思路来简化复杂软件中的错误处理。让我们来看一个理想化的 regexp 包版本，它通过使用一个本地错误类型调用 panic 来报告解析错误。这里是 Error 的定义、一个 error 方法和 Compile 函数。 

// Error 是解析错误的类型；它满足 error 接口。
type Error string
func (e Error) Error() string {
    return string(e)
}

// error 是 *Regexp 的一个方法，它通过用一个 Error 触发恐慌来报告解析错误。
func (regexp *Regexp) error(err string) {
    panic(Error(err))
}

// Compile 返回正则表达式的解析表示。
func Compile(str string) (regexp *Regexp, err error) {
    regexp = new(Regexp)
    // 如果存在解析错误，doParse 将触发恐慌。
    defer func() {
        if e := recover(); e != nil {
            regexp = nil    // 清除返回值。
            err = e.(Error) // 如果不是解析错误，将重新触发恐慌。
        }
    }()
    return regexp.doParse(str), nil
}

如果 doParse 发生恐慌，恢复代码块将把返回值设置为 nil—延迟函数可以修改命名的返回值。然后，在对 err 的赋值中，它会通过断言该值具有本地类型 Error 来检查问题是否为解析错误。如果不是，类型断言将失败，导致一个运行时错误，这会像没有中断过一样继续栈展开过程。这个检查意味着，如果发生了意外情况，例如索引越界，代码将会失败，尽管我们正在使用 panic 和 recover 来处理解析错误。

有了错误处理机制，error 方法（因为它是一个绑定到类型的方法，所以它与内置的 error 类型同名是合适的，甚至是自然的）使得报告解析错误变得容易，而无需担心手动展开解析栈： 

if pos == 0 {
    re.error("'*' illegal at start of expression")
}

尽管这种模式很有用，但它只应在包内部使用。Parse 将其内部的 panic 调用转换为 error 值；它不会向客户端暴露 panic。这是一个值得遵循的好规则。

顺便说一下，这个重新恐慌的惯用法会在实际错误发生时改变恐慌的值。然而，原始的和新的失败都会在崩溃报告中呈现，因此问题的根本原因仍然可见。所以这种简单的重新恐慌方法通常是足够的—毕竟这是一次崩溃—但如果你只想显示原始值，你可以编写更多的代码来过滤意外问题，并用原始错误重新恐慌。这留作读者的练习。
