引言

Go 是一种开源编程语言，旨在简洁、可靠和高效，专门设计用于轻松构建大规模软件。虽然它借鉴了现有语言的思想，但其独特的特性使得高效的 Go 程序在性质上与它的近亲语言所编写的程序有所不同。将 C++ 或 Java 程序简单直接地翻译成 Go 不太可能产生令人满意的结果——Java 程序是用 Java 写的，而不是用 Go。另一方面，从 Go 的视角来思考问题，则可能产出一个成功且截然不同的程序。换句话说，要写好 Go 代码，理解其特性和惯用法至关重要。同样重要的是了解 Go 编程的既定规范，如命名、格式、程序结构等，以便其他 Go 程序员能够轻松理解你编写的程序。

本文档提供编写清晰、地道 Go 代码的技巧。它对 语言规范、Go 语言之旅 以及 如何编写 Go 代码 进行了补充，你应该首先阅读这些内容。

格式化

格式化问题是最有争议但影响最小的问题。人们可以适应不同的格式化风格，但最好不必如此；如果每个人都遵循相同的风格，就能在这个话题上花费更少的时间。问题在于，如何在没有冗长规范风格指南的情况下接近这种理想状态。

在 Go 中，我们采取了一种不同寻常的方法，让机器处理大部分格式化问题。`gofmt` 程序（也可以通过 `go fmt` 调用，它在包级别而非源文件级别运作）读取一个 Go 程序，并以标准缩进和垂直对齐的风格输出源代码，同时保留并在必要时重新格式化注释。如果你想知道如何处理某种新的布局情况，运行 `gofmt`；如果结果看起来不对，调整你的程序（或提交一个关于 `gofmt` 的 bug），而不是变通处理。

例如，无需花费时间去对齐结构体字段的注释。`gofmt` 会为你完成这项工作。给定如下声明： 

type T struct {
    name string // 对象的名称
    value int // 对象的值
}

`gofmt` 会对齐这些列： 

type T struct {
    name    string // 对象的名称
    value   int    // 对象的值
}

标准包中的所有 Go 代码均已使用 `gofmt` 格式化。

还有一些格式化细节。简要说明如下：

缩进
我们使用制表符进行缩进，`gofmt` 默认也输出制表符。仅在必须时才使用空格。
行长度
Go 没有行长度限制。不用担心超出打孔卡宽度。如果一行感觉太长，请换行并使用一个额外的制表符缩进。
圆括号
Go 需要的圆括号比 C 和 Java 少：控制结构（`if`、`for`、`switch`）的语法中没有圆括号。 此外，运算符优先级层次更简短清晰，因此 
x<<8 + y<<16
的含义如其空格所示，这与其他语言不同。

注释

Go 提供 C 风格的 `/* */` 块注释和 C++ 风格的 `//` 行注释。行注释是常态；块注释主要作为包注释出现，但在表达式内部或用于禁用大段代码时也很有用。

出现在顶层声明之前（中间没有空行）的注释被视为对该声明本身的文档。这些“文档注释”是给定 Go 包或命令的主要文档。关于文档注释的更多信息，请参阅“Go 文档注释”。

命名

命名在 Go 中和在其他任何语言中一样重要。它们甚至具有语义效果：名称在包外的可见性由其首字母是否大写决定。因此，值得花点时间谈谈 Go 程序中的命名规范。

包名

当一个包被导入时，包名就成为其内容的访问器。在 

import "bytes"

导入包的代码可以通过 bytes.Buffer 来访问该包的内容。为了让所有使用该包的人都能用相同的名称来指代其内容，包名应当简洁、贴切、易记。按照惯例，包名通常采用全小写、单个单词的形式，无需使用下划线或混合大小写。 应尽量选用简短的包名，因为所有使用你的包的人都要频繁输入这个名字。 不必事先担心命名冲突。包名仅仅是导入时的默认名称，无需在所有源代码中唯一；即使偶尔发生冲突，导入方也可以在本地选择不同的别名。无论如何，冲突很少见，因为导入路径中的文件名决定了实际使用的是哪个包。

另一个惯例是：包名应与其源代码目录的基名一致；例如 src/encoding/base64 中的包被导入为 "encoding/base64"，但其包名为 base64，而非 encoding_base64 或 encodingBase64。

包的使用者将通过包名来引用其内容，因此包内的导出名称可利用这一点避免重复。 （应避免使用 import . 符号，它虽可简化必须在测试包外部运行的测试场景，但其他情况下应避免使用。）例如，bufio 包中的带缓冲读取器类型名为 Reader，而非 BufReader，因为用户会将其视为 bufio.Reader——这是一个清晰简洁的名称。此外，由于导入的实体总是通过包名访问，bufio.Reader 不会与 io.Reader 冲突。类似地，创建 ring.Ring 新实例的函数——在 Go 中称为“构造函数”——通常命名为 NewRing，但由于 Ring 是该包唯一导出的类型，且包名为 ring，因此它直接称为 New，包的使用者会看到 ring.New。 善用包结构有助于选择优秀的名称。

另一个简短例子是 once.Do；once.Do(setup) 可读性很好，写成 once.DoOrWaitUntilDone(setup) 并不会更优。长名称并不自动提升可读性，有时一条有用的文档注释比冗长的名称更有价值。

Getters

Go 不会自动提供 getter 和 setter。自行实现 getter 和 setter 并无不妥，且常常是合适的，但不必在 getter 方法名中加入 Get 前缀。若字段名为 owner（小写、未导出），其 getter 方法应命名为 Owner（大写、导出），而非 GetOwner。通过大写名称导出即可区分字段与方法。若需 setter，通常命名为 SetOwner。这两个名称在实际使用中可读性良好： 

owner := obj.Owner()
if owner != user {
    obj.SetOwner(user)
}

接口名称

按照惯例，单方法接口的命名通常采用方法名加 -er 后缀或类似修饰，以构成表示行为的名词：如 Reader、Writer、Formatter、CloseNotifier 等。

这类名称已有众多范例，遵循它们及其所表达的函数含义是有效的做法。Read、Write、Close、Flush、String 等方法具有规范的签名和语义。为避免混淆，除非你的方法具有相同的签名和语义，否则不要使用这些名称。反之，若你的类型实现了与知名类型中同义的方法，应赋予相同的名称和签名；例如将字符串转换方法命名为 String 而非 ToString。

MixedCaps

最后，Go 的惯例是使用 MixedCaps 或 mixedCaps（混合大小写）而非下划线来书写多单词名称。

分号

与 C 类似，Go 的形式文法使用分号终止语句，但不同于 C 的是，这些分号不会出现在源代码中。相反，词法分析器采用简单规则在扫描时自动插入分号，因此源文本通常不含显式分号。

规则如下：若换行前的最后一个标记是标识符（包括 int、float64 等基本类型）、基本字面量（如数字或字符串常量），或以下标记之一： 

break continue fallthrough return ++ -- ) }

则词法分析器总在该标记后插入分号。可概括为“若换行出现在可能结束语句的标记后，则插入分号”。

在右花括号前也可省略分号，因此如下语句： 

    go func() { for { dst <- <-src } }()

无需添加分号。惯用的 Go 程序仅在 for 循环子句等处使用分号，以分隔初始化语句、条件判断和后置语句。若将多条语句写在同一行，也需要分号分隔。

分号插入规则的一个重要结论是：不能将控制结构（if、for、switch 或 select）的左花括号置于下一行。否则，花括号前会被插入分号，可能导致意外错误。应这样书写： 

if i < f() {
    g()
}

而不是这样写 

if i < f()  // 错误！
{           // 错误！
    g()
}

控制结构

Go 的控制结构与 C 相关，但在重要方面有所不同。Go 没有 do 或 while 循环，只有一种略微泛化的 for；switch 更加灵活；if 和 switch 可以接受一个可选的初始化语句，类似于 for；break 和 continue 语句可以带一个可选标签来指明要中断或继续的目标；此外还有一些新的控制结构，包括类型开关和多路通信复用器 select。语法也略有不同：没有圆括号，并且主体必须用花括号括起来。

If

在 Go 中，一个简单的 if 语句如下所示： 

if x > 0 {
    return y
}

强制使用花括号鼓励将简单的 if 语句写在多行。无论如何，这都是良好的风格，特别是当主体包含 return 或 break 等控制语句时。

由于 if 和 switch 接受初始化语句，通常会看到它们用来设置一个局部变量。 

if err := file.Chmod(0664); err != nil {
    log.Print(err)
    return err
}

在 Go 库中，你会发现，当一个 if 语句不会流入下一个语句时——即主体以 break、continue、goto 或 return 结束——不必要的 else 就会被省略。 

f, err := os.Open(name)
if err != nil {
    return err
}
codeUsing(f)

这是代码需要防护一系列错误条件的常见情况的一个例子。如果成功的控制流沿着页面向下运行，随着错误的出现而排除它们，代码会更易读。由于错误情况通常以 return 语句结束，因此生成的代码不需要 else 语句。 

f, err := os.Open(name)
if err != nil {
    return err
}
d, err := f.Stat()
if err != nil {
    f.Close()
    return err
}
codeUsing(f, d)

重新声明和重新赋值

附带一提：上一节的最后一个例子演示了 := 短声明形式工作方式的一个细节。调用 os.Open 的声明如下： 

f, err := os.Open(name)

这条语句声明了两个变量：f 和 err。几行之后，对 f.Stat 的调用如下： 

d, err := f.Stat()

这看起来像是声明了 d 和 err。但请注意，err 在两条语句中都出现了。这种重复是合法的：err 由第一条语句声明，但在第二条语句中只是重新赋值。这意味着对 f.Stat 的调用使用了上面声明的现有 err 变量，只是给它赋了一个新值。

在 := 声明中，即使变量 v 已经被声明，它也可以再次出现，前提是：

  • 此声明与 v 的现有声明位于同一作用域（如果 v 已在外部作用域中声明，该声明将创建一个新变量 §），
  • 初始化中对应的值可以赋值给 v，并且
  • 该声明至少创建了一个其他变量。

这种不寻常的特性纯粹是实用主义，例如，它使得在很长的 if-else 链中使用单个 err 值变得容易。你会经常看到它的使用。

§ 值得注意的是，在 Go 中，函数参数和返回值的作用域与函数体相同，即使它们在词法上位于包含主体的花括号之外。

For

Go 的 for 循环与 C 的相似——但并不相同。它统一了 for 和 while，并且没有 do-while。它有三种形式，其中只有一种使用分号。 

// 类似 C 的 for
for init; condition; post { }

// 类似 C 的 while
for condition { }

// 类似 C 的 for(;;)
for { }

短声明使得在循环中直接声明索引变量变得容易。 

sum := 0
for i := 0; i < 10; i++ {
    sum += i
}

如果你正在遍历数组、切片、字符串或映射，或者从通道读取，range 子句可以管理循环。 

for key, value := range oldMap {
    newMap[key] = value
}

如果你只需要范围中的第一个元素（键或索引），请省略第二个： 

for key := range m {
    if key.expired() {
        delete(m, key)
    }
}

如果你只需要范围中的第二个元素（值），请使用空白标识符，即下划线，来丢弃第一个： 

sum := 0
for _, value := range array {
    sum += value
}

空白标识符有许多用途，如后续章节所述。

对于字符串，range 会为你完成更多工作，通过解析UTF-8来分解出单个Unicode码点。 错误的编码会消耗一个字节并产生替换符文U+FFFD。 （名称（及关联的内置类型）rune 是Go术语中表示单个Unicode码点的方式。 详情请参见语言规范。） 这个循环 

for pos, char := range "日本\x80語" { // \x80 是一个非法的UTF-8编码
    fmt.Printf("character %#U starts at byte position %d\n", char, pos)
}

会打印 

character U+65E5 '日' starts at byte position 0
character U+672C '本' starts at byte position 3
character U+FFFD '�' starts at byte position 6
character U+8A9E '語' starts at byte position 7

最后，Go没有逗号运算符，并且 ++ 和 -- 是语句而不是表达式。 因此，如果你想在 for 中运行多个变量， 应使用并行赋值（尽管这不能使用 ++ 和 --）。 

// 反转 a
for i, j := 0, len(a)-1; i < j; i, j = i+1, j-1 {
    a[i], a[j] = a[j], a[i]
}

Switch

Go的 switch 比C语言的更为通用。 表达式不必是常量，甚至不必是整数， 各个分支从上到下求值，直到找到匹配项， 如果 switch 没有表达式，则切换到 true。 因此，将 if-else-if-else 链写成 switch 是可行且惯用的做法。 

func unhex(c byte) byte {
    switch {
    case '0' <= c && c <= '9':
        return c - '0'
    case 'a' <= c && c <= 'f':
        return c - 'a' + 10
    case 'A' <= c && c <= 'F':
        return c - 'A' + 10
    }
    return 0
}

没有自动贯穿（fall through），但可以使用逗号分隔的列表来呈现多个分支。 

func shouldEscape(c byte) bool {
    switch c {
    case ' ', '?', '&', '=', '#', '+', '%':
        return true
    }
    return false
}

尽管在Go中 break 语句不像在其他一些类C语言中那么常见，但它可用于提前终止 switch。 然而，有时需要跳出的是外围的循环，而不是switch本身，在Go中可以通过在循环上放置标签并“break”到该标签来实现。 这个例子展示了两种用法。 

Loop:
    for n := 0; n < len(src); n += size {
        switch {
        case src[n] < sizeOne:
            if validateOnly {
                break
            }
            size = 1
            update(src[n])

        case src[n] < sizeTwo:
            if n+1 >= len(src) {
                err = errShortInput
                break Loop
            }
            if validateOnly {
                break
            }
            size = 2
            update(src[n] + src[n+1]<<shift)
        }

当然，continue 语句也接受可选的标签，但它只适用于循环。

在结束本节时，这里有一个使用两个 switch 语句对字节切片进行比较的例程： 

// Compare 返回一个整数，按字典序比较两个字节切片。
// 如果 a == b 则结果为 0，如果 a < b 则为 -1，如果 a > b 则为 +1
func Compare(a, b []byte) int {
    for i := 0; i < len(a) && i < len(b); i++ {
        switch {
        case a[i] > b[i]:
            return 1
        case a[i] < b[i]:
            return -1
        }
    }
    switch {
    case len(a) > len(b):
        return 1
    case len(a) < len(b):
        return -1
    }
    return 0
}

类型开关

switch也可以用于发现接口变量的动态类型。 这种 类型开关 使用了类型断言的语法，并在括号内使用关键字 type。 如果switch在表达式中声明了一个变量，该变量将在每个分支中具有相应的类型。 在这种情况下，重新使用变量名称也是惯用做法，实际上在每个分支中声明了一个同名但类型不同的新变量。 

var t interface{}
t = functionOfSomeType()
switch t := t.(type) {
default:
    fmt.Printf("unexpected type %T\n", t)     // %T 打印 t 的实际类型
case bool:
    fmt.Printf("boolean %t\n", t)             // t 的类型是 bool
case int:
    fmt.Printf("integer %d\n", t)             // t 的类型是 int
case *bool:
    fmt.Printf("pointer to boolean %t\n", *t) // t 的类型是 *bool
case *int:
    fmt.Printf("pointer to integer %d\n", *t) // t 的类型是 *int
}

多返回值

Go的一个不寻常的特性是函数和方法可以返回多个值。 这种形式可用于改进C程序中一些笨拙的惯用法：带内的错误返回（例如用 -1 表示 EOF） 以及修改通过地址传递的参数。

在C中，写入错误通过负计数来指示，错误代码隐藏在一个volatile位置中。 在Go中，Write 可以返回计数 和 一个错误：“是的，你写入了一些字节，但不是全部，因为你填满了设备”。 os 包中文件的 Write 方法签名为： 

func (file *File) Write(b []byte) (n int, err error)

正如文档所说，当 n != len(b) 时，它返回写入的字节数和一个非nil的 error。 这是一种常见的风格；有关更多示例，请参见关于错误处理的章节。

类似的方法可以避免向返回值传递指针来模拟引用参数。 这里是一个简单的函数，用于从字节切片中的某个位置抓取一个数字，返回该数字和下一个位置。 

func nextInt(b []byte, i int) (int, int) {
    for ; i < len(b) && !isDigit(b[i]); i++ {
    }
    x := 0
    for ; i < len(b) && isDigit(b[i]); i++ {
        x = x*10 + int(b[i]) - '0'
    }
    return x, i
}

你可以像这样使用它来扫描输入切片 b 中的数字： 

    for i := 0; i < len(b); {
        x, i = nextInt(b, i)
        fmt.Println(x)
    }

具名结果参数

Go 函数的返回或结果 "参数" 可以被命名，并像常规变量一样使用，就像传入参数一样。 当被命名时，它们在函数开始时会被初始化为其类型的零值；如果函数执行一个不带参数的 return 语句，则结果参数的当前值将用作返回值。

这些名称并非强制要求，但它们可以使代码更简短、更清晰：它们是文档。 如果我们命名 nextInt 的结果，那么哪个返回的 int 对应什么就变得显而易见了。 

func nextInt(b []byte, pos int) (value, nextPos int) {

因为具名结果是初始化的，并且与无参数的 return 关联，所以它们既能简化代码也能使其更清晰。 这是一个良好运用了它们的 io.ReadFull 版本： 

func ReadFull(r Reader, buf []byte) (n int, err error) {
    for len(buf) > 0 && err == nil {
        var nr int
        nr, err = r.Read(buf)
        n += nr
        buf = buf[nr:]
    }
    return
}

延迟执行

Go 的 defer 语句安排一个函数调用（延迟函数）在执行 defer 的函数返回之前立即运行。 这是一种不寻常但有效的方式，用于处理诸如资源释放之类的场景，无论函数通过哪条路径返回都必须释放这些资源。 典型的例子是解锁互斥锁或关闭文件。 

// Contents 返回文件内容作为字符串。
func Contents(filename string) (string, error) {
    f, err := os.Open(filename)
    if err != nil {
        return "", err
    }
    defer f.Close()  // f.Close 将在我们完成时运行。

    var result []byte
    buf := make([]byte, 100)
    for {
        n, err := f.Read(buf[0:])
        result = append(result, buf[0:n]...) // append 将在后面讨论。
        if err != nil {
            if err == io.EOF {
                break
            }
            return "", err  // f 会在这里返回时关闭。
        }
    }
    return string(result), nil // f 会在这里返回时关闭。
}

延迟调用 Close 这样的函数有两个优点。首先，它保证你永远不会忘记关闭文件，如果你后来编辑函数以添加新的返回路径，这是一个容易犯的错误。其次，它意味着关闭操作紧挨着打开操作，这比将其放在函数末尾要清晰得多。

延迟函数的参数（如果函数是方法，则包括接收器）在 defer 执行时求值，而不是在 call 执行时求值。除了不用担心变量在函数执行过程中改变值之外，这意味着一个单一的延迟调用点可以延迟多个函数执行。这里有一个简单的例子。 

for i := 0; i < 5; i++ {
    defer fmt.Printf("%d ", i)
}

延迟函数以 LIFO（后进先出）顺序执行，因此此代码将导致函数返回时打印 4 3 2 1 0。 一个更实际的例子是跟踪程序中函数执行的简单方法。我们可以编写几个简单的跟踪例程，像这样： 

func trace(s string)   { fmt.Println("entering:", s) }
func untrace(s string) { fmt.Println("leaving:", s) }

// 像这样使用它们：
func a() {
    trace("a")
    defer untrace("a")
    // 做些事情....
}

我们可以通过利用延迟函数的参数在 defer 执行时求值这一事实来做得更好。跟踪例程可以设置反跟踪例程的参数。 这个例子： 

func trace(s string) string {
    fmt.Println("entering:", s)
    return s
}

func un(s string) {
    fmt.Println("leaving:", s)
}

func a() {
    defer un(trace("a"))
    fmt.Println("in a")
}

func b() {
    defer un(trace("b"))
    fmt.Println("in b")
    a()
}

func main() {
    b()
}

打印 

entering: b
in b
entering: a
in a
leaving: a
leaving: b

对于习惯于其他语言基于代码块的资源管理的程序员来说，defer 可能看起来很特别，但它最有趣和最强大的应用恰恰来自于它不是基于代码块而是基于函数这一事实。 在关于 panic 和 recover 的章节中，我们将看到它的可能性的另一个例子。

数据

使用 new 进行分配

Go 有两种分配原语：内建函数 new 和 make。 它们所做的事情不同，适用的类型也不同，这可能会让人感到困惑，但规则很简单。 我们先讨论 new。 它是一个分配内存的内建函数，但与某些其他语言中的同名函数不同，它不会初始化内存，而只会将其清零。 也就是说，new(T) 为类型 T 的新变量分配清零的存储空间，并返回其地址，即类型为 *T 的值。 用 Go 的术语来说，它返回一个指向新分配的类型 T 零值的指针。

从 Go 1.26 开始，new 也接受一个（值）表达式作为参数，用于指定变量的初始值。 例如，new(int64(300)) 会分配一个类型为 int64 的新变量，将其初始化为 300，并返回其地址。

由于 new 返回的内存是清零的，因此在设计数据结构时，安排每种类型的零值无需进一步初始化即可使用会很有帮助。这意味着数据结构的用户可以使用 new 创建一个实例并立即开始使用。 例如，bytes.Buffer 的文档说明：“Buffer 的零值是一个已准备好使用的空缓冲区。” 同样，sync.Mutex 没有显式的构造函数或 Init 方法。 相反，sync.Mutex 的零值被定义为一个未锁定的互斥锁。

零值可用这一特性是传递性的。考虑以下类型声明。 

type SyncedBuffer struct {
    lock    sync.Mutex
    buffer  bytes.Buffer
}

类型为 SyncedBuffer 的值在分配或声明后也立即可用。在下面的代码片段中，p 和 v 都无需进一步设置即可正常工作。 

p := new(SyncedBuffer)  // type *SyncedBuffer
var v SyncedBuffer      // type  SyncedBuffer

构造函数和复合字面量

有时零值不够好，需要一个初始化构造函数，正如下面这个从 os 包派生而来的示例所示。 

func NewFile(fd int, name string) *File {
    if fd < 0 {
        return nil
    }
    f := new(File)
    f.fd = fd
    f.name = name
    f.dirinfo = nil
    f.nepipe = 0
    return f
}

这里有很多样板代码。我们可以使用复合字面量来简化它，复合字面量是一种每次求值时都会创建新实例的表达式。 

func NewFile(fd int, name string) *File {
    if fd < 0 {
        return nil
    }
    f := File{fd, name, nil, 0}
    return &f
}

请注意，与 C 语言不同，返回局部变量的地址是完全可以的；与变量关联的存储空间在函数返回后仍然存在。 实际上，对复合字面量取地址会在每次求值时分配一个新的实例，因此我们可以将最后两行合并。 

    return &File{fd, name, nil, 0}

复合字面量的字段按顺序排列，并且必须全部存在。 然而，通过将元素显式标记为 field:value 对，初始化器可以以任何顺序出现，缺失的字段将保留其各自的零值。因此，我们可以这样写： 

    return &File{fd: fd, name: name}

作为一种极端情况，如果复合字面量根本不包含任何字段，它会创建该类型的零值。表达式 new(File) 和 &File{} 是等价的。

复合字面量也可以用于数组、切片和映射，其字段标签分别是索引或映射键。 在这些示例中，只要 Enone、Eio 和 Einval 的值不同，初始化就能正常工作。 

a := [...]string   {Enone: "no error", Eio: "Eio", Einval: "invalid argument"}
s := []string      {Enone: "no error", Eio: "Eio", Einval: "invalid argument"}
m := map[int]string{Enone: "no error", Eio: "Eio", Einval: "invalid argument"}

使用 make 进行分配

回到分配的话题。 内建函数 make(T, args) 的用途与 new(T) 不同。 它只用于创建切片、映射和通道，并且返回一个已初始化（而非清零）的类型 T（而非 *T）的值。 之所以有这种区别，是因为这三种类型在底层代表了必须在使用前初始化的数据结构的引用。 例如，切片是一个包含三个字段的描述符：一个指向数据（在数组内）的指针、长度和容量，在这些字段初始化之前，切片是 nil。 对于切片、映射和通道，make 会初始化内部数据结构并为值的使用做好准备。 例如， 

make([]int, 10, 100)

会分配一个包含 100 个 int 的数组，然后创建一个长度为 10、容量为 100 的切片结构，指向该数组的前 10 个元素。 （创建切片时，容量可以省略；有关更多信息，请参阅切片章节。） 相比之下，new([]int) 返回一个指向新分配的、已清零的切片结构的指针，即一个指向 nil 切片值的指针。

这些示例说明了 new 和 make 之间的区别。 

var p *[]int = new([]int)       // allocates slice structure; *p == nil; rarely useful
var v  []int = make([]int, 100) // the slice v now refers to a new array of 100 ints
// 不必要的复杂写法： var p *[]int = new([]int) *p = make([]int, 100, 100) // 惯用写法： v := make([]int, 100)

请记住，make 只适用于映射（map）、切片（slice）和通道（channel），且不会返回指针。 若需获得显式指针，请使用 new 分配或直接取变量的地址。

数组

数组在规划内存详细布局时很有用，有时能帮助避免内存分配，但它们主要是作为构建切片的基础——切片将是下一节的主题。 为该主题奠定基础，这里先简要介绍一下数组。

Go 语言中数组的工作方式与 C 语言存在重大差异。 在 Go 中：

  • 数组是值类型。将一个数组赋值给另一个数组会复制所有元素。
  • 特别是，如果将数组传递给函数，函数接收的是该数组的副本，而非指向它的指针。
  • 数组的大小是其类型的一部分。[10]int 和 [20]int 是不同的类型。

值属性可能很有用，但也可能开销很大；如果您希望获得类似 C 的行为和效率，可以传递指向数组的指针。 

func Sum(a *[3]float64) (sum float64) {
    for _, v := range *a {
        sum += v
    }
    return
}

array := [...]float64{7.0, 8.5, 9.1}
x := Sum(&array)  // 注意显式的取地址运算符

但即便这种风格也不是惯用的 Go 写法。请改用切片。

切片

切片是对数组的包装，为数据序列提供了更通用、更强大和更便捷的接口。 除了像变换矩阵这类具有显式维度的项目外，Go 中的大多数数组编程都是通过切片而非简单数组完成的。

切片持有一个指向底层数组的引用，如果将一个切片赋值给另一个切片，两者将引用同一个数组。 如果一个函数接收切片参数，它对切片元素所做的修改对调用者是可见的，这类似于传递指向底层数组的指针。 因此，一个 Read 函数可以接收切片参数而非指针和计数；切片内的长度设置了要读取数据的上限。 以下是 os 包中 File 类型的 Read 方法的签名： 

func (f *File) Read(buf []byte) (n int, err error)

该方法返回读取的字节数以及一个错误值（如果有的话）。 要读取较大的缓冲区 buf 的前 32 个字节，请切片（此处用作动词）该缓冲区。 

    n, err := f.Read(buf[0:32])

这种切片操作既常见又高效。事实上，暂时抛开效率不谈，下面的代码片段也能读取缓冲区的前 32 个字节。 

    var n int
    var err error
    for i := 0; i < 32; i++ {
        nbytes, e := f.Read(buf[i:i+1])  // 读取一个字节。
        n += nbytes
        if nbytes == 0 || e != nil {
            err = e
            break
        }
    }

只要仍在底层数组的范围内，就可以更改切片的长度；只需将其赋值给它自身的一个切片即可。 切片的容量可通过内置函数 cap 访问，它报告了切片可以具有的最大长度。 下面是一个向切片追加数据的函数。如果数据超过容量，切片会被重新分配。 函数返回结果切片。该函数利用了这样一个事实：将 len 和 cap 应用于 nil 切片是合法的，并会返回 0。 

func Append(slice, data []byte) []byte {
    l := len(slice)
    if l + len(data) > cap(slice) {  // 重新分配
        // 分配所需空间的两倍，以备将来增长。
        newSlice := make([]byte, (l+len(data))*2)
        // copy 函数是预声明的，适用于任何切片类型。
        copy(newSlice, slice)
        slice = newSlice
    }
    slice = slice[0:l+len(data)]
    copy(slice[l:], data)
    return slice
}

我们必须返回该切片，因为尽管 Append 可以修改 slice 的元素，但切片本身（运行时持有的包含指针、长度和容量的数据结构）是按值传递的。

向切片追加数据这一思想非常有用，以至于它已被内置函数 append 所捕获。 然而，要理解该函数的设计，我们需要更多信息，因此稍后会再讨论它。

二维切片

Go 的数组和切片是一维的。 要创建等效的二维数组或切片，必须定义数组的数组或切片的切片，如下所示： 

type Transform [3][3]float64  // 一个 3x3 的数组，实际上是数组的数组。
type LinesOfText [][]byte     // 字节切片的切片。

由于切片是可变长度的，因此每个内部切片可以具有不同的长度。 这可能是常见的情况，就像我们的 LinesOfText 示例：每行都有独立的长度。 

text := LinesOfText{
    []byte("Now is the time"),
    []byte("for all good gophers"),
    []byte("to bring some fun to the party."),
}

有时需要分配一个二维切片，例如在处理像素扫描线时可能会遇到这种情况。 实现方式有两种。 一种是独立分配每个切片；另一种是分配一个单独的数组，并将各个切片指向其中。 使用哪种方式取决于您的应用程序。 如果切片可能增长或缩小，则应独立分配以避免覆盖下一行；如果不会，使用单次分配构建对象可能更高效。 为方便参考，以下是两种方法的简要概述。 首先，一次分配一行： 

// 分配顶层切片。
picture := make([][]uint8, YSize) // y 方向上每单位一行。
// 循环遍历每一行，为每行分配切片。
for i := range picture {
    picture[i] = make([]uint8, XSize)
}

现在改为单次分配，再将其切分为各行： 

// 分配顶层切片，与之前相同。
picture := make([][]uint8, YSize) // y 方向上每单位一行。
// 分配一个大切片以容纳所有像素。
pixels := make([]uint8, XSize*YSize) // 类型是 []uint8，即使 picture 是 [][]uint8。
// 循环遍历每一行，从剩余的 pixels 切片前部切出每一行。
for i := range picture {
    picture[i], pixels = pixels[:XSize], pixels[XSize:]
}

映射

映射是一种便捷且强大的内置数据结构，它将一种类型（键）的值与另一种类型（元素或值）的值关联起来。 键可以是任何定义了等式运算符的类型，例如整数、浮点数和复数、字符串、指针、接口（只要动态类型支持等式比较）、结构体和数组。 切片不能用作映射的键，因为切片没有定义等式比较。 与切片类似，映射持有对底层数据结构的引用。 如果你将一个映射传递给一个函数，并且该函数更改了映射的内容，那么这些更改对调用者是可见的。

映射可以使用常规的复合字面量语法构建，其中键值对由冒号分隔，因此在初始化过程中构建映射很容易。 

var timeZone = map[string]int{
    "UTC":  0*60*60,
    "EST": -5*60*60,
    "CST": -6*60*60,
    "MST": -7*60*60,
    "PST": -8*60*60,
}

映射值的赋值和获取在语法上与数组和切片的操作相同，只是索引不必是整数。 

offset := timeZone["EST"]

尝试使用映射中不存在的键来获取值，将返回该映射条目类型的零值。例如，如果映射包含整数，查找一个不存在的键将返回 0。 一个集合可以实现为值类型为 bool 的映射。将映射条目设置为 true 以将该值放入集合，然后通过简单的索引来测试它。 

attended := map[string]bool{
    "Ann": true,
    "Joe": true,
    ...
}

if attended[person] { // 如果 person 不在映射中，则为 false
    fmt.Println(person, "was at the meeting")
}

有时你需要区分缺失的条目和零值。"UTC" 是否有条目，或者它返回 0 是因为根本不在映射中？ 你可以通过一种多重赋值的形式来区分。 

var seconds int
var ok bool
seconds, ok = timeZone[tz]

出于显而易见的原因，这被称为“逗号 ok”惯用法。 在这个例子中，如果 tz 存在，seconds 将被适当地设置，ok 为 true；否则，seconds 将设置为零，ok 为 false。 这里有一个函数将它与一个友好的错误报告结合在一起： 

func offset(tz string) int {
    if seconds, ok := timeZone[tz]; ok {
        return seconds
    }
    log.Println("unknown time zone:", tz)
    return 0
}

要测试映射中是否存在某个键而不关心实际值，可以使用空白标识符（_）代替通常用于存储值的变量。 

_, present := timeZone[tz]

要删除映射条目，请使用内置函数 delete，其参数是映射和要删除的键。即使该键在映射中已不存在，这样做也是安全的。 

delete(timeZone, "PDT")  // 现在使用标准时间

打印

Go 语言中的格式化打印采用类似于 C 语言 printf 系列函数的风格，但功能更丰富、更通用。这些函数位于 fmt 包中，并且首字母大写：fmt.Printf、fmt.Fprintf、fmt.Sprintf 等等。字符串函数（如 Sprintf）返回一个字符串，而不是填充到提供的缓冲区中。

您无需提供格式字符串。对于 Printf、Fprintf 和 Sprintf，每个都有另一对函数，例如 Print 和 Println。这些函数不接受格式字符串，而是为每个参数生成默认格式。Println 版本还会在参数之间插入空格，并在输出末尾追加换行符，而 Print 版本仅在两侧的操作数都不是字符串时才添加空格。在本例中，每一行都产生相同的输出。 

fmt.Printf("Hello %d\n", 23)
fmt.Fprint(os.Stdout, "Hello ", 23, "\n")
fmt.Println("Hello", 23)
fmt.Println(fmt.Sprint("Hello ", 23))

格式化打印函数 fmt.Fprint 及其同类函数接受实现了 io.Writer 接口的任何对象作为第一个参数；变量 os.Stdout 和 os.Stderr 是常见的实例。

接下来，内容开始与 C 语言有所不同。首先，像 %d 这样的数字格式不接受用于表示有符号性或大小的标志；相反，打印例程使用参数的类型来决定这些属性。 

var x uint64 = 1<<64 - 1
fmt.Printf("%d %x; %d %x\n", x, x, int64(x), int64(x))

打印结果为 

18446744073709551615 ffffffffffffffff; -1 -1

如果您只需要默认转换，例如整数的十进制表示，可以使用万能格式 %v（代表“value”）；其结果与 Print 和 Println 产生的完全相同。此外，该格式可以打印任何值，甚至包括数组、切片、结构体和映射。这里是上一节中定义的时区映射的打印语句。 

fmt.Printf("%v\n", timeZone)  // 或者直接 fmt.Println(timeZone)

输出如下： 

map[CST:-21600 EST:-18000 MST:-25200 PST:-28800 UTC:0]

对于映射，Printf 及其同类函数按键的字典序对输出进行排序。

当打印结构体时，修改后的格式 %+v 会用字段名注释结构体的字段，而对于任何值，替代格式 %#v 会以完整的 Go 语法打印该值。 

type T struct {
    a int
    b float64
    c string
}
t := &T{ 7, -2.35, "abc\tdef" }
fmt.Printf("%v\n", t)
fmt.Printf("%+v\n", t)
fmt.Printf("%#v\n", t)
fmt.Printf("%#v\n", timeZone)

打印结果为 

&{7 -2.35 abc   def}
&{a:7 b:-2.35 c:abc     def}
&main.T{a:7, b:-2.35, c:"abc\tdef"}
map[string]int{"CST":-21600, "EST":-18000, "MST":-25200, "PST":-28800, "UTC":0}

（注意其中的 & 符号。） 这种带引号的字符串格式也可在应用于 string 或 []byte 类型的值时通过 %q 使用。替代格式 %#q 会在可能的情况下使用反引号代替双引号。（%q 格式也适用于整数和 rune，产生单引号的 rune 常量。）另外，%x 不仅可用于整数，也可用于字符串、字节数组和字节切片，生成一长串十六进制字符串，并且如果在格式中包含空格（% x），它会在字节之间插入空格。

另一个方便的格式是 %T，它打印值的类型。 

fmt.Printf("%T\n", timeZone)

打印结果为 

map[string]int

如果您想控制自定义类型的默认格式，只需在该类型上定义一个签名为 String() string 的方法即可。对于我们简单的类型 T，可能看起来像这样。 

func (t *T) String() string {
    return fmt.Sprintf("%d/%g/%q", t.a, t.b, t.c)
}
fmt.Printf("%v\n", t)

以以下格式打印 

7/-2.35/"abc\tdef"

（如果您需要打印类型为 T 的值以及指向 T 的指针，那么 String 方法的接收者必须是值类型；此示例使用指针是因为这对于结构体类型更高效且更符合习惯。有关更多信息，请参见下面关于指针与值接收者的部分。）

我们的 String 方法能够调用 Sprintf，因为打印例程是完全可重入的，并且可以这样包装。然而，关于这种方法有一个重要的细节需要理解：不要通过以一种会无限递归调用您自己的 Sprintf 方法的方式来构造 String 方法。如果 Sprintf 调用尝试直接将接收者作为字符串打印，就会发生这种情况，这反过来又会再次调用该方法。这是一个常见且容易犯的错误，如下例所示。 

type MyString string

func (m MyString) String() string {
    return fmt.Sprintf("MyString=%s", m) // 错误：将无限递归。
}

修复也很简单：将参数转换为基本的字符串类型，该类型没有该方法。 

type MyString string
func (m MyString) String() string {
    return fmt.Sprintf("MyString=%s", string(m)) // 正确：注意类型转换。
}

在初始化部分，我们将看到另一种避免这种递归的技术。

另一种打印技巧是将打印例程的参数直接传递给另一个此类例程。 Printf 的签名使用 ...interface{} 类型作为其最后一个参数，以指定在格式字符串之后可以出现任意数量（任意类型）的参数。 

func Printf(format string, v ...interface{}) (n int, err error) {

在函数 Printf 内部，v 的行为类似于类型为 []interface{} 的变量，但如果将其传递给另一个可变参数函数，它则像一个普通的参数列表。 这里是我们上面使用过的函数 log.Println 的实现。它将其参数直接传递给 fmt.Sprintln 进行实际的格式化。 

// Println 以 fmt.Println 的方式将输出打印到标准日志记录器。
func Println(v ...interface{}) {
    std.Output(2, fmt.Sprintln(v...))  // Output 接受参数 (int, string)
}

我们在嵌套调用 Sprintln 时，在 v 后面写上 ...，是为了告诉编译器将 v 视为一个参数列表；否则它只会将 v 作为一个单独的切片参数传递。

关于打印，还有比我们这里涵盖的更多内容。详情请参阅 fmt 包的 godoc 文档。

顺便说一下，... 参数可以是特定类型，例如 ...int，用于一个从整数列表中选择最小值的 Min 函数： 

func Min(a ...int) int {
    min := int(^uint(0) >> 1)  // 最大的 int 值
    for _, i := range a {
        if i < min {
            min = i
        }
    }
    return min
}

Append

现在我们有了缺失的部分，可以解释内置函数 append 的设计了。append 的签名与我们上面自定义的 Append 函数不同。其示意如下： 

func append(slice []T, elements ...T) []T

其中 T 是任何给定类型的占位符。实际上，你无法在 Go 中编写一个类型 T 由调用者决定的函数。这就是 append 是内置的原因：它需要编译器的支持。

append 所做的是将元素追加到切片的末尾并返回结果。需要返回结果是因为，与我们手写的 Append 一样，底层的数组可能会改变。这个简单的例子： 

x := []int{1,2,3}
x = append(x, 4, 5, 6)
fmt.Println(x)

会打印出 [1 2 3 4 5 6]。所以 append 的工作方式有点像 Printf，收集任意数量的参数。

但是如果我们想像我们的 Append 那样，将一个切片追加到另一个切片呢？很简单：在调用处使用 ...，就像我们在上面调用 Output 时所做的那样。以下代码片段产生的输出与上面的例子相同。 

x := []int{1,2,3}
y := []int{4,5,6}
x = append(x, y...)
fmt.Println(x)

如果没有那个 ...，代码将无法编译，因为类型会不对；y 的类型不是 int。

初始化

尽管表面上看起来与 C 或 C++ 中的初始化没有太大的不同，但 Go 中的初始化更加强大。复杂的结构体可以在初始化期间构建，并且被初始化对象之间（甚至不同包之间）的顺序问题会被正确处理。

常量

Go 中的常量就是常量。它们是在编译时创建的，即使是在函数中作为局部变量定义时也是如此，并且只能是数字、字符（rune）、字符串或布尔值。由于编译时限制，定义它们的表达式必须是常量表达式，能够由编译器求值。例如，1<<3 是一个常量表达式，而 math.Sin(math.Pi/4) 不是，因为对 math.Sin 的函数调用需要在运行时发生。

方法

指针与值

正如我们在 ByteSize 中看到的，方法可以为任何命名类型定义（指针或接口除外）；接收器不必是结构体。

在前面关于切片的讨论中，我们编写了一个 Append 函数。我们也可以将它定义为切片的一个方法。为此，我们首先声明一个命名类型来绑定方法，然后将该类型的一个值作为方法的接收器。 

type ByteSlice []byte

func (slice ByteSlice) Append(data []byte) []byte {
    // 函数体与上面定义的 Append 函数完全相同。
}

这仍然要求方法返回更新后的切片。我们可以通过重新定义该方法，使其接收器为 ByteSlice 的指针来消除这种笨拙，这样方法就可以覆盖调用者的切片。 

func (p *ByteSlice) Append(data []byte) {
    slice := *p
    // 函数体如上，但没有返回语句。
    *p = slice
}

事实上，我们可以做得更好。如果我们修改函数，使其看起来像一个标准的 Write 方法，如下所示， 

func (p *ByteSlice) Write(data []byte) (n int, err error) {
    slice := *p
    // 再次与上面相同。
    *p = slice
    return len(data), nil
}

那么类型 *ByteSlice 就满足了标准接口 io.Writer，这非常方便。例如，我们可以向其中写入内容。 

    var b ByteSlice
    fmt.Fprintf(&b, "这一天有 %d 小时\n", 7)

我们传递了 ByteSlice 的地址，因为只有 *ByteSlice 满足 io.Writer。 关于接收器是指针还是值的规则是：值方法可以被指针和值调用，但指针方法只能被指针调用。

这条规则源于指针方法可以修改接收器；如果在一个值上调用它们，方法将接收到该值的一个副本，因此任何修改都会被丢弃。 因此语言禁止这种错误。 不过，有一个方便的例外情况。当值是可寻址的时，语言会自动插入取地址操作符，以处理在值上调用指针方法的常见情况。 在我们的例子中，变量 b 是可寻址的，因此我们可以直接用 b.Write 调用其 Write 方法。编译器会为我们将其重写为 (&b).Write。

顺便提一下，在字节切片上使用 Write 的思想是 bytes.Buffer 实现的核心。

接口

Go 中的接口提供了一种指定对象行为的方式：如果某个东西能做这件事，那么它就可以在这里使用。我们已经看过几个简单的例子；自定义的打印机可以通过 String 方法实现，而 Fprintf 可以将输出生成到任何拥有 Write 方法的东西上。 只有一个或两个方法的接口在 Go 代码中很常见，通常以方法命名，例如实现 Write 的 io.Writer。

一个类型可以实现多个接口。 例如，一个集合如果实现了 sort.Interface（包含 Len()、Less(i, j int) bool 和 Swap(i, j int) 方法），就可以使用 sort 包中的例程进行排序，同时它也可以拥有自定义的格式化器。 在这个构造的例子中，Sequence 同时满足两者。

转换

Sequence 的 String 方法正在重新实现 Sprint 为切片所做的工作。（它的时间复杂度为 O(N²)，性能较差。）如果在调用 Sprint 之前将 Sequence 转换为普通的 []int，我们就可以共享这部分工作（并同时加速）。 

func (s Sequence) String() string {
    s = s.Copy()
    sort.Sort(s)
    return fmt.Sprint([]int(s))
}

此方法是展示如何在 String 方法中安全调用 Sprintf 的转换技巧的另一个示例。 因为两种类型（Sequence 和 []int）在忽略类型名称时是相同的，所以它们之间的转换是合法的。 这种转换不会创建新值，它只是临时性地将现有值视为具有新类型。 （存在其他会产生新值的合法转换，例如从整数到浮点数的转换。）

在 Go 程序中，将一个表达式的类型转换以访问不同方法集是一种惯用法。例如，我们可以使用现有的类型 sort.IntSlice 将整个示例简化为： 

type Sequence []int

// Method for printing - sorts the elements before printing
func (s Sequence) String() string {
    s = s.Copy()
    sort.IntSlice(s).Sort()
    return fmt.Sprint([]int(s))
}

现在，我们不再让 Sequence 实现多个接口（排序和打印），而是利用数据项可以转换为多种类型（Sequence、sort.IntSlice 和 []int）的能力，每种类型各司其职。这在实践中不太常见，但可能很有效。

接口转换与类型断言

类型开关是一种转换形式：它们接受一个接口，并且在 switch 语句的每个分支中，在某种意义上将其转换为该分支的类型。 以下是 fmt.Printf 下代码如何使用类型开关将值转换为字符串的简化版本。 如果它已经是字符串，我们希望获得接口持有的实际字符串值；而如果它具有 String 方法，我们希望获得调用该方法的结果。 

type Stringer interface {
    String() string
}

var value interface{} // Value provided by caller.
switch str := value.(type) {
case string:
    return str
case Stringer:
    return str.String()
}

第一个分支找到了一个具体值；第二个分支将一个接口转换为另一个接口。这样混合类型是完全可以的。

如果我们只关心一种类型怎么办？如果我们知道该值包含一个 string 并且只是想提取它？ 单分支的类型开关可以做到，但 类型断言 也可以。 类型断言接受一个接口值并从中提取指定显式类型的值。 其语法借鉴了类型开关的开头子句，但使用显式类型而非 type 关键字： 

value.(typeName)

其结果是一个具有静态类型 typeName 的新值。 该类型必须是接口持有的具体类型，或者是该值可以转换到的另一个接口类型。 要提取我们已知存在于该值中的字符串，可以这样写： 

str := value.(string)

但如果最终该值并不包含字符串，程序将在运行时错误中崩溃。 为了防止这种情况，可以使用 "comma, ok" 惯用法安全地测试该值是否为字符串： 

str, ok := value.(string)
if ok {
    fmt.Printf("string value is: %q\n", str)
} else {
    fmt.Printf("value is not a string\n")
}

如果类型断言失败，str 仍然存在且类型为 string，但它的值将是零值，即空字符串。

为了说明此功能，这里有一个 if-else 语句，它等效于本节开头的类型开关。 

if str, ok := value.(string); ok {
    return str
} else if str, ok := value.(Stringer); ok {
    return str.String()
}

通用性

如果一个类型仅仅是为了实现一个接口而存在，并且除了该接口定义的方法外永远不会有导出的方法，那么就不需要导出该类型本身。 仅导出接口可以明确表示，该值除了接口描述的内容外，没有其他值得关注的行为。 这也避免了在每个通用方法实例上重复编写文档的需要。

在这种情况下，构造函数应返回接口值，而不是实现类型。 例如，在哈希库中，crc32.NewIEEE 和 adler32.New 都返回接口类型 hash.Hash32。 在 Go 程序中将 Adler-32 算法替换为 CRC-32 算法，只需更改构造函数调用；代码的其余部分不受算法变更的影响。

类似的方法允许各种 crypto 包中的流密码算法与它们链接的块密码分离。 crypto/cipher 包中的 Block 接口指定了块密码的行为，它提供单个数据块的加密。 然后，类似于 bufio 包，实现此接口的密码包可用于构建流密码（由 Stream 接口表示），而无需了解块加密的细节。

crypto/cipher 接口如下所示： 

type Block interface {
    BlockSize() int
    Encrypt(dst, src []byte)
    Decrypt(dst, src []byte)
}

type Stream interface {
    XORKeyStream(dst, src []byte)
}

这里是计数器模式（CTR）流的定义，它将分组密码转化为流密码；注意分组密码的细节已被抽象化： 

// NewCTR 返回一个 Stream，该 Stream 使用给定的 Block 以计数器模式进行加密/解密。
// iv 的长度必须与 Block 的块大小相同。
func NewCTR(block Block, iv []byte) Stream

NewCTR 不仅适用于某个特定的加密算法和数据源，而是适用于 Block 接口的任何实现以及任何 Stream。因为它们返回接口值，所以将 CTR 加密替换为其他加密模式是一个局部化的更改。必须修改构造函数的调用，但由于周围的代码只需将结果视为 Stream，因此它不会察觉到差异。

接口与方法

由于几乎任何东西都可以拥有方法，因此几乎任何东西都可以满足一个接口。一个说明性的例子在 http 包中，它定义了 Handler 接口。任何实现了 Handler 的对象都可以处理 HTTP 请求。 

type Handler interface {
    ServeHTTP(ResponseWriter, *Request)
}

ResponseWriter 本身是一个接口，它提供了将响应返回给客户端所需的方法访问。这些方法包括标准的 Write 方法，因此 http.ResponseWriter 可以在任何可以使用 io.Writer 的地方使用。 Request 是一个结构体，包含从客户端接收到的请求的解析表示。

为了简洁，我们忽略 POST 请求，并假设 HTTP 请求总是 GET 请求；这种简化不会影响处理器的设置方式。这是一个用于统计页面访问次数的简单处理器实现。 

// 简单的计数器服务器。
type Counter struct {
    n int
}

func (ctr *Counter) ServeHTTP(w http.ResponseWriter, req *http.Request) {
    ctr.n++
    fmt.Fprintf(w, "counter = %d\n", ctr.n)
}

（延续我们的主题，注意 Fprintf 如何能够打印到 http.ResponseWriter。） 在真实的服务器中，对 ctr.n 的访问需要并发访问保护。有关建议，请参阅 sync 和 atomic 包。

作为参考，以下是将此类服务器附加到 URL 树中某个节点的方法。 

import "net/http"
...
ctr := new(Counter)
http.Handle("/counter", ctr)

但为什么要把 Counter 做成结构体呢？一个整数就足够了。（接收者必须是一个指针，以便递增对接收者可见。） 

// 更简单的计数器服务器。
type Counter int

func (ctr *Counter) ServeHTTP(w http.ResponseWriter, req *http.Request) {
    *ctr++
    fmt.Fprintf(w, "counter = %d\n", *ctr)
}

如果你的程序有一些内部状态需要在页面被访问时得到通知怎么办？将一个通道与网页绑定。 

// 每次访问时发送通知的通道。
// （可能希望该通道是有缓冲的。）
type Chan chan *http.Request

func (ch Chan) ServeHTTP(w http.ResponseWriter, req *http.Request) {
    ch <- req
    fmt.Fprint(w, "notification sent")
}

最后，假设我们想在 /args 上展示调用服务器二进制文件时使用的参数。 编写一个打印参数的函数很容易。 

func ArgServer() {
    fmt.Println(os.Args)
}

我们如何将其变成一个 HTTP 服务器呢？我们可以让 ArgServer 成为某个类型的方法，而忽略该类型的值，但有一种更简洁的方法。 由于我们可以为除指针和接口以外的任何类型定义方法，因此我们可以为一个函数编写方法。 http 包中包含以下代码： 

// HandlerFunc 类型是一个适配器，允许将普通函数用作 HTTP 处理器。
// 如果 f 是一个具有适当签名的函数，则 HandlerFunc(f) 是一个调用 f 的 Handler 对象。
type HandlerFunc func(ResponseWriter, *Request)

// ServeHTTP 调用 f(w, req)。
func (f HandlerFunc) ServeHTTP(w ResponseWriter, req *Request) {
    f(w, req)
}

HandlerFunc 是一个具有 ServeHTTP 方法的类型，因此该类型的值可以处理 HTTP 请求。看看该方法的实现：接收者是一个函数 f，而方法调用 f。这看起来可能有点奇怪，但它与接收者是一个通道并在通道上发送消息并无太大不同。

为了将 ArgServer 变成一个 HTTP 服务器，我们首先修改它以具有正确的签名。 

// 参数服务器。
func ArgServer(w http.ResponseWriter, req *http.Request) {
    fmt.Fprintln(w, os.Args)
}

ArgServer 现在与 HandlerFunc 具有相同的签名，因此可以将其转换为该类型以访问其方法，就像我们将 Sequence 转换为 IntSlice 以访问 IntSlice.Sort 一样。 设置它的代码很简洁： 

http.Handle("/args", http.HandlerFunc(ArgServer))

当有人访问页面 /args 时，安装在该页面的处理器的值为 ArgServer，类型为 HandlerFunc。 HTTP 服务器将调用该类型的 ServeHTTP 方法，以 ArgServer 作为接收者，该方法又会（通过 HandlerFunc.ServeHTTP 内部的调用 f(w, req)）调用 ArgServer。 然后，参数将被显示出来。

在本节中，我们使用一个结构体、一个整数、一个通道和一个函数创建了一个 HTTP 服务器，这都归功于接口只是方法的集合，而方法可以为（几乎）任何类型定义。

空白标识符

我们之前在for range 循环和映射的上下文中已经多次提到过空白标识符。空白标识符可以赋值或声明为任何类型的任意值，其值会被安全地丢弃。它有点像写入 Unix 的 /dev/null 文件：它代表一个只写值，用作占位符，在需要变量但实际值无关紧要的地方使用。它的用途远不止我们之前看到的那些。

多重赋值中的空白标识符

在 for range 循环中使用空白标识符是一种更通用情况——多重赋值的特例。

如果赋值语句左侧需要多个值，但其中某个值不会被程序使用，那么在赋值语句左侧使用空白标识符就避免了创建虚拟变量的需要，并明确表示该值将被丢弃。例如，当调用一个返回值和错误信息的函数，但只有错误信息重要时，可以使用空白标识符来丢弃无关的值。 

if _, err := os.Stat(path); os.IsNotExist(err) {
    fmt.Printf("%s does not exist\n", path)
}

偶尔你会看到为了忽略错误而丢弃错误值的代码；这是糟糕的做法。始终应该检查错误返回值；它们被设计出来是有原因的。 

// 糟糕！如果 path 不存在，此代码将崩溃。
fi, _ := os.Stat(path)
if fi.IsDir() {
    fmt.Printf("%s is a directory\n", path)
}

为副作用导入

像前面例子中未使用的 fmt 或 io 导入，最终应该被使用或移除：空白赋值标识代码为半成品。但有时，仅仅为了其副作用而导入一个包是有用的，而无需任何显式使用。例如，在它的 init 函数期间，net/http/pprof 包注册了提供调试信息的 HTTP 处理器。它有一个导出的 API，但大多数客户端只需要处理器注册，并通过网页访问数据。为了仅为其副作用导入该包，可以将包名重命名为空白标识符：

这种导入形式清楚地表明该包是为其副作用而导入的，因为不可能有其他用途：在这个文件中，它没有名称。（如果它有名称，而我们未使用该名称，编译器会拒绝该程序。）

接口检查

正如我们在前面讨论接口时看到的，一个类型无需显式声明它实现了某个接口。相反，一个类型只需实现该接口的方法即可实现该接口。实际上，大多数接口转换是静态的，因此在编译时进行检查。例如，将一个 *os.File 传递给一个期望 io.Reader 的函数，除非 *os.File 实现了 io.Reader 接口，否则无法编译。

不过，有些接口检查确实发生在运行时。一个例子是 encoding/json 包，它定义了一个 Marshaler 接口。当 JSON 编码器接收到一个实现了该接口的值时，编码器会调用该值的序列化方法将其转换为 JSON，而不是执行标准转换。编码器在运行时通过类似这样的类型断言来检查此属性： 

m, ok := val.(json.Marshaler)

如果只需要询问一个类型是否实现了某个接口，而无需实际使用该接口本身，例如作为错误检查的一部分，可以使用空白标识符来忽略类型断言的值： 

if _, ok := val.(json.Marshaler); ok {
    fmt.Printf("value %v of type %T implements json.Marshaler\n", val, val)
}

这种情况的一个出现场景是：在实现该类型的包内，需要确保该类型确实满足了某个接口的要求。 例如，若某个类型（比如 json.RawMessage）需要自定义 JSON 表示方式，它就应当实现 json.Marshaler 接口，但编译器通常不会通过静态转换来自动验证这一点。 如果该类型无意中未能满足接口要求，JSON 编码器依然可以工作，但不会使用自定义的实现。 为了确保实现的正确性，可以在包中使用空白标识符进行全局声明： 

var _ json.Marshaler = (*RawMessage)(nil)

在此声明中，涉及将 *RawMessage 转换为 Marshaler 的赋值操作，要求 *RawMessage 必须实现 Marshaler 接口，并且此特性将在编译时得到检查。 倘若 json.Marshaler 接口发生变更，此包将无法再通过编译，从而提示我们需要对其进行更新。

此结构中出现的空白标识符表明，该声明仅用于类型检查，而非为了创建变量。 不过，不必对满足接口的每种类型都如此操作。按照惯例，这类声明通常只在代码中尚不存在静态转换时才会使用，而这种情况并不常见。

嵌入

Go 语言没有提供典型的、基于类型的子类化概念，但它具备通过嵌入类型到结构体或接口中，从而“借用”部分实现的能力。

接口嵌入非常简单。我们之前已经提及过 io.Reader 和 io.Writer 接口；以下是它们的定义。 

type Reader interface {
    Read(p []byte) (n int, err error)
}```go
type Writer interface {
    Write(p []byte) (n int, err error)
}
```

`io` 包也导出了其他几个接口，这些接口规定了对象可以实现的多个方法。例如，`io.ReadWriter` 接口同时包含 `Read` 和 `Write` 方法。我们可以通过显式列出这两个方法来定义 `io.ReadWriter`，但更简洁且更直观的做法是通过嵌入这两个接口来构成新的接口，如下所示：

```go
// ReadWriter 接口组合了 Reader 和 Writer 接口。
type ReadWriter interface {
    Reader
    Writer
}
```

这正如它看起来那样：一个 `ReadWriter` 可以执行 `Reader` 能做的*以及* `Writer` 能做的；它是被嵌入接口的联合体。只有接口才能被嵌入到其他接口中。

同样的基本思想也适用于结构体，但其影响更为深远。`bufio` 包有两个结构体类型：`bufio.Reader` 和 `bufio.Writer`，它们各自实现了 `io` 包中对应的接口。`bufio` 包还实现了一个带缓冲的读写器，它通过嵌入将读取器和写入器组合到一个结构体中实现：它在结构体内部列出这些类型，但不为它们指定字段名称。

```go
// ReadWriter 存储了指向 Reader 和 Writer 的指针。
// 它实现了 io.ReadWriter 接口。
type ReadWriter struct {
    *Reader  // *bufio.Reader
    *Writer  // *bufio.Writer
}
```

被嵌入的元素是指向结构体的指针，因此在使用前必须被初始化指向有效的结构体。`ReadWriter` 结构体也可以写成：

```go
type ReadWriter struct {
    reader *Reader
    writer *Writer
}
```

但如果这样做，为了提升字段的方法并满足 `io` 接口的要求，我们还需要提供转发方法，如下所示：

```go
func (rw *ReadWriter) Read(p []byte) (n int, err error) {
    return rw.reader.Read(p)
}
```

通过直接嵌入结构体，我们避免了这种记账工作。被嵌入类型的方法会自动获得，这意味着 `bufio.ReadWriter` 不仅拥有 `bufio.Reader` 和 `bufio.Writer` 的方法，它同时也满足所有这三个接口：`io.Reader`、`io.Writer` 和 `io.ReadWriter`。

嵌入与子类化有一个重要的区别。当我们嵌入一个类型时，该类型的方法会成为外部类型的方法，但当这些方法被调用时，方法的接收者是内部类型，而不是外部类型。在我们的例子中，当调用一个 `bufio.ReadWriter` 的 `Read` 方法时，其效果与上面写出的转发方法完全相同；接收者是 `ReadWriter` 的 `reader` 字段，而不是 `ReadWriter` 本身。

嵌入也可以是一个简单的便利。此示例展示了一个嵌入字段与一个常规的命名字段并存的情况。

```go
type Job struct {
    Command string
    *log.Logger
}
```

现在，`Job` 类型拥有 `*log.Logger` 的 `Print`、`Printf`、`Println` 等方法。我们当然可以给 `Logger` 指定一个字段名称，但没有必要这样做。现在，一旦初始化，我们就可以将日志记录到 `Job`：

```go
job.Println("starting now...")
```

`Logger` 是 `Job` 结构体的一个常规字段，因此我们可以在 `Job` 的构造函数中用通常的方式初始化它，如下所示：

```go
func NewJob(command string, logger *log.Logger) *Job {
    return &Job{command, logger}
}
```

或者使用复合字面量：

```go
job := &Job{command, log.New(os.Stderr, "Job: ", log.Ldate)}
```

如果我们需要直接引用一个嵌入字段，该字段的类型名称（忽略包限定符）就充当字段名称，就像在 `ReadWriter` 结构体的 `Read` 方法中那样。这里，如果我们需要访问 `Job` 变量 `job` 的 `*log.Logger`，我们可以写作 `job.Logger`，这在我们想要细化 `Logger` 的方法时会很有用。

```go
func (job *Job) Printf(format string, args ...interface{}) {
    job.Logger.Printf("%q: %s", job.Command, fmt.Sprintf(format, args...))
}
```

嵌入类型引入了名称冲突的问题，但解决冲突的规则很简单。
首先，一个字段或方法 `X` 会隐藏在该类型更深层嵌套部分中的任何其他同名项 `X`。如果 `log.Logger` 包含一个名为 `Command` 的字段或方法，那么 `Job` 的 `Command` 字段将占据主导地位。

其次，如果相同的名称出现在相同的嵌套层级，这通常是一个错误；如果 `Job` 结构体已经包含另一个名为 `Logger` 的字段或方法，那么再嵌入 `log.Logger` 将是错误的。
然而，如果这个重复的名称在类型定义之外从未在程序中被提及，那么这是允许的。这个限定条件为应对从外部嵌入的类型可能发生的变化提供了一些保护；如果添加了一个字段与另一个子类型中的另一个字段冲突，但两个字段都从未被使用，那么就没有问题。


并发



通过通信来共享

并发编程是一个很大的主题，这里只能介绍一些 Go 特有的亮点。




在许多环境中，并发编程的困难之处在于，要实现对共享变量的正确访问需要非常小心。Go 鼓励一种不同的方法，即在通道上传递共享的值，实际上，从不允许单独的执行线程主动共享。在任何给定时间，只有一个 goroutine 可以访问该值。从设计上讲，数据竞争不可能发生。为了鼓励这种思维方式，我们将其总结为一句口号：




不要通过共享内存来通信；相反，通过通信来共享内存。




这种方法也可能被过度使用。例如，引用计数可能最适合通过在整数变量周围放置一个互斥锁来实现。但作为一种高级方法，使用通道来控制访问可以使编写清晰、正确的程序变得更加容易。




理解这种模型的一种方式是，考虑一个在单个 CPU 上运行的典型单线程程序。它不需要任何同步原语。现在运行另一个这样的实例；它也不需要同步。现在让这两个实例进行通信；如果通信本身就是同步器，那么仍然不需要其他同步。例如，Unix 管道就完美地符合这个模型。虽然 Go 的并发方法起源于 Hoare 的通信顺序进程（CSP），但它也可以被视为 Unix 管道的一种类型安全的泛化。




Goroutines




之所以叫它们 goroutines，是因为现有的术语——线程、协程、进程等等——传达了不准确的含义。一个 goroutine 有一个简单的模型：它是一个函数，与同一地址空间中的其他 goroutine 并发执行。它非常轻量，开销几乎只相当于分配栈空间。而且栈的初始大小很小，因此很廉价，并在需要时通过分配（和释放）堆存储来增长。




Goroutines 被多路复用到多个操作系统线程上，因此如果一个 goroutine 阻塞了（例如在等待 I/O 时），其他的 goroutine 会继续运行。它们的设计隐藏了许多线程创建和管理的复杂性。




在函数或方法调用前加上 go 关键字，就可以在一个新的 goroutine 中运行该调用。当调用完成时，该 goroutine 会静默地退出。（其效果类似于 Unix shell 中的 & 符号，用于在后台运行命令。）



go list.Sort()  // 并发运行 list.Sort；不等待它完成。




在 goroutine 调用中使用函数字面量会很方便。



func Announce(message string, delay time.Duration) {
    go func() {
        time.Sleep(delay)
        fmt.Println(message)
    }()  // 注意括号——必须调用该函数。
}




在 Go 中，函数字面量就是闭包：其实现确保函数引用的变量只要处于活动状态就会一直存在。




这些例子不太实用，因为这些函数没有办法发出完成信号。为此，我们需要通道。




通道




与映射类似，通道通过 make 分配，生成的值充当对底层数据结构的引用。如果提供了一个可选的整数参数，它会设置通道的缓冲区大小。默认值是零，表示无缓冲或同步通道。



ci := make(chan int)            // 整型的无缓冲通道
cj := make(chan int, 0)         // 整型的无缓冲通道
cs := make(chan *os.File, 100)  // 指向文件的指针的缓冲通道




无缓冲通道将通信（交换一个值）与同步（保证两个计算/ goroutine 处于已知状态）结合在一起。




使用通道有很多很好的惯用法。这里介绍一个入门示例。在上一节中，我们在后台启动了一个排序操作。通道可以让启动排序的 goroutine 等待排序完成。



c := make(chan int)  // 分配一个通道。
// 在一个 goroutine 中启动排序；当它完成时，在通道上发出信号。
go func() {
    list.Sort()
    c <- 1  // 发送一个信号；值不重要。
}()
doSomethingForAWhile()
<-c   // 等待排序完成；丢弃发送的值。




接收方总是阻塞，直到有数据可接收。如果通道是无缓冲的，发送方会阻塞，直到接收方接收到值。如果通道有缓冲，发送方仅阻塞到值被复制到缓冲区为止；如果缓冲区已满，这意味着要等待直到某个接收方取走一个值。




缓冲通道可以用作信号量，例如用来限制吞吐量。在这个例子中，传入的请求被传递给 handle，该函数向通道发送一个值，处理请求，然后从通道接收一个值，以便为下一个消费者“释放”信号量。通道缓冲区的容量限制了对 process 的并发调用次数。
var sem = make(chan int, MaxOutstanding) func handle(r *Request) { sem <- 1 // 等待活动队列排空。 process(r) // 可能需要很长时间。 <-sem // 完成；允许下一个请求运行。 } func Serve(queue chan *Request) { for { req := <-queue go handle(req) // 不等待 handle 完成。 } }

一旦有 MaxOutstanding 个处理器正在执行 process，任何更多的处理器在尝试向已满的通道缓冲区发送数据时都会阻塞，直到现有处理器之一完成并从缓冲区接收数据。

但这个设计有个问题：Serve 会为每个传入的请求创建一个新的协程，尽管同一时刻最多只能有 MaxOutstanding 个协程在运行。如果请求来得太快，程序可能会消耗无限的资源。我们可以通过修改 Serve 函数来限制协程的创建，以此解决这个缺陷： 

func Serve(queue chan *Request) {
    for req := range queue {
        sem <- 1
        go func() {
            process(req)
            <-sem
        }()
    }
}

（请注意，在 Go 1.22 版本之前，这段代码存在一个 bug：循环变量在所有协程之间共享。详情请参阅 Go wiki。）

另一种能良好管理资源的方法是启动固定数量的 handle 协程，这些协程都从请求通道读取数据。协程的数量限制了同时调用 process 的次数。这个 Serve 函数还接受一个通道，通过该通道通知它退出；在启动协程后，它会阻塞在从该通道接收数据上。 

func handle(queue chan *Request) {
    for r := range queue {
        process(r)
    }
}

func Serve(clientRequests chan *Request, quit chan bool) {
    // 启动处理器
    for i := 0; i < MaxOutstanding; i++ {
        go handle(clientRequests)
    }
    <-quit  // 等待退出指令。
}

通道的通道

Go 语言最重要的特性之一是通道是一等值，可以像任何其他值一样被分配和传递。这个特性的常见用途是实现安全、并行的解复用。

在上一节的例子中，handle 是一个理想化的请求处理器，但我们没有定义它处理的类型。如果该类型包含一个用于回复的通道，那么每个客户端就可以提供自己接收答案的路径。下面是类型 Request 的示意性定义。 

type Request struct {
    args        []int
    f           func([]int) int
    resultChan  chan int
}

客户端提供一个函数及其参数，并在请求对象内部提供一个通道来接收答案。 

func sum(a []int) (s int) {
    for _, v := range a {
        s += v
    }
    return
}

request := &Request{[]int{3, 4, 5}, sum, make(chan int)}
// 发送请求
clientRequests <- request
// 等待响应。
fmt.Printf("answer: %d\n", <-request.resultChan)

在服务器端，唯一变化的是处理函数。 

func handle(queue chan *Request) {
    for req := range queue {
        req.resultChan <- req.f(req.args)
    }
}

显然，要使其具备实用性还有很多工作要做，但这段代码是一个限流、并行、非阻塞的 RPC 系统的框架，并且其中看不到任何互斥锁。

并行化

这些思想的另一个应用是将计算并行化到多个 CPU 核心上。如果计算可以被分解为可以独立执行的独立部分，那么它就可以被并行化，并使用一个通道来通知每个部分何时完成。

假设我们要对一个向量中的元素执行一个耗时操作，并且对每个元素的操作值是独立的，就像下面这个理想化的例子。 

type Vector []float64

// 对 v[i], v[i+1] ... 直到 v[n-1] 应用操作。
func (v Vector) DoSome(i, n int, u Vector, c chan int) {
    for ; i < n; i++ {
        v[i] += u.Op(v[i])
    }
    c <- 1    // 通知这部分已完成
}

我们在一个循环中独立启动各个部分，每个 CPU 核心一个。它们可以按任意顺序完成，但这无关紧要；我们只需要在启动所有协程后，排空通道来统计完成信号。 

const numCPU = 4 // CPU 核心数

func (v Vector) DoAll(u Vector) {
    c := make(chan int, numCPU)  // 缓冲可选但合理。
    for i := 0; i < numCPU; i++ {
        go v.DoSome(i*len(v)/numCPU, (i+1)*len(v)/numCPU, u, c)
    }
    // 排空通道。
    for i := 0; i < numCPU; i++ {
        <-c    // 等待一个任务完成
    }
    // 全部完成。
}

与其为 numCPU 创建一个常量值，我们可以询问运行时哪个值是合适的。函数 runtime.NumCPU 返回机器的硬件 CPU 核心数，所以我们可以写： 

var numCPU = runtime.NumCPU()

还有一个函数 runtime.GOMAXPROCS，它报告（或设置）用户指定的 Go 程序可以同时运行的协程数量。其默认值为 runtime.NumCPU，但可以通过设置同名的 shell 环境变量或通过调用函数并传入一个正数来覆盖。传入零则只是查询该值。因此，如果我们希望遵从用户的资源请求，我们应该写： 

var numCPU = runtime.GOMAXPROCS(0)

请务必不要混淆并发与并行的概念——并发是将程序构建为独立执行的组件，而并行是为了在多核 CPU 上提高效率而并行执行计算。尽管 Go 的并发特性可以让一些问题很容易地构建为并行计算，但 Go 是一种并发语言，而不是一种并行语言，并非所有并行化问题都适合 Go 的模型。关于这一区别的讨论，请参阅 这篇博客文章 中引用的演讲。

一个有泄漏的缓冲区

并发编程的工具甚至能让非并发思想的表达更简洁。这里有一个从某个 RPC 包中抽象出的示例：客户端协程不断从某个来源（可能是网络）接收数据。为了避免反复分配和释放缓冲区，它维护了一个空闲列表，并使用有缓冲的通道来表示它。当通道为空时，就会分配一个新的缓冲区。消息缓冲区准备好后，便通过 serverChan 发送给服务器。 

var freeList = make(chan *Buffer, 100)
var serverChan = make(chan *Buffer)

func client() {
    for {
        var b *Buffer
        // 尝试从空闲列表获取缓冲区；若无则分配一个。
        select {
        case b = <-freeList:
            // 成功获取；无需其他操作。
        default:
            // 没有空闲缓冲区，分配一个新的。
            b = new(Buffer)
        }
        load(b)              // 从网络读取下一条消息。
        serverChan <- b      // 发送给服务器。
    }
}

服务器循环从客户端接收每条消息，进行处理，然后将缓冲区归还给空闲列表。 

func server() {
    for {
        b := <-serverChan    // 等待工作。
        process(b)
        // 如果空闲列表有空间，则重用缓冲区。
        select {
        case freeList <- b:
            // 缓冲区已放回空闲列表；无需其他操作。
        default:
            // 空闲列表已满，直接继续。
        }
    }
}

客户端尝试从 freeList 中检索缓冲区；如果没有可用的，则分配一个新的。服务器发送到 freeList 的操作会将 b 放回空闲列表，除非列表已满，此时缓冲区会被丢弃，等待垃圾回收器回收。（select 语句中的 default 子句在其他情况均未就绪时执行，这意味着这些 select 语句永远不会阻塞。）这个实现仅用几行代码就构建了一个“有漏洞的桶”式空闲列表，其簿记工作完全依赖有缓冲的通道和垃圾回收器。

错误

库函数通常需要向调用者返回某种错误指示。如前所述，Go 语言的多返回值特性使得在返回正常值的同时附带详细的错误描述变得非常简单。良好的风格是利用这一特性提供详细的错误信息。例如，我们将会看到，os.Open 在失败时不仅会返回一个 nil 指针，还会返回一个描述问题所在的错误值。

按照惯例，错误的类型为 error，这是一个简单的内置接口。 

type error interface {
    Error() string
}

库作者可以自由地使用更丰富的模型在底层实现这个接口，这样不仅能查看错误，还能提供一些上下文信息。如前所述，除了常见的 *os.File 返回值，os.Open 还会返回一个错误值。如果文件打开成功，错误将是 nil，但当出现问题时，它会持有一个 os.PathError： 

// PathError 记录了一个错误以及导致该错误的操作和文件路径。
type PathError struct {
    Op string    // "open", "unlink" 等。
    Path string  // 相关的文件。
    Err error    // 由系统调用返回。
}

func (e *PathError) Error() string {
    return e.Op + " " + e.Path + ": " + e.Err.Error()
}

PathError 的 Error 方法会生成类似这样的字符串： 

open /etc/passwx: no such file or directory

这样一个包含了问题文件名、操作以及所触发的操作系统错误的错误信息，即使在远离导致其发生的调用处打印出来，也非常有用；它比单纯的“no such file or directory”提供了更多信息。

在可行的情况下，错误字符串应能标识其来源，例如通过添加前缀来指明生成错误的操作或包。例如，在 image 包中，由于未知格式导致的解码错误，其字符串表示是 "image: unknown format"。

关心错误具体细节的调用者可以使用类型开关或类型断言来查找特定的错误并提取详细信息。对于 PathErrors，这可能包括检查内部的 Err 字段以确定是否为可恢复的失败。 

for try := 0; try < 2; try++ {
    file, err = os.Create(filename)
    if err == nil {
        return
    }
    if e, ok := err.(*os.PathError); ok && e.Err == syscall.ENOSPC {
        deleteTempFiles()  // 回收一些空间。
        continue
    }
    return
}

这里的第二个 if 语句是另一个 类型断言。如果断言失败，ok 将为 false，且 e 为 nil。如果成功，ok 将为 true，这意味着该错误类型是 *os.PathError，而 e 也是该类型，我们可以检查它以获取更多关于错误的信息。

Panic

向调用者报告错误的常规方法是返回一个 error 作为额外的返回值。经典的 Read 方法就是一个众所周知的例子；它返回一个字节数和一个 error。但如果错误是无法恢复的呢？有时程序确实无法继续执行。

为此，有一个内置函数 panic，它实际上会创建一个运行时错误来停止程序（但请参阅下一节）。该函数接受一个任意类型的参数（通常是一个字符串），在程序终止时打印出来。它也是一种指示发生了不可能事件（例如退出一个无限循环）的方式。 

// 使用牛顿法实现立方根的玩具示例。
func CubeRoot(x float64) float64 {
    z := x/3   // 任意初始值
    for i := 0; i < 1e6; i++ {
        prevz := z
        z -= (z*z*z-x) / (3*z*z)
        if veryClose(z, prevz) {
            return z
        }
    }
    // 一百万次迭代仍未收敛；说明存在问题。
    panic(fmt.Sprintf("CubeRoot(%g) did not converge", x))
}

这只是一个示例，但实际的库函数应避免使用 panic。如果问题可以被掩盖或绕过，让程序继续运行总是比终止整个程序更好。一个可能的反例是在初始化期间：如果库确实无法完成自身设置，此时使用 panic 是合理的。 

var user = os.Getenv("USER")

func init() {
    if user == "" {
        panic("no value for $USER")
    }
}

恢复 (Recover)

当 panic 被调用时（包括因运行时错误如切片索引越界或类型断言失败而隐式触发），它会立即停止当前函数的执行，并开始展开该 goroutine 的栈，沿途执行所有延迟函数。如果栈展开到达该 goroutine 的栈顶，程序就会终止。然而，可以使用内置函数 recover 来重新获取对该 goroutine 的控制权并恢复正常执行。

调用 recover 会停止栈展开，并返回传递给 panic 的参数。因为栈展开期间运行的唯一代码位于延迟函数内部，所以 recover 只在延迟函数内部有用。

recover 的一个应用是：在服务器中关闭一个发生故障的 goroutine，而不影响其他正在执行的 goroutine。 

func server(workChan <-chan *Work) {
    for work := range workChan {
        go safelyDo(work)
    }
}

func safelyDo(work *Work) {
    defer func() {
        if err := recover(); err != nil {
            log.Println("work failed:", err)
        }
    }()
    do(work)
}

在这个例子中，如果 do(work) 发生 panic，结果会被记录日志，并且该 goroutine 会干净地退出，不会干扰其他 goroutine。在延迟闭包中不需要做任何其他事情；调用 recover 就完全处理了该情况。

因为除非直接从延迟函数中调用，否则 recover 总是返回 nil，所以延迟代码可以调用那些内部也使用了 panic 和 recover 的库例程而不会失败。例如，safelyDo 中的延迟函数可以在调用 recover 之前调用一个日志记录函数，并且该日志记录代码的运行不会受到 panic 状态的影响。

有了我们的恢复模式后，do 函数（及其调用的任何函数）可以通过调用 panic 干净地摆脱任何糟糕的状况。我们可以利用这个想法来简化复杂软件中的错误处理。让我们看一个理想化的 regexp 包版本，它通过使用局部错误类型调用 panic 来报告解析错误。以下是 Error 的定义、一个 error 方法以及 Compile 函数。 

// Error 是解析错误的类型；它满足 error 接口。
type Error string
func (e Error) Error() string {
    return string(e)
}

// error 是 *Regexp 的一个方法，它通过使用 Error 类型的 panic 来报告解析错误。
func (regexp *Regexp) error(err string) {
    panic(Error(err))
}

// Compile 返回正则表达式的已解析表示形式。
func Compile(str string) (regexp *Regexp, err error) {
    regexp = new(Regexp)
    // doParse 在解析错误时会 panic。
    defer func() {
        if e := recover(); e != nil {
            regexp = nil    // 清除返回值。
            err = e.(Error) // 如果不是解析错误，将重新 panic。
        }
    }()
    return regexp.doParse(str), nil
}

如果 doParse 发生 panic，恢复块会将返回值设置为 nil——延迟函数可以修改具名返回值。然后，在将错误赋值给 err 时，它会通过类型断言检查问题是否是解析错误（即断言其具有局部类型 Error）。 如果不是，类型断言将失败，导致一个运行时错误，从而像没有被中断过一样继续栈展开。 这个检查意味着如果发生了意外情况，例如索引越界，代码将失败，即使我们正在使用 panic 和 recover 来处理解析错误。

有了错误处理机制，error 方法（因为它是一个绑定到类型的方法，所以与内置的 error 类型同名是合适的，甚至很自然）使得报告解析错误变得容易，无需手动考虑解析栈的展开： 

if pos == 0 {
    re.error("'*' illegal at start of expression")
}

尽管这个模式很有用，但它应只在包内部使用。 Parse 将其内部的 panic 调用转换为 error 值；它不会向其客户端暴露 panic。 这是一个值得遵循的好规则。

顺便说一下，这种重新 panic 的惯用法会在发生实际错误时更改 panic 的值。然而，原始错误和新的错误都会出现在崩溃报告中，因此问题的根本原因仍然可见。所以这种简单的重新 panic 方法通常是足够的——毕竟这是一次崩溃——但如果你只想显示原始值，你可以编写更多代码来过滤意外问题，并使用原始错误重新 panic。 这留作读者的练习。
