简介

以下是关于为 GCC 的 gccgo 前端做贡献的一些说明。 有关为 Go 项目中除 gccgo 以外的其他部分做贡献的信息，请参见 为 Go 项目做贡献。 有关自行构建 gccgo 的信息，请参见 安装和使用 gccgo。 关于使用 gccgo 前端进行开发的更多详细过程信息，请参见 gofrontend 代码仓库中的 HACKING 文件。

法律前提

您必须遵守 Go 版权规则，这些规则适用于对 gccgo 前端及关联的 libgo 库所做的所有更改。属于 GCC 而非 gccgo 部分的代码，必须遵循通用的 GCC 贡献规则。

代码

gccgo 前端的主源代码仓库位于 https://go.googlesource.com/gofrontend。 该仓库在 https://github.com/golang/gofrontend 进行了镜像。 主源代码本身无法独立构建，只能与 GCC 一起构建（将来可能会支持其他编译器）。对 gccgo 前端所做的更改也会被应用到托管在 gcc.gnu.org 的 GCC 源代码仓库中。在 gofrontend 仓库中，go 目录会被镜像到 GCC 仓库中的 gcc/go/gofrontend 目录，而 gofrontend 仓库中的 libgo 目录则会被镜像到 GCC 的 libgo 目录。此外，来自 Go 主仓库 的 test 目录也会被镜像到 GCC 仓库中的 gcc/testsuite/go.test/test 目录。

这些目录的更改始终从主源代码仓库流向 GCC 仓库。除了通过在主源代码中进行更改然后进行镜像之外，绝不应该在 GCC 仓库中直接修改这些文件。

gccgo 前端是用 C++ 编写的。它遵循 GNU 和 GCC 的 C++ 编码标准。在为前端编写代码时，请遵循周围代码的格式。几乎所有特定于 GCC 的代码都不在前端本身，而是在 GCC 源代码的 gcc/go 目录中。

gccgo 的运行时库与 Go 主仓库 中的库大部分相同。Go 仓库中的库代码会定期使用 shell 脚本 libgo/merge.sh 合并到 gofrontend 仓库的 libgo/go 目录中，然后再合并到 GCC 仓库中。因此，大多数库的更改应在 Go 主仓库中进行。libgo/go 目录之外的文件是 gccgo 特有的；不过，libgo/runtime 中的一些文件是基于 Go 主仓库中 src/runtime 目录下的文件。

测试

所有补丁都必须经过测试。引入新测试失败的补丁是不可接受的。

要运行 gccgo 测试套件，请在您的构建目录中运行 make check-go。这将运行 gcc/testsuite/go.* 下的各种测试，并且还会运行 libgo 测试套件。这份来自 Go 主仓库的测试副本是通过 gcc/testsuite/go.test/go-test.exp 中的 DejaGNU 脚本来运行的。

大多数新的测试应该提交到 Go 主仓库，以便稍后镜像到 GCC 仓库中。如果有针对 gccgo 的特定测试需求，它们应放在 GCC 仓库的 gcc/testsuite/go.go-torture 或 gcc/testsuite/go.dg 目录中。

提交更改

对 Go 前端的更改应遵循与 Go 主仓库相同的流程，只是针对的是 gofrontend 项目和 gofrontend-dev@googlegroups.com 邮件列表，而不是 go 项目和 golang-dev@googlegroups.com 邮件列表。然后，这些更改将被合并到 GCC 源代码中。
