## 起源 {#Origins}

### 这个项目的目的是什么？ {#What_is_the_purpose_of_the_project}

在2007年Go语言诞生之初，编程世界与今天大不相同。
生产环境软件通常使用C++或Java编写，
GitHub尚未出现，大多数计算机还不是多处理器系统，
除了Visual Studio和Eclipse之外，几乎没有可用的IDE或其他高级工具，
更不用说能免费从互联网上获取了。

与此同时，我们对于使用当时编程语言及其关联构建系统来开发大型软件项目所面临的过度复杂性感到沮丧。
自从C、C++和Java等语言首次被开发以来，计算机速度已大大提升，但编程行为本身并未取得同等程度的进步。
此外，尽管多处理器正变得普遍，但大多数语言在高效且安全地为其编程方面提供的帮助甚少。

我们决定退一步思考，随着技术发展，哪些主要问题将在未来几年主导软件工程，以及一门新语言如何能帮助应对这些问题。
例如，多核CPU的兴起表明，语言应为某种形式的并发或并行提供一等支持。
并且，为了让大型并发程序中的资源管理切实可行，需要垃圾回收，或至少是某种安全的自动内存管理。

这些思考促成了[一系列讨论](https://commandcenter.blogspot.com/2017/09/go-ten-years-and-climbing.html)，
Go语言由此诞生，起初是一些想法和需求，后来成为一门语言。
一个首要目标是，Go能通过赋能工具、自动化常规任务（如代码格式化）以及消除处理大型代码库的障碍，为一线程序员提供更多帮助。

关于Go语言的目标以及这些目标如何被实现（或至少在努力达成），一篇更详尽的描述可见于文章
[《Go在谷歌：服务于软件工程的语言设计》](/talks/2012/splash.article)。

### 项目的历史是怎样的？ {#history}

罗伯特·格瑞史默、罗布·派克和肯·汤普森于2007年9月21日开始在白板上勾勒一门新语言的目标。
几天之内，目标就具体化为一个行动计划以及对这门语言的大致构想。
设计工作在与其他无关工作并行的同时，业余时间继续进行。
到2008年1月，肯开始编写一个编译器来探索想法；它生成C代码作为输出。
到年中，这门语言已成为一个全职项目，并且足够稳定，可以尝试开发一个生产编译器。
2008年5月，伊恩·泰勒根据草案规范，独立开始为Go语言开发一个GCC前端。
拉克斯·考克斯在2008年末加入，并帮助将语言和库从原型推进到现实。

Go语言于2009年11月10日成为一个公开的开源项目。
社区中的无数贡献者贡献了想法、讨论和代码。

如今，全世界有数百万Go程序员——Gophers——而且这个数字每天都在增长。
Go的成功远远超出了我们的预期。

### 这门语言是叫Go还是Golang？ {#go_or_golang}

这门语言叫Go。
“Golang”的别名源于其网站最初是*golang.org*。
（当时还没有*.dev*域名。）
然而，许多人使用Golang这个名字，而且它作为一个标签很方便。
例如，这门语言在社交媒体上的标签就是“#golang”。
无论怎样，这门语言的名字就是Go。

附注：尽管[官方标志](/blog/go-brand)有两个大写字母，但这门语言的名字写作Go，而不是GO。

### 为什么你们要创造一门新语言？ {#creating_a_new_language}

Go诞生于对我们在谷歌工作时所使用的现有语言和环境的沮丧。
编程变得过于困难，而语言的选择是部分原因。
人们必须在高效编译、高效执行和易于编程之间做出选择；这三者无法在同一种主流语言中兼得。
那些可以自由选择的程序员，正通过转向Python和JavaScript等动态类型语言（而非C++或，程度较轻的Java）来追求易用性而牺牲安全性和效率。

对此感到困扰的不止我们。
在编程语言领域沉寂多年之后，Go是几门新语言中最早问世的之一——
Rust、Elixir、Swift等等——它们让编程语言开发重新成为一个活跃的、几乎主流的领域。Go通过尝试将解释型、动态类型语言的编程便捷性与静态类型、编译型语言的高效性和安全性相结合来解决这些问题。它还致力于更好地适应当前硬件，支持网络化和多核计算。最后，使用Go的体验应该是*快速的*：在单台计算机上构建大型可执行文件最多只需几秒钟。为了实现这些目标，我们重新思考了现有编程语言中的一些方法，从而产生了：组合式而非层次结构的类型系统；对并发和垃圾回收的支持；严格依赖规范；等等。这些无法由库或工具妥善处理，因此需要一门新的编程语言。

文章[《谷歌内部的Go》](/talks/2012/splash.article)讨论了Go语言设计背后的背景和动机，并更详细地介绍了本常见问题解答中提供的许多答案。

### Go语言的祖先是什么？ {#ancestors}

Go主要属于C语言家族（基本语法），同时吸收了Pascal/Modula/Oberon语言家族（声明、包）的重要元素，并借鉴了受Tony Hoare的CSP启发的语言（如Newsqueak和Limbo）中的一些并发理念。然而，它是一门全新的语言。在各个方面，这门语言的设计都基于对程序员行为和如何使编程（至少是我们所做的这类编程）更高效的思考，这意味着编程要变得更有趣。

### 设计中的指导原则是什么？ {#principles}

在设计Go时，Java和C++是编写服务器最常用的语言，至少在谷歌是如此。我们认为这些语言需要过多的记录工作和重复代码。一些程序员的应对方式是转向更动态、更灵活的语言，如Python，但代价是牺牲了效率和类型安全。我们认为完全可以在一门语言中同时实现高效性、安全性和灵活性。

Go尝试减少输入量（该词有两种含义）。在整个设计过程中，我们努力减少冗余和复杂性。没有前向声明，没有头文件；所有内容都只声明一次。初始化具有表达力、自动化且易于使用。语法简洁，关键字精简。通过使用`:=`声明并初始化构造进行简单的类型推导，减少了重复（如`foo.Foo* myFoo = new(foo.Foo)`）。或许最激进的是，没有类型层次结构：类型只是*存在*，它们无需声明彼此的关系。这些简化使得Go既具有表达力又易于理解，同时不牺牲生产力。

另一个重要原则是保持概念的正交性。方法可以为任何类型实现；结构体表示数据，而接口表示抽象；等等。正交性使得理解事物组合时会发生什么变得更加容易。

## 用法 {#Usage}

### 谷歌内部是否在使用Go？ {#internal_usage}

是的。Go在谷歌内部的生产环境中被广泛使用。一个例子是谷歌的下载服务器`dl.google.com`，它用于分发Chrome二进制文件和其他大型可安装程序（如`apt-get`包）。

Go并不是谷歌使用的唯一语言，远非如此，但它是包括[网站可靠性工程（SRE）](/talks/2013/go-sreops.slide)和大规模数据处理在内的多个领域的关键语言。它也是运行谷歌云软件的关键组成部分。

### 还有哪些公司使用Go？ {#external_usage}

Go的使用在全球范围内不断增长，尤其是在云计算领域（当然不仅限于此）。用Go编写的几个主要云基础设施项目包括Docker和Kubernetes，但还有更多。

不过，不仅仅局限于云计算，你可以从[go.dev网站](/)上的公司列表以及一些[成功案例](/solutions/case-studies)中看到。此外，Go Wiki包含一个定期更新的[页面](/wiki/GoUsers)，列出了许多使用Go的公司。

Wiki上还有一个页面，链接了更多关于使用该语言的公司和项目的[成功案例](/wiki/SuccessStories)。

### Go程序能否与C/C++程序链接？ {#Do_Go_programs_link_with_Cpp_programs}

可以在同一地址空间内同时使用C和Go，但这并非自然契合，可能需要特殊的接口软件。此外，将C代码与Go代码链接会放弃Go提供的内存安全和栈管理特性。有时确实需要使用C库来解决问题，但这样做总会引入纯Go代码所没有的风险元素，因此需谨慎使用。

如果确实需要将C与Go结合使用，具体做法取决于Go编译器的实现。谷歌Go团队支持的Go工具链中的“标准”编译器称为`gc`。此外，还有基于GCC的编译器（`gccgo`）和基于LLVM的编译器（`gollvm`），以及越来越多服务于不同目的、有时实现语言子集的非标准编译器，例如[TinyGo](https://tinygo.org/)。

`Gc`使用与C不同的调用约定和链接器，因此不能直接从C程序中调用，反之亦然。[`cgo`](/cmd/cgo/)程序提供了一种“外部函数接口”机制，允许从Go代码中安全地调用C库。SWIG将此功能扩展到了C++库。

你也可以将`cgo`和SWIG与`gccgo`和`gollvm`一起使用。由于它们使用传统ABI，在非常谨慎的情况下，也可以将这些编译器生成的代码直接与GCC/LLVM编译的C或C++程序链接。然而，安全地进行此操作需要理解所有相关语言的调用约定，以及在从Go调用C或C++时对栈限制的考虑。

### Go支持哪些IDE？ {#ide}Go 项目并未包含定制的 IDE，但该语言及其库的设计使得分析源代码变得简单。因此，大多数知名编辑器和 IDE 都能很好地支持 Go，无论是原生支持还是通过插件实现。

Go 团队还为 LSP 协议提供了一个 Go 语言服务器，称为 [`gopls`](https://pkg.go.dev/golang.org/x/tools/gopls#section-readme)。支持 LSP 的工具可以使用 `gopls` 来集成语言特定的支持。

提供良好 Go 支持的知名 IDE 和编辑器列表包括 Emacs、Vim、VSCode、Atom、Eclipse、Sublime、IntelliJ（通过名为 GoLand 的定制变体）等等。很可能您喜爱的编程环境也能高效地用于 Go 语言开发。

### Go 支持 Google 的 protocol buffers 吗？ {#protocol_buffers}

一个独立的开源项目提供了必要的编译器插件和库。它位于 [github.com/golang/protobuf/](https://github.com/golang/protobuf)。

## 设计 {#Design}

### Go 有运行时吗？ {#runtime}

Go 拥有一个庞大的运行时库，通常简称为 *runtime*，它是每个 Go 程序的一部分。这个库实现了垃圾回收、并发、栈管理以及 Go 语言的其它关键特性。尽管它在语言中更为核心，Go 的运行时类似于 C 语言的 `libc` 库。

然而，需要理解的重要一点是，Go 的运行时并不包含虚拟机（例如 Java 运行时所提供的那种）。Go 程序会被提前编译为本地机器码（或者对于某些变体实现，编译为 JavaScript 或 WebAssembly）。因此，尽管“运行时”一词通常用于描述程序运行的虚拟环境，但在 Go 中，它仅仅是提供关键语言服务的库的名称。

### Unicode 标识符是怎么回事？ {#unicode_identifiers}

在设计 Go 时，我们希望确保它不会过度以 ASCII 为中心，这意味着需要将标识符空间扩展到 7 位 ASCII 的限制之外。Go 的规则——标识符字符必须是 Unicode 定义的字母或数字——易于理解和实现，但也有限制。例如，组合字符被设计排除在外，这同时也排除了某些语言（如天城体）。

这条规则还有另一个不幸的后果。由于导出的标识符必须以大写字母开头，根据定义，由某些语言的字符创建的标识符就无法被导出。目前唯一的解决方案是使用类似 `X日本語` 的方式，但这显然不尽如人意。

自从该语言的最早版本以来，人们已经进行了大量思考，以寻求最佳方式扩展标识符空间，从而容纳使用其他母语的程序员。具体做法仍然是一个活跃的讨论话题，未来的语言版本可能会在标识符的定义上更加宽松。例如，它可能会采纳 Unicode 组织关于标识符的[一些建议](http://unicode.org/reports/tr31/)。无论发生什么变化，都必须在保持（或可能扩展）字母大小写决定标识符可见性这一我们最喜爱的 Go 特性的前提下，以兼容的方式进行。

目前，我们有一条简单的规则，可以在未来进行扩展而不会破坏现有程序，并且避免了因使用可能产生歧义的标识符规则而必然会导致的 bug。

### 为什么 Go 没有 X 特性？ {#Why_doesnt_Go_have_feature_X}

每种语言都包含新颖的特性，同时也会省略某些人喜欢的特性。Go 的设计着眼于编程的愉悦性、编译速度、概念的正交性，以及支持并发和垃圾回收等特性的需求。您喜爱的特性之所以可能缺失，可能是因为它不合适，因为它会影响编译速度或设计的清晰度，或者因为它会使基础系统模型变得过于复杂。

如果 Go 缺少 *X* 特性让您感到困扰，请原谅我们，并研究一下 Go 确实拥有的特性。您可能会发现它们以有趣的方式弥补了 *X* 的缺失。

### Go 何时获得了泛型类型？ {#generics}

Go 1.18 版本为语言添加了类型参数。这允许一种多态或泛型编程的形式。详情请参阅[语言规范](/ref/spec)和[提案](/design/43651-type-parameters)。

### 为什么 Go 最初发布时没有泛型类型？ {#beginning_generics}

Go 的定位是一种编写服务器程序的语言，这些程序应易于长期维护。（有关更多背景信息，请参阅[此文](/talks/2012/splash.article)）。其设计专注于可扩展性、可读性和并发性等方面。当时，多态编程似乎并非语言目标的核心，因此为了简洁性而最初被省略。

泛型很方便，但它们会在类型系统和运行时带来复杂性的代价。我们花了一些时间来开发一种设计，我们认为它带来的价值与其复杂性是相称的。

### 为什么 Go 没有异常？ {#exceptions}

我们认为，将异常与控制结构耦合（如 `try-catch-finally` 习惯用法）会导致代码混乱。这也容易促使程序员将许多普通错误（例如无法打开文件）标记为异常情况。

Go 采用了不同的方法。对于普通的错误处理，Go 的多值返回使得报告错误变得简单，而不会使返回值超载。[一个规范的错误类型，结合 Go 的其他特性](/doc/articles/error_handling.html)，使得错误处理变得愉快，但与其他语言中的方式截然不同。Go 还提供了一些内建函数来标记和处理真正的异常状况。这种恢复机制仅在函数状态因错误而需要清理时执行，这足以应对灾难性情况，但无需额外的控制结构，且使用得当时，能产生简洁的错误处理代码。

详情请参阅《[Defer、Panic 与 Recover](/doc/articles/defer_panic_recover.html)》一文。此外，《[错误即值](/blog/errors-are-values)》这篇博文展示了 Go 中处理错误的一种简洁方法，其演示了既然错误只是值，那么 Go 的全部能力都可以用于错误处理。

### 为什么 Go 没有断言？ {#assertions}

Go 不提供断言。它们无疑很方便，但我们的经验是，程序员会将其作为拐杖，从而避免思考正确的错误处理和报告。正确的错误处理意味着服务器在出现非致命错误后继续运行而不是崩溃。正确的错误报告意味着错误信息直接明了，省去了程序员解读庞大崩溃追踪日志的麻烦。当查看错误的程序员不熟悉代码时，精确的错误信息尤为重要。

我们理解这是一个争议点。Go 语言和库中许多与现代实践不同的做法，仅仅是因为我们认为有时值得尝试不同的方法。

### 为什么在 CSP 思想基础上构建并发？ {#csp}

并发和多线程编程长期以来一直以难度著称。我们认为部分原因是像 [pthreads](https://en.wikipedia.org/wiki/POSIX_Threads) 这样的复杂设计，部分是对底层细节（如互斥锁、条件变量和内存屏障）的过度强调。更高级别的接口可以使代码简单得多，即使底层仍然存在互斥锁等机制。

为并发提供高级语言支持的最成功模型之一，来自霍尔（Hoare）的通信顺序进程（Communicating Sequential Processes，即 CSP）。Occam 和 Erlang 是两个源自 CSP 的知名语言。Go 的并发原语源于这个技术树的不同分支，其主要贡献是将 channel（通道）作为一等公民的强大概念。几种早期语言的经验表明，CSP 模型非常适合过程式语言框架。

### 为什么使用 goroutine 而不是线程？ {#goroutines}

Goroutine（协程）是让并发易于使用的一部分。这个思想（已存在一段时间）是将独立执行的函数（即协程）多路复用到一组线程上。当一个协程阻塞时（例如通过调用阻塞系统调用），运行时会自动将同一线程上的其他协程移动到其他可运行的线程，这样它们就不会被阻塞。程序员看不到这些细节，这正是关键。其结果，我们称之为 goroutine，可以非常轻量：除了栈内存（仅几KB）外，开销很小。

为了使栈保持较小，Go 的运行时使用可调整大小的有界栈。新创建的 goroutine 会被分配几KB 内存，这通常足够使用。当不够时，运行时会自动增长（和缩减）用于存储栈的内存，从而允许许多 goroutine 占用适度的内存量。每次函数调用的平均 CPU 开销大约相当于三条廉价指令。在同一个地址空间中创建数十万个 goroutine 是可行的。如果 goroutine 只是线程，系统资源会在更小的数量级耗尽。

### 为什么 map 操作不是原子的？ {#atomic_maps}

经过长时间讨论，我们认为 map 的典型用法并不需要来自多个 goroutine 的安全访问，而在那些确实需要的场景中，map 可能是某个更大的、已同步的数据结构或计算的一部分。因此，要求所有 map 操作都获取互斥锁会拖慢大多数程序，却只为少数情况增加安全性。然而，这并非一个轻松的决定，因为它意味着不受控制的 map 访问可能会使程序崩溃。

该语言并未禁止原子 map 更新。在需要时（例如托管不受信任的程序时），实现可以锁定 map 访问。

Map 访问只有在发生更新时才不安全。只要所有 goroutine 都只是读取（查找 map 中的元素，包括使用 `for` `range` 循环遍历）而不通过对元素赋值或进行删除操作来更改 map，那么它们并发访问 map 而无需同步就是安全的。

为了辅助正确使用 map，该语言的某些实现包含一个特殊检查，会在运行时自动报告 map 被并发执行不安全地修改。此外，`sync` 库中有一个名为 [`sync.Map`](https://pkg.go.dev/sync#Map) 的类型，对于某些使用模式（如静态缓存）效果很好，尽管它不适合作为内建 map 类型的通用替代品。

### 你们会接受我的语言变更建议吗？ {#language_changes}

人们经常建议改进语言——[邮件列表](https://groups.google.com/group/golang-nuts)包含了此类讨论的丰富历史——但其中只有极少数变更被接受。尽管Go是一个开源项目，但语言和库受到[兼容性承诺](/doc/go1compat.html)的保护，该承诺禁止做出破坏现有程序的变更——至少在源代码层面如此（程序可能需要偶尔重新编译以保持最新状态）。如果你的提案违反了Go 1规范，我们甚至无法考虑，无论其价值如何。

未来的主要Go版本可能与Go 1不兼容，但相关讨论才刚刚开始，有一点可以确定：在此过程中引入的不兼容性将会非常少。此外，兼容性承诺鼓励我们为旧程序提供自动迁移路径，以防这种情况发生。

即使你的提案与Go 1规范兼容，也可能不符合Go设计目标的精神。文章*[Go at Google：服务于软件工程的语言设计](/talks/2012/splash.article)*解释了Go的起源及其设计背后的动机。

## 类型 {#types}

### Go是面向对象的语言吗？ {#Is_Go_an_object-oriented_language}

既是也不是。虽然Go拥有类型和方法，并支持面向对象风格的编程，但不存在类型层次结构。Go中的"interface"概念提供了一种不同的方法，我们认为它易于使用且在某些方面更具通用性。Go还提供了将类型嵌入其他类型的方式，可以提供类似于（但并非完全等同于）子类化的功能。此外，Go中的方法比C++或Java中的更通用：它们可以为任何类型的数据定义，甚至是基本的、"未装箱"的整数这样的内置类型。它们不限于结构体（类）。

同时，由于缺乏类型层次结构，Go中的"对象"比C++或Java等语言中的对象感觉轻量得多。

### 如何实现方法的动态分发？ {#How_do_I_get_dynamic_dispatch_of_methods}

实现动态分发方法的唯一方式是通过接口。结构体或任何其他具体类型上的方法始终是静态解析的。

### 为什么没有类型继承？ {#inheritance}

面向对象编程，至少在众所周知的语言中，涉及大量关于类型之间关系的讨论，而这些关系通常可以自动推导。Go采取了不同的方法。

Go不要求程序员预先声明两个类型相关，而是让类型自动满足任何指定了其方法子集的接口。除了减少记录工作，这种方法还具有实际优势：类型可以同时满足多个接口，而无需传统多重继承的复杂性；接口可以非常轻量——一个甚至零个方法的接口就能表达有用的概念；如果出现新想法或为了测试，可以在事后添加接口——而无需修改原始类型。由于类型和接口之间没有显式关系，因此不存在需要管理或讨论的类型层次结构。

这些理念可用于构建类似于类型安全Unix管道的结构。例如，可以看看`fmt.Fprintf`如何实现格式化打印到任意输出（而不仅仅是文件），或者`bufio`包如何与文件I/O完全分离，或者`image`包如何生成压缩图像文件。所有这些理念都源于一个接口（`io.Writer`），它代表一个单一方法（`Write`）。而这只是皮毛。Go的接口对程序结构产生深远影响。

这需要一些时间来适应，但这种隐式的类型依赖关系是Go最具生产力的特点之一。

### 为什么`len`是函数而不是方法？ {#methods_on_basics}

我们曾争论过这个问题，但最终决定将`len`及类似函数实现为函数在实践中是可行的，并且不会复杂化基本类型（在Go类型意义上）的接口问题。

### 为什么Go不支持方法和运算符重载？ {#overloading}

如果方法分发不需要同时进行类型匹配，就会变得更简单。其他语言的经验告诉我们，拥有多种同名但签名不同的方法偶尔有用，但在实践中也可能令人困惑且脆弱。仅按名称匹配并要求类型一致性是Go类型系统中的一个主要简化决策。

关于运算符重载，它更像是一种便利而非绝对需求。同样，没有它，事情会更简单。

### 为什么Go没有"implements"声明？ {#implements_interface}

Go类型通过实现接口的方法来实现该接口，仅此而已。这一特性使得可以在不修改现有代码的情况下定义和使用接口。它促成了一种[结构化类型](https://en.wikipedia.org/wiki/Structural_type_system)，促进了关注点分离并提高了代码重用性，并且使得在代码发展过程中出现的模式上构建更加容易。接口的语义是Go轻巧、灵活的主要原因之一。

更多详情请参阅[关于类型继承的问题](#inheritance)。

### 如何保证我的类型满足某个接口？ {#guarantee_satisfies_interface}

你可以通过尝试使用`T`的零值或指向`T`的指针（视情况而定）进行赋值，来要求编译器检查类型`T`是否实现了接口`I`：```go
type T struct{}
var _ I = T{}       // 验证 T 是否实现了 I。
var _ I = (*T)(nil) // 验证 *T 是否实现了 I。
```如果`T`（或相应的`*T`）未实现`I`，这个错误将在编译时被捕获。

如果你希望接口的使用者显式声明他们实现了该接口，可以向接口的方法集中添加一个带描述性名称的方法。例如：```
type Fooer interface {
    Foo()
    ImplementsFooer()
}
```一个类型必须实现 `ImplementsFooer` 方法才能成为 `Fooer`，从而明确记录这一事实，并在 [go doc](/cmd/go/#hdr-Show_documentation_for_package_or_symbol) 的输出中进行声明。```
type Bar struct{}
func (b Bar) ImplementsFooer() {}
func (b Bar) Foo() {}
```大多数代码不会使用这种约束，因为它们会限制接口的实用性。但有时，为了消解相似接口之间的歧义，这些约束是必要的。

### 为什么类型 T 不满足 Equal 接口？ {#t_and_equal_interface}

考虑一个简单的接口，用于表示能够将自身与另一个值进行比较的对象：```
type Equaler interface {
    Equal(Equaler) bool
}
```而这个类型，`T`：```go
type T int
func (t T) Equal(u T) bool { return t == u } // 不满足Equaler接口
```与某些多态类型系统中的类似情况不同，`T` 并未实现 `Equaler` 接口。`T.Equal` 的参数类型是 `T`，而非严格的必需类型 `Equaler`。

在 Go 语言中，类型系统不会自动提升 `Equal` 方法的参数类型；这是程序员的责任，正如实现了 `Equaler` 接口的类型 `T2` 所展示的那样：```go
type T2 int
func (t T2) Equal(u Equaler) bool { return t == u.(T2) }  // 满足Equaler接口
```然而，即使这样也与其他类型系统不同，因为在Go中，*任何*实现了 `Equaler` 的类型都可以作为 `T2.Equal` 的参数传递，并且在运行时我们必须检查该参数是否为 `T2` 类型。有些语言会安排在编译时做出这种保证。

一个相关的例子则反过来说明问题：```
type Opener interface {
   Open() Reader
}

func (t T3) Open() *os.File
```在Go语言中，`T3`并不满足`Opener`接口，尽管在其他语言中它可能满足。

虽然Go的类型系统在此类情况下为程序员提供的功能确实较少，但缺乏子类型化使得关于接口满足的规则变得非常易于陈述：函数的名称和签名是否与接口完全一致？Go的规则也易于高效实现。我们认为这些优势弥补了缺乏自动类型提升的不足。

### 能否将 []T 转换为 []interface{}？ {#convert_slice_of_interface}

不能直接转换。
语言规范不允许这样做，因为这两种类型在内存中的表示方式不同。
必须逐个将元素复制到目标切片。以下示例将`int`切片转换为`interface{}`切片：```
t := []int{1, 2, 3, 4}
s := make([]interface{}, len(t))
for i, v := range t {
    s[i] = v
}
```### 如果 T1 和 T2 具有相同的底层类型，能否将 []T1 转换为 []T2？ {#convert_slice_with_same_underlying_type}

以下代码示例中的最后一行无法编译。```go
type T1 int
type T2 int
var t1 T1
var x = T2(t1) // OK
var st1 []T1
var sx = ([]T2)(st1) // 不允许
```在 Go 语言中，类型与方法紧密相关，因为每个命名类型都有一个（可能为空的）方法集。
一般的规则是，您可以更改正在转换的类型名称（从而可能更改其方法集），但不能更改复合类型元素的名称（及其方法集）。
Go 语言要求您在类型转换时必须明确指定。

### 为什么我的 nil error 值不等于 nil？ {#nil_error}

在底层，接口由两个元素实现：一个类型 `T` 和一个值 `V`。
`V` 是一个具体的值，例如一个 `int`、`struct` 或指针，它本身绝不是接口，并且具有类型 `T`。
例如，如果我们将 `int` 值 3 存储在一个接口中，那么得到的接口值在原理上就是 (`T=int`, `V=3`)。
值 `V` 也被称为接口的*动态*值，因为在程序执行过程中，给定的接口变量可能持有不同的值 `V`（以及对应的类型 `T`）。

一个接口值只有当 `V` 和 `T` 都未设置时（`T=nil`，`V` 未设置），才是 `nil`。
特别地，一个 `nil` 接口将始终持有 `nil` 类型。
如果我们在一个接口值内部存储了一个类型为 `*int` 的 `nil` 指针，无论该指针的值如何，其内部类型都将是 `*int`：(`T=*int`, `V=nil`)。
因此，这样一个接口值将是非 `nil` 的*，即使其内部的指针值 `V` 是* `nil`。

这种情况可能会令人困惑，它出现在将一个 `nil` 值（例如一个 `error` 返回值）存储到接口值中时：```go
func returnsError() error {
    var p *MyError = nil
    if bad() {
        p = ErrBad
    }
    return p // 总会返回一个非nil的错误。
}
```如果一切顺利，函数将返回一个 `nil` 的 `p`，因此返回值是一个持有 (`T=*MyError`, `V=nil`) 的 `error` 接口值。这意味着，如果调用者将返回的错误与 `nil` 进行比较，即使没有发生任何错误，它看起来也总是像发生了错误一样。为了向调用者返回一个正确的 `nil` `error`，函数必须显式地返回一个 `nil`：```
func returnsError() error {
    if bad() {
        return ErrBad
    }
    return nil
}
```返回错误的函数最好在其签名中使用 `error` 类型（如我们上面所做的那样），而不是像 `*MyError` 这样的具体类型，以帮助确保错误被正确创建。例如，[`os.Open`](/pkg/os/#Open) 返回的是一个 `error`，即使当它不为 `nil` 时，它也总是具体类型 [`*os.PathError`](/pkg/os/#PathError)。

每当使用接口时，都可能出现与此处描述的类似情况。请记住，如果任何具体值已被存储在接口中，该接口就不会是 `nil`。更多信息，请参阅 [反射定律](/doc/articles/laws_of_reflection.html)。

### 为什么零大小类型的行为如此奇怪？ {#zero_size_types}

Go 支持零大小类型，例如没有字段的结构体 (`struct{}`) 或没有元素的数组 (`[0]byte`)。你无法在零大小类型中存储任何内容，但当不需要值时，这些类型有时很有用，例如 `map[int]struct{}` 或者一个拥有方法但没有值的类型。

具有零大小类型的不同变量可能会被放置在内存中的同一位置。这是安全的，因为这些变量中无法存储任何值。

此外，语言并未保证指向两个不同零大小变量的指针是否相等。这类比较甚至可能在程序的某个时刻返回 `true`，然后在另一个时刻返回 `false`，具体取决于程序是如何编译和执行的。

零大小类型的另一个问题是，指向零大小结构体字段的指针不得与指向内存中其他对象的指针重叠。这可能会在垃圾回收器中引起混淆。这意味着，如果一个结构体的最后一个字段是零大小的，那么该结构体将被填充，以确保指向该最后一个字段的指针不会与紧跟在结构体之后的内存重叠。因此，这个程序：

```go
[待翻译: var s struct {
    f1 byte
    f2 struct{}
}
fmt.Println(unsafe.Sizeof(s)) // 输出 2
]
``````
func main() {
	type S struct {
		f1 byte
		f2 struct{}
	}
	fmt.Println(unsafe.Sizeof(S{}))
}
```在大多数Go实现中，将输出 `2` 而不是 `1`。

### 为什么Go没有像C那样的无标记联合？ {#unions}

无标记联合会违反Go的内存安全保证。

### 为什么Go没有变体类型？ {#variant_types}

变体类型（也称为代数类型）提供了一种方式来指定一个值可能属于一组类型中的某一个，但仅限于这些类型。系统编程中的一个常见例子是指定一个错误是网络错误、安全错误或应用程序错误，并允许调用者通过检查错误的类型来区分问题的来源。另一个例子是语法树，其中每个节点可以是不同的类型：声明、语句、赋值等。

我们曾考虑将变体类型添加到Go中，但经过讨论后决定不采纳，因为它们与接口的重叠方式令人困惑。如果变体类型的元素本身就是接口，会发生什么？

此外，变体类型处理的某些内容在语言中已有涵盖。使用接口值存储错误并通过类型开关来区分情况，可以很容易地表达错误处理的示例。语法树的例子虽然不那么优雅，但也是可行的。

### 为什么Go没有协变结果类型？ {#covariant_types}

协变结果类型意味着一个像这样的接口：```
type Copyable interface {
    Copy() interface{}
}
```由该方法满足```
func (v Value) Copy() Value
```因为 `Value` 实现了空接口。
但在 Go 中方法类型必须精确匹配，所以 `Value` 并未实现 `Copyable`。
Go 将类型的行为（即其方法）与类型实现分离。
如果两个方法返回不同类型，它们执行的就不是同一操作。
渴望协变结果类型的开发者，往往试图通过接口表达类型继承体系。
在 Go 中，保持接口与实现之间的清晰分离更为自然。

## 值 {#Values}

### 为什么 Go 不提供隐式数值转换？ {#conversions}

C 语言中数值类型自动转换的便利性，远不及其引发的困惑。表达式何时为无符号？数值范围多大？是否会发生溢出？结果是否具有可移植性（不依赖运行机器架构）？
这也增加了编译器复杂度；C 语言的"常规算术转换"不仅难以实现，在不同架构间也存在不一致性。
出于可移植性考虑，我们决定通过代码中显式转换的代价，换取清晰明确的语义。
不过，Go 中常量的定义——具有任意精度且不受符号性和尺寸约束——极大地缓解了这个问题。

一个相关细节是：与 C 语言不同，即使 `int` 是 64 位类型，`int` 和 `int64` 仍是不同类型。`int` 类型具有通用性；如果你关注整数的具体位宽，Go 鼓励进行显式声明。

### Go 中的常量如何工作？ {#constants}

虽然 Go 对不同数值类型变量间的转换要求严格，但语言中的常量要灵活得多。
字面常量如 `23`、`3.14159` 以及 [`math.Pi`](/pkg/math/#pkg-constants) 等，存在于一种任意精度、无溢出或下溢的理想数字空间中。
例如，`math.Pi` 的值在源码中被指定到小数点后 63 位，涉及该值的常量表达式能保持远超 `float64` 所能承载的精度。
只有当常量或常量表达式被赋值给变量——即程序中的内存位置时，它才成为具有常规浮点特性和精度的"计算机数字"。

此外，
由于常量本质上是数字而非类型化值，Go 中的常量比变量使用更自由，这在一定程度上软化了严格转换规则带来的不便。
开发者可以编写诸如```
sqrt2 := math.Sqrt(2)
```编译器不会抱怨，因为理想数字 `2` 可以安全且精确地转换为 `float64` 类型，以供 `math.Sqrt` 函数调用。

标题为 [Constants（常量）](/blog/constants) 的博客文章更详细地探讨了这个主题。

### 为什么地图是内置的？ {#builtin_maps}

原因与字符串相同：它们是如此强大和重要的数据结构，以至于提供一个优秀的实现并辅以语法支持，能让编程变得更加愉快。我们相信 Go 的地图实现足够强大，可以满足绝大多数使用场景。如果特定应用能从自定义实现中获益，你当然可以编写一个，但它在语法上不会那么方便；这似乎是一个合理的权衡。

### 为什么地图不允许切片作为键？ {#map_keys}

地图查找需要一个等值比较运算符，而切片没有实现该运算符。它们没有实现等值比较，是因为在这种类型上等值的定义并不明确；这其中涉及诸多考量，比如浅比较与深比较、指针比较与值比较、如何处理递归类型等等。我们可能会重新审视这个问题——而且为切片实现等值比较不会使任何现有程序失效——但在没有明确定义切片等值比较应该意味着什么之前，目前将其排除在外更为简单。

结构体和数组是定义了等值比较的，因此它们可以用作地图键。

### 为什么地图、切片和通道是引用类型，而数组是值类型？ {#references}

这个话题有很多历史渊源。早期，地图和通道在语法上就是指针，不可能声明或使用一个非指针实例。同时，我们也曾为数组应该如何工作而纠结。最终我们得出结论，指针与值的严格分离使得语言更难使用。将这些类型更改为行为上引用关联的共享数据结构解决了这些问题。这一改变给语言增加了一些令人遗憾的复杂性，但对可用性产生了巨大影响：当 Go 引入这一特性时，它变成了一种更高效、更舒适的语言。

## 编写代码 {#Writing_Code}

### 如何为库编写文档？ {#How_are_libraries_documented}

若要从命令行访问文档，[go](/pkg/cmd/go/) 工具提供了一个 [doc](/pkg/cmd/go/#hdr-Show_documentation_for_package_or_symbol) 子命令，它为声明、文件、包等提供了一个文本界面来查看文档。

全局包发现页面 [pkg.go.dev/pkg/](/pkg/) 运行着一个服务器，该服务器从互联网上任何地方的 Go 源代码中提取包文档，并将其作为带有指向声明和相关元素的链接的 HTML 页面来提供。这是了解现有 Go 库最简便的方式。

在该项目早期，有一个类似的程序 `godoc`，它也可以运行来提取本地机器上文件的文档；[pkg.go.dev/pkg/](/pkg/) 本质上是它的一个后继者。另一个后继者是 [`pkgsite`](https://pkg.go.dev/golang.org/x/pkgsite/cmd/pkgsite) 命令，它像 `godoc` 一样可以在本地运行，尽管它尚未集成到 `go` `doc` 命令显示的结果中。

### 为什么 “go get” 在克隆仓库时使用 HTTPS？ {#git_https}

公司通常只允许通过标准 TCP 端口 80 (HTTP) 和 443 (HTTPS) 进行出站通信，而阻塞其他端口的出站通信，包括 TCP 端口 9418 (git) 和 TCP 端口 22 (SSH)。当使用 HTTPS 而非 HTTP 时，`git` 默认会强制进行证书验证，从而提供对中间人攻击、窃听和篡改的防护。因此，`go get` 命令为了安全而使用 HTTPS。

`Git` 可以配置为通过 HTTPS 进行身份验证，或者使用 SSH 代替 HTTPS。要通过 HTTPS 进行身份验证，你可以在 git 会读取的 `$HOME/.netrc` 文件中添加一行：```
machine github.com login *USERNAME* password *APIKEY*
```对于 GitHub 账户，密码可以是一个[个人访问令牌](https://help.github.com/articles/creating-a-personal-access-token-for-the-command-line/)。

`Git` 也可以配置为对匹配特定前缀的 URL 使用 SSH 代替 HTTPS。例如，若要为所有 GitHub 访问使用 SSH，请将以下内容添加到你的 `~/.gitconfig` 文件中：```
[url "ssh://在github.com上"]
	insteadOf = https://[github.com/]
```当处理私有模块但使用公共模块代理获取依赖时，可能需要设置 `GOPRIVATE`。
详情和附加设置请参阅[私有模块](/ref/mod#private-modules)章节。

### 如何使用 "go get" 管理软件包版本？ {#get_version}

Go 工具链内置了管理版本化相关软件包集合的系统，称为**模块**。
模块在 [Go 1.11](/doc/go1.11#modules) 中引入，自 [1.14](/doc/go1.14#introduction) 起已可用于生产环境。

要使用模块创建项目，请运行 [`go mod init`](/ref/mod#go-mod-init)。
此命令会创建一个用于跟踪依赖版本的 `go.mod` 文件。```
go mod init example/project
```要添加、升级或降级依赖项，请运行 [`go get`](/ref/mod#go-get)：```
go get golang.org/x/text@v0.3.5
```请参阅 [教程：创建模块](/doc/tutorial/create-module.html) 了解入门操作的更多信息。

请参阅 [模块开发指南](/doc/#developing-modules) 了解使用模块管理依赖项的指南。

模块内的包在演进过程中应遵循[导入兼容性规则](https://research.swtch.com/vgo-import)保持向后兼容：

> 若旧包与新包具有相同的导入路径，\
> 则新包必须与旧包保持向后兼容。

此处可参考 [Go 1 兼容性准则](/doc/go1compat.html)：
不要移除已导出的标识符，鼓励使用带标签的复合字面量等。
若需要不同功能，应添加新名称而非修改旧名称。

模块通过[语义版本控制](https://semver.org/)和语义化导入版本管理对此进行规范化。
若必须破坏兼容性，则需以新的主版本号发布模块。
主版本号为2及以上的模块要求在其路径中包含[主版本号后缀](/ref/mod#major-version-suffixes)（如 `/v2`）。
这既保留了导入兼容性规则：同一模块的不同主版本具有独立路径。

## 指针与内存分配 {#Pointers}

### 函数参数何时按值传递？ {#pass_by_value}

与所有C语言家族类似，Go中所有参数均按值传递。
即函数始终接收所传入值的副本，如同通过赋值语句将值赋予参数。
例如，向函数传递 `int` 值会创建该 `int` 的副本，而传递指针值则创建指针的副本（但不会复制其指向的数据）。
（关于此机制如何影响方法接收者的讨论，请参阅[后续章节](/doc/faq#methods_on_values_or_pointers)）

映射（map）和切片（slice）值的行为类似指针：它们是指向底层数据的描述符。
复制映射或切片值不会复制其指向的数据。
复制接口（interface）值会创建其存储内容的副本：若接口值持有结构体（struct），则复制该结构体；若持有指针，则复制指针（但同样不复制指针指向的数据）。

请注意，此处讨论的是操作语义。
实际实现可能采用优化手段避免复制，前提是不改变操作语义。

### 何时应使用指向接口的指针？ {#pointer_to_interface}

几乎从不需要。指向接口值的指针仅出现在罕见且复杂的场景中，通常涉及为延迟求值而伪装接口值类型。

常见错误是向期望接口的函数传递指向接口值的指针。
编译器会报错，但该情况仍可能令人困惑，因为有时[指针是实现接口的必要条件](#different_method_sets)。
关键认知在于：虽然具体类型的指针可以满足接口要求，但**除一个特例外，指向接口的指针永远不能满足接口要求**。

考虑变量声明```
var w io.Writer
```打印函数 `fmt.Fprintf` 的第一个参数是一个满足 `io.Writer` 接口的值——即实现了标准 `Write` 方法的对象。因此我们可以这样写：```
fmt.Fprintf(w, "hello, world\n")
```然而，如果我们传递 `w` 的地址，程序将无法编译。```
fmt.Fprintf(&w, "hello, world\n") // [编译时错误。]
```唯一特例是：任何值（即使是接口的指针）都可以被赋值给空接口类型（`interface{}`）的变量。
即便如此，若该值是指向接口的指针，几乎可以肯定是编程错误；其结果往往令人困惑。

### 方法该定义在值类型还是指针类型上？ {#methods_on_values_or_pointers}```go
func (s *MyStruct) pointerMethod() { } // 指针方法
func (s MyStruct)  valueMethod()   { } // 值方法
```对于不习惯使用指针的程序员来说，这两个示例之间的区别可能令人困惑，但实际情况非常简单。在为类型定义方法时，接收器（在上文示例中为 `s`）的行为与作为方法参数时完全相同。因此，是将接收器定义为值还是指针，与函数参数应该是值还是指针的问题是相同的。

有几个需要考虑的因素。

首先，也是最重要的一点，方法是否需要修改接收器？
如果需要，接收器*必须*是指针。
（切片和映射充当引用，因此它们的情况更为微妙，但例如要在方法中更改切片的长度，接收器仍然必须是指针。）
在上述示例中，如果 `pointerMethod` 修改了 `s` 的字段，
调用者将看到这些更改，但 `valueMethod` 是使用调用者参数的副本调用的（这是传递值的定义），因此它所做的更改对调用者是不可见的。

顺便提一下，在 Java 中，方法接收器始终是指针，尽管它们的指针性质在某种程度上是隐藏的（并且最近的发展正在将值接收器引入 Java）。Go 中的值接收器反而是不寻常的。

其次是效率方面的考虑。如果接收器很大，例如一个大的 `struct`，使用指针接收器可能开销更小。

接下来是一致性。如果类型的某些方法必须有指针接收器，那么其余方法也应该如此，这样无论类型如何使用，方法集都是一致的。
详情请参阅关于[方法集](#different_method_sets)的章节。

对于基本类型、切片和小型 `struct` 等类型，
值接收器的开销非常小，因此除非方法的语义要求使用指针，否则值接收器既高效又清晰。

### `new` 和 `make` 有什么区别？ {#new_and_make}

简而言之：`new` 分配内存，而 `make` 用于初始化切片、映射和通道类型。

更多详情请参阅 [Effective Go 的相关章节](/doc/effective_go.html#allocation_new)。

### 在 64 位机器上 `int` 的大小是多少？ {#q_int_sizes}

`int` 和 `uint` 的大小是特定于实现的，但在给定平台上它们彼此大小相同。
为了可移植性，依赖于特定值大小的代码应使用显式大小的类型，如 `int64`。
在 32 位机器上，编译器默认使用 32 位整数，
而在 64 位机器上，整数具有 64 位。
（历史上，情况并非总是如此。）

另一方面，浮点标量和复数类型始终是确定大小的（没有 `float` 或 `complex` 基本类型），
因为程序员在使用浮点数时应该意识到精度问题。
（无类型）浮点常量使用的默认类型是 `float64`。
因此 `foo` `:=` `3.0` 声明了一个类型为 `float64` 的变量 `foo`。
对于由（无类型）常量初始化的 `float32` 变量，变量类型必须在变量声明中显式指定：```
var foo float32 = 3.0
```或者，必须通过转换为常量指定类型，如 `foo := float32(3.0)`。

### 如何判断变量是分配在堆还是栈上？ {#stack_or_heap}

从正确性的角度来看，你不需要知道。
在 Go 中，只要存在对变量的引用，该变量就会一直存在。
实现所选择的存储位置与语言的语义无关。

存储位置确实会影响高效程序的编写。
在可能的情况下，Go 编译器会将函数局部的变量分配到该函数的栈帧中。
但是，如果编译器无法证明该变量在函数返回后不会被引用，
则编译器必须将变量分配到垃圾回收的堆上，以避免悬空指针错误。
此外，如果局部变量非常大，将其存储在堆上而不是栈上可能更合理。

在当前的编译器中，如果一个变量的地址被取用，该变量就是堆分配的候选者。
然而，基本的*逃逸分析*能够识别出一些情况，即这些变量不会存活到函数返回之后，因此可以驻留在栈上。

### 为什么我的 Go 进程使用了这么多虚拟内存？ {#Why_does_my_Go_process_use_so_much_virtual_memory}

Go 内存分配器预留了一大块虚拟内存区域作为分配的 arena。
这块虚拟内存是特定 Go 进程本地的；该预留不会剥夺其他进程的内存。

要查看分配给 Go 进程的实际内存量，请使用 Unix 的 `top` 命令，并查看 `RES`（Linux）或 `RSIZE`（macOS）列。

## 并发 {#Concurrency}

### 哪些操作是原子操作？那么互斥锁呢？ {#What_operations_are_atomic_What_about_mutexes}

关于 Go 中操作的原子性描述，可以在 [Go 内存模型](/ref/mem) 文档中找到。

底层的同步和原子原语可以在 [sync](/pkg/sync) 和 [sync/atomic](/pkg/sync/atomic) 包中找到。
这些包适用于简单的任务，例如增加引用计数或保证小范围的互斥。

对于更高级的操作，例如并发服务器之间的协调，更高级的技术可以产生更优雅的程序，Go 通过其 goroutine 和 channel 支持这种方法。
例如，你可以这样设计你的程序：在任何时刻，只有一个 goroutine 负责处理特定的一段数据。
这种方法被总结在最初的 [Go 谚语](https://www.youtube.com/watch?v=PAAkCSZUG1c) 中：

> 不要通过共享内存来通信。相反，通过通信来共享内存。

请参阅 [通过通信共享内存](/doc/codewalk/sharemem/) 代码漫游及其[相关文章](/blog/share-memory-by-communicating)，以详细了解这个概念。

大型的并发程序可能会借鉴这两种工具包。

### 为什么我的程序在使用更多 CPU 时没有变得更快？ {#parallel_slow}

一个程序在使用更多 CPU 时是否会变得更快，取决于它正在解决的问题。
Go 语言提供了并发原语，例如 goroutine 和 channel，但并发只有在底层问题本质上是并行的时才能启用并行性。
本质上顺序的问题无法通过添加更多 CPU 来加速，而那些可以被分解成可并行执行的片段的问题则可以加速，有时效果显著。

有时，添加更多 CPU 甚至可能使程序变慢。
实际上，那些花在同步或通信上的时间多于实际计算的程序，在使用多个操作系统线程时可能会经历性能下降。
这是因为在线程之间传递数据涉及上下文切换，这会产生显著的成本，并且该成本可能会随着 CPU 数量的增加而增加。
例如，[Go 规范中的素数筛示例](/ref/spec#An_example_package) 虽然启动了许多 goroutine，但并没有显著的并行性；增加线程（CPU）的数量更可能使其变慢，而不是变快。

有关此主题的更多详细信息，请参阅题为[并发不是并行](/blog/concurrency-is-not-parallelism)的演讲。

### 如何控制 CPU 的数量？ {#number_cpus}

可供执行 goroutine 同时使用的 CPU 数量由 `GOMAXPROCS` shell 环境变量控制，其默认值是可用的 CPU 核心数。
因此，具有并行执行潜力的程序在多 CPU 机器上默认应能达到并行。
要更改使用的并行 CPU 数量，请设置该环境变量或使用运行时包中同名的[函数](/pkg/runtime/#GOMAXPROCS)来配置运行时支持以使用不同数量的线程。
将其设置为 1 会消除真正并行的可能性，迫使独立的 goroutine 轮流执行。

运行时可能会分配比 `GOMAXPROCS` 值更多的线程来服务多个未完成的 I/O 请求。
`GOMAXPROCS` 只影响有多少 goroutine 可以同时实际执行；任意更多的 goroutine 可能在系统调用中被阻塞。

Go 的 goroutine 调度器在平衡 goroutine 和线程方面做得很好，甚至可以抢占一个 goroutine 的执行，以确保同一线程上的其他 goroutine 不会饿死。
然而，它并不完美。
如果你遇到性能问题，在每个应用程序的基础上设置 `GOMAXPROCS` 可能会有所帮助。

### 为什么没有 goroutine ID？ {#no_goroutine_id}

Goroutine 没有名字；它们只是匿名的工作者。
它们不向程序员暴露任何唯一标识符、名称或数据结构。
有些人对此感到惊讶，他们期望 `go` 语句能返回某个项目，以便稍后用来访问和控制该 goroutine。goroutine 保持匿名的根本原因，是为了在编写并发代码时能使用 Go 语言的全部特性。相比之下，当线程和 goroutine 被命名后形成的使用模式，可能会限制使用它们的库所能实现的功能。

下面通过一个示例说明其困难之处。一旦给某个 goroutine 命名并围绕它构建模型，它就变得特殊化，人们倾向于将所有计算都与该 goroutine 关联起来，而忽略了使用多个（可能共享的）goroutine 进行处理的可能性。如果 `net/http` 包将每个请求的状态与某个 goroutine 绑定，那么客户端在服务请求时将无法使用更多的 goroutine。

此外，诸如图形系统库（要求所有处理必须在"主线程"上执行）的经验表明，在并发语言中采用这种方式会显得笨拙且局限性强。特殊线程或 goroutine 的存在，迫使程序员扭曲程序结构以避免因无意间在错误线程上操作而导致的崩溃和其他问题。

对于那些确实需要特定 goroutine 的情况，Go 语言提供了如通道等特性，可通过灵活的方式与之交互。

## 函数和方法 {#Functions_methods}

### 为什么 T 和 *T 有不同的方法集？ {#different_method_sets}

正如 [Go 语言规范](/ref/spec#Types) 所述，类型 `T` 的方法集由所有接收者类型为 `T` 的方法组成，而对应的指针类型 `*T` 的方法集则包含所有接收者为 `*T` 或 `T` 的方法。这意味着 `*T` 的方法集包含了 `T` 的方法集，但反之则不然。

这种区分源于以下原因：如果一个接口值包含指针 `*T`，方法调用可以通过解引用指针来获取值；但如果接口值包含值 `T`，方法调用就没有安全的方式获取指针。（如果允许这样做，方法将能修改接口内部值的内存，这违反了语言规范。）

即使在某些情况下编译器可以获取值的地址并传递给方法，如果方法修改了该值，这些修改在调用方也将丢失。

举个例子，如果以下代码是有效的：```
var buf bytes.Buffer
io.Copy(buf, os.Stdin)
```如果这样做，它会将标准输入复制到 `buf` 的一个**副本**中，而不是 `buf` 本身。这几乎从来不是期望的行为，因此被语言所禁止。

### 闭包作为 goroutine 运行时会发生什么？ {#closures_and_goroutines}

由于循环变量的工作方式，在 Go 1.22 版本之前（本节末尾有更新说明），在使用并发的闭包时可能会产生一些混淆。请看以下程序：

{{raw `
<pre>
func main() {
    done := make(chan bool)

    values := []string{"a", "b", "c"}
    for _, v := range values {
        go func() {
            fmt.Println(v)
            done &lt;- true
        }()
    }

    // 等待所有 goroutine 完成后再退出
    for _ = range values {
        &lt;-done
    }
}
</pre>
`}}

你可能会错误地期望输出为 `a, b, c`。但你实际看到的很可能是 `c, c, c`。这是因为循环的每次迭代都使用同一个 `v` 变量实例，因此每个闭包共享这同一个变量。当闭包运行时，它会打印执行 `fmt.Println` 时 `v` 的值，但 `v` 可能在 goroutine 启动后已被修改。为了帮助在问题发生前检测到它以及其他问题，请运行 [`go vet`](/cmd/go/#hdr-Run_go_tool_vet_on_packages)。

要在启动闭包时将 `v` 的当前值绑定到每个闭包，必须修改内部循环，使其每次迭代都创建一个新变量。一种方法是将变量作为参数传递给闭包：

{{raw `
<pre>
    for _, v := range values {
        go func(<b>u</b> string) {
            fmt.Println(<b>u</b>)
            done &lt;- true
        }(<b>v</b>)
    }
</pre>
`}}

在这个例子中，`v` 的值被作为参数传递给匿名函数。然后，该值在函数内部作为变量 `u` 可访问。

更简单的方法是直接创建一个新变量，使用一种看起来可能有点奇怪但在 Go 中完全有效的声明风格：

{{raw `
<pre>
    for _, v := range values {
        <b>v := v</b> // 创建一个新的 'v'。
        go func() {
            fmt.Println(<b>v</b>)
            done &lt;- true
        }()
    }
</pre>
`}}

这种语言行为——即不为每次迭代定义新变量——事后看来被认为是一个失误，并在 [Go 1.22](/wiki/LoopvarExperiment) 中得到了解决，该版本确实为每次迭代创建一个新变量，从而消除了这个问题。

## 控制流 {#Control_flow}

### 为什么 Go 没有 `?:` 运算符？ {#Does_Go_have_a_ternary_form}

Go 中没有三元测试运算符。你可以使用以下方式来达到相同的效果：```
if expr {
    n = trueVal
} else {
    n = falseVal
}
```Go中没有`?:`运算符的原因是，该语言的设计者们发现这个运算符经常被用来创建极其复杂的表达式。尽管`if-else`形式更长，但它无疑是更清晰的。一种语言只需要一种条件控制流结构。

## 类型参数 {#Type_Parameters}

### 为什么Go语言要有类型参数？ {#why_generics}

类型参数允许所谓的泛型编程，即在定义函数和数据结构时使用后续在使用这些函数和数据结构时才指定的类型。例如，它们使得编写一个返回任意有序类型的两个值中的最小值的函数成为可能，而无需为每种可能的类型编写单独的版本。更多深入的解释和示例，请参阅博客文章[为什么需要泛型？](/blog/why-generics)。

### Go语言中的泛型是如何实现的？ {#generics_implementation}

编译器可以选择为每个实例化分别编译，或者将相似的实例化编译为单一实现。单一实现的方法类似于一个带有接口参数的函数。不同的编译器会根据不同的情况做出不同的选择。标准Go编译器通常会为每个形状相同的类型参数生成一个单一实例化，其中形状由类型的属性（如大小及其包含的指针位置）决定。未来的版本可能会尝试在编译时间、运行时效率和代码大小之间进行权衡。

### Go语言中的泛型与其他语言中的泛型相比如何？ {#generics_comparison}

所有语言的基本功能是相似的：可以使用后续指定的类型来编写类型和函数。尽管如此，还是存在一些差异。

* **Java**

	在Java中，编译器在编译时检查泛型类型，但在运行时移除这些类型。这被称为[类型擦除](https://en.wikipedia.org/wiki/Generics_in_Java#Problems_with_type_erasure)。例如，一个在编译时称为`List<Integer>`的Java类型，在运行时将变成非泛型的`List`类型。这意味着，例如，在使用Java形式的类型反射时，无法区分类型`List<Integer>`的值和类型`List<Float>`的值。在Go中，泛型类型的反射信息包含完整的编译时类型信息。

	Java使用类型通配符，如 `List<? extends Number>` 或 `List<? super Number>` 来实现泛型的协变和逆变。Go没有这些概念，这使得Go中的泛型类型更加简单。

* **C++**

	传统上，C++模板不对类型参数施加任何约束，尽管C++20通过[概念](https://en.wikipedia.org/wiki/Concepts_(C%2B%2B))支持可选的约束。在Go中，约束对所有类型参数都是强制性的。C++20概念表示为必须能与类型参数一起编译的小型代码片段。Go的约束是定义了所有允许类型参数集合的接口类型。

	C++支持模板元编程；而Go不支持。实际上，所有C++编译器都是在模板实例化的地方编译每个模板；如上所述，Go可以并且确实对不同的实例化使用不同的方法。

* **Rust**

	Rust中约束的版本称为特征约束。在Rust中，特征约束与类型之间的关联必须明确定义，要么在定义特征约束的crate中，要么在定义类型的crate中。在Go中，类型参数隐式地满足约束，就像Go类型隐式地实现接口类型一样。Rust标准库为比较或加法等操作定义了标准特征；Go标准库没有这样做，因为这些可以通过用户代码中的接口类型来表达。唯一的例外是Go的`comparable`预定义接口，它捕获了类型系统中无法表达的属性。

* **Python**

	Python不是一种静态类型语言，因此可以合理地说，所有Python函数默认都是泛型的：它们始终可以使用任何类型的值调用，并且任何类型错误都会在运行时检测到。

### 为什么Go语言使用方括号作为类型参数列表？ {#generic_brackets}

Java和C++使用尖括号作为类型参数列表，如Java的`List<Integer>`和C++的`std::vector<int>`。然而，这个选项对Go来说不可用，因为它会导致一个语法问题：当在函数内部解析代码时，例如`v := F<T>`，在看到`<`时，无法确定我们是在看一个实例化还是在看一个使用`<`运算符的表达式。这在没有类型信息的情况下非常难以解决。

例如，考虑如下语句

{{raw `
<pre>
    a, b = w &lt; x, y &gt; (z)
</pre>
`}}

如果没有类型信息，就不可能决定赋值的右侧是一对表达式（`w < x` 和 `y > z`），还是一个泛型函数实例化并返回两个结果值的调用（`(w<x, y>)(z)`）。

Go的一个关键设计决策是解析可以在没有类型信息的情况下进行，而使用尖括号作为泛型语法时，这似乎是不可能的。

Go使用方括号并非独创或原创；还有其他语言，如Scala，也在泛型代码中使用方括号。

### 为什么不支持带类型参数的方法？ {#generic_methods}

Go允许泛型类型拥有方法，但是除了接收者之外，这些方法的参数不能使用参数化类型。我们不预期Go会添加泛型方法。问题在于如何实现它们。
具体而言，考虑检查一个接口中的某个值是否实现了另一个具有额外方法的接口。
例如，考虑以下类型：一个空结构体，它带有一个泛型 `Nop` 方法，该方法可以返回其参数，适用于任何可能的类型：```
type Empty struct{}

func (Empty) Nop[T any](x T) T {
	return x
}
```现在假设一个 `Empty` 值被存储在 `any` 变量中，并传递给其他代码来检查它能做什么：```
func TryNops(x any) {
	if x, ok := x.(interface{ Nop(string) string }); ok {
		fmt.Printf("string %s\n", x.Nop("hello"))
	}
	if x, ok := x.(interface{ Nop(int) int }); ok {
		fmt.Printf("int %d\n", x.Nop(42))
	}
	if x, ok := x.(interface{ Nop(io.Reader) io.Reader }); ok {
		data, err := io.ReadAll(x.Nop(strings.NewReader("hello world")))
		fmt.Printf("reader %q %v\n", data, err)
	}
}
```如果 `x` 是一个 `Empty` 类型，这段代码是如何工作的？
看起来 `x` 必须同时满足所有三个测试条件，以及任何其他具有任意类型的形式。

当这些方法被调用时，会执行什么代码？
对于非泛型方法，编译器会为所有方法实现生成代码，并将其链接到最终的程序中。
但对于泛型方法，可能存在无限数量的方法实现，因此需要不同的策略。

有四个选项：

1.  在链接阶段，列出所有可能的动态接口检查，然后找出那些满足这些检查但缺少已编译方法的类型，再重新调用编译器来添加这些方法。
    这会使构建过程显著变慢，因为需要在链接后停止并重复部分编译。这尤其会拖慢增量构建。更糟糕的是，新编译的方法代码本身可能包含新的动态接口检查，这个过程可能需要重复进行。可以构造出一些例子，使得这个过程甚至永远无法完成。

2.  实现某种即时编译（JIT），在运行时编译所需的方法代码。
    Go 语言因其纯粹的预先编译（AOT）所带来的简单性和可预测性能而受益匪浅。仅仅为了实现一个语言特性而引入 JIT 的复杂性，我们并不情愿。

3.  为每个泛型方法安排一个慢速的回退方案，该方案针对类型参数上的每种可能的语言操作都使用一个函数表，然后将该回退实现用于动态测试。
    这种方法会使一个被意外类型参数化的泛型方法，比用编译时已观察到的类型参数化的同一方法慢得多。这将使性能变得非常不可预测。

4.  定义泛型方法根本不能用于满足接口。
    接口是 Go 语言编程中不可或缺的一部分。从设计角度来看，禁止泛型方法满足接口是不可接受的。

以上选项没有一个是好的，因此我们选择了“以上都不是”。

我们选择不使用带类型参数的方法，而是使用带类型参数的顶层函数，或者将类型参数添加到接收器类型上。

更多细节和示例，请参见 [提案](/design/43651-type-parameters#no-parameterized-methods)。

### 为什么我不能为参数化类型的接收器使用更具体的类型？ {#types_in_method_declaration}

泛型类型的方法声明在书写时，其接收器会包含类型参数的名称。
或许是因为这与在调用点指定类型的语法有些相似，有些人认为这提供了一种机制，可以通过在接收器中命名一个具体类型（例如 `string`）来为特定的类型参数生成定制化的方法：

```go
// [待翻译: 这是一个错误的例子]
func (s string) Method() {
    // 这里的 s 是 string 类型，不是泛型参数
    // 这会导致编译器错误
}
``````
type S[T any] struct { f T }

func (s S[string]) Add(t string) string {
    return s.f + t
}
```这是因为编译器将`string`解读为该方法中类型参数的名称。编译器错误信息通常类似"`operator + not defined on s.f (variable of type string)`"。这可能令人困惑，因为`+`运算符对预声明类型`string`完全有效，但此处的声明覆盖了该方法中`string`的定义，而运算符无法在这个不相关的`string`版本上工作。覆盖像`string`这样的预声明名称是合法的，但这种做法很奇怪，通常是编程错误。

### 为什么编译器无法推断程序中的类型参数？ {#type_inference}

虽然程序员通常能很容易看出泛型类型或函数的类型参数应该是什么，但语言规范不允许编译器进行此类推断。类型推断功能被有意限制，以确保推断的类型永远不会产生歧义。其他语言的经验表明，意外的类型推断可能导致程序阅读和调试时产生严重混淆。始终可以在调用时显式指定要使用的类型参数。未来可能会支持新的推断形式，但前提是这些规则保持简单清晰。

## 包与测试 {#Packages_Testing}

### 如何创建多文件包？ {#How_do_I_create_a_multifile_package}

将包的所有源文件放在单独目录中。源文件可以自由引用其他文件中的项目，无需前向声明或头文件。

除了拆分为多个文件外，该包的编译和测试方式与单文件包完全相同。

### 如何编写单元测试？ {#How_do_I_write_a_unit_test}

在包源文件所在目录中创建以`_test.go`结尾的新文件。在该文件内导入`"testing"`包，并编写如下形式的函数：

```go
func TestXxx(*testing.T) { /* ... */ }
``````
func TestFoo(t *testing.T) {
    ...
}
```在该目录中运行 `go test`。该脚本会查找所有 `Test` 函数，构建测试二进制文件并执行。

更多详情请参阅[如何编写Go代码](/doc/code.html)文档、[`testing`](/pkg/testing/) 包以及 [`go test`](/cmd/go/#hdr-Test_packages) 子命令说明。

### 我最喜欢的测试辅助函数在哪里？ {#testing_framework}

Go 语言的标准 [`testing`](/pkg/testing/) 包使得编写单元测试变得简单，但它缺乏其他语言测试框架提供的功能，例如断言函数。本文[前一章节](#assertions)解释了为什么 Go 没有断言，同样的论点也适用于测试中 `assert` 的使用。恰当的错误处理意味着在一个测试失败后让其他测试继续运行，这样调试失败的人就能全面了解问题所在。一个测试报告 `isPrime` 对 2、3、5 和 7（或对 2、4、8 和 16）给出了错误答案，比报告 `isPrime` 对 2 给出了错误答案从而不再运行后续测试要有用得多。触发测试失败的程序员可能不熟悉出错的代码。现在投入时间编写良好的错误信息，将在测试中断时带来回报。

一个相关的观点是，测试框架往往会发展成自己的迷你语言，包含条件控制和打印机制，但 Go 本身已经具备所有这些能力；何必重新发明呢？我们宁愿用 Go 编写测试；这样少学一种语言，而且这种方法使得测试直接明了、易于理解。

如果为编写良好的错误信息而需要的额外代码量看起来重复且繁重，那么采用表驱动测试可能效果更好，即遍历一个在数据结构中定义的输入和输出列表（Go 对数据结构字面量有出色的支持）。这样，编写优秀测试和良好错误信息的工作就能分摊到许多测试用例中。Go 标准库中充满了示范性示例，例如[`fmt` 包的格式化测试](/src/fmt/fmt_test.go)。

### 为什么 *X* 不在标准库中？ {#x_in_std}

标准库的目的是支撑运行时库、连接操作系统，并提供许多 Go 程序所需的关键功能，例如格式化 I/O 和网络。它还包含对网络编程重要的元素，包括密码学以及对 HTTP、JSON 和 XML 等标准的支持。

没有明确的标准来定义哪些内容应包含在内，因为在很长一段时间内，这是*唯一的* Go 库。然而，现在有标准来定义哪些内容可以被添加。

对标准库的新添加是罕见的，纳入的门槛也很高。包含在标准库中的代码承担着巨大的持续维护成本（通常由原作者之外的人承担），受制于 [Go 1 兼容性承诺](/doc/go1compat.html)（这会阻碍对 API 中任何缺陷的修复），并且受制于 Go 的[发布时间表](/s/releasesched)，这使得错误修复无法快速提供给用户。

大多数新代码应位于标准库之外，并通过 [`go` 工具](/cmd/go/) 的 `go get` 命令访问。此类代码可以有自己的维护者、发布周期和兼容性保证。用户可以在 [pkg.go.dev](https://pkg.go.dev/) 上查找包并阅读其文档。

尽管标准库中确实存在一些不太合适的内容，例如 `log/syslog`，但由于 Go 1 兼容性承诺，我们会继续维护库中的所有内容。但我们鼓励大多数新代码放在别处。

## 实现 {#Implementation}

### 构建编译器使用了什么编译器技术？ {#What_compiler_technology_is_used_to_build_the_compilers}

Go 有多个生产编译器，还有其他多个针对不同平台的编译器正在开发中。

默认编译器 `gc` 作为支持 `go` 命令的一部分包含在 Go 发行版中。`Gc` 最初是用 C 编写的，因为自举（bootstrapping）很困难——你需要一个 Go 编译器来搭建 Go 环境。但情况有所进展，自 Go 1.5 版本起，编译器本身就是一个 Go 程序。该编译器是使用自动翻译工具从 C 转换为 Go 的，如[设计文档](/s/go13compiler)和[演讲](/talks/2015/gogo.slide#1)中所述。因此，编译器现在是“自托管的”，这意味着我们需要面对自举问题。解决方案是事先拥有一个可工作的 Go 安装，就像通常拥有一个可工作的 C 安装一样。如何从源代码构建新的 Go 环境的故事在[这里](/s/go15bootstrap)和[这里](/doc/install/source)有描述。

`Gc` 是用 Go 编写的，使用递归下降解析器，并使用一个同样是用 Go 编写、但基于 Plan 9 加载器的自定义加载器来生成 ELF/Mach-O/PE 二进制文件。

`Gccgo` 编译器是一个用 C++ 编写的前端，采用递归下降解析器，并耦合到标准的 GCC 后端。一个实验性的 [LLVM 后端](https://go.googlesource.com/gollvm/) 正在使用同一个前端。

在项目初期，我们曾考虑为 `gc` 使用 LLVM，但认为它太大且太慢，无法满足我们的性能目标。现在回想起来，更重要的是，一开始就使用 LLVM 会更难引入 Go 所需的一些 ABI 和相关更改，例如栈管理，这些并非标准 C 环境的一部分。

事实证明，Go 是一种很好的语言来实现 Go 编译器，尽管这并非其最初的目标。一开始就实现自托管将使 Go 的设计能够专注于其最初的使用场景，即联网服务器。如果我们早期就决定 Go 应该自我编译，那么我们最终可能会得到一种更针对编译器构建的语言，这是一个有价值的目标，但并非我们最初的目标。虽然 `gc` 有其自身的实现，但 [`go/parser`](/pkg/go/parser/) 包提供了原生的词法分析器和解析器，并且也有原生的[类型检查器](/pkg/go/types)。`gc` 编译器使用的是这些库的变体。

### 运行时支持是如何实现的？ {#How_is_the_run_time_support_implemented}

同样由于引导启动问题，运行时代码最初主要是用 C 编写的（含少量汇编），但后来已被翻译为 Go（除了部分汇编代码）。`Gccgo` 的运行时支持使用 `glibc`。`gccgo` 编译器通过一种称为分段栈的技术来实现 goroutine，该技术得到了近期对 gold 链接器修改的支持。`Gollvm` 同样构建在相应的 LLVM 基础设施之上。

### 为什么我的简单程序生成的二进制文件如此之大？ {#Why_is_my_trivial_program_such_a_large_binary}

`gc` 工具链中的链接器默认创建静态链接的二进制文件。因此，所有 Go 二进制文件都包含 Go 运行时，以及支持动态类型检查、反射乃至 panic 时栈跟踪所需的运行时类型信息。

一个简单的 C "hello, world" 程序在 Linux 上使用 gcc 编译并静态链接后大小约为 750 kB，其中包含了 `printf` 的实现。而一个使用 `fmt.Printf` 的等效 Go 程序则有数兆字节，但它包含了更强大的运行时支持以及类型和调试信息。

使用 `gc` 编译的 Go 程序可以通过 `-ldflags=-w` 标志禁用 DWARF 生成，从而从二进制文件中移除调试信息，而不会损失其他功能。这可以显著减小二进制文件的大小。

### 我能停止这些关于未使用变量/导入的报错吗？ {#unused_variables_and_imports}

存在未使用的变量可能意味着程序有 bug，而未使用的导入只是会减慢编译速度，随着程序代码和开发人员的不断增加，这种影响可能会变得相当显著。基于这些原因，Go 拒绝编译包含未使用变量或导入的程序，这是在短期便利性与长期构建速度和程序清晰度之间做出的权衡。

不过，在开发代码时，暂时出现这些情况很常见，在程序能够编译之前必须编辑掉它们可能会很烦人。

有人曾请求添加一个编译器选项来关闭这些检查，或者至少将它们降级为警告。然而，这样的选项并未被添加，因为编译器选项不应影响语言的语义，而且 Go 编译器不报告警告，只报告阻碍编译的错误。

不设置警告有两个原因。首先，如果值得抱怨，就值得在代码中修复。（反之，如果值得修复，就值得一提。）其次，让编译器生成警告会鼓励实现去警告那些可能使编译输出嘈杂、掩盖真正*应该*被修复的错误的边缘情况。

不过，要解决这个问题很简单。在开发期间使用空白标识符来让未使用的内容得以保留。```
import "unused"

// 通过引用包中的项来标记此导入已使用。
var _ = unused.Item  // TODO: Delete before committing!

func main() {
    debugData := debug.Profile()
    _ = debugData // 仅在调试期间使用。
    ....
}
```如今，大多数Go程序员会使用一个名为[goimports](https://godoc.org/golang.org/x/tools/cmd/goimports)的工具，它会自动重写Go源文件以包含正确的导入，从而在实际中消除了未使用导入的问题。这个程序可以轻松连接到大多数编辑器和IDE，以便在编写Go源文件时自动运行。此功能也已内置在`gopls`中，如[上文所述](/doc/faq#ide)。

### 为什么我的病毒扫描软件认为我的Go发行版或编译后的二进制文件被感染了？ {#virus}

这很常见，尤其是在Windows机器上，几乎总是误报。商业病毒扫描程序常常被Go二进制文件的结构所困惑，因为它们不像其他语言编译的二进制文件那样常见。

如果你刚安装了Go发行版，系统就报告它被感染了，那肯定是个错误。为了彻底确认，你可以通过与[下载页面](/dl/)上的校验和进行比较来验证下载内容。

无论如何，如果你认为报告有误，请向病毒扫描软件供应商报告一个bug。也许病毒扫描程序最终能学会理解Go程序。

## 性能 {#Performance}

### 为什么Go在基准测试X中表现不佳？ {#Why_does_Go_perform_badly_on_benchmark_x}

Go的设计目标之一是在可比程序中接近C的性能，但在某些基准测试中它表现相当差，包括[golang.org/x/exp/shootout](https://go.googlesource.com/exp/+/master/shootout/)中的一些测试。最慢的测试依赖于一些Go中没有可比性能版本的库。例如，[pidigits.go](https://go.googlesource.com/exp/+/master/shootout/pidigits.go)依赖于一个多精度数学包，而C版本（不同于Go版本）使用了[GMP](https://gmplib.org/)（它是用优化的汇编语言编写的）。依赖正则表达式的基准测试（例如[regex-dna.go](https://go.googlesource.com/exp/+/master/shootout/regex-dna.go)）本质上是将Go原生的[regexp包](/pkg/regexp)与PCRE等成熟、高度优化的正则表达式库进行比较。

基准测试的胜负取决于大量的调优，而大多数基准测试的Go版本需要更多关注。如果你测量真正可比的C和Go程序（[reverse-complement.go](https://go.googlesource.com/exp/+/master/shootout/reverse-complement.go)就是一个例子），你会发现两种语言在原始性能上比这个测试套件所显示的要接近得多。

尽管如此，仍有改进空间。编译器不错但可以更好，许多库需要重大的性能改进，而且垃圾回收器还不够快。（即使它足够快，注意不生成不必要的垃圾也会产生巨大影响。）

无论如何，Go通常非常有竞争力。随着语言和工具的发展，许多程序的性能已有了显著提高。请参阅关于[分析Go程序](/blog/profiling-go-programs)的博客文章以获取一个信息丰富的例子。它虽然有些陈旧，但仍包含有用的信息。

## 与C的差异 {#change_from_c}

### 为什么语法与C如此不同？ {#different_syntax}

除了声明语法外，差异并不大，这源于两个愿望。首先，语法应该感觉轻量，没有太多强制关键字、重复或晦涩之处。其次，该语言被设计为易于分析，并且可以在没有符号表的情况下进行解析。这使得构建调试器、依赖分析器、自动文档提取器、IDE插件等工具变得更加容易。C及其后代在这方面是出了名的困难。

### 为什么声明是反向的？ {#declarations_backwards}

只有当你习惯C时它们才是反向的。在C中，变量声明的概念类似于表示其类型的表达式，这是一个好主意，但类型和表达式语法混合得不太好，结果可能令人困惑；考虑函数指针。Go在很大程度上将表达式和类型语法分开，这简化了事情（对指针使用前缀`*`是证明规则的例外）。在C中，声明```
    int* a, b;
```在Go中，声明`a`为指针但`b`不是；```
    var a, b *int
```声明二者均为指针。这样更加清晰规范。此外，`:=` 短声明形式表明完整的变量声明应当保持与 `:=` 相同的顺序，因此```
    var a uint64 = 1
```与`:=`短声明形式具有相同效果```
    a := uint64(1)
```解析也通过为类型设置不同于表达式语法的独立语法而得到简化；像`func`和`chan`这样的关键字使事情更清晰。

更多详情请参阅关于
[Go声明语法](/doc/articles/gos_declaration_syntax.html)的文章。

### 为什么没有指针运算？ {#no_pointer_arithmetic}

出于安全性考虑。没有指针运算，就可以创建一种永远无法派生出错误地址并成功运行的语言。编译器和硬件技术已经发展到使用数组索引的循环可以与使用指针运算的循环同样高效。此外，缺乏指针运算可以简化垃圾收集器的实现。

### 为什么`++`和`--`是语句而不是表达式？以及为什么是后缀而不是前缀？ {#inc_dec}

没有指针运算，前缀和后缀自增运算符的便利性价值就降低了。通过将它们完全从表达式层次结构中移除，表达式语法得到了简化，围绕`++`和`--`求值顺序的混乱问题（考虑`f(i++)`和`p[i] = q[++i]`）也随之消除。这种简化意义重大。至于后缀与前缀，两者都可以很好地工作，但后缀版本更为传统；坚持使用前缀源于STL，而STL是一个库，其所属语言的名称中讽刺地包含了后缀自增。

### 为什么有大括号却没有分号？以及为什么不能把开括号放在下一行？ {#semicolons}

Go使用大括号进行语句分组，这种语法对于使用过C系列语言的程序员来说很熟悉。然而，分号是给解析器用的，而不是给人用的，我们希望尽可能地消除它们。为了实现这个目标，Go从BCPL借鉴了一个技巧：在形式语法中，分隔语句的分号是存在的，但它们是由词法分析器在任何可能是语句结尾的行尾自动插入的，无需前瞻。这在实践中效果很好，但其效果是强制了一种大括号风格。例如，函数的开括号不能单独出现在一行上。

有人认为词法分析器应该进行前瞻，以允许大括号放在下一行。我们不同意。既然Go代码旨在通过
[`gofmt`](/cmd/gofmt/)自动格式化，
那么*必须*选择某种风格。这种风格可能不同于你在C或Java中使用的，但Go是一种不同的语言，`gofmt`的风格与其他任何风格一样好。更重要的是——非常重要的是——为所有Go程序提供单一的、由程序强制执行的格式，其优势远远超过了特定风格的任何感知劣势。还要注意，Go的风格意味着Go的交互式实现可以一次使用标准语法执行一行代码，而无需特殊规则。

### 为什么需要垃圾回收？它不会太昂贵吗？ {#garbage_collection}

系统程序中最大的簿记来源之一是管理已分配对象的生命周期。在像C这样需要手动管理的语言中，这可能会消耗程序员大量的时间，并且通常是造成严重错误的根源。即使在像C++或Rust这样提供辅助机制的语言中，这些机制也可能对软件设计产生重大影响，常常增加自身的编程开销。我们认为消除这种程序员开销至关重要，而过去几年垃圾收集技术的进步让我们相信，它可以足够廉价地实现，并且延迟足够低，从而可以成为联网系统的可行方案。

并发编程的大部分困难都根源于对象生命周期问题：随着对象在线程之间传递，保证它们被安全释放变得很繁琐。自动垃圾回收使得编写并发代码变得容易得多。当然，在并发环境中实现垃圾回收本身就是一个挑战，但一次性解决这个问题，而不是在每个程序中都解决，对所有人都有帮助。

最后，并发性暂且不论，垃圾回收使接口更简单，因为它们不需要指定内存如何在它们之间进行管理。

这并不是说最近在像Rust这样的语言中，为管理资源问题带来新想法的工作是错误的；我们鼓励这项工作，并很高兴看到它的发展。但Go采取了一种更传统的方法，即通过垃圾回收——且仅通过垃圾回收——来解决对象生命周期问题。

当前的实现是一个标记清除收集器。如果机器是多处理器的，收集器将在单独的CPU核心上与主程序并行运行。近年来在收集器上的主要工作已将暂停时间（即使是对于大型堆）通常减少到亚毫秒范围，这几乎消除了对联网服务器中垃圾收集的主要反对意见之一。工作仍在继续，以改进算法，进一步减少开销和延迟，并探索新方法。Go团队的Rick Hudson在2018年的
[ISMM主题演讲](/blog/ismmkeynote)中
描述了迄今为止的进展，并提出了一些未来的方法。

关于性能问题，请记住，Go为程序员提供了对内存布局和分配的相当大的控制权，这比典型的垃圾收集语言要多得多。一个细心的程序员可以通过良好地使用语言来显著降低垃圾收集开销；请参阅关于
[分析Go程序](/blog/profiling-go-programs)的文章，其中包含一个实际示例，包括Go性能分析工具的演示。
