## 起源 {#Origins}

### 项目的目的是什么？ {#What_is_the_purpose_of_the_project}

在 Go 语言于 2007 年诞生之初，编程世界与今日大不相同。
生产级软件通常用 C++ 或 Java 编写，
GitHub 尚未出现，大多数计算机还不是多处理器系统，
除 Visual Studio 和 Eclipse 外，几乎没有什么 IDE 或高级工具可用，
更不用说在互联网上免费提供的工具了。

与此同时，我们对于使用现有语言及其关联构建系统开发大型软件项目所需的过度复杂性感到沮丧。
自 C、C++ 和 Java 等语言诞生以来，计算机速度已大幅提升，
但编程行为本身并未取得同等程度的进步。
此外，虽然多处理器正变得普及，但大多数语言在帮助程序员高效、安全地编写并行程序方面几乎无能为力。

我们决定退一步思考：随着技术发展，未来几年哪些主要问题将主导软件工程，
以及一门新语言如何帮助解决这些问题。
例如，多核 CPU 的兴起表明语言应提供某种并发或并行的一流支持。
为了让大型并发程序中的资源管理变得可行，垃圾回收——或至少某种安全的自动内存管理——是必需的。

这些思考引发了
[一系列讨论](https://commandcenter.blogspot.com/2017/09/go-ten-years-and-climbing.html)，
Go 语言由此诞生。它最初是一套想法和需求，
随后演变为一门语言。
一个核心目标是：Go 要通过支持工具化、自动化日常任务（如代码格式化），
并消除处理大型代码库的障碍，来为一线程序员提供更多帮助。

关于 Go 的目标以及如何实现（或至少接近）这些目标，更全面的描述可参见文章
[Go at Google: Language Design in the Service of Software Engineering](/talks/2012/splash.article)。

### 项目的历史是怎样的？ {#history}

2007年9月21日，Robert Griesemer、Rob Pike 和 Ken Thompson 开始在白板上勾勒新语言的目标。
几天之内，目标逐渐明确，形成了一个具体的计划和初步构想。
设计工作与其他无关工作并行进行，作为兼职项目持续开展。
到2008年1月，Ken 已开始编写一个编译器来探索想法；该编译器以生成 C 代码作为输出。
到年中，该语言已成为全职项目，并趋于稳定，足以尝试开发生产级编译器。
2008年5月，Ian Taylor 根据草案规范，独立开始为 Go 开发 GCC 前端。
Russ Cox 于2008年末加入，帮助将语言和库从原型推进到现实。

Go 于2009年11月10日成为公开的开源项目。
社区中无数人贡献了想法、讨论和代码。

如今，全球已有数百万 Go 程序员——gopher——且每天都在增加。
Go 的成功远超我们的预期。

### 这门语言叫 Go 还是 Golang？ {#go_or_golang}

这门语言叫 Go。
"golang" 的别名源于网站最初域名为 *golang.org*。
（当时还没有 *.dev* 域名。）
尽管很多人使用 golang 这个名字，它确实很方便作为标签。
例如，该语言在社交媒体上的标签是 "#golang"。
但无论如何，语言的正式名称就是 Go。

附注：尽管
[官方标志](/blog/go-brand) 有两个大写字母，但语言名称写作 Go，而非 GO。

### 你们为什么创造一门新语言？ {#creating_a_new_language}

Go 诞生于对现有语言和开发环境的不满，这些语言和环境无法满足我们在谷歌的工作需求。
编程变得过于困难，部分原因在于语言选择。开发者不得不在高效编译、高效执行和易于编程之间做出取舍；
在主流语言中，三者无法兼得。
能够选择的程序员为了追求易用性而牺牲了安全性和效率，转向了 Python 和 JavaScript 等动态类型语言，
而非 C++ 或（在较小程度上的）Java。

我们并非唯一有此关切的人。
在编程语言领域多年相对平静之后，
Go 是第一批新语言之一——随后还有 Rust、Elixir、Swift 等——
它们使编程语言开发重新成为一个活跃、甚至接近主流的领域。Go 试图通过将解释型、动态类型语言的编程易用性与静态类型、编译型语言的效率和安全性相结合来解决这些问题。
它还旨在更好地适应当前的硬件环境，支持网络化和多核计算。
最后，使用 Go 旨在实现*快速*：在单台计算机上构建一个大型可执行文件最多应只需几秒钟。
实现这些目标促使我们重新思考现有语言中的一些编程方法，最终形成了：
一种组合式而非层次化的类型系统；
对并发和垃圾回收的支持；严格的依赖规范；
等等。
这些无法通过库或工具很好地处理，因此需要一种新的语言。

文章《[Go at Google](/talks/2012/splash.article)》讨论了 Go 语言设计背后的背景和动机，
并提供了关于本常见问题中许多答案的更多细节。

### Go 的祖先有哪些？ {#ancestors}

Go 主要属于 C 语言家族（基本语法），
同时从 Pascal/Modula/Oberon 家族（声明、包）获得了显著影响，
并吸收了一些受 Tony Hoare 的 CSP（通信顺序进程）启发的语言的思想，
例如 Newsqueak 和 Limbo（并发）。
然而，它全面地是一门新语言。
在各方面，这门语言的设计都是基于思考程序员的所作所为以及如何使编程——至少是我们所做的那种编程——更高效，也就是更有趣。

### 设计中的指导原则是什么？ {#principles}

在设计 Go 时，Java 和 C++ 是编写服务器最常用的语言，至少在 Google 是如此。
我们觉得这些语言需要过多的记录和重复。
一些程序员的反应是转向更动态、更灵活的语言，如 Python，但代价是牺牲了效率和类型安全性。
我们认为，应该可以在一门语言中同时拥有效率、安全性和灵活性。

Go 试图减少两种意义上的“打字”（typing）工作量。
在整个设计过程中，我们努力减少冗余和复杂性。
没有前向声明，也没有头文件；所有内容只声明一次。
初始化富有表现力、自动化且易于使用。
语法简洁，关键字使用轻巧。
通过使用 `:=` 声明并初始化构造进行简单的类型推导，减少了重复（例如 `foo.Foo* myFoo = new(foo.Foo)`）。
也许最根本的是，没有类型层次结构：类型就*是*它们本身，无需宣告其关系。
这些简化使得 Go 既富有表现力又易于理解，同时不会牺牲生产力。

另一个重要原则是保持概念的正交性。
方法可以为任何类型实现；结构体表示数据，而接口表示抽象；等等。
正交性使得更容易理解事物组合在一起时会发生什么。

## 用法 {#Usage}

### Google 内部是否在使用 Go？ {#internal_usage}

是的。Go 在 Google 内部的生产环境中被广泛使用。
一个例子是 Google 的下载服务器 `dl.google.com`，
它负责分发 Chrome 二进制文件和其他大型可安装程序，如 `apt-get` 包。

Go 并非 Google 使用的唯一语言，远非如此，但它是多个领域的关键语言，
包括[网站可靠性工程 (SRE)](/talks/2013/go-sreops.slide) 和大规模数据处理。
它也是运行 Google Cloud 的软件的关键组成部分。

### 还有哪些公司在使用 Go？ {#external_usage}

Go 的使用在全球范围内不断增长，尤其是在但不限于云计算领域。
用 Go 编写的几个主要云基础设施项目是 Docker 和 Kubernetes，
但还有更多。

不仅仅是云计算，你也可以从 [go.dev 网站](/) 上的公司列表以及一些[成功案例](/solutions/case-studies)中看到。
此外，Go Wiki 包含一个定期更新的[页面](/wiki/GoUsers)，列出了众多使用 Go 的公司中的一部分。

该 Wiki 还有一个页面，链接到更多关于使用该语言的公司和项目的[成功案例](/wiki/SuccessStories)。

### Go 程序能否与 C/C++ 程序链接？ {#Do_Go_programs_link_with_Cpp_programs}

可以在同一地址空间中同时使用 C 和 Go，但这并非自然匹配，可能需要特殊的接口软件。
此外，将 C 与 Go 代码链接会放弃 Go 提供的内存安全性和栈管理特性。
有时确实必须使用 C 库来解决问题，但这样做总是会引入纯 Go 代码不存在的风险元素，因此请谨慎操作。

如果确实需要在 Go 中使用 C，如何操作取决于 Go 编译器的实现。
由 Google Go 团队支持的 Go 工具链中的“标准”编译器称为 `gc`。
此外，还有基于 GCC 的编译器 (`gccgo`) 和基于 LLVM 的编译器 (`gollvm`)，
以及为不同目的服务的、数量不断增长的非主流编译器，有时实现语言子集，
例如 [TinyGo](https://tinygo.org/)。

`gc` 使用与 C 不同的调用约定和链接器，因此不能从 C 程序直接调用，反之亦然。
[`cgo`](/cmd/cgo/) 程序提供了一种“外部函数接口”机制，允许从 Go 代码中安全地调用 C 库。
SWIG 将此能力扩展到了 C++ 库。

你也可以将 `cgo` 和 SWIG 与 `gccgo` 和 `gollvm` 一起使用。
由于它们使用传统的 ABI，在非常小心的情况下，也可以将这些编译器编译的代码直接与 GCC/LLVM 编译的 C 或 C++ 程序链接。
然而，安全地这样做需要理解所有相关语言的调用约定，并且在从 Go 调用 C 或 C++ 时需要注意栈限制。

### Go 支持哪些 IDE？ {#ide}Go 项目并未包含自定义 IDE，但其语言和库的设计使得分析源代码变得简单。因此，大多数知名编辑器和 IDE 都能良好支持 Go，无论是直接支持还是通过插件支持。

Go 团队还支持一个符合 LSP 协议的 Go 语言服务器，称为 [`gopls`](https://pkg.go.dev/golang.org/x/tools/gopls#section-readme)。支持 LSP 的工具可以使用 `gopls` 来集成特定语言的支持。

提供良好 Go 支持的知名 IDE 和编辑器列表包括 Emacs、Vim、VSCode、Atom、Eclipse、Sublime、IntelliJ（通过名为 GoLand 的自定义变体）等等。很可能你喜爱的环境也能成为高效进行 Go 编程的选择。

### Go 是否支持 Google 的协议缓冲区？ {#protocol_buffers}

一个独立的开源项目提供了必要的编译器插件和库。它位于 [github.com/golang/protobuf/](https://github.com/golang/protobuf)。

## 设计 {#Design}

### Go 是否有运行时？ {#runtime}

Go 有一个庞大的运行时库，通常简称为 *运行时*，它是每个 Go 程序的一部分。这个库实现了垃圾回收、并发、栈管理以及 Go 语言的其他关键特性。尽管它在语言中更为核心，但 Go 的运行时类似于 C 语言库 `libc`。

然而，重要的是要理解，Go 的运行时并不包含虚拟机，不像 Java 运行时所提供的那样。Go 程序会被提前编译为本地机器码（对于某些变体实现，也可编译为 JavaScript 或 WebAssembly）。因此，尽管术语“运行时常”用于描述程序运行的虚拟环境，但在 Go 中，“运行时”只是指代提供关键语言服务的库的名称。

### Unicode 标识符是怎么回事？ {#unicode_identifiers}

在设计 Go 时，我们希望确保它不会过度以 ASCII 为中心，这意味着需要将标识符的范围从 7 位 ASCII 的限制中扩展出来。Go 的规则——标识符字符必须是 Unicode 定义的字母或数字——易于理解和实现，但也存在限制。例如，组合字符在设计上被排除在外，这导致一些语言（如天城文）无法使用。

这条规则还有另一个令人遗憾的后果。由于导出的标识符必须以大写字母开头，那么根据定义，由某些语言的字符创建的标识符就无法被导出。目前唯一的解决方案是使用类似 `X日本語` 的名称，这显然不尽如人意。

自该语言的最早版本起，关于如何最好地扩展标识符空间以适应使用其他母语的程序员，已经进行了大量思考。具体该怎么做仍然是一个活跃的讨论话题，该语言的未来版本可能会在标识符的定义上更加宽松。例如，它可能会采纳 Unicode 组织关于标识符的[一些建议](http://unicode.org/reports/tr31/)。无论发生什么变化，都必须在保持（或可能扩展）大小写决定标识符可见性的方式的前提下兼容地进行，这仍然是我们最喜欢 Go 的特性之一。

目前，我们有一条简单的规则，未来可以在不破坏现有程序的前提下进行扩展，这条规则避免了那些因包含模糊标识符而必然产生的错误。

### 为什么 Go 没有特性 X？ {#Why_doesnt_Go_have_feature_X}

每种语言都包含新颖的特性，同时也省略了某些人钟爱的特性。Go 的设计着眼于编程的愉悦性、编译速度、概念的正交性以及支持并发和垃圾回收等特性的需求。你钟爱的特性可能被省略，因为它不适用，或者因为它影响了编译速度或设计的清晰度，或者因为它会使基础系统模型变得过于复杂。

如果 Go 缺少特性 *X* 让你困扰，请原谅我们，并探究一下 Go 确实拥有的特性。你可能会发现它们以有趣的方式弥补了缺少 *X* 的不足。

### Go 何时引入了泛型？ {#generics}

Go 1.18 版本为该语言添加了类型参数。这使得多态或泛型编程成为可能。详情请参阅[语言规范](/ref/spec)和[提案](/design/43651-type-parameters)。

### 为什么 Go 最初发布时没有泛型？ {#beginning_generics}

Go 的初衷是作为一种易于长期维护的服务器程序编写语言。（更多背景信息请参阅[这篇文章](/talks/2012/splash.article)。）设计重点放在了可扩展性、可读性和并发性等方面。当时，多态编程似乎并非该语言目标的核心，因此为了简洁起见，最初未将其纳入。

泛型很方便，但它们在类型系统和运行时复杂度方面带来了代价。我们花了相当长的时间来设计一个我们认为其价值与复杂度相称的方案。

### 为什么 Go 没有异常？ {#exceptions}

我们认为，像 `try-catch-finally` 习惯用法那样将异常耦合到控制结构中，会导致代码混乱。它也倾向于鼓励程序员将过多的普通错误（例如无法打开文件）标记为异常。

Go 采用了不同的方法。对于简单的错误处理，Go 的多值返回使得报告错误变得容易，而不会使返回值过载。[一个规范的错误类型，结合 Go 的其他特性](/doc/articles/error_handling.html)，使得错误处理变得愉悦，但与其他语言中的方式截然不同。Go 语言也提供了一些内置函数，用于发出真正异常状况的信号并从中恢复。恢复机制仅在函数因错误而被终止的状态下执行，这足以处理灾难性情况，且无需额外的控制结构。若使用得当，能够编写出简洁清晰的错误处理代码。

详情请参阅《[Defer, Panic, and Recover](/doc/articles/defer_panic_recover.html)》一文。此外，博文《[Errors are values](/blog/errors-are-values)》展示了一种在 Go 中清晰处理错误的方法：由于错误本身就是值，因此可以运用 Go 的全部能力来处理错误。

### 为什么 Go 没有断言？ {#assertions}

Go 不提供断言功能。断言确实方便，但我们的经验表明，程序员会依赖断言来回避对恰当错误处理和报告的思考。恰当的错误处理意味着服务器在遇到非致命错误后能继续运行而非崩溃。恰当的错误报告意味着错误信息直接明了，省去程序员解读冗长崩溃堆栈的麻烦。当查看错误的程序员不熟悉代码时，精确的错误信息尤为重要。

我们理解这是一个存在争议的点。Go 语言和标准库中存在许多与现代实践不同的设计，这纯粹是因为我们有时认为值得尝试不同的方法。

### 为什么基于 CSP 理念构建并发？ {#csp}

并发和多线程编程历来以复杂著称。我们认为这部分归因于像 [pthreads](https://en.wikipedia.org/wiki/POSIX_Threads) 这样的复杂设计，部分归因于对互斥锁、条件变量、内存屏障等底层细节的过度强调。更高层次的接口能够实现更简洁的代码，即使底层仍存在互斥锁等机制。

为并发提供高级语言支持的最成功模型之一来自霍尔的通信顺序进程（CSP）。Occam 和 Erlang 是两种源自 CSP 的知名语言。Go 的并发原语源自该家族树的另一分支，其主要贡献是将通道作为一等公民对象这一强大概念。早期多种语言的实践表明，CSP 模型能很好地融入过程式语言框架。

### 为什么用 goroutine 而非线程？ {#goroutines}

Goroutine 是使并发易于使用的关键部分。其核心思想（已存在多年）是将独立执行的函数（即协程）多路复用到一组线程上。当某个协程阻塞时（例如调用阻塞系统调用），运行时会自动将同一操作系统线程上的其他协程转移到不同的可运行线程，避免它们被阻塞。程序员无需感知这些细节——这正是设计的意义所在。由此产生的机制（我们称为 goroutine）开销极低：除栈内存（仅几KB）外几乎无额外成本。

为保持栈空间小巧，Go 运行时采用可调整的有界栈。新创建的 goroutine 通常分配几KB内存，这几乎总是够用的。当栈空间不足时，运行时会自动增长（或收缩）栈内存，使得大量 goroutine 能在有限内存中生存。每个函数调用的平均 CPU 开销约为三条廉价指令。在同一地址空间中创建数十万个 goroutine 是切实可行的。如果 goroutine 仅是普通线程，系统资源将在远小于此的数量时耗尽。

### 为什么 map 操作未定义为原子操作？ {#atomic_maps}

经过长期讨论，我们决定：map 的典型使用场景无需从多个 goroutine 安全访问；而在确实需要安全访问的场景中，map 往往已是某个更大同步数据结构或计算的一部分。因此，要求所有 map 操作都获取互斥锁会拖慢大多数程序，却仅能为少数情况增加安全性。然而这并非易事，因为不受控的 map 访问可能导致程序崩溃。

语言规范并不排除原子 map 更新。在必要时（如托管不可信程序），实现层面可以对 map 访问进行加锁同步。

仅当发生更新时，map 访问才不安全。只要所有 goroutine 只进行读取操作（包括使用 `for` `range` 遍历查找元素），而不通过赋值元素或删除操作修改 map，则无需同步即可安全地并发访问 map。

为辅助正确使用 map，某些语言实现包含特殊检查机制：当 map 被并发执行不安全地修改时，会在运行时自动报告。此外，sync 库中的 [`sync.Map`](https://pkg.go.dev/sync#Map) 类型适用于特定使用模式（如静态缓存），尽管它不能作为内置 map 类型的通用替代方案。

### 你们会接受我的语言修改建议吗？ {#language_changes}

人们常提出语言改进建议——[邮件列表](https://groups.google.com/group/golang-nuts)中保存着丰富的相关讨论历史——但其中极少被采纳。尽管Go是一个开源项目，但其语言和库受到[兼容性承诺](/doc/go1compat.html)的保护，该承诺防止了至少在源代码层面会破坏现有程序的变更（程序可能偶尔需要重新编译以保持最新状态）。如果您的提案违反了Go 1规范，无论其价值如何，我们都无法考虑。

Go的未来主要版本可能与Go 1不兼容，但关于该主题的讨论才刚刚开始，且有一点是确定的：在此过程中引入的不兼容性将非常少。此外，兼容性承诺鼓励我们为旧程序提供自动化的适应路径，以应对此种情况。

即使您的提案与Go 1规范兼容，也可能不符合Go的设计目标精神。文章*[Go at Google: Language Design in the Service of Software Engineering](/talks/2012/splash.article)*解释了Go的起源及其设计背后的动机。

## 类型 {#types}

### Go是面向对象的语言吗？ {#Is_Go_an_object-oriented_language}

既是也不是。尽管Go拥有类型和方法，并允许面向对象风格的编程，但它没有类型层次结构。Go中“接口”的概念提供了一种不同的方法，我们认为它更易于使用，并且在某些方面更具通用性。还有将类型嵌入其他类型的方法，以提供类似于（但不完全相同）子类化的功能。此外，Go中的方法比C++或Java中的更通用：它们可以为任何类型的数据定义，甚至包括内置类型，如普通的、“未装箱”的整数。它们不限于结构体（类）。

此外，缺乏类型层次结构使得Go中的“对象”比C++或Java等语言中的对象感觉更轻量。

### 如何实现方法的动态派发？ {#How_do_I_get_dynamic_dispatch_of_methods}

唯一能实现方法动态派发的方式是通过接口。结构体或任何其他具体类型上的方法总是静态解析的。

### 为什么没有类型继承？ {#inheritance}

面向对象编程，至少在大多数知名语言中，涉及太多关于类型之间关系的讨论，而这些关系通常可以自动推导。Go采取了一种不同的方法。

Go不要求程序员预先声明两个类型是相关的，而是让一个类型自动满足任何指定了其方法子集的接口。除了减少记录工作外，这种方法还具有实际优势。类型可以同时满足多个接口，而没有传统多继承的复杂性。接口可以非常轻量——一个包含一个甚至零个方法的接口就能表达一个有用的概念。接口可以在事后添加，如果出现新想法或用于测试——而无需注释原始类型。由于类型和接口之间没有显式的关系，因此没有需要管理或讨论的类型层次结构。

可以使用这些思想来构建类似于类型安全Unix管道的东西。例如，请参阅`fmt.Fprintf`如何实现格式化打印到任何输出（而不仅仅是文件），或`bufio`包如何能与文件I/O完全分离，或`image`包如何生成压缩的图像文件。所有这些思想都源于一个单一的接口（`io.Writer`），它代表一个单一的方法（`Write`）。而这只是冰山一角。Go的接口对程序的结构方式有着深远的影响。

这需要一些时间来适应，但这种隐式的类型依赖风格是Go最具生产力的特点之一。

### 为什么 `len` 是函数而不是方法？ {#methods_on_basics}

我们曾就此问题进行讨论，但最终决定将 `len` 及其类似功能实现为函数在实践中是可行的，并且不会使基本类型在接口（Go类型意义上的）方面的问题复杂化。

### 为什么Go不支持方法和运算符重载？ {#overloading}

如果方法派发不需要进行类型匹配，那么它可以被简化。其他语言的经验告诉我们，拥有多种同名但签名不同的方法有时是有用的，但在实践中也可能令人困惑且脆弱。仅通过名称匹配并要求类型一致性是Go类型系统中的一个重大简化决策。

关于运算符重载，它似乎更多是一种便利而非绝对需求。同样，没有它事情会更简单。

### 为什么Go没有“implements”声明？ {#implements_interface}

Go类型通过实现该接口的方法来实现一个接口，仅此而已。这一特性允许接口的定义和使用无需修改现有代码。它实现了一种[结构化类型](https://en.wikipedia.org/wiki/Structural_type_system)，促进了关注点分离，提高了代码重用，并使得在代码开发过程中出现的模式上进行构建变得更容易。接口的语义是Go语言感觉灵活轻量的主要原因之一。

更多详情请参见[关于类型继承的问题](#inheritance)。

### 如何保证我的类型满足一个接口？ {#guarantee_satisfies_interface}

你可以要求编译器检查类型 `T` 是否实现了接口 `I`，方法是尝试使用 `T` 或 `T` 的指针（视情况而定）的零值进行赋值：```
type T struct{}
var _ I = T{}       // 验证 T 实现了接口 I。
var _ I = (*T)(nil) // 验证 *T 实现了接口 I。
```如果 `T`（或相应地 `*T`）未实现接口 `I`，则错误将在编译时被捕捉到。

若希望接口的使用者显式声明其已实现该接口，可在接口的方法集中添加一个具有描述性名称的方法。例如：```
type Fooer interface {
    Foo()
    ImplementsFooer()
}
```为了成为 `Fooer` 接口，一个类型必须实现 `ImplementsFooer` 方法，从而在 [go doc](/cmd/go/#hdr-Show_documentation_for_package_or_symbol) 的输出中清楚地记录并声明这一事实。```
type Bar struct{}
func (b Bar) ImplementsFooer() {}
func (b Bar) Foo() {}
```大多数代码不会使用这种约束，因为它们会限制接口的通用性。但有时，这种约束对于解决相似接口之间的歧义是必要的。

### 为什么类型T不满足Equal接口？ {#t_and_equal_interface}

考虑这个表示可与另一个值进行比较的对象的简单接口：```
type Equaler interface {
    Equal(Equaler) bool
}
```以及这个类型 `T`：```go
type T int
func (t T) Equal(u T) bool { return t == u } // 不满足Equaler接口
```与一些多态类型系统中的类似情况不同，
`T` 并未实现 `Equaler` 接口。
`T.Equal` 方法的参数类型是 `T`，
而非字面上要求的类型 `Equaler`。

在 Go 中，类型系统不会自动提升 `Equal` 方法的参数；
这是程序员的责任，如实现了 `Equaler` 接口的类型 `T2` 所示：```go
type T2 int
func (t T2) Equal(u Equaler) bool { return t == u.(T2) }  // 满足Equaler接口
```不过即便如此，这依然与其他类型系统不同，因为在 Go 中**任何**满足 `Equaler` 接口的类型都可以作为 `T2.Equal` 方法的参数传递，而我们需要在运行时检查参数是否为 `T2` 类型。某些语言会尝试在编译时确保这种类型安全。

另一个相关的示例则呈现相反的情况：```
type Opener interface {
   Open() Reader
}

func (t T3) Open() *os.File
```在Go中，`T3`并不满足`Opener`接口，尽管在其他语言中可能是满足的。

虽然Go的类型系统在此类情况下确实为程序员做得更少，但缺乏子类型化使得关于接口满足的规则非常容易表述：函数的名称和签名是否与接口完全一致？Go的规则也易于高效实现。我们认为这些优势弥补了缺乏自动类型提升的不足。

### 能否将 `[]T` 转换为 `[]interface{}`？ {#convert_slice_of_interface}

不能直接转换。  
语言规范禁止这种操作，因为这两种类型在内存中的表示不同。必须将元素逐个复制到目标切片。以下示例将 `int` 类型的切片转换为 `interface{}` 类型的切片：```
t := []int{1, 2, 3, 4}
s := make([]interface{}, len(t))
for i, v := range t {
    s[i] = v
}
```### 如果T1和T2的底层类型相同，我能将[]T1转换为[]T2吗？ {#convert_slice_with_same_underlying_type}

此代码示例的最后一行无法通过编译。```go
type T1 int
type T2 int
var t1 T1
var x = T2(t1) // 可行
var st1 []T1
var sx = ([]T2)(st1) // 不可行
```在Go语言中，类型与方法紧密相关，每个命名类型都拥有一个（可能为空的）方法集。

通用规则是：您可以更改被转换类型的名称（从而可能改变其方法集），但无法更改复合类型中元素的名称（及其方法集）。Go语言要求您在类型转换时保持显式声明。

### 为什么我的nil error值不等于nil？ {#nil_error}

在底层实现中，interface（接口）由两个元素构成：类型 `T` 和值 `V`。`V` 是具体值（如 `int`、`struct` 或指针），其本身绝不会是接口，且具有类型 `T`。例如，当我们将 `int` 值3存储到接口中时，形成的接口值结构可示意为：(`T=int`, `V=3`)。值 `V` 也被称为接口的*动态*值，因为在程序执行过程中，同一个接口变量可能持有不同的 `V` 值（及对应的类型 `T`）。

接口值仅在 `V` 和 `T` 均未设置时才为 `nil`（即 `T=nil` 且 `V` 未设置）。特别要强调的是，`nil` 接口始终会持有 `nil` 类型。若我们将类型为 `*int` 的 `nil` 指针存入接口值，无论指针值如何，其内部类型都将保持为 `*int`：(`T=*int`, `V=nil`)。因此，这样的接口值*即使内部指针值 `V` 为 `nil`*，该接口本身也**不是** `nil`。

这种情况容易引起混淆，尤其在 `nil` 值被存储于接口值（例如 `error` 返回值）时：```go
func returnsError() error {
    var p *MyError = nil
    if bad() {
        p = ErrBad
    }
    return p // 总是会返回一个非空错误。
}
```如果一切顺利，该函数返回一个 `nil` 值的 `p`，因此返回值是一个 `error` interface（接口）值，其内部持有（`T=*MyError`，`V=nil`）。这意味着，如果调用方将返回的 error 与 `nil` 进行比较，即使实际上没有发生任何错误，它也**总是看起来像是出现了错误**。

为了向调用方返回一个正确的 `nil` `error`，该函数必须返回一个显式的 `nil`：

---
**改写说明**：
- **准确翻译技术细节和关键对比**：确保函数返回的错误接口值内部状态及调用方比较结果的表述准确。
- **遵循术语规范和常见表达**：采用中文技术社区普遍接受的 error、nil、interface 等术语译法，保持行文专业流畅。
- **理顺逻辑衔接与语气**：合理组织语句，突出函数返回错误的常见误区和正确做法。

如果您需要更简洁或更详细的句式，我可以继续为您优化调整。```
func returnsError() error {
    if bad() {
        return ErrBad
    }
    return nil
}
```对返回错误的函数而言，最好在签名中始终使用 `error` 接口类型（如我们上面所做的），而不是诸如 `*MyError` 这样的具体类型，以帮助保证错误被正确创建。例如，[`os.Open`](/pkg/os/#Open) 返回的是 `error` 类型，即使当它不为 `nil` 时，实际上也总是具体类型 [`*os.PathError`](/pkg/os/#PathError)。

这里描述的类似情况，在使用接口时都可能发生。
请记住，如果接口中存储了任何具体值，该接口就不会是 `nil`。
更多信息请参见 [反射定律](/doc/articles/laws_of_reflection.html)。

### 为什么零大小类型的行为很特殊？ {#zero_size_types}

Go 支持零大小类型，例如没有字段的结构体（`struct{}`）或没有元素的数组（`[0]byte`）。
零大小类型中无法存储任何内容，但当不需要值时，这些类型有时很有用，例如在 `map[int]struct{}` 中，或用于定义拥有方法但没有值的类型。

具有零大小类型的不同变量可能被放置在内存中的相同位置。
这是安全的，因为这些变量中无法存储任何值。

此外，语言并不保证指向两个不同零大小变量的指针是否会被比较为相等。
这种比较甚至可能在程序的一个时刻返回 `true`，而在另一个时刻返回 `false`，具体取决于程序的确切编译和执行方式。

零大小类型的另一个问题是，指向零大小结构体字段的指针绝不能与内存中指向另一个对象的指针重叠。
这可能会导致垃圾回收器混淆。
这意味着，如果结构体的最后一个字段是零大小的，该结构体会被填充，以确保指向最后一个字段的指针不会与紧跟该结构体之后的内存重叠。
因此，以下程序：

```go
type S struct {
    A struct{}
    B int
}
```

将在内存中为 `S` 分配空间，其布局可能使得 `S` 的大小大于 `B` 的大小加上一个零大小字段的大小（即大于 `B` 的大小）。```
func main() {
	type S struct {
		f1 byte
		f2 struct{}
	}
	fmt.Println(unsafe.Sizeof(S{}))
}
```将会输出 `2` 而非 `1`，在大多数 Go 语言实现中都是如此。

### 为什么 Go 没有像 C 那样的无标签联合体？ {#unions}

无标签联合体会违反 Go 的内存安全保证。

### 为什么 Go 没有变体类型？ {#variant_types}

变体类型（也称为代数类型）提供了一种方法，可以指定一个值只能是某组特定类型之一，不能是其他类型。在系统编程中，一个常见例子是：指定一个错误可能是网络错误、安全错误或应用程序错误，并允许调用者通过检查错误的类型来辨别问题的根源。另一个例子是语法树，其中每个节点可以是不同类型：声明、语句、赋值等。

我们曾考虑向 Go 添加变体类型，但经过讨论后决定不加入，因为它们与接口在概念上存在令人困惑的重叠。如果变体类型的元素本身就是接口，会发生什么？

此外，变体类型所解决的部分问题，Go 语言已经以其他方式覆盖了。错误处理的例子可以通过使用接口值来存储错误，并使用类型开关来区分情况，很容易表达。语法树的例子虽然不够优雅，但也可以实现。

### 为什么 Go 没有协变结果类型？ {#covariant_types}

协变结果类型将意味着像下面这样的接口```
type Copyable interface {
    Copy() interface{}
}
```此方法将满足该接口。```
func (v Value) Copy() Value
```因为`Value`类型实现了空接口。
但在Go语言中，方法类型必须精确匹配，所以`Value`并没有实现`Copyable`接口。
Go语言将类型的**行为**（即其方法）与类型的**实现**分离开来。
如果两个方法返回不同的类型，那么它们所做的就不是同一件事。
那些想要协变结果类型的程序员，往往是想通过接口来表达类型层级关系。
在Go语言中，更自然的方式是让接口与实现保持清晰的分离。

## 值 {#Values}

### 为什么Go不提供隐式数值转换？ {#conversions}

C语言中数值类型之间自动转换的便利性，远不及其所带来的困惑。一个表达式何时是无符号的？值有多大？会溢出吗？结果是否可移植，不依赖于执行它的机器？
这也使编译器复杂化；C语言的“常规算术转换”不易实现，并且在不同架构间存在不一致性。
出于可移植性的考虑，我们决定以代码中增加一些显式转换为代价，来换取清晰和直接。
不过，Go语言中常量的定义——具有任意精度、无符号和大小标注的值——极大地缓解了这一问题。

一个相关的细节是，与C语言不同，即使`int`是一个64位类型，`int`和`int64`也是不同的类型。`int`类型是通用的；如果你关心一个整数占据多少位，Go鼓励你明确指定。

### Go语言中的常量是如何工作的？ {#constants}

尽管Go对不同数值类型变量之间的转换要求严格，但语言中的常量要灵活得多。
字面常量如`23`、`3.14159`和[`math.Pi`](/pkg/math/#pkg-constants)占据一种理想的数域，具有任意精度，不会溢出或下溢。
例如，`math.Pi`的值在源代码中被指定到小数点后63位，涉及该值的常量表达式保持的精度超过了`float64`所能容纳的范围。
只有当常量或常量表达式被赋给一个变量——程序中的一个内存位置时——它才会成为一个具有通常浮点属性和精度的“计算机”数字。

此外，由于它们仅仅是数字，而非具类型的值，Go中的常量比变量使用起来更自由，从而缓和了严格转换规则带来的一些不便。
你可以写出这样的表达式，比如`math.Sqrt(2)`，而编译器不会报错，因为理想数`2`可以安全准确地转换为`float64`以调用`math.Sqrt`函数。

一篇标题为[常量](/blog/constants)的博文更深入地探讨了这个主题。

### 为什么映射是内置的？ {#builtin_maps}

同样的道理，切```
sqrt2 := math.Sqrt(2)
```编译器不会报错，因为理想数 `2` 可以安全准确地转换为 `float64` 来调用 `math.Sqrt`。

一篇标题为[常量](/blog/constants)的博文更深入地探讨了这个话题。

### 为什么映射是内置的？ {#builtin_maps}

原因与字符串相同：它们是非常强大且重要的数据结构，提供一种优秀的实现并辅以语法支持能让编程更加愉悦。我们相信 Go 的映射实现足够强大，能满足绝大多数使用场景。如果特定应用能从自定义实现中获益，当然可以编写一个，但在语法上不会那么方便；这似乎是一个合理的折衷。

### 为什么映射不允许切片作为键？ {#map_keys}

映射查找需要相等运算符，而切片并未实现它。切片未实现相等性是因为相等性在这种类型上定义得并不完善；存在诸多考量，涉及浅层比较与深层比较、指针比较与值比较、如何处理递归类型等等。我们可能会重新审视这个问题——为切片实现相等性不会使任何现有程序失效——但在切片相等性的含义没有明确概念之前，暂时搁置此问题更为简单。

结构体和数组定义了相等性，因此它们可以作为映射键。

### 为什么映射、切片和通道是引用，而数组是值？ {#references}

这个话题有很多历史因素。早期，映射和通道在语法上就是指针，无法声明或使用非指针实例。此外，数组的工作方式也让我们绞尽脑汁。最终我们认识到，指针和值的严格分离使语言更难使用。通过让这些类型表现为对关联共享数据结构的引用，这些问题得以解决。这一改动为语言增加了一些令人遗憾的复杂性，但对可用性影响巨大：引入后 Go 变成了一种更高效、更舒适的语言。

## 指针与分配 {#Pointers}

### 函数参数何时按值传递？ {#pass_by_value}

与所有 C 家族的语言一样，Go 中的一切都是按值传递的。
也就是说，函数总是获得被传递事物的副本，就像有一个赋值语句将该值赋给参数一样。例如，将一个 `int` 值传递给函数会创建该 `int` 的副本，而传递一个指针值则会创建该指针的副本，但不会复制它所指向的数据。
（有关这如何影响方法接收者的讨论，请参阅 [后续章节](/doc/faq#methods_on_values_or_pointers)）。

Map 和 slice 的行为类似于指针：它们是包含指向底层 map 或 slice 数据的指针的描述符。复制一个 map 或 slice 值并不会复制它所指向的数据。复制一个 interface 值会复制存储在 interface 值中的东西。如果 interface 值持有一个 struct，复制 interface 值会复制该 struct。如果 interface 值持有一个指针，复制 interface 值会复制该指针，但同样不会复制它所指向的数据。

请注意，此讨论是关于操作的语义。
实际的实现可能会应用优化以避免复制，只要这些优化不改变语义。

### 何时应使用指向接口的指针？ {#pointer_to_interface}

几乎从不使用。指向接口值的指针仅出现在罕见且棘手的情况中，涉及隐藏接口值的类型以进行延迟求值。

一个常见的错误是将指向接口值的指针传递给期望接口的函数。编译器会对此错误发出抱怨，但这种情况可能仍然令人困惑，因为有时[指针对于满足接口是必需的](#different_method_sets)。
关键在于，虽然指向具体类型的指针可以满足一个接口，但有一个例外——*指向接口的指针永远不能满足接口*。

考虑变量声明，```
var w io.Writer
```打印函数 `fmt.Fprintf` 的第一个参数是一个满足 `io.Writer` 接口的值——即实现了标准 `Write` 方法的类型。因此我们可以这样写：```
fmt.Fprintf(w, "hello, world\n")
```然而，如果我们传递 `w` 的地址，程序将无法编译。```go
fmt.Fprintf(&w, "hello, world\n") // 编译时错误。
```唯一的例外是，任何值，甚至是指向接口的指针，都可以赋值给空接口类型（`interface{}`）的变量。  
即便如此，如果该值是指向接口的指针，这几乎肯定是一个错误；其结果可能会令人困惑。

### 我应该在值类型还是指针类型上定义方法？ {#methods_on_values_or_pointers}```go
func (s *MyStruct) pointerMethod() { } // 指针上的方法
func (s MyStruct)  valueMethod()   { } // 值上的方法
```对于不习惯指针的程序员来说，这两个示例之间的区别可能令人困惑，但实际情况其实非常简单。在类型上定义方法时，接收器（即上述示例中的 `s`）的行为与方法的参数完全一致。因此，将接收器定义为值类型还是指针类型，与函数参数应使用值类型还是指针类型是同一问题。

这里有几点需要考虑：

首先，也是最重要的一点：该方法是否需要修改接收器？如果是，接收器**必须**是指针类型。（切片和映射的行为类似于引用，因此它们的处理略有不同，但例如要在方法中改变切片的长度，接收器仍必须是指针。）在上述示例中，如果 `pointerMethod` 修改了 `s` 的字段，调用者将看到这些更改；但 `valueMethod` 是用调用者参数的副本调用的（这是传递值的定义），因此它所做的更改对调用者不可见。

顺便说一句，在 Java 中方法接收器一直都是指针，尽管其指针特性有些隐蔽（而且近期的发展正在将值接收器引入 Java）。Go 中的值接收器才是不同寻常的。

其次是效率方面的考虑。如果接收器很大（例如一个大型结构体），使用指针接收器可能更划算。

接下来是一致性。如果类型的某些方法必须使用指针接收器，那么其余方法也应该如此，这样无论类型如何使用，其方法集都是一致的。有关详细信息，请参阅关于[方法集](#different_method_sets)的部分。

对于基本类型、切片和小型结构体等类型，值接收器的成本非常低，因此除非方法的语义需要指针，否则值接收器既高效又清晰。

### `new` 和 `make` 有什么区别？ {#new_and_make}

简而言之：`new` 分配内存，而 `make` 用于初始化 slice、map 和 channel 类型。

更多详细信息，请参阅[《Effective Go》的相关章节](/doc/effective_go.html#allocation_new)。

### 在 64 位机器上 `int` 的大小是多少？ {#q_int_sizes}

`int` 和 `uint` 的大小取决于具体实现，但在同一平台上它们的大小相同。为了可移植性，依赖特定大小值的代码应使用显式指定大小的类型，例如 `int64`。在 32 位机器上，编译器默认使用 32 位整数；而在 64 位机器上，整数为 64 位。（历史上，情况并非总是如此。）

另一方面，浮点标量和复数类型总是有大小的（没有 `float` 或 `complex` 基本类型），因为程序员在使用浮点数时应注意精度问题。（无类型）浮点常量使用的默认类型是 `float64`。因此，`foo := 3.0` 会声明一个类型为 `float64` 的变量 `foo`。对于由（无类型）常量初始化的 `float32` 变量，必须在变量声明中显式指定变量类型：

```go
foo := float32(3.0)
``````
var foo float32 = 3.0
```或者，必须通过类型转换为常量指定类型，如 `foo := float32(3.0)`。

### 我如何知道一个变量是分配在堆上还是栈上？ {#stack_or_heap}

从正确性的角度来看，你不需要知道这一点。在 Go 语言中，只要变量存在引用，它就会一直存在。实现选择的存储位置与语言的语义无关。

然而，存储位置对编写高效程序确实有影响。在可能的情况下，Go 编译器会将函数局部变量分配在该函数的栈帧中。但是，如果编译器无法证明变量在函数返回后不会被引用，那么编译器就必须将该变量分配在垃圾回收的堆上，以避免悬垂指针错误。此外，如果一个局部变量非常大，将其存储在堆上可能比存储在栈上更合理。

在当前的编译器中，如果一个变量被取了地址（&），那么该变量就是堆分配的候选者。然而，一种基本的*逃逸分析*能够识别某些情况，即这些变量不会存活到函数返回之后，因此可以驻留在栈上。

### 为什么我的 Go 进程使用了这么多虚拟内存？ {#Why_does_my_Go_process_use_so_much_virtual_memory}

Go 的内存分配器保留了一大块虚拟内存作为分配的 arena（竞技场）。这块虚拟内存是特定 Go 进程本地的；保留它并不会剥夺其他进程的内存。

要查看分配给一个 Go 进程的实际内存量，请使用 Unix 的 `top` 命令，并查看 `RES`（Linux）或 `RSIZE`（macOS）列。

## 并发 {#Concurrency}

### 哪些操作是原子的？那互斥锁呢？ {#What_operations_are_atomic_What_about_mutexes}

关于 Go 语言中操作原子性的描述，可以在 [Go 内存模型](/ref/mem) 文档中找到。

低级的同步和原子操作原语在 [sync](/pkg/sync) 和 [sync/atomic](/pkg/sync/atomic) 包中可用。这些包适用于简单的任务，例如增加引用计数或保证小范围的互斥。

对于更高级别的操作，例如并发服务器之间的协调，使用更高级别的技术可以产生更优雅的程序，Go 通过其 goroutine 和 channel 支持这种方法。例如，你可以构建你的程序，使得一次只有一个 goroutine 负责特定的数据片段。这种方法被原初的 [Go 谚语](https://www.youtube.com/watch?v=PAAkCSZUG1c) 所总结：

“不要通过共享内存来通信。相反，通过通信来共享内存。”

关于这个概念的详细讨论，请参阅[通过通信共享内存](/doc/codewalk/sharemem/) 代码漫游及其[相关文章](/blog/share-memory-by-communicating)。

大型并发程序很可能会同时借鉴这两种工具包。

### 为什么我的程序没有因为 CPU 增多而运行得更快？ {#parallel_slow}

一个程序是否会因为 CPU 增多而运行得更快，取决于它所解决的问题。Go 语言提供了并发原语，如 goroutine 和 channel，但并发只有在所解决的问题本身是并行的情况下，才能实现并行。本质上顺序的问题无法通过增加 CPU 来加速，而那些可以被拆分成可以并行执行的块的问题则可以被加速，有时效果显著。

有时增加 CPU 反而会使程序变慢。实际上，花在同步或通信上的时间比执行有用计算的时间更多的程序，在使用多个操作系统线程时可能会遇到性能下降。这是因为在线程之间传递数据涉及上下文切换，其成本很高，并且这个成本可能会随着 CPU 数量的增加而增加。例如，Go 规范中的[素数筛例子](/ref/spec#An_example_package)虽然启动了许多 goroutine，但没有显著的并行性；增加线程（CPU）数量更可能使其变慢，而不是变快。

关于此主题的更多细节，请参阅题为[并发不是并行](/blog/concurrency-is-not-parallelism)的演讲。

### 如何控制 CPU 的数量？ {#number_cpus}

可以同时执行 goroutine 的 CPU 数量由 `GOMAXPROCS` shell 环境变量控制，其默认值是可用的 CPU 核心数。因此，有并行执行潜力的程序在多 CPU 机器上默认应能实现并行。要更改使用的并行 CPU 数量，可以设置该环境变量，或使用 runtime 包中同名的[函数](/pkg/runtime/#GOMAXPROCS)来配置运行时支持，以使用不同数量的线程。将其设置为 1 会消除真正并行的可能性，迫使独立的 goroutine 轮流执行。

运行时可能会分配比 `GOMAXPROCS` 值更多的线程来处理多个未完成的 I/O 请求。`GOMAXPROCS` 仅影响一次可以实际执行的 goroutine 数量；更多的 goroutine 可能阻塞在系统调用中。

Go 的 goroutine 调度器在平衡 goroutine 和线程方面做得很好，甚至可以抢占一个 goroutine 的执行，以确保同一线程上的其他 goroutine 不会被饿死。然而，它并不完美。如果你遇到性能问题，按每个应用程序设置 `GOMAXPROPS` 可能会有所帮助。

### 为什么没有 goroutine ID？ {#no_goroutine_id}

Goroutine 没有名字；它们只是匿名的工作单元。它们不会向程序员暴露任何唯一的标识符、名称或数据结构。有些人对此感到惊讶，他们期望 `go` 语句会返回一些项目，这些项目可以用来在之后访问和控制该 goroutine。Goroutine 设计为匿名的根本原因在于，这样在编写并发代码时能够使用完整的 Go 语言特性。相反，当线程和 goroutine 被命名时，其使用模式可能会限制使用它们的库所能实现的功能。

这里有一个说明其困难之处的示例。一旦命名了一个 goroutine 并围绕它构建了模型，它就会变得特殊化，人们往往会将所有计算都关联到该 goroutine，而忽视了利用多个可能共享的 goroutine 进行处理的可能性。如果 `net/http` 包将每个请求的状态与某个 goroutine 关联起来，那么客户端在处理请求时就无法使用更多的 goroutine。

此外，那些要求所有处理都在"主线程"上进行的库（例如图形系统库）的经验表明，当部署在并发语言中时，这种方法可能会显得笨拙且限制重重。特殊线程或 goroutine 的存在本身就会迫使程序员扭曲程序结构，以避免因无意中在错误的线程上操作而导致的崩溃和其他问题。

对于那些确实需要某个特定 goroutine 发挥特殊作用的情况，该语言提供了如 channel（通道）等特性，可以灵活地用于与之进行交互。

## 函数与方法 {#Functions_methods}

### 为什么 T 和 *T 拥有不同的方法集？ {#different_method_sets}

正如 [Go 语言规范](/ref/spec#Types) 所述，类型 `T` 的方法集由所有接收者类型为 `T` 的方法组成，而对应指针类型 `*T` 的方法集则由所有接收者为 `*T` 或 `T` 的方法组成。这意味着 `*T` 的方法集包含了 `T` 的方法集，反之则不然。

这种区别源于以下原因：如果一个接口值包含指针 `*T`，方法调用可以通过解引用该指针来获取一个值；但如果一个接口值包含一个值 `T`，则没有安全的方法能让方法调用获取一个指针。（如果允许这样做，方法将能够修改接口内部值的内容，这是语言规范所不允许的。）

即使在编译器可以获取值的地址并将其传递给方法的情况下，如果方法修改了该值，这些更改在调用者中也会丢失。

例如，如果下面的代码是有效的：```
var buf bytes.Buffer
io.Copy(buf, os.Stdin)
```它将会把标准输入复制到 `buf` 的**副本**中，而不是 `buf` 本身。  
这几乎从来都不是期望的行为，因此该语言禁止这样做。

### 作为 goroutine 运行的闭包会发生什么？ {#closures_and_goroutines}

由于循环变量的工作方式，在 Go 1.22 版本之前（关于此的更新见本节末尾），  
在并发环境下使用闭包时可能会出现一些混淆。  
考虑以下程序：

{{raw `
<pre>
func main() {
    done := make(chan bool)

    values := []string{"a", "b", "c"}
    for _, v := range values {
        go func() {
            fmt.Println(v)
            done &lt;- true
        }()
    }

    // 在退出前等待所有 goroutine 完成
    for _ = range values {
        &lt;-done
    }
}
</pre>
`}}

人们可能会错误地期望看到 `a, b, c` 作为输出。  
而实际可能看到的却是 `c, c, c`。  
这是因为循环的每次迭代都使用了变量 `v` 的同一个实例，  
因此每个闭包都共享了这一个变量。当闭包运行时，它打印的是  
`fmt.Println` 执行时 `v` 的值，  
但自 goroutine 启动后，`v` 可能已经被修改了。  
为了帮助在问题发生前检测到这类及其他问题，请运行  
[`go vet`](/cmd/go/#hdr-Run_go_tool_vet_on_packages)。

为了在每次启动时将 `v` 的当前值绑定到每个闭包，  
必须修改内部循环，使其在每次迭代中创建一个新变量。  
一种方法是将该变量作为参数传递给闭包：

{{raw `
<pre>
    for _, v := range values {
        go func(<b>u</b> string) {
            fmt.Println(<b>u</b>)
            done &lt;- true
        }(<b>v</b>)
    }
</pre>
`}}

在这个例子中，`v` 的值作为参数传递给了  
匿名函数。该值随后在函数内部作为变量 `u` 可访问。

更简单的方法是直接创建一个新变量，使用一种  
在 Go 中可行、但可能显得有些奇怪的声明样式：

{{raw `
<pre>
    for _, v := range values {
        <b>v := v</b> // 创建一个新的 'v'。
        go func() {
            fmt.Println(<b>v</b>)
            done &lt;- true
        }()
    }
</pre>
`}}

这种语言行为——即不为每次迭代定义新变量——  
事后看来被认为是一个错误，  
并在 [Go 1.22](/wiki/LoopvarExperiment) 中得到了解决，该版本  
确实为每次迭代创建新变量，从而消除了这个问题。

## 控制流 {#Control_flow}

### 为什么 Go 没有 `?:` 运算符？ {#Does_Go_have_a_ternary_form}

Go 中没有三元测试运算符。  
你可以使用以下方式达到相同的结果：```
if expr {
    n = trueVal
} else {
    n = falseVal
}
```Go 语言中不存在 `?:` 运算符的原因是，该语言的设计者们曾看到这一运算符被过度使用，从而导致代码变得难以理解地复杂。虽然 `if-else` 形式的代码更长，但它无疑更清晰。一种编程语言只需要一种条件控制流构造。

## 类型参数 {#Type_Parameters}

### 为什么 Go 语言有类型参数？ {#why_generics}

类型参数使得所谓的泛型编程成为可能。在泛型编程中，函数和数据结构是根据稍后使用时才指定的类型来定义的。例如，它们使得编写一个能够返回任意有序类型中两个值的最小值的函数成为可能，而无需为每种可能的类型都编写单独的版本。有关带示例的更深入解释，请参阅博客文章 [Why Generics?](/blog/why-generics)。

### Go 语言中的泛型是如何实现的？ {#generics_implementation}

编译器可以选择是为每次实例化单独编译，还是将类似的实例化编译为单一实现。单一实现的方式类似于一个带有接口参数的函数。不同的编译器会针对不同情况做出不同的选择。标准 Go 编译器通常为具有相同形状（shape）的每个类型参数生成一个实例化，其中形状由类型的属性决定，例如大小及其包含的指针的位置。未来的版本可能会在编译时间、运行时效率和代码大小之间进行权衡实验。

### Go 语言的泛型与其他语言中的泛型相比如何？ {#generics_comparison}

所有语言中的基本功能都是相似的：可以使用稍后指定的类型来编写类型和函数。尽管如此，也存在一些差异。

* **Java**

    在 Java 中，编译器在编译时检查泛型类型，但在运行时会移除这些类型。这被称为
    [类型擦除](https://en.wikipedia.org/wiki/Generics_in_Java#Problems_with_type_erasure)。
    例如，一个在编译时已知为 `List<Integer>` 的 Java 类型，在运行时将成为非泛型类型 `List`。
    这意味着，例如，在使用 Java 形式的类型反射时，无法区分 `List<Integer>` 类型的值和
    `List<Float>` 类型的值。在 Go 中，泛型类型的反射信息包含完整的编译时类型信息。

    Java 使用类型通配符，如 {{raw `<code>List&lt;? extends Number&gt;</code>`}}
    或 {{raw `<code>List<? super Number></code>`}} 来实现泛型的协变和逆变。
    Go 没有这些概念，这使得 Go 中的泛型类型简单得多。

* **C++**

    传统上，C++ 模板不对类型参数强制施加任何约束，尽管 C++20 通过
    [concepts](https://en.wikipedia.org/wiki/Concepts_(C%2B%2B)) 支持可选的约束。
    在 Go 中，约束对于所有类型参数都是强制性的。C++20 的 concepts 表达为必须能与类型参数一起编译的小代码片段。
    Go 的约束是接口类型，它们定义了所有允许的类型参数的集合。

    C++ 支持模板元编程；Go 不支持。在实践中，所有 C++ 编译器都在模板被实例化的地方编译它；如上所述，Go 可以并且确实对不同的实例化采用不同的方法。

* **Rust**

    Rust 中约束的版本称为 trait bounds（特征约束）。在 Rust 中，trait bound 与类型之间的关联必须显式定义，要么在定义 trait bound 的 crate 中，要么在定义类型的 crate 中。在 Go 中，类型参数隐式地满足约束，就像 Go 类型隐式地实现接口类型一样。Rust 标准库为比较或加法等操作定义了标准 trait；Go 标准库没有这样做，因为这些可以通过接口类型在用户代码中表达。唯一的例外是 Go 的 `comparable` 预定义接口，它捕获了一个在类型系统中无法表达的属性。

* **Python**

    Python 不是一种静态类型语言，因此可以合理地说，所有 Python 函数默认情况下总是泛型的：它们总是可以用任何类型的值调用，任何类型错误都在运行时检测。

### 为什么 Go 语言对类型参数列表使用方括号？ {#generic_brackets}

Java 和 C++ 使用尖括号作为类型参数列表，如 Java 的 `List<Integer>` 和 C++
的 `std::vector<int>`。
然而，这个选项对于 Go 来说不可用，因为它会导致语法问题：在解析函数内部的代码时，例如 `v := F<T>`，在看到 {{raw `<code>&lt;</code>`}} 的那一刻，无法确定我们看到的是一个实例化还是一个使用 {{raw `<code>&lt;</code>`}} 运算符的表达式。没有类型信息，这非常难以解决。

例如，考虑如下语句：

{{raw `
<pre>
    a, b = w &lt; x, y &gt; (z)
</pre>
`}}

如果没有类型信息，就无法确定赋值右侧是一对表达式（{{raw `<code>w &lt; x</code>`}}
和 {{raw `<code>y &gt; z</code>`}}），还是一个泛型函数实例化和调用，该调用返回两个结果值
（{{raw `<code>(w&lt;x, y&gt;)(z)</code>`}}）。

Go 的一个关键设计决策是解析可以在没有类型信息的情况下进行，而使用尖括号作为泛型的语法似乎无法做到这一点。

Go 使用方括号并非独创或独有；还有其他语言，如 Scala，也使用方括号表示泛型代码。

### 为什么 Go 语言不支持带有类型参数的方法？ {#generic_methods}

Go 允许泛型类型拥有方法，但是，除了接收者之外，这些方法的参数不能使用参数化类型。
我们预计 Go 不会添加泛型方法。问题在于如何实现它们。
具体来说，考虑检查一个接口中的值是否实现了另一个拥有额外方法的接口。
例如，考虑这个类型：一个空结构体，带有一个泛型 `Nop` 方法，该方法返回其参数（针对任何可能的类型）。```
type Empty struct{}

func (Empty) Nop[T any](x T) T {
	return x
}
```现在假设一个 `Empty` 值被存储在任意类型中，并传递给其他代码进行检查：

``````
func TryNops(x any) {
	if x, ok := x.(interface{ Nop(string) string }); ok {
		fmt.Printf("string %s\n", x.Nop("hello"))
	}
	if x, ok := x.(interface{ Nop(int) int }); ok {
		fmt.Printf("int %d\n", x.Nop(42))
	}
	if x, ok := x.(interface{ Nop(io.Reader) io.Reader }); ok {
		data, err := io.ReadAll(x.Nop(strings.NewReader("hello world")))
		fmt.Printf("reader %q %v\n", data, err)
	}
}
```这段代码如果 `x` 是 `Empty` 类型会如何工作？
看起来 `x` 必须满足所有三个测试条件，以及任何其他类型形式的任何其他测试。

当这些方法被调用时，会执行什么代码？
对于非泛型方法，编译器会为所有方法实现生成代码，并将它们链接到最终程序中。
但对于泛型方法，可能存在无限多的方法实现，因此需要不同的策略。

有四种选择：

 1. 在链接时，列出所有可能的动态接口检查，然后查找满足这些检查但缺少已编译方法的类型，接着重新调用编译器添加这些方法。

    这将显著减慢构建速度，因为需要在链接后停止并重复部分编译过程。尤其会减慢增量构建的速度。更糟糕的是，新编译的方法代码本身可能包含新的动态接口检查，这个过程将不得不重复进行。可以构造出一些例子，使这个过程甚至永远无法完成。

 2. 实现某种 JIT（即时编译），在运行时编译所需的方法代码。

    Go 语言从纯粹的提前编译所带来的简洁性和可预测性能中受益匪浅。
    我们不愿意仅仅为了实现一个语言特性就引入 JIT 的复杂性。

 3. 为每个泛型方法安排一个较慢的后备实现，该实现使用一个函数表来处理类型参数上所有可能的语言操作，然后对动态测试使用这个后备实现。

    这种方法将使一个由非预期类型参数化的泛型方法比由编译时观察到的类型参数化的相同方法慢得多。
    这会使性能变得非常不可预测。

 4. 定义泛型方法根本不能用于满足接口。

    接口是 Go 编程的重要组成部分。
    从设计角度来看，不允许泛型方法满足接口是不可接受的。

这些选择都不是好办法，因此我们选择了“以上都不是”。

代替带类型参数的方法，使用带类型参数的顶级函数，或者将类型参数添加到接收者类型中。

更多详细信息和示例，请参见[提案](/design/43651-type-parameters#no-parameterized-methods)。

### 为什么我不能为参数化类型的接收者使用更具体的类型？ {#types_in_method_declaration}

泛型类型的方法声明使用包含类型参数名称的接收者来编写。
或许是因为其语法与在调用点指定类型的语法相似，
有些人认为这提供了一种机制，可以通过在接收者中指定一个具体类型（例如 `string`）来为某些类型参数生成定制的方法：```
type S[T any] struct { f T }

func (s S[string]) Add(t string) string {
    return s.f + t
}
```这段代码会编译失败，因为编译器将 `string` 视为方法中类型参数的名称。编译器错误信息通常类似于 "`operator + not defined on s.f (variable of type string)`"。这可能会令人困惑，因为 `+` 运算符在预定义的 `string` 类型上完全可以正常使用，但该声明在此方法中覆盖了 `string` 的定义，导致运算符无法作用于这个无关的 `string` 版本。虽然覆盖像 `string` 这样的预定义名称是合法的，但这种做法较为异常，通常属于编程错误。

### 为什么编译器无法推断程序中的类型参数？ {#type_inference}

在许多情况下，程序员可以很容易看出泛型类型或函数的类型参数必须是什么，但语言不允许编译器进行推断。类型推断被有意限制，以确保永远不会对推断出的类型产生混淆。其他语言的经验表明，意外的类型推断可能导致在阅读和调试程序时产生相当大的困惑。始终可以通过调用时显式指定类型参数来解决。未来可能会支持新的推断形式，只要规则保持简单清晰。

## 包与测试 {#Packages_Testing}

### 如何创建多文件包？ {#How_do_I_create_a_multifile_package}

将包的所有源文件放在一个独立的目录中。源文件可以自由引用其他文件中的项，无需前置声明或头文件。

除了被拆分为多个文件外，该包的编译和测试方式与单文件包完全相同。

### 如何编写单元测试？ {#How_do_I_write_a_unit_test}

在与包源文件相同的目录中，创建一个以 `_test.go` 结尾的新文件。在该文件中，`import "testing"` 并编写如下形式的函数```
func TestFoo(t *testing.T) {
    ...
}
```在该目录下运行 `go test` 命令。该脚本会查找 `Test` 函数，构建测试二进制文件并执行。

更多详情请参阅[《如何编写Go代码》](/doc/code.html)文档、[`testing`](/pkg/testing/)包以及[`go test`](/cmd/go/#hdr-Test_packages)子命令。

### 我常用的测试辅助函数在哪里？ {#testing_framework}

Go语言标准库中的[`testing`](/pkg/testing/)包能轻松编写单元测试，但它缺少其他语言测试框架提供的断言函数等功能。本文档[前文](#assertions)已解释过Go语言不支持断言的原因，这些论点同样适用于测试中的`assert`使用。恰当的错误处理意味着单个测试失败后其他测试仍应继续运行，这样调试者才能全面了解问题所在。测试报告`isPrime`对2、3、5、7（或2、4、8、16）返回错误结果，比报告`isPrime`对2返回错误且停止后续测试更有价值。触发测试失败的程序员可能不熟悉失败的代码，现在投入精力编写清晰的错误提示，将在测试失败时带来回报。

相关要点是，测试框架往往会发展成自带条件判断、控制流和打印机制的微型语言，但Go语言已具备这些能力，何必重复构建？我们更愿意用Go编写测试——少学一种语言，且能让测试保持简洁直观。

如果编写高质量错误提示所需代码量显得冗长繁琐，采用表驱动测试可能更合适：将输入输出定义在数据结构中并迭代测试（Go对数据结构字面量的支持很完善）。这样编写优秀测试和错误提示的工作量就能分摊到多个测试用例中。Go标准库中有很多典型示例，例如[`fmt`包的格式化测试](/src/fmt/fmt_test.go)。

### 为什么标准库没有包含*X*？ {#x_in_std}

标准库的职责是支撑运行时库、对接操作系统，并提供众多Go程序所需的关键功能（如格式化I/O和网络）。它还包含Web编程相关要素（如加密支持、HTTP/JSON/XML标准实现）。

历史上标准库是Go语言唯一的库，因此没有明确的纳入标准。但如今已有新的收录准则。

标准库新增项目极其稀少，收录门槛很高。纳入标准库的代码需承担长期维护成本（常由原作者之外的人维护），受[Go 1兼容性承诺](/doc/go1compat.html)约束（阻塞API缺陷修复），且受Go[发布周期](/s/releasesched)限制，使得错误修复无法快速触达用户。

大多数新代码应存放于标准库之外，通过[`go`工具](/cmd/go/)的`go get`命令获取。这类代码可拥有独立的维护者、发布周期和兼容性保障。用户可在[pkg.go.dev](https://pkg.go.dev/)查找包并阅读文档。

尽管标准库中存在某些不属其职责范畴的内容（如`log/syslog`），但基于Go 1兼容性承诺，我们会继续维护库中所有内容。但我们建议将多数新代码存放于其他位置。

## 实现原理 {#Implementation}

### 构建编译器使用了什么技术？ {#What_compiler_technology_is_used_to_build_the_compilers}

Go语言存在多个生产级编译器，另有针对不同平台的编译器正在开发中。

默认编译器`gc`随Go发行版分发，作为`go`命令的支持组件。`Gc`最初用C语言编写，因为自举存在困难——需要先有Go编译器才能搭建Go环境。但随着技术进步，自Go 1.5版本起该编译器已成为Go程序。该编译器通过自动翻译工具从C转换为Go，相关设计详见[设计文档](/s/go13compiler)和[演讲](/talks/2015/gogo.slide#1)。这意味着编译器现已实现"自举"，需要解决自举问题。解决方案是预先存在可用的Go安装环境，如同C语言环境搭建流程。从源码构建全新Go环境的详细过程参见[此处](/s/go15bootstrap)和[此处](/doc/install/source)。

`Gc`采用Go语言编写，使用递归下降解析器和基于Plan 9加载器开发的自定义加载器（同样用Go实现）来生成ELF/Mach-O/PE二进制文件。

`Gccgo`编译器是C++编写的前端，采用递归下降解析器对接标准GCC后端。实验性的[LLVM后端](https://go.googlesource.com/gollvm/)使用相同的前端。

项目初期曾考虑为`gc`采用LLVM，但因其过于庞大缓慢且无法满足性能目标而作罢。回溯来看更重要的是，若从LLVM起步，将更难引入Go所需的ABI及栈管理等改造（这些不属于标准C环境范畴）。

实践证明Go语言很适合实现Go编译器，尽管这并非其初始设计目标。早期未实现自举使Go设计能专注于原始用例（网络服务器）。若早期就决定Go应自我编译，可能会演变成更偏向编译器构建的语言——这固然有价值，但非我们最初的目标。尽管 `gc` 拥有独立的实现，但标准库中也提供了原生的 [`go/parser`](/pkg/go/parser/) 包（包含词法分析器和解析器）以及[类型检查器](/pkg/go/types)。`gc` 编译器使用的正是这些库的变体。

### 运行时支持是如何实现的？ {#How_is_the_run_time_support_implemented}

同样由于引导问题，运行时代码最初主要采用 C 语言编写（配合少量汇编代码），但后来已全部转换为 Go 语言（仅保留必要的汇编部分）。  
`Gccgo` 的运行时支持依赖于 `glibc`。`gccgo` 编译器通过一种称为“分段栈”的技术实现协程，这一技术得到了 gold 链接器近期修改的支持。`Gollvm` 同样基于相应的 LLVM 基础架构构建。

### 为什么我的简单程序编译后文件这么大？ {#Why_is_my_trivial_program_such_a_large_binary}

`gc` 工具链中的链接器默认生成静态链接的二进制文件。因此所有 Go 二进制文件都会包含 Go 运行时系统，以及支持动态类型检查、反射甚至 panic 时栈追踪所必需的运行时类型信息。

在 Linux 上使用 gcc 静态编译链接的简单 C 语言 "hello, world" 程序大小约为 750 kB（包含 `printf` 的实现）。而功能等效的 Go 程序使用 `fmt.Printf` 后体积会增加到数兆字节，但其中包含了更强大的运行时支持、类型信息和调试信息。

使用 `gc` 编译的 Go 程序可以通过 `-ldflags=-w` 标志禁用 DWARF 调试信息生成，这将移除二进制文件中的调试信息且不影响其他功能，能显著减小文件体积。

### 能否让编译器不再抱怨未使用的变量/导入？ {#unused_variables_and_imports}

未使用的变量可能意味着存在程序缺陷，而未使用的导入只会拖慢编译速度——随着代码量和开发人员的增加，这种影响会日益显著。基于这些原因，Go 会拒绝编译包含未使用变量或导入的程序，以此牺牲短期便利性来换取长期的构建速度和代码清晰度。

然而在开发过程中，暂时出现这类情况十分常见，若必须在编译前手动清理这些代码，确实会令人困扰。

有人曾提议添加编译器选项来关闭这些检查（或至少降级为警告），但该选项始终未被加入。原因在于编译器选项不应影响语言语义，且 Go 编译器的设计原则是不产生警告信息，只报告阻碍编译的错误。

不采用警告机制有两个原因：其一，若某个问题值得指出，就应当在代码中修正（反之若无需修正，则更无需提醒）；其二，若允许编译器输出警告，容易导致实现层面针对模糊场景频繁告警，这些噪音可能掩盖那些**真正需要修复**的错误。

其实处理这类情况很简单：在开发阶段使用空白标识符即可让未使用的变量或导入暂时保留。```go
import "unused"

// 此声明通过引用包中的一个项来标记该导入为已使用。
var _ = unused.Item  // TODO: Delete before committing!

func main() {
    debugData := debug.Profile()
    _ = debugData // 仅在调试期间使用。
    ....
}
```如今，大多数Go程序员使用一个工具
[goimports](https://godoc.org/golang.org/x/tools/cmd/goimports)，
它可以自动重写Go源文件以确保导入正确，
从而实际消除未使用导入的问题。
该程序可以轻松连接到大多数编辑器和IDE，以便在编写Go源文件时自动运行。
该功能也内置于 `gopls` 中，如
[上文所述](/doc/faq#ide)。

### 为什么我的病毒扫描软件认为我的Go发行版或编译后的二进制文件被感染了？ {#virus}

这种情况很常见，尤其是在Windows机器上，而且几乎总是误报。
商业病毒扫描程序经常被Go二进制文件的结构搞糊涂，因为它们不像其他语言编译的二进制文件那样常见。

如果你刚安装了Go发行版，系统报告它被感染了，那肯定是个错误。
要非常彻底的话，你可以通过将校验和与
[下载页面](/dl/)上的进行比较来验证下载内容。

无论如何，如果你认为报告有误，请向你的病毒扫描软件供应商报告错误。
也许病毒扫描程序最终能学会理解Go程序。

## 性能 {#Performance}

### 为什么Go在基准测试X中表现不佳？ {#Why_does_Go_perform_badly_on_benchmark_x}

Go的设计目标之一是使可比较程序的性能接近C语言，
但在某些基准测试中它表现得相当差，包括
[golang.org/x/exp/shootout](https://go.googlesource.com/exp/+/master/shootout/) 中的几个。
最慢的那些依赖于一些库，而Go中没有性能相当的版本。
例如，[pidigits.go](https://go.googlesource.com/exp/+/master/shootout/pidigits.go)
依赖于一个多精度数学包，而C语言版本
（与Go不同）使用了[GMP](https://gmplib.org/)（它是
用优化的汇编语言编写的）。
依赖正则表达式的基准测试
（例如[regex-dna.go](https://go.googlesource.com/exp/+/master/shootout/regex-dna.go)）
本质上是在将Go原生的[regexp包](/pkg/regexp)与
像PCRE这样成熟、高度优化的正则表达式库进行比较。

基准测试的胜负取决于大量的调优，而大多数
基准测试的Go版本需要更多关注。如果你测量真正可比较的C和Go程序
（[reverse-complement.go](https://go.googlesource.com/exp/+/master/shootout/reverse-complement.go)
是一个例子），你会看到这两种语言在原始性能上比
这个套件所显示的要接近得多。

尽管如此，仍有改进空间。编译器很好但可以
更好，许多库需要重大性能改进，而且垃圾回收器
还不够快。（即使它够快，注意不产生不必要的
垃圾也可能产生巨大影响。）

无论如何，Go通常可以非常具有竞争力。
随着语言和工具的发展，许多程序的性能
都有了显著提高。
参见关于
[分析Go程序](/blog/profiling-go-programs)的博客文章，以获取一个信息丰富的示例。
它虽然有些旧，但仍包含有用的信息。

## 与C的差异 {#change_from_c}

### 为什么语法与C如此不同？ {#different_syntax}

除了声明语法外，差异并不大，源于
两个愿望。首先，语法应该感觉轻量，没有太多
强制关键字、重复或晦涩之处。其次，该语言
被设计为易于分析，
并且可以在没有符号表的情况下进行解析。这使得构建
工具（如调试器、依赖分析器、自动
文档提取器、IDE插件等）变得更加容易。C语言及其
后代在这方面是出了名的困难。

### 为什么声明是反向的？ {#declarations_backwards}

只有当你习惯C语言时，它们才是反向的。在C语言中，概念是
变量像表示其类型的表达式一样声明，这是一个
好主意，但类型和表达式语法混合得不好，
结果可能令人困惑；考虑函数指针。Go
大多将表达式和类型语法分开，这简化了事情（使用
前缀 `*` 表示指针是证明规则的一个例外）。在C语言中，声明```
    int* a, b;
```在Go中声明 `a` 为一个指针而 `b` 不是；```
    var a, b *int
```声明两者均为指针。这种方式更清晰、更规范。此外，短变量声明形式 `:=` 表明，完整的变量声明应该与 `:=` 保持相同的顺序，因此```
    var a uint64 = 1
```具有相同效果```
    a := uint64(1)
```解析工作也因拥有独立的类型语法（而非仅复用表达式语法）而得到简化；`func` 和 `chan` 等关键字使语义更加清晰。

更多详细信息请参阅关于 [Go 的声明语法](/doc/articles/gos_declaration_syntax.html) 的文章。

### 为什么没有指针算术？ {#no_pointer_arithmetic}

为了安全。没有指针算术，就有可能创造一种语言，使得任何非法地址都无法被成功地错误访问。编译器和硬件技术已经发展到这样的程度：使用数组索引的循环可以和使用指针算术的循环一样高效。此外，缺乏指针算术还可以简化垃圾回收器的实现。

### 为什么 `++` 和 `--` 是语句而不是表达式？为什么是后缀而不是前缀？ {#inc_dec}

没有了指针算术，前缀和后缀自增运算符的便利性价值就降低了。通过将它们完全从表达式层次结构中移除，表达式语法得以简化，同时关于 `++` 和 `--` 求值顺序的复杂问题（例如 `f(i++)` 和 `p[i] = q[++i]`）也得以消除。这种简化意义重大。至于后缀与前缀，两者都能很好地工作，但后缀形式更为传统；坚持前缀形式始于STL，那是一个用于名称中讽刺地包含一个后缀自增的语言的库。

### 为什么有花括号却没有分号？为什么不能将左花括号放在下一行？  {#semicolons}

Go 使用花括号进行语句分组，这种语法对于使用过任何 C 家族语言的程序员来说都很熟悉。然而，分号是为解析器准备的，而非人类，我们希望尽可能消除它们。为了实现这个目标，Go 借鉴了 BCPL 的一个技巧：在正式语法中，分隔语句的分号是存在的，但它们会被词法分析器在任何可能是语句结尾的行末自动注入，无需向前看。这在实践中运行良好，但其效果是强制了一种花括号风格。例如，函数的左花括号不能单独出现在一行上。

有人认为词法分析器应该进行向前看，以允许花括号放在下一行。我们不同意。由于 Go 代码旨在通过 [`gofmt`](/cmd/gofmt/) 进行自动格式化，因此*必须*选择某种风格。这种风格可能与你在 C 或 Java 中使用的不同，但 Go 是一门不同的语言，`gofmt` 的风格不亚于任何其他风格。更重要的是——远为重要的是——为所有 Go 程序强制采用单一的、由程序规定的格式，其优势远大于对特定风格可能存在的任何负面看法。还要注意，Go 的风格意味着 Go 的交互式实现可以一次使用标准语法处理一行，无需特殊规则。

### 为什么要进行垃圾回收？代价会不会太高？ {#garbage_collection}

系统编程中，最大的记录管理开销来源之一是管理已分配对象的生命周期。
在诸如 C 语言中，这项工作需要手动完成，这会消耗程序员大量时间，并且常常是导致严重 bug 的根源。
即使在像 C++ 或 Rust 这样提供辅助机制的语言中，这些机制也可能对软件设计产生重大影响，通常会增加额外的编程开销。
我们认为消除这种程序员开销至关重要，而近年来垃圾回收技术的进步让我们相信，可以以足够低的开销和足够低的延迟实现它，使其成为网络化系统的可行方案。

并发编程的大部分困难根源于对象生命周期问题：
随着对象在线程间传递，要保证它们被安全地释放变得非常繁琐。
自动垃圾回收使得编写并发代码变得容易得多。
当然，在并发环境中实现垃圾回收本身就是一个挑战，但解决一次这个问题，而不是在每个程序中都解决，将使所有人受益。

最后，并发问题暂且不论，垃圾回收使接口更简单，因为接口无需规定如何管理其间的内存。

这并不是说最近在 Rust 等语言中，为解决资源管理问题带来新思想的工作是误导；我们鼓励这项工作，并很高兴看到它的发展。
但 Go 采取了更传统的方法，即通过垃圾回收——而且仅仅是垃圾回收——来解决对象生命周期问题。

当前的实现是一个标记清除回收器。
如果机器是多处理器的，回收器会运行在一个单独的 CPU 核心上，与主程序并行。
近年来对回收器的主要改进已经将暂停时间减少到亚毫秒级，即使对于大堆也是如此，这基本上消除了网络服务器中反对垃圾回收的主要理由之一。
工作仍在继续，以优化算法，进一步降低开销和延迟，并探索新方法。
2018 年 Go 团队的 Rick Hudson 所做的 [ISMM 主题演讲](/blog/ismmkeynote) 描述了迄今为止的进展，并提出了一些未来方法。

关于性能，请记住 Go 赋予程序员对内存布局和分配相当大的控制权，远超典型的垃圾回收语言。细心的程序员可以通过善用语言特性来显著降低垃圾回收开销；
请参阅关于 [Go 程序性能分析](/blog/profiling-go-programs) 的文章，其中包含一个实际示例，包括对 Go 性能分析工具的演示。
