以下说明适用于标准工具链（gc Go编译器及其工具）。 Gccgo具有原生的GDB支持。

请注意，当调试使用标准工具链构建的Go程序时， Delve 是比GDB更好的替代工具。 它比GDB更了解Go的运行时、数据结构和表达式。 Delve目前支持 amd64 架构下的Linux、OSX和Windows系统。 有关支持平台的最新列表，请参阅 Delve文档。

GDB对Go程序的理解程度有限。 其栈管理、线程机制和运行时包含了许多与GDB预期执行模型显著不同的方面， 这些差异可能会使调试器产生混淆，即使在使用gccgo编译程序时也可能导致错误结果。 因此，尽管GDB在某些场景下（例如调试Cgo代码或调试运行时本身）可能有用， 但它并非Go程序（特别是高度并发的程序）可靠的调试器。 此外，解决这些困难的问题并非Go项目的优先事项。

简而言之，以下说明仅应作为在GDB能正常工作时的使用指南， 而非成功的保证。 除了本概述外，您可能还需要查阅 GDB手册。

简介

当您在Linux、macOS、FreeBSD或NetBSD系统上使用 gc 工具链编译和链接Go程序时， 生成的二进制文件包含DWARFv4调试信息，新版本（≥7.5）的GDB调试器可以使用这些信息 来检查活动进程或核心转储。

向链接器传递 '-w' 标志可省略调试信息 （例如，go build -ldflags=-w prog.go）。

gc 编译器生成的代码包含函数调用的内联优化和变量的寄存器化。 这些优化有时会使使用 gdb 进行调试变得更加困难。 如果您发现需要禁用这些优化， 请使用 go build -gcflags=all="-N -l" 来构建程序。

如果您想使用gdb检查核心转储，可以在程序崩溃时触发转储 （在允许的系统上），方法是在环境中设置 GOTRACEBACK=crash（更多信息请参阅 运行时包文档）。

常用操作

  • 显示代码的文件名和行号、设置断点和反汇编： 
    (gdb) list
(gdb) list 行号
(gdb) list 文件名.go:行号
(gdb) break 行号
(gdb) break 文件名.go:行号
(gdb) disas
  • 显示回溯信息和展开栈帧： 
    (gdb) bt
(gdb) frame n
  • 显示栈帧上局部变量、参数和返回值的名称、类型和位置： 
    (gdb) info locals
(gdb) info args
(gdb) p 变量名
(gdb) whatis 变量名
  • 显示全局变量的名称、类型和位置： 
    (gdb) info variables 正则表达式

Go扩展

GDB最近的一个扩展机制允许其为给定的二进制文件加载扩展脚本。 工具链利用此机制扩展GDB，添加了若干命令以检查运行时代码的内部结构 （如goroutine）并美观打印内置的map、slice和channel类型。

  • 美观打印字符串、slice、map、channel或interface： 
    (gdb) p 变量
  • 用于字符串、slice和map的 $len() 和 $cap() 函数： 
    (gdb) p $len(变量)
  • 将interface转换为其动态类型的函数： 
    (gdb) p $dtype(变量)
(gdb) iface 变量

    已知问题：如果interface值的长名称与短名称不同， GDB无法自动找到其动态类型（在打印堆栈跟踪时很烦人，美观打印器会退回到打印 短类型名称和指针）。

  • 检查goroutine： 
    (gdb) info goroutines
(gdb) goroutine n 命令
(gdb) help goroutine
    例如： 
    (gdb) goroutine 12 bt
    您可以通过传递 all 代替特定的goroutine ID来检查所有goroutine。 例如： 
    (gdb) goroutine all bt

如果您想了解其工作原理或希望扩展它，请查看Go源代码分发中的 src/runtime/runtime-gdb.py。 它依赖于一些特殊的魔术类型（hash<T,U>）和变量 （runtime.m 和 runtime.g），链接器 （src/cmd/link/internal/ld/dwarf.go） 确保这些在DWARF代码中得到描述。

如果您对调试信息的格式感兴趣，请运行 objdump -W a.out 并浏览 .debug_* 各节。

已知问题

  1. 字符串美观打印功能仅对 type string 生效，不适用于从此类型派生的其他类型。
  2. 运行时库的 C 语言部分缺少类型信息。
  3. GDB 无法理解 Go 的名称限定符，会将 "fmt.Print" 视为包含 "." 的非结构化字面量（该点号需要被引号包裹）。对于形如 pkg.(*MyType).Meth 的方法名，GDB 的处理会更加困难。
  4. 自 Go 1.11 版本起，调试信息默认进行压缩。较旧版本的 GDB（例如 macOS 默认提供的版本）无法理解这种压缩格式。您可以使用 go build -ldflags=-compressdwarf=false 来生成未压缩的调试信息。（为方便起见，您可以将 -ldflags 选项写入 GOFLAGS 环境变量，这样就不必每次手动指定了。）

命名规范

变量和函数名必须使用其所属包名进行限定。regexp 包的 Compile 函数在 GDB 中表示为 'regexp.Compile'。

方法必须使用其接收者类型名限定。例如，*Regexp 类型的 String 方法表示为 'regexp.(*Regexp).String'。

在调试信息中，遮蔽其他变量的变量会自动添加数字后缀。被闭包引用的变量将以指针形式显示，并自动添加 '&' 前缀。

美化打印

GDB 的美化打印机制由类型名称上的正则匹配触发。以切片为例： 

(gdb) p utf
$22 =  []uint8 = {0 '\000', 0 '\000', 0 '\000', 0 '\000'}

由于切片、数组和字符串不是 C 指针，GDB 无法为你解释下标操作，但你可以查看其运行时内部表示来做到这一点（这里的制表符补全很有帮助）： 


(gdb) p slc
$11 =  []int = {0, 0}
(gdb) p slc-><TAB>
array  slc    len
(gdb) p slc->array
$12 = (int *) 0xf84057af00
(gdb) p slc->array[1]
$13 = 0

扩展函数 $len 和 $cap 适用于字符串、数组和切片： 

(gdb) p $len(utf)
$23 = 4
(gdb) p $cap(utf)
$24 = 4

通道（Channel）和映射（map）是“引用”类型，GDB 将它们显示为指向类似 C++ 类型的指针，如 hash*。解引用将触发美化打印。

接口在运行时中表示为指向类型描述符的指针和指向值的指针。Go 的 GDB 运行时扩展能解码此结构，并自动触发针对运行时类型的美化打印。扩展函数 $dtype 可为您解码动态类型（示例取自 regexp.go 第 293 行的断点处）。 

(gdb) p i
$4 = {str = "cbb"}
(gdb) whatis i
type = regexp.input
(gdb) p $dtype(i)
$26 = (struct regexp.inputBytes *) 0xf8400b4930
(gdb) iface i
regexp.input: struct regexp.inputBytes *
