以下说明适用于标准工具链（即gc Go编译器及工具）。 Gccgo具有原生gdb支持。

请注意，在调试使用标准工具链构建的Go程序时，Delve是比GDB更好的替代工具。它对Go运行时、数据结构和表达式的理解优于GDB。Delve目前支持Linux、OSX以及Windows平台的amd64架构。 若要获取最新支持平台列表，请参阅Delve文档。

GDB无法很好地理解Go程序。其栈管理、线程机制和运行时环境包含诸多与GDB预期执行模型不同的方面，这可能导致调试器混淆并产生错误结果——即使使用gccgo编译的程序也不例外。 因此，尽管GDB在某些场景下仍有用处（例如调试Cgo代码或运行时本身），但它并非Go程序的可靠调试器，尤其对于高并发程序。此外，解决这些难题并非Go项目的优先事项。

简而言之，下文所述操作仅应视为GDB在可用时的参考指南，而非成功调试的保证。 除本概述外，您可能还需查阅GDB手册。

简介

在Linux、macOS、FreeBSD或NetBSD系统上使用gc工具链编译链接Go程序时，生成的二进制文件包含DWARFv4调试信息，最新版本（≥7.5）的GDB调试器可利用该信息检查活动进程或核心转储。

向链接器传递'-w'标志可省略调试信息（例如go build -ldflags=-w prog.go）。

gc编译器生成的代码包含函数内联和变量寄存器化优化。这些优化有时会使gdb调试变得困难。 如需禁用这些优化，请使用go build -gcflags=all="-N -l"构建程序。

若需使用gdb检查核心转储，可在允许该操作的系统中通过设置环境变量GOTRACEBACK=crash触发程序崩溃时的转储（详见运行时包文档）。

常规操作

  • 显示代码文件名与行号、设置断点及反汇编： 
    (gdb) list
(gdb) list 行号
(gdb) list 文件名.go:行号
(gdb) break 行号
(gdb) break 文件名.go:行号
(gdb) disas
  • 显示回溯信息并展开栈帧： 
    (gdb) bt
(gdb) frame 帧编号
  • 显示栈帧中局部变量、参数和返回值的名称、类型及位置： 
    (gdb) info locals
(gdb) info args
(gdb) p 变量名
(gdb) whatis 变量名
  • 显示全局变量的名称、类型及位置： 
    (gdb) info variables 正则表达式

Go扩展功能

GDB近期推出的扩展机制允许为特定二进制文件加载扩展脚本。工具链利用此机制扩展GDB功能，添加了若干用于检查运行时代码内部结构（如goroutine）的命令，以及对内置map、slice和channel类型的美化打印功能。

  • 美化打印字符串、slice、map、channel或interface： 
    (gdb) p 变量
  • 用于字符串、slice和map的$len()及$cap()函数： 
    (gdb) p $len(变量)
  • 将interface转换为其动态类型的函数： 
    (gdb) p $dtype(变量)
(gdb) iface 变量

    已知问题：当interface值的完整名称与简短名称不同时，GDB无法自动识别其动态类型（这在打印堆栈跟踪时很麻烦，美化打印器会退而使用简短类型名和指针）。

  • 检查goroutine： 
    (gdb) info goroutines
(gdb) goroutine 编号 命令
(gdb) help goroutine
    示例： 
    (gdb) goroutine 12 bt
    可通过传递all代替特定goroutine ID来检查所有goroutine。 例如： 
    (gdb) goroutine all bt

如需了解实现原理或进行扩展，请查看Go源码分发包中的src/runtime/runtime-gdb.py文件。该文件依赖某些特殊类型（hash<T,U>）和变量（runtime.m与runtime.g），链接器（src/cmd/link/internal/ld/dwarf.go）会确保这些内容在DWARF代码中得到描述。

如需查看调试信息格式，请执行objdump -W a.out并浏览.debug_*各节。

已知问题

  1. 字符串美化打印仅对 string 类型触发，不适用于从其派生的类型。
  2. 运行时库的 C 部分缺少类型信息。
  3. GDB 无法理解 Go 语言的名称限定符，会将 "fmt.Print" 视为包含 "." 的非结构化字面量，因此需要加上引号。对于形如 pkg.(*MyType).Meth 的方法名，GDB 反应更为强烈。
  4. 从 Go 1.11 起，调试信息默认采用压缩格式。较旧版本的 GDB（例如 macOS 默认提供的版本）无法理解这种压缩。您可以通过使用 go build -ldflags=-compressdwarf=false 生成未压缩的调试信息。（为方便起见，您可以将 -ldflags 选项放入 GOFLAGS 环境变量，这样无需每次手动指定。）

命名规范

变量和函数名必须包含其所属包的名称限定。来自 regexp 包的 Compile 函数在 GDB 中表示为 'regexp.Compile'。

方法名必须包含其接收者类型的名称。例如，*Regexp 类型的 String 方法表示为 'regexp.(*Regexp).String'。

遮蔽其他变量的变量会在调试信息中自动添加数字后缀。闭包引用的变量会显示为带有 '&' 前缀的指针。

美化打印

GDB 的美化打印机制由类型名称上的正则表达式匹配触发。一个关于切片的示例： 

(gdb) p utf
$22 =  []uint8 = {0 '\000', 0 '\000', 0 '\000', 0 '\000'}

由于切片、数组和字符串不是 C 指针，GDB 无法为你解释下标操作，但你可以查看其运行时表示来做到这一点（此处制表符补全很有帮助）： 


(gdb) p slc
$11 =  []int = {0, 0}
(gdb) p slc-><TAB>
array  slc    len
(gdb) p slc->array
$12 = (int *) 0xf84057af00
(gdb) p slc->array[1]
$13 = 0

扩展函数 $len 和 $cap 适用于字符串、数组和切片： 

(gdb) p $len(utf)
$23 = 4
(gdb) p $cap(utf)
$24 = 4

通道（channel）和映射（map）是“引用”类型，GDB 将其显示为指向类似 C++ 类型 hash<int,string>* 的指针。解引用将触发美化打印。

接口在运行时表示为指向类型描述符的指针和指向值的指针。Go 的 GDB 运行时扩展会解码此信息，并自动触发对运行时类型的美化打印。扩展函数 $dtype 可为您解码动态类型（以下示例取自 regexp.go 第 293 行的断点处）。 

(gdb) p i
$4 = {str = "cbb"}
(gdb) whatis i
type = regexp.input
(gdb) p $dtype(i)
$26 = (struct regexp.inputBytes *) 0xf8400b4930
(gdb) iface i
regexp.input: struct regexp.inputBytes *
