Go 项目欢迎所有贡献者。

本文档旨在帮助您了解为 Go 项目贡献代码的流程，该流程与其他开源项目略有不同。 我们假设您对 Git 和 Go 有基本的了解。

除了此处的信息外，Go 社区还维护了一个 代码评审 维基页面。 在您学习评审流程的过程中，欢迎随时为该维基做出贡献。

请注意，gccgo 前端位于其他地方； 请参阅 为 gccgo 做贡献。

成为贡献者

概览

第一步是注册为 Go 贡献者并配置您的开发环境。 以下是需要遵循的步骤清单：

  • 第 0 步：确定您将用于为 Go 做贡献的单一 Google 帐户。 在后续所有步骤中使用该帐户，并确保 git 配置为使用该帐户的电子邮件地址创建提交。
  • 第 1 步：签署并提交一份 CLA（贡献者许可协议）。
  • 第 2 步：为 Go Git 仓库配置认证凭据。 访问 go.googlesource.com， 点击页面右上角菜单栏中的 "Generate Password"，并按照说明操作。
  • 第 3 步：通过访问此页面， 注册 Gerrit——Go 团队使用的代码评审工具。 CLA 和注册只需为您的帐户完成一次。
  • 第 4 步：通过运行以下命令安装 git-codereview： go install golang.org/x/review/git-codereview@latest

如果您愿意，也有一个自动化工具可以引导您完成这些步骤。 只需运行： 

$ go install golang.org/x/tools/cmd/go-contrib-init@latest
$ cd /code/to/edit
$ go-contrib-init

本章后续内容将详细说明这些说明。 如果您已完成上述步骤（无论是手动还是通过工具），请跳转至 贡献代码之前。

第 0 步：选择一个 Google 帐户

对 Go 的贡献是通过具有特定电子邮件地址的 Google 帐户进行的。 请确保在整个过程中以及后续所有贡献中都使用同一帐户。 您可能需要决定是使用个人地址还是公司地址。 该选择将取决于您将编写和提交的代码的版权所有者是谁。 您可能希望在决定使用哪个帐户之前与您的雇主讨论此话题。

Google 帐户可以是 Gmail 电子邮件帐户、G Suite 组织帐户或与外部电子邮件地址关联的帐户。 例如，如果您需要使用一个不由 G Suite 管理的现有公司电子邮件，您可以创建一个 与您现有电子邮件地址关联的帐户。

您还需要确保您的 Git 工具配置为使用您选择的电子邮件地址创建提交。 您可以全局配置 Git（作为所有项目的默认设置），也可以局部配置（针对单个特定项目）。 您可以使用以下命令检查当前配置： 

$ git config --global user.email  # 检查当前全局配置
$ git config user.email           # 检查当前局部配置

要更改已配置的地址： 

$ git config --global user.email name@example.com   # 更改全局配置
$ git config user.email name@example.com            # 更改局部配置

第 1 步：贡献者许可协议

在向 Go 项目发送您的第一个变更之前，您必须完成以下两个 CLA 之一。 您应该签署哪个 CLA 取决于您的作品版权归谁所有。

  • 如果您是版权所有者，您需要同意 个人贡献者许可协议， 该协议可以在线完成。
  • 如果您的组织是版权所有者，则该组织需要同意 公司贡献者许可协议。

您可以在 Google 开发者贡献者许可协议 网站上查看您当前已签署的协议并签署新协议。 如果您的贡献的版权所有者已经就另一个 Google 开源项目完成了该协议， 则无需再次完成。

如果您提交代码的版权所有者发生变化——例如，如果您开始代表一家新公司贡献代码——请发送邮件至 golang-dev 邮件列表。 这将让我们了解情况，以便确保完成相应的协议。

第 2 步：配置 git 认证

Go 的主代码仓库位于 go.googlesource.com， 这是一个由 Google 托管的 Git 服务器。 网络服务器上的认证通过您的 Google 帐户进行，但您还需要在计算机上配置 git 以便访问它。 请按照以下步骤操作：

  1. 访问 go.googlesource.com 并点击页面右上方菜单栏中的“生成密码”。 您将被重定向到 accounts.google.com 进行登录。
  2. 登录后，您将进入一个标题为“配置 Git”的页面。 该页面包含一个个性化脚本，在本地运行该脚本后，它将配置 Git 以保存您的唯一认证密钥。 此密钥与服务器上生成并存储的密钥配对，类似于 SSH 密钥的工作原理。
  3. 在您的本地终端中复制并运行此脚本，将您的秘密认证令牌存储在 .gitcookies 文件中。 如果您使用的是 Windows 电脑并且运行的是 cmd，则应改为按照黄色方框中的说明运行命令；否则运行常规脚本。

第 3 步：创建 Gerrit 账户

Gerrit 是 Go 维护人员用于讨论和审查代码提交的开源工具。

要注册您的账户，请访问 go-review.googlesource.com/login/，并使用您在上面使用的同一个 Google 账户登录一次。

第 4 步：安装 git-codereview 命令

对 Go 的更改在接受之前都必须经过审查，无论更改者是谁。 一个名为 git-codereview 的自定义 git 命令简化了向 Gerrit 发送更改的过程。

通过运行以下命令安装 git-codereview 命令： 

$ go install golang.org/x/review/git-codereview@latest

确保 git-codereview 已安装在您的 shell 路径中，以便 git 命令能够找到它。 检查以下命令 

$ git codereview help

是打印帮助文本，而不是错误信息。如果打印错误，请确保 $GOPATH/bin 在您的 $PATH 中。

在 Windows 上使用 git-bash 时，您必须确保 git-codereview.exe 在您的 git 执行路径中。 运行 git --exec-path 来发现正确的位置，然后创建一个符号链接，或者只是将可执行文件从 $GOPATH/bin 复制到此目录。

贡献代码之前

该项目欢迎代码补丁，但为确保事项协调良好，您应在开始工作前讨论任何重大更改。 建议您在问题跟踪器中标记您的贡献意向，可以通过 提交新问题 或认领 现有问题 来实现。

在哪里贡献

Go 项目由主要的 go 代码仓库组成，其中包含 Go 语言的源代码，以及许多 golang.org/x/... 代码仓库。 这些代码仓库包含支持 Go 的各种工具和基础设施。例如，golang.org/x/pkgsite 用于 pkg.go.dev， golang.org/x/playground 用于 Go 演示场，而 golang.org/x/tools 包含各种 Go 工具，包括 Go 语言服务器 gopls。您可以在 go.googlesource.com 上查看所有 golang.org/x/... 代码仓库的列表。

检查问题跟踪器

无论您已经知道要贡献什么，还是正在寻找想法，问题跟踪器 始终是首先要去的地方。 问题会被分类以进行分类和管理工作流程。

大多数 golang.org/x/... 仓库也使用主要的 Go 问题跟踪器。但是，其中一些仓库会单独管理其问题，因此请务必为您想要贡献的仓库检查正确的问题跟踪器。

大多数问题将标记有以下工作流标签之一：

  • NeedsInvestigation：问题尚未完全理解，需要进行分析以了解根本原因。
  • NeedsDecision：问题相对理解良好，但 Go 团队尚未决定处理它的最佳方式。最好在编写代码之前等待决定。如果您对处理此状态的问题感兴趣，如果一段时间后仍未做出决定，请随时在问题评论中“ping”维护人员。
  • NeedsFix：问题已完全理解，可以编写代码来修复它。

您可以使用 GitHub 的搜索功能来寻找可以帮助解决的问题。示例：

  • 需要调查的问题：is:issue is:open label:NeedsInvestigation
  • 需要修复的问题：is:issue is:open label:NeedsFix
  • 需要修复且有建议更改的问题：is:issue is:open label:NeedsFix ("golang.org/cl" OR "go.dev/cl")
  • 需要修复但无建议更改的问题：is:issue is:open label:NeedsFix NOT "golang.org/cl" NOT "go.dev/cl"

为任何新问题创建议题

除极其微小的改动外，所有贡献都应关联到现有问题。 欢迎随时创建问题并讨论您的计划。 这个流程让每个人都有机会验证设计方案， 有助于避免工作重复， 并确保想法符合该语言和工具的目标。 它还能在编写代码前检查设计是否合理； 代码审查工具不适用于高层讨论。

规划工作时，请注意Go项目对主Go仓库遵循六个月开发周期。每个周期的后半段是为期三个月的功能冻结期，期间仅接受错误修复和文档更新。在功能冻结期间可提交新贡献，但它们不会在冻结期结束前被合并。冻结适用于整个主仓库以及构建发布版本所需的golang.org/x/...仓库中的代码。请参见已纳入 标准库的包列表 和go命令的包列表。

对语言、库或工具的重大更改 （包括主仓库及所有golang.org/x仓库的API变更， 以及对go命令的命令行修改） 必须通过 变更提案流程 才能被接受。

仅限敏感的安全相关问题应报告至security@golang.org。

通过Gerrit提交变更

我们鼓励使用Gerrit提交变更以供审查。虽然对于仅使用过GitHub工作流的用户来说存在一定的学习曲线，但它是Go项目贡献者使用的主要工作流，是一个更顺畅且功能更完善的工作流。

以下各节简要介绍了通过Gerrit提交变更的流程。 有关与Gerrit交互的更多详情，请参阅 Gerrit文档。 特别是，我们建议阅读 审查界面概览 和 Gerrit基础教程 — 适用于GitHub用户页面。

概览

这是整体流程的概览：

  • 步骤 1： 从 go.googlesource.com 克隆源代码，并通过一次编译和测试确保其状态稳定。

    如果您要对主Go仓库进行更改：

    $ git clone https://go.googlesource.com/go
$ cd go/src
$ ./all.bash                                # 编译并测试

    如果您要对某个 golang.org/x/... 仓库进行更改 （例如本例中的 golang.org/x/tools）： 

    $ git clone https://go.googlesource.com/tools
$ cd tools
$ go test ./...                             # 编译并测试
  • 步骤 2： 在基于 master 分支创建的新分支中准备更改。 要提交更改，请使用 git codereview change；这将在分支中创建或修改一个单一的提交。 
    $ git checkout -b mybranch
$ [编辑文件...]
$ git add [文件...]
$ git codereview change   # 在分支中创建提交
$ [再次编辑...]
$ git add [文件...]
$ git codereview change   # 用新的更改修改现有的提交
$ [等等]
  • 步骤 3： 测试您的更改，可以通过运行您编辑的包中的测试，或者重新运行 all.bash。

    在主Go仓库中：

    $ ./all.bash    # 重新编译并测试

    在 golang.org/x/... 仓库中：

    $ go test ./... # 重新编译并测试
  • 步骤 4： 使用 git codereview mail 将更改发送到 Gerrit 进行审查（尽管名称中有 mail，但它不使用电子邮件）。 
    $ git codereview mail     # 将更改发送到 Gerrit
  • 步骤 5： 审查后，将更改应用到同一个提交并再次发送到 Gerrit： 
    $ [编辑文件...]
$ git add [文件...]
$ git codereview change   # 更新同一个提交
$ git codereview mail     # 再次发送到 Gerrit

本节的剩余部分将更详细地描述这些步骤。

步骤 1：克隆源代码

除了最近版本的Go安装，您还需要从正确的仓库检出一份源代码本地副本。 您可以将Go源代码仓库克隆到本地文件系统上的任何位置，只要在您的GOPATH （默认为用户主目录下的go目录）之外。 请从 go.googlesource.com（而非GitHub）克隆：

主Go仓库：

$ git clone https://go.googlesource.com/go
$ cd go

golang.org/x/... 仓库

（本例为 golang.org/x/tools）： 
$ git clone https://go.googlesource.com/tools
$ cd tools

步骤 2：在新分支中准备更改

每次Go更改都必须在一个单独的分支中进行，该分支基于 master 分支创建。 您可以使用普通的 git 命令创建分支并将更改添加到暂存区： 

$ git checkout -b mybranch
$ [编辑文件...]
$ git add [文件...]

要提交更改，请使用 git codereview change，而不是 git commit。 

$ git codereview change
（打开 $EDITOR）

您可以像往常一样在您喜欢的编辑器中编辑提交描述。 git codereview change 命令会自动在底部附近添加一个唯一的 Change-Id 行。 该行被 Gerrit 用来匹配同一更改的连续上传。 请勿编辑或删除它。 Change-Id 看起来像这样： 

Change-Id: I2fbdbffb3aab626c4b6f56348861b7909e3e8990

该工具还会检查您是否在源代码上运行了 go fmt，以及提交信息是否遵循了建议的格式。

如果您需要再次编辑文件，可以暂存新的更改并重新运行 git codereview change： 后续每次运行都会修改现有的提交，同时保留 Change-Id。

请确保在每个分支中始终只保留一个提交。 如果您错误地添加了多个提交，可以使用 git rebase 来将它们合并为一个提交。

步骤 3：测试您的更改

您已经编写并测试了代码，但在提交代码供审查之前，请运行整个代码树的所有测试，以确保您的更改不会破坏其他包或程序。

在主 Go 仓库中

对于标准库包，包内的所有测试必须通过： 

$ go test

可以使用 all.bash 运行整个代码树的简短测试套件（在 Windows 下构建请使用 all.bat）： 

$ cd go/src
$ ./all.bash

在运行一段时间并打印出大量测试输出后，该命令应以如下内容结束： 

ALL TESTS PASSED

您可以使用 make.bash 代替 all.bash，只构建编译器和标准库而不运行测试套件。 一旦 go 工具构建完成，它将被安装在您克隆 Go 仓库目录下的 bin/go，您可以直接从那里运行它。 另请参阅关于如何快速测试您的更改的部分。

在 golang.org/x/... 仓库中

运行整个仓库的测试（以此示例中的 golang.org/x/tools 为例）： 

$ cd tools
$ go test ./...

如果您担心构建状态，可以查看构建状态面板。 测试失败也可能被代码审查中的 TryBots 捕获。

一些仓库，如 golang.org/x/vscode-go，将拥有不同的测试基础设施，因此请务必查看您正在工作的仓库的文档。仓库根目录下的 README 文件通常包含此信息。

步骤 4：提交更改进行审查

当更改准备就绪并在整个代码树上测试通过后，即可提交审查。 这是通过 mail 子命令完成的，尽管其名称如此，但它并不会直接发送任何邮件；它只是将更改发送到 Gerrit： 

$ git codereview mail

Gerrit 会为您的更改分配一个编号和 URL，git codereview mail 命令将打印出来，类似于： 

remote: New Changes:
remote:   https://go-review.googlesource.com/99999 math: improved Sin, Cos and Tan precision for very large arguments

如果出现错误，请查看排查邮件错误部分。

如果您的更改与一个未解决的 GitHub issue 相关，并且您遵循了建议的提交信息格式，几分钟内，一个机器人将通过评论更新该 issue，将您的 Gerrit 更改链接到它。

步骤 5：根据审查意见修改更改

Go 维护者将在 Gerrit 上审查您的代码，您将通过电子邮件收到通知。 您可以在 Gerrit 上查看审查意见并在那里进行评论。 如果您愿意，也可以使用电子邮件回复。

如果您需要在审查后修改更改，请在您之前创建的相同分支中编辑文件，将它们添加到 Git 暂存区，然后使用 git codereview change 修改提交： 

$ git codereview change     # 修改当前提交
（打开 $EDITOR）
$ git codereview mail       # 将新更改发送到 Gerrit

如果您不需要更改提交描述，只需保存并退出编辑器即可。 请记住不要触动特殊的 Change-Id 行。

再次提醒，请确保在每个分支中始终只保留一个提交。 如果您错误地添加了多个提交，可以使用 git rebase 来将它们合并为一个提交。

通过 GitHub 提交更改

尽管推荐使用 Gerrit 工作流并且其支持更好， 但已经熟悉 GitHub 工作流的贡献者 也能够使用相同的流程进行 Go 贡献。 即使 Go 维护者使用 Gerrit 进行代码审查， 也创建了一个名为 GerritBot 的工具， 用于将 GitHub 拉取请求同步到 Gerrit。

按照常规方式提交GitHub拉取请求即可。 GerritBot会创建对应的Gerrit变更列表（即"CL"），并在您的GitHub拉取请求中发布链接；对拉取请求的更新也会同步到Gerrit的CL中。 当有人在该CL上发表评论时，评论也会同步发布到您的拉取请求中，以便您收到通知。

需要注意以下几点：

  • 您需要拥有一个Gerrit账户来回复审阅者， 包括根据建议实现后将反馈标记为"已完成"。 建议您熟悉Gerrit的使用，例如浏览开放状态的CL、 订阅感兴趣的CL更新（通过星标图标）， 或审阅或给予+1评价他人的CL。
  • 若要更新拉取请求中的代码，只需推送到对应分支即可；您可以选择新增提交， 或进行变基后强制推送（两种方式均可接受）。
  • 如果请求被接受，所有提交将被压缩合并， 最终提交描述将由拉取请求的标题和描述拼接而成。 各个独立提交的描述信息将被舍弃。 具体建议请参阅编写规范的提交消息。
  • 更多详情请查看GerritBot常见问题。

规范的提交消息

Go项目的提交消息需遵循特定规范，本节将详细说明。

以下是一个规范的示例： 

math: 改进超大参数下的Sin、Cos和Tan精度

现有实现在处理大参数时数值特性较差，
现采用McGillicutty算法提升1e10以上数值的计算精度。

该算法详见 https://wikipedia.org/wiki/McGillicutty_Algorithm

Fixes #159

首行说明

变更描述的首行通常是对本次变更的简短总结，需以受影响的主要包名作为前缀。

判断标准是：该行内容应能完善"本变更修改了Go的_____"这个句子。 这意味着它不应以大写字母开头，不是完整句子，且需准确概括变更带来的结果。

首行后必须空一行。

正文内容

后续描述应详细阐述，提供变更背景并说明具体作用。 请使用完整句子和正确标点，如同编写Go代码注释一样规范。 禁止使用HTML、Markdown或其他标记语言。 文本建议控制在72列宽度左右。 更多细节请参阅提交消息规范。

可补充相关信息，例如影响性能的变更需提供基准测试数据。 规范做法是使用benchstat工具格式化变更描述中的基准数据。

关联问题追踪

使用特殊标记"Fixes #12345"可将变更与Go问题追踪器中的12345号问题关联。 当变更最终合入后，问题追踪器会自动将该问题标记为已解决。

若变更仅是解决问题过程中的阶段性步骤，请改用"Updates #12345"。 这会在问题中留下指向Gerrit变更的评论，但变更合入时不会关闭该问题。

若您向golang.org/x/...仓库提交变更，必须使用GitHub支持的完整语法格式， 以确保变更关联至主仓库而非x/仓库的问题。绝大多数问题均通过主仓库的问题追踪器管理。 正确格式应为"Fixes golang/go#159"。

代码审查流程

本节将详细说明代码审查流程，以及变更提交后的处理方式。

初学者常见错误

变更提交至Gerrit后，通常会在几天内进行初步筛选。 维护者将查看代码并进行初步审查，对首次贡献者通常侧重于基础格式和常见问题检查，包括但不限于：

  • 提交消息未遵循建议格式。
  • 缺少关联的GitHub问题。绝大多数变更需要关联具体问题， 用以说明所修复的缺陷或实现的功能，且需在问题追踪器中达成共识后方可进行。 Gerrit审查仅讨论实现方案，不评估变更价值。
    仅微小或格式性修改可豁免此要求。
  • 在开发周期的冻结阶段提交变更（此时主干暂停接受常规变更）。 此类情况下，维护者可能会标注R=go1.12进行审查， 表示将在下一个开发窗口期重新审查。 若您知晓当前非合适时间窗口，可自行添加R=go1.XX注释。

测试机器人集群

初步审阅后，维护者将触发测试机器人集群（Trybots）， 该集群将在多种不同架构上运行完整测试套件。 多数测试在几分钟内完成，结果链接将发布在Gerrit中供查看。

若测试机器人运行失败，请点击链接查看失败平台的完整日志。 尝试分析问题原因，更新您的补丁以修复问题后重新上传。 维护者将触发新一轮测试机器人运行以确认问题是否已解决。

某些情况下，主干代码可能在部分平台上出现数小时的构建故障。 若测试机器人报告的失败与您的补丁无关，请访问 构建信息面板，检查同一平台上其他近期提交是否出现相同故障。 此时，您可在Gerrit中留言说明该故障与您的变更无关，以帮助维护者了解情况。 您也可在GitHub Issues中搜索故障错误信息，或浏览 近期更新的watchflakes标签问题。 若您的变更基于较早的提交，或可能已被他人修复，请尝试通过git rebase变基至最新的主干提交。

代码审查

Go社区重视彻底严谨的代码审查。 请将每条审查意见视为待处理工单：您需要采取行动（实施建议或说服审查者）来"关闭"它。

更新变更后，请逐条回复所有审查意见。 您可点击"Done"按钮表示已采纳建议； 否则请点击"Reply"解释未采纳原因或替代方案。

变更经历多轮审查是正常现象：审查者每次提出新意见后，会等待更新后的变更再次审查。 即使经验丰富的贡献者也会经历此过程，请勿因此气馁。

投票规范

临近决策时，审查者将对变更进行代码审查投票。投票类型包括：

  • +2：变更获准合并。仅Go维护者（亦称"审批者"）可投+2票。
  • +1：变更基本合格，但审查者要求小幅修改后方可批准； 或非维护者无法批准但支持批准。

变更必须获得维护者的代码审查+2票方可提交。

维护者还可投出Hold +1票，标记当前不应提交的变更 （例如变更涉及的新API正处提案审查阶段）。

变更不得存在任何维护者的Hold +1票方可提交。

最后，变更必须有两名谷歌员工参与（作为提交者或至少投出代码审查+1票的审查者）。 此要求出于合规性与供应链安全考虑。

提交已批准的变更

当变更准备就绪时，维护者将提交变更，将其作为提交记录添加至Gerrit仓库。

批准与提交分两步进行，因为某些情况下维护者可能批准但暂不提交（例如主干临时冻结）。

提交变更将使其入库。变更说明中将包含代码审查链接， 该链接会更新仓库中的变更地址。 由于集成采用Git的"Cherry Pick"方式， 提交操作会改变仓库中的提交哈希值。

若您的变更获批数日后仍未提交，可在Gerrit中留言请求提交。

版权声明

Go仓库文件不列作者姓名，既为保持简洁也免维护更新。 您的贡献信息将体现在变更日志中。

您提交的新文件应使用标准版权声明： 

// Copyright {{now.UTC.Year}} The Go Authors. All rights reserved.
// Use of this source code is governed by a BSD-style
// license that can be found in the LICENSE file.

仓库文件以添加年份为版权年份。请勿更新所修改文件的版权年份。

邮件错误排查

git codereview mail命令失败最常见的原因是 提交记录中的邮箱地址与注册流程中使用的地址不一致。
如果您看到类似这样的提示... 

remote: Processing changes: refs: 1, done
remote:
remote: ERROR:  In commit ab13517fa29487dcf8b0d48916c51639426c5ee9
remote: ERROR:  author email address XXXXXXXXXXXXXXXXXXX
remote: ERROR:  does not match your user account.

您需要为此仓库配置 Git，使其使用您注册时使用的电子邮箱地址。 要更改电子邮箱地址以避免再次出现此问题，请运行： 

$ git config user.email email@address.com

然后使用以下命令将提交的作者信息更改为这个备用电子邮箱地址： 

$ git commit --amend --author="Author Name <email@address.com>"

然后通过运行以下命令重试： 

$ git codereview mail

快速测试您的更改

每次更改代码树都运行 all.bash 会很繁琐。 虽然强烈建议在提交更改前运行它，但在正常的开发周期中，您可能只想编译和测试您正在开发的包。

  • 通常，您可以运行 make.bash 而不是 all.bash， 这样只会重新构建 Go 工具链，而不会运行整个测试套件。 或者您可以运行 run.bash，这样只会运行整个测试套件而不会重新构建工具链。 您可以将 all.bash 视为 make.bash 后接 run.bash。
  • 在本节中，我们将您克隆 Go 仓库的目录称为 $GOROOT。 由 $GOROOT/src/make.bash 构建的 go 工具将安装在 $GOROOT/bin/go，您可以调用它来测试您的代码。 例如，如果您修改了编译器并想测试它如何影响您自己项目的测试套件， 只需使用它运行 go test： 
    $ cd <MYPROJECTDIR>
$ $GOROOT/bin/go test
  • 如果您正在更改标准库，您可能不需要重新构建编译器：只需运行您已更改包的测试即可。 您可以使用您通常使用的 Go 版本，也可以使用从您的克隆构建的 Go 编译器 （这有时是必要的，因为您正在修改的标准库代码可能需要比您已安装的稳定版本更新的版本）。 
    $ cd $GOROOT/src/crypto/sha1
$ [make changes...]
$ $GOROOT/bin/go test .
  • 如果您正在修改编译器本身，您可以只重新编译 compile 工具 （这是 go build 调用的内部二进制文件，用于编译每个单独的包）。 之后，您需要通过编译或运行某些东西来测试它。 
    $ cd $GOROOT/src
$ [make changes...]
$ $GOROOT/bin/go install cmd/compile
$ $GOROOT/bin/go build [something...]   # 测试新编译器
$ $GOROOT/bin/go run [something...]     # 测试新编译器
$ $GOROOT/bin/go test [something...]    # 测试新编译器
    这同样适用于 Go 工具链的其他内部工具，例如 asm、cover、link 等。 只需使用 go install cmd/<TOOL> 重新编译并安装该工具，然后使用构建的 Go 二进制文件进行测试。
  • 除了标准的按包测试外，$GOROOT/test 中还有一个顶层测试套件，包含多个黑盒测试和回归测试。 该测试套件由 all.bash 运行，但您也可以手动运行它： 
    $ $GOROOT/bin/go test cmd/internal/testdir

指定审查者 / 抄送其他人

除非有明确说明（例如在提交更改前的讨论中），否则最好不要指定审查者。 所有更改都会自动抄送到 golang-codereviews@googlegroups.com 邮件列表。 如果这是您的第一次更改，在防止垃圾邮件的机制下，它可能会在出现在邮件列表之前有一个审核延迟。

您可以使用 -r 或 -cc 选项指定审查者或抄送相关方。 两者都接受逗号分隔的电子邮箱地址列表： 

$ git codereview mail -r joe@golang.org -cc mabel@example.com,math-nuts@swtch.com

同步您的客户端

在您工作期间，其他人可能已向仓库提交了更改。要更新您的本地分支，请运行 

$ git codereview sync

（在底层，这运行 git pull -r。）

审查他人的代码

作为审查过程的一部分，审查者可以直接提出更改（在 GitHub 工作流中，这相当于其他人向拉取请求添加提交）。 Gerrit 提供了访问命令的功能，这些命令可帮助您导入另一位开发者提议的更改，以便您在本地审查/测试它们。 从您想要导入的 CL 的 Gerrit 页面，打开“⋮”菜单，点击“下载补丁”链接。 根据您首选的 git 工作流，选择适当的命令。选项看起来像这样： 

$ git fetch https://go.googlesource.com/review refs/changes/21/13245/1 && git checkout FETCH_HEAD

要撤消，切换回您之前工作的分支。

设置 git 别名

git-codereview 命令可以直接从 shell 运行，例如通过输入 

$ git codereview sync

但更方便的做法是为 git-codereview 自己的子命令设置别名，这样上面的命令就变成了 

$ git sync

git-codereview 的子命令经过精心设计，与 Git 自身命令有所区别，因此可以安全地设置这些别名。 要安装这些别名，请将以下文本复制到您的 Git 配置文件中（通常是位于主目录下的 .gitconfig）： 

[alias]
	change = codereview change
	gofmt = codereview gofmt
	mail = codereview mail
	pending = codereview pending
	submit = codereview submit
	sync = codereview sync

发送多个相互依赖的更改

高级用户可能希望将相关的多个提交堆叠在一个分支上。 Gerrit 允许更改之间相互依赖，从而形成这样的依赖链。 每个更改都需要被单独批准和提交，但其依赖关系对审查者是可见的。

要发送一组相互依赖的更改，请将每个更改作为不同的提交保留在同一个分支下，然后运行： 

$ git codereview mail HEAD

请注意要显式指定 HEAD，这在发送单个更改时通常不是必需的。更多详细信息可在 git-codereview 文档 中找到。
