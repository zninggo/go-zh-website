你可以通过将SQL参数值作为`sql`包函数的参数来避免SQL注入风险。`sql`包中的许多函数为SQL语句以及该语句参数中使用的值提供了参数（其他函数则为预编译语句和参数提供参数）。

以下示例中的代码使用`?`符号作为`id`参数的占位符，该参数作为函数参数提供：```go
// 执行带参数SQL语句的正确格式。
rows, err := db.Query("SELECT * FROM user WHERE id = ?", id)
```执行数据库操作的`sql`包函数会根据你提供的参数创建预处理语句。运行时，`sql`包将SQL语句转换为预处理语句，并将其与分离的参数一同发送。

**注意：** 参数占位符的形式取决于你使用的数据库管理系统和驱动程序。例如，PostgreSQL的[pq驱动程序](https://pkg.go.dev/github.com/lib/pq)接受`$1`这样的占位符形式，而非`?`。

你可能想要使用`fmt`包中的函数，将SQL语句与参数拼接成字符串——像这样：```
// 安全风险！
rows, err := db.Query(fmt.Sprintf("SELECT * FROM user WHERE id = %s", id))
```**这不是安全的做法！** 当您这样做时，Go 会在发送完整的语句到数据库管理系统（DBMS）之前，将整个 SQL 语句组装起来，用参数值替换 `%s` 格式动词。这带来了[SQL 注入](https://en.wikipedia.org/wiki/SQL_injection)的风险，因为调用此代码的用户可能会将一段意料之外的 SQL 代码片段作为 `id` 参数传入。该片段可能以对您的应用程序危险的、无法预测的方式完成 SQL 语句的构造。

例如，通过传递某个特定的 `%s` 值，您可能会得到类似下面的结果，这可能会返回数据库中所有用户的记录：```
SELECT * FROM user WHERE id = 1 OR 1=1;
```
