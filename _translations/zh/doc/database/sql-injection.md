通过将SQL参数值作为 `sql` 包函数的参数提供，您可以避免SQL注入风险。`sql` 包中的许多函数为SQL语句以及该语句中使用的参数值提供了参数（其他函数则为预编译语句及其参数提供参数）。

以下示例中的代码使用 `?` 符号作为 `id` 参数的占位符，该参数作为函数参数提供：```
// 执行带参数的SQL语句的正确格式。
rows, err := db.Query("SELECT * FROM user WHERE id = ?", id)
````sql` 包中执行数据库操作的函数会根据您提供的参数创建预处理语句。在运行时，`sql` 包会将 SQL 语句转换为预处理语句，并将其与参数分开发送。

**注意：** 参数占位符的格式取决于您使用的数据库管理系统（DBMS）和驱动程序。例如，[用于 Postgres 的 pq 驱动程序](https://pkg.go.dev/github.com/lib/pq)接受 `$1` 形式的占位符，而不是 `?`。

您可能会想要使用 `fmt` 包中的函数来将 SQL 语句和包含的参数拼接成一个字符串——就像这样：```
// 安全风险！
rows, err := db.Query(fmt.Sprintf("SELECT * FROM user WHERE id = %s", id))
```这是不安全的操作！执行此操作时，Go 会在将完整语句发送给数据库管理系统（DBMS）之前，就组装整个 SQL 语句并用参数值替换 `%s` 格式动词。这会导致 [SQL 注入](https://en.wikipedia.org/wiki/SQL_injection) 风险，因为代码调用者可能将非预期的 SQL 片段作为 `id` 参数发送。该片段可能以对您的应用程序构成危险的方式，在不可预测的程度上补全 SQL 语句。

例如，通过传递特定的 `%s` 值，最终可能生成类似以下语句，导致返回数据库中的所有用户记录：```
SELECT * FROM user WHERE id = 1 OR 1=1;
```
