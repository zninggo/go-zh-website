引言

本指南旨在帮助高级 Go 用户通过了解 Go 垃圾回收器的工作原理，从而更好地理解其应用程序的成本。 它还提供指导，说明 Go 用户如何利用这些洞见来改善其应用程序的资源利用率。 本指南不假设读者具备任何垃圾回收知识，但假定读者熟悉 Go 编程语言。

Go 语言负责安排 Go 值的存储； 在大多数情况下，Go 开发者无需关心这些值存储在哪里，或者为什么需要存储（如果需要的话）。 然而，在实践中，这些值通常需要存储在计算机的物理内存中，而物理内存是有限的资源。 因为它是有限的，所以在执行 Go 程序时，必须仔细管理内存并进行回收，以避免耗尽内存。 按需分配和回收内存是 Go 实现的工作。

自动回收内存的另一个术语是 垃圾回收。 从高层次看，垃圾回收器（或简称 GC）是一个系统，它通过识别内存中哪些部分不再需要，来代表应用程序回收内存。 Go 标准工具链提供了一个随每个应用程序一起发布的运行时库，该运行时库包含一个垃圾回收器。

请注意，本指南所描述的垃圾回收器的存在并非由 Go 语言规范 保证，规范仅保证 Go 值的底层存储由语言自身管理。 这种省略是有意为之的，它允许使用完全不同的内存管理技术。

因此，本指南是关于 Go 编程语言的一个特定实现，可能不适用于其他实现。 具体来说，以下指南适用于标准工具链（gc Go 编译器和工具）。 Gccgo 和 Gollvm 都使用非常相似的 GC 实现，因此许多相同的概念适用，但细节可能有所不同。

此外，本文档是一个动态更新的文档，会随着时间推移而变化，以最好地反映 Go 的最新版本。 本文档目前描述的是 Go 1.19 版本的垃圾回收器。

Go 值存储在哪里

在我们深入了解 GC 之前，让我们先讨论一下不需要 GC 管理的内存。

例如，存储在局部变量中的非指针 Go 值很可能根本不会由 Go GC 管理，Go 会安排将内存分配绑定到该值所创建的词法作用域。 通常，这比依赖 GC 更高效，因为 Go 编译器能够预先确定该内存何时可以被释放，并发出清理的机器指令。 通常，我们将这种为 Go 值分配内存的方式称为“栈分配”，因为该空间存储在 goroutine 栈上。

其内存无法以这种方式分配的 Go 值，因为 Go 编译器无法确定其生命周期，被称为逃逸到堆。 “堆”可以看作是内存分配的“万能收纳处”，用于存放需要放置在某处的 Go 值。 在堆上分配内存的行为通常被称为“动态内存分配”，因为编译器和运行时对这块内存的使用方式和清理时间几乎无法做任何假设。 这就是 GC 发挥作用的地方：它是一个专门识别和清理动态内存分配的系统。

Go 值可能需要逃逸到堆的原因有很多。 一个原因可能是其大小是动态确定的。 例如，考虑一个切片的底层数组，其初始大小是由一个变量而非常量决定的。 请注意，逃逸到堆必须是传递性的：如果一个对 Go 值的引用被写入另一个已经确定会逃逸的 Go 值中，那么该值也必须逃逸。

Go 值是否逃逸取决于其使用的上下文以及 Go 编译器的逃逸分析算法。 试图精确枚举值何时逃逸是脆弱且困难的：算法本身相当复杂，并且在不同的 Go 版本之间会变化。 有关如何识别哪些值逃逸、哪些不逃逸的更多详细信息，请参阅消除堆分配部分。

追踪式垃圾回收

垃圾回收可以指代多种不同的自动内存回收方法；例如引用计数法。在本文档中，垃圾回收特指追踪式垃圾回收——它通过递归地跟踪指针来识别正在使用的对象，这些对象被称为存活对象。

让我们更严格地定义这些术语。

  • 对象——对象是一块动态分配的内存区域，包含一个或多个 Go 值。

  • 指针——引用对象中任意值的内存地址。这自然包括 *T 形式的 Go 值，但也包括内置 Go 值的部分。字符串、切片、通道、映射和接口值都包含 GC 必须跟踪的内存地址。

对象和指向其他对象的指针共同构成了对象图。为了识别存活内存，GC 从程序的根开始遍历对象图——这些根是指向程序肯定正在使用的对象的指针。根的两个例子是局部变量和全局变量。遍历对象图的过程称为扫描。你可能在 Go 文档中看到的另一个短语是对象是否可达，这仅仅意味着该对象可以通过扫描过程被发现。另外请注意，除了一个例外，一旦内存变得不可达，它将始终保持不可达状态。

这个基本算法是所有追踪式垃圾回收器所共有的。追踪式垃圾回收器的不同之处在于它们发现内存存活后采取的行动。Go 的垃圾回收器使用标记-清扫技术，这意味着为了跟踪进度，GC 还会将其遇到的值标记为存活。追踪完成后，GC 随后会遍历堆中的所有内存，并将所有未标记的内存标记为可供分配。这个过程称为清扫。

你可能熟悉的另一种替代技术是将对象移动到内存的新位置，并留下一个转发指针，该指针随后用于更新应用程序的所有指针。我们将这种移动对象的 GC 称为移动式垃圾回收器；Go 拥有一个非移动式垃圾回收器。

垃圾回收周期

由于 Go 的垃圾回收器是标记-清扫式垃圾回收器，它大致分为两个阶段运行：标记阶段和清扫阶段。虽然这个说法看似同义反复，但它包含一个重要见解：在所有内存都被追踪之前，不可能将内存释放回以供分配，因为可能仍有未扫描的指针使对象保持存活。因此，清扫行为必须与标记行为完全分开。此外，当没有垃圾回收相关工作可做时，GC 也可能完全处于非活动状态。GC 在清扫、关闭和标记这三个阶段中持续轮转，这被称为垃圾回收周期。就本文档而言，请考虑垃圾回收周期从清扫开始，然后关闭，接着进入标记。

接下来的几节将重点建立对 GC 成本的直观理解，以帮助用户为自身利益调整 GC 参数。

理解成本

垃圾回收本质上是一个复杂的软件，构建在更复杂的系统之上。在试图理解 GC 并调整其行为时，很容易陷入细节的泥潭。本节旨在提供一个推理 Go GC 成本及其调优参数的框架。

首先，考虑这个基于三个简单公理的 GC 成本模型。

  1. 垃圾回收只涉及两种资源：物理内存和 CPU 时间。

  2. 垃圾回收的内存成本由存活的堆内存、标记阶段之前分配的新堆内存以及元数据空间组成，这些元数据空间即使与前述成本成正比，相比之下也很小。

    周期 N 的 GC 内存成本 = 来自周期 N-1 的存活堆 + 新堆

    存活堆内存是由前一个 GC 周期确定为存活的内存，而新堆内存是当前周期中分配的任何内存，这些内存在周期结束时可能存活也可能不存活。在任何给定时间点有多少内存存活是程序的属性，而不是 GC 可以直接控制的。

  3. 垃圾回收的 CPU 成本被建模为每个周期的固定成本，以及与存活堆大小成比例的边际成本。

    周期 N 的 GC CPU 时间 = 每个周期的固定 CPU 时间成本 + 每字节的平均 CPU 时间成本 * 周期 N 中发现的存活堆内存

    每个周期的固定 CPU 时间成本包括每个周期发生固定次数的事情，例如为下一个 GC 周期初始化数据结构。这个成本通常很小，仅为完整性而包含。

    垃圾回收的大部分 CPU 成本是标记和扫描，这由边际成本体现。标记和扫描的平均成本取决于 GC 实现，但也取决于程序的行为。例如，更多的指针意味着更多的 GC 工作，因为 GC 至少需要访问程序中的所有指针。像链表和树这样的结构也更难让 GC 并行遍历，从而增加了每字节的平均成本。

    该模型忽略了清扫成本，该成本与总堆内存成正比，包括已失效的内存（必须将其标记为可供分配）。对于 Go 当前的 GC 实现，清扫比标记和扫描快得多，因此相比之下成本可以忽略不计。

这个模型简单但有效：它准确地分类了 GC 的主要成本。 它也告诉我们，垃圾回收器的总 CPU 成本取决于给定时间段内的 GC 周期总数。 最后，这个模型内含了 GC 的一个基本的时间/空间权衡。

要理解其中原因，让我们探讨一个受限但有用的场景：稳态。 从 GC 的角度来看，应用程序的稳态由以下属性定义：

  • 应用程序分配新内存的速率（以字节/秒计）是恒定的。

    这意味着，从 GC 的角度看，应用程序的工作负载在一段时间内看起来大致相同。 例如，对于一个网络服务，这将是恒定的请求速率，平均来看，处理的是相同类型的请求，并且每个请求的平均生命周期大致保持不变。

  • GC 的边际成本是恒定的。

    这意味着对象图的统计数据，例如对象大小的分布、指针的数量和数据结构的平均深度，在每个周期之间保持不变。

让我们通过一个例子来理解。 假设某个应用程序处于稳态，以 10 MiB/s 的速率分配内存，而 GC 可以以 100 MiB/核·秒的速度扫描内存（此为假设值）。 稳态不对活跃堆的大小做任何假设，但为简单起见，我们假设这个应用程序的活跃堆始终为 10 MiB。 同样，为简单起见，我们假设 GC 的固定成本为零。 让我们来调整 GC 周期的间隔。

假设每个 GC 周期恰好在 1 个核·秒后发生。 那么，在每个 GC 周期结束时，我们的示例应用程序将分配了 10 MiB 的额外内存，导致总堆大小为 20 MiB。 并且在每个 GC 周期中，GC 将花费 0.1 个核·秒来扫描 10 MiB 的活跃堆，从而产生 10% 的 CPU 开销。 记住，GC 只需要遍历活跃堆，而不是整个堆。 （注意：活跃堆恒定并不意味着所有新分配的内存都会失效。 这意味着，GC 运行后，新旧堆内存以某种组合失效，而最终结果只是每个周期找到 10 MiB 的活跃内存。）

现在假设每个 GC 周期发生频率降低，每 2 个核·秒一次。 那么，在稳态下，我们的示例应用程序在每个 GC 周期将拥有 30 MiB 的总堆大小，因为它将在此期间分配 20 MiB。 但在每个 GC 周期中，GC 仍然只需要 0.1 个核·秒来扫描 10 MiB 的活跃内存。 同样，我们假设活跃堆大小保持不变，无论分配了多少内存。 因此，这意味着我们的 GC 开销从 10% 降至 5%，代价是内存使用量增加了 50%。

这种开销的变化就是前面提到的基本时间/空间权衡。 而 GC 频率正是这个权衡的核心： 如果我们更频繁地执行 GC，那么我们使用的内存就更少，反之亦然。 但 GC 实际执行的频率是多少呢？ 在 Go 中，决定 GC 何时启动是用户可以控制的主要参数。

GOGC

在较高层次上，GOGC 决定了 GC CPU 和内存之间的权衡。

它的工作原理是在每个 GC 周期后确定目标堆大小，即下一个周期的总堆大小的目标值。 GC 的目标是在总堆大小超过目标堆大小之前完成一次收集周期。 总堆大小定义为上一个周期结束时的活跃堆大小，加上应用程序自上一个周期以来分配的任何新堆内存。 同时，目标堆内存定义为：

目标堆内存 = 活跃堆 + (活跃堆 + GC 根) * GOGC / 100

例如，考虑一个活跃堆大小为 8 MiB、goroutine 栈占 1 MiB、全局变量中的指针占 1 MiB 的 Go 程序。 那么，当 GOGC 值为 100 时，在下一次 GC 运行之前将分配的新内存量为 10 MiB，即 10 MiB 工作量的 100%，总堆内存占用为 18 MiB。 当 GOGC 值为 50 时，新分配量将为 50%，即 5 MiB。 当 GOGC 值为 200 时，新分配量将为 200%，即 20 MiB。

注意：仅从 Go 1.18 开始，GOGC 才包含根集。 以前，它只计算活跃堆。 通常，goroutine 栈中的内存量相当小，活跃堆大小主导了 GC 工作的所有来源，但在程序拥有数十万个 goroutine 的情况下，GC 做出的判断会很差。

堆目标控制着 GC 频率：目标越大，GC 可以等待开始下一个标记阶段的时间就越长，反之亦然。 虽然精确的公式对于估算很有用，但最好从其根本目的来理解 GOGC：一个在 GC CPU 和内存权衡中选择一个点的参数。 关键要点是，将 GOGC 翻倍将使堆内存开销翻倍，并大致将 GC CPU 成本减半，反之亦然。 （要查看完整解释，请参阅附录。）

注意：目标堆大小只是一个目标，有多种原因可能导致 GC 周期未能恰好在该目标点完成。 例如，一次足够大的堆分配可以简单地超出目标。 然而，其他原因出现在 GC 实现中，超出了本指南迄今使用的GC 模型。 有关更多细节，请参阅延迟部分，但完整的细节可以在额外资源中找到。

可以通过 GOGC 环境变量（所有 Go 程序都识别此变量）或通过 runtime/debug 包中的 SetGCPercent API 来配置 GOGC。

请注意，GOGC 也可用于完全关闭垃圾回收器（前提是内存限制不适用），方法是将 GOGC 设置为 off 或调用 SetGCPercent(-1)。 从概念上讲，此设置等同于将 GOGC 设置为无穷大，因为触发垃圾回收之前的新内存分配量是无界的。

内存限制

直到 Go 1.19，GOGC 是唯一可用于修改垃圾回收器行为的参数。 虽然它作为一种设定权衡的方式效果很好，但它没有考虑到可用内存是有限的。 考虑当存活堆大小发生瞬时峰值时会发生什么：因为垃圾回收器会选择一个与该存活堆大小成比例的总堆大小，所以 GOGC 必须针对峰值存活堆大小进行配置，即使在通常情况下，更高的 GOGC 值会提供更好的权衡。

正因如此，在 1.19 版本中，Go 增加了设置运行时内存限制的支持。内存限制可以通过所有 Go 程序都识别的 GOMEMLIMIT 环境变量来配置，也可以通过 runtime/debug 包中提供的 SetMemoryLimit 函数来配置。

此内存限制设置了 Go 运行时可使用的总内存量 的上限。所包含的具体内存集合是根据 runtime.MemStats 定义的，表达式为

Sys - HeapReleased

或者等价地，根据 runtime/metrics 包，

/memory/classes/total:bytes - /memory/classes/heap/released:bytes

由于 Go 的 GC 对其使用的堆内存有明确的控制权，它会根据此内存限制以及 Go 运行时使用的其他内存量来设定总堆大小。

这一观察引出了另一个有趣的细节：即使当 GOGC 设置为 off（关闭）时，内存限制仍然会被遵守！实际上，这种特定配置代表了 资源利用率的最大化，因为它设定了维持某个内存限制所需的最小 GC 频率。在这种情况下，程序 所有 的执行阶段都让堆大小增长到触及内存限制。

现在，虽然内存限制显然是一个强大的工具，但 使用内存限制并非没有代价，而且当然也不会使 GOGC 失去作用。

考虑当活动堆增长到足以使总内存使用接近内存限制时会发生什么。 在上面的稳态可视化中，尝试关闭 GOGC，然后慢慢降低内存限制，观察会发生什么。 注意，随着 GC 不断执行以维持一个不可能实现的内存限制，应用程序所花费的总时间将开始无限增长。

这种由于频繁的 GC 周期导致程序无法取得合理进展的情况被称为 颠簸。 它尤其危险，因为它实际上会使程序停滞。 更糟糕的是，这可能恰恰发生在我们试图通过 GOGC 避免的相同场景中：足够大的瞬时堆尖峰可能导致程序无限期地停滞！ 尝试在瞬时堆尖峰可视化中降低内存限制（大约 30 MiB 或更低），并注意最糟糕的行为是如何特别从堆尖峰开始的。

在许多情况下，无限期的停滞比内存不足的情况更糟，后者往往会导致更快的失败。

因此，内存限制被定义为 软性。 Go 运行时不保证在所有情况下都会维持这个内存限制；它只承诺做出一些合理的努力。 对内存限制的这种放宽对于避免颠簸行为至关重要，因为它给了 GC 一个出路：让内存使用超过限制，以避免在 GC 上花费过多时间。

其内部工作原理是，GC 为其在某个时间窗口内可使用的 CPU 时间量设定了一个上限（对非常短暂的 CPU 使用瞬时尖峰带有一些滞后性）。 此限制目前设定在大约 50%，时间窗口为 2 * GOMAXPROCS 个 CPU 秒。 限制 GC CPU 时间的后果是 GC 的工作被延迟，与此同时 Go 程序可能继续分配新的堆内存，甚至超过内存限制。

50% GC CPU 限制背后的直觉是基于对具有充足可用内存的程序的最坏情况影响。 在内存限制设置错误、设得过低的情况下，程序最多会减速 2 倍，因为 GC 不能占用其超过 50% 的 CPU 时间。

注意：本页的可视化未模拟 GC CPU 限制。

建议用法

虽然内存限制是一个强大的工具，并且 Go 运行时采取措施来缓解误用导致的最坏行为，但深思熟虑地使用它仍然很重要。 以下是一些关于内存限制在哪些场景下最有用、最适用，以及在哪些场景下可能弊大于利的建议摘要。

  • 应该在您的 Go 程序执行环境完全由您控制，并且该 Go 程序是唯一能访问某些资源（即某种内存预留，例如容器内存限制）的程序时，利用内存限制。

    一个很好的例子是将 Web 服务部署到具有固定可用内存量的容器中。

    在这种情况下，一个很好的经验法则是额外预留 5-10% 的余量，以考虑 Go 运行时不知道的内存来源。

  • 应该随时调整内存限制以适应变化的情况。

    一个很好的例子是 cgo 程序，其中 C 库暂时需要使用大量内存。

  • 不要在 Go 程序可能与其他程序共享其部分有限内存，且这些程序通常与 Go 程序解耦的情况下，将 GOGC 设置为 off 并同时设置内存限制。 应保留内存限制，因为它可能有助于抑制不良的瞬时行为，但应将 GOGC 设置为针对平均情况更小、更合理的值。

    虽然尝试为“共同租户”程序“预留”内存可能很诱人，但除非程序完全同步（例如，Go 程序调用某个子进程，并在其被调用者执行时阻塞），否则结果将不太可靠，因为不可避免地两个程序都需要更多内存。 让 Go 程序在不需要时使用更少的内存，总体上会产生更可靠的结果。 这个建议也适用于超额承诺的情况，即在一台机器上运行的容器的内存限制总和可能超过该机器实际可用的物理内存。

  • 不要在部署到您无法控制的执行环境时使用内存限制，尤其是当您的程序的内存使用与其输入成正比时。

    一个很好的例子是命令行工具或桌面应用程序。 在不清楚程序可能接收什么样的输入，或系统可用内存多少的情况下，在程序中硬编码一个内存限制，可能会导致令人困惑的崩溃和性能低下。 此外，高级最终用户总是可以根据需要自行设置内存限制。

  • 不要在程序已接近其环境的内存限制时，设置内存限制以避免内存不足情况。

    这实际上是用严重应用减速的风险取代了内存不足的风险，这通常不是一种有利的权衡，即使 Go 做出了缓解颠簸的努力。 在这种情况下，增加环境的内存限制（然后再可能设置内存限制）或降低 GOGC（这比颠簸缓解提供了一种更清晰的权衡）会更有效。

延迟

本文档中的可视化模型将GC执行期间的应用程序模拟为暂停状态。 确实存在这种行为的GC实现，它们被称为"全局暂停"GC。

然而，Go的GC并非完全全局暂停，它在与应用程序并发执行的过程中完成了大部分工作。 这主要是为了降低应用程序的延迟。 具体而言，即单个计算单元（例如一个Web请求）的端到端持续时间。 至此，本文档主要关注应用程序的吞吐量（例如每秒处理的Web请求数）。 请注意，GC周期一节中的每个示例都侧重于执行程序的总CPU时间。 然而，对于网络服务而言，这样的持续时间意义较小。 虽然吞吐量对于网络服务仍然重要（即每秒查询数），但通常每个单独请求的延迟更为关键。

就延迟而言，一个全局暂停的GC可能需要相当长的时间来执行其标记和清除阶段，在此期间应用程序无法取得进一步进展，在网络服务的上下文中，任何正在处理的请求也是如此。 相反，Go的GC避免使任何全局应用程序暂停的长度与堆的大小成比例，并且核心跟踪算法是在应用程序积极执行时完成的。 （暂停时间在算法上更强烈地与GOMAXPROCS成正比，但最常见的主导因素是停止运行中的goroutine所需的时间。） 并发收集并非没有代价：实际上，它通常导致设计的吞吐量低于等效的全局暂停垃圾回收器。 然而，重要的是要注意到更低的延迟并不固有地意味着更低的吞吐量，并且Go垃圾回收器的性能随着时间的推移在延迟和吞吐量方面都稳步提高。

Go当前GC的并发特性并不使本文档到目前为止讨论的任何内容失效：没有任何陈述依赖于这个设计选择。 GC频率仍然是GC在CPU时间和内存之间权衡以实现吞吐量的主要方式，事实上，它对于延迟也承担着同样的作用。 这是因为GC的大部分开销是在标记阶段活跃时产生的。

那么，关键要点是降低GC频率也可能带来延迟的改进。 这不仅适用于通过修改调优参数（如增加GOGC和/或内存限制）来降低GC频率的情况，也适用于优化指南中描述的优化。

然而，延迟通常比吞吐量更难理解，因为它是程序逐瞬间执行的产物，而不仅仅是成本的总和。 因此，延迟与GC频率之间的联系不那么直接。 以下是为那些希望深入研究的人列出的可能延迟来源：

  1. GC在标记和清除阶段之间转换时的短暂全局暂停，
  2. GC在标记阶段占据25% CPU资源时的调度延迟，
  3. 用户goroutine为响应高分配速率而协助GC，
  4. GC处于标记阶段时指针写入需要额外工作，
  5. 正在运行的goroutine必须被暂停以扫描其根。

这些延迟来源在执行跟踪中可见，除了指针写入需要额外工作这一项。

终结器、清理函数与弱指针

垃圾回收使用有限的内存提供了无限内存的假象。 内存被分配但从未被显式释放，这使得API和并发算法比原始的手动内存管理更为简单。 （一些具有手动内存管理的语言使用替代方法，如"智能指针"和编译时所有权跟踪来确保对象被释放，但这些特性深深嵌入到这些语言的API设计约定中。）

只有活跃对象——即可以从全局变量或某个goroutine中的计算可达的对象——才能影响程序的行为。 一旦对象变得不可达（"死亡"），任何时候它都可以被GC安全地回收。 这允许多种多样的GC设计，例如Go今天使用的跟踪式设计。 对象的死亡在语言层面不是一个可观察事件。

然而，Go的运行时库提供了三个打破这种假象的特性：清理函数、弱指针和终结器。 这些特性中的每一个都提供了观察和响应对象死亡的某种方式，对于终结器而言，甚至可以逆转死亡。 这当然使Go程序复杂化，并给GC实现增加了额外的负担。 尽管如此，这些特性之所以存在，是因为它们在多种情况下非常有用，并且Go程序一直使用它们并从中受益。

关于每个特性的详细信息，请参考其包文档（runtime.AddCleanup、weak.Pointer、runtime.SetFinalizer）。 以下是使用这些特性的一些通用建议、每个特性可能遇到的常见问题概述，以及测试这些特性用法的建议。

通用建议

  • 编写单元测试。

    清理函数、弱指针和析构器的确切执行时机可能难以预测，即使在多次连续执行后，也很容易让你误以为一切正常。但这也容易产生细微的错误。为它们编写测试可能比较棘手，但鉴于这些特性使用起来如此微妙，测试就显得比以往更加重要。

  • 避免在典型的 Go 代码中直接使用这些特性。

    这些是具有微妙限制和行为的底层特性。例如，无法保证清理函数或析构器会在程序退出时运行，甚至根本无法保证它们一定会运行。它们 API 文档中的长段注释应被视为一种警告。绝大多数 Go 代码不会从直接使用这些特性中受益，而只是间接受益。

  • 将这些机制的使用封装在 package 内部。

    在可能的情况下，不要让这些机制的使用泄露到你 package 的公共 API 中；提供接口使用户难以或不可能误用它们。例如，与其要求用户为某些 C 分配的内存设置清理函数来释放它，不如编写一个包装 package 并将该细节隐藏在其中。

  • 将具有析构器、清理函数和弱指针的对象的访问权限限制在创建并应用它们的 package 内。

    这一点与上一点相关，但值得特别指出，因为这是一种非常强大的模式，可以以更不易出错的方式使用这些特性。例如，unique 包在底层使用了弱指针，但完全封装了被弱指针引用的对象。这些值永远不能被应用程序的其他部分修改，只能通过Value 方法复制，从而为 package 用户营造出内存无限的假象。

  • 尽可能以确定性的方式清理非内存资源，将析构器和清理函数作为后备手段。

    清理函数和析构器非常适合内存资源，例如外部（如从 C）分配的内存，或对 `mmap` 映射的引用。C 的 malloc 分配的内存最终必须由 C 的 free 释放。将调用 `free` 的析构器附加到包装 C 内存的对象上，是确保 C 内存最终因垃圾回收而被回收的一种合理方式。

    然而，非内存资源（如文件描述符）往往受到系统限制，而 Go 运行时通常并不了解这些限制。此外，在给定的 Go 程序中，垃圾回收器的时机通常是 package 作者难以控制的（例如，GC 运行的频率由 GOGC 控制，实际上操作员可以将其设置为多种不同的值）。这两个事实共同导致清理函数和析构器不适合作为释放非内存资源的唯一机制。

    如果你是一名 package 作者，提供了包装某些非内存资源的 API，请考虑提供一个显式的 API 来确定性地释放资源（通过 `Close` 方法或类似方式），而不是依赖于通过清理函数或析构器使用垃圾回收器。相反，最好将清理函数和析构器用作针对程序员错误的最佳努力处理器，要么像 os.File 那样无论如何清理资源，要么向用户报告未能确定性清理资源的错误。

  • 优先使用清理函数而非析构器。

    历史上，析构器的添加是为了简化 Go 代码与 C 代码之间的接口，并清理非内存资源。其预期用途是将其应用于拥有 C 内存或其他非内存资源的包装对象，以便在 Go 代码使用完毕后可以释放资源。这些原因至少部分解释了为什么析构器的作用域狭窄，为什么任何给定对象只能有一个析构器，以及为什么该析构器必须仅附加到对象的第一个字节。这一限制已经抑制了一些用例。例如，任何希望内部缓存有关传递给它的对象信息的 package，都无法在对象消失后清理这些信息。

    但更糟糕的是，由于析构器会复活它们所附加的对象（以便将其传递给析构器函数，甚至可能使其在之后继续存活），因此它们效率低下且容易出错。这个简单的事实意味着，如果对象是引用循环的一部分，它就永远无法被释放，并且支持该对象的内存至少要到下一个垃圾回收周期才能被重用。

    不过，由于析构器会复活对象，它们确实比清理函数具有更明确的执行顺序。因此，对于清理具有复杂销毁顺序要求的结构体，析构器仍然可能（尽管很少）有用。

    但在 Go 1.24 及更高版本的所有其他用途中，我们建议你使用清理函数，因为它们比析构器更灵活、不易出错且效率更高。

常见的清理函数问题

  • 附加了清理函数的对象不得从清理函数内部可达（例如，通过被捕获的局部变量）。否则将阻止对象被回收，清理函数也永远不会执行。 

    • f := new(myFile)
f.fd = syscall.Open(...)
runtime.AddCleanup(f, func(fd int) {
	syscall.Close(f.fd) // 错误：我们引用了 f，所以这个清理函数不会执行！
}, f.fd)

  • 附加了清理函数的对象不得从清理函数的参数可达。否则将阻止对象被回收，清理函数也永远不会执行。 

    • f := new(myFile)
f.fd = syscall.Open(...)
runtime.AddCleanup(f, func(f *myFile) {
	syscall.Close(f.fd)
}, f) // 错误：我们引用了 f，所以这个清理函数永远不会执行。这个特定情况还会导致恐慌。

  • 析构器有明确定义的执行顺序，但清理函数没有。清理函数还可以彼此并发执行。

  • 耗时较长的清理函数应创建一个 goroutine，以避免阻塞其他清理函数的执行。

  • runtime.GC 不会等待不可达对象的清理函数执行完毕，只等待它们全部入队。

常见的弱指针问题

  • 弱指针可能在其 Value 方法开始返回 nil 的时间点出人意料。务必用 nil 检查保护对 Value 的调用，并准备好备选方案。

  • 当弱指针用作映射键时，它们不影响映射值的可达性。因此，如果一个弱指针映射键指向的对象也能从映射值访问到，那么该对象仍将被视为可达的。

常见的析构器问题

  • 附加了析构器的对象不得通过任何路径从自身可达（换言之，它们不能处于引用循环中）。否则将阻止对象被回收，析构器也永远不会执行。 

    • f := new(myCycle)
f.self = f // 错误：f 从 f 可达，所以这个析构器永远不会执行。
runtime.SetFinalizer(f, func(f *myCycle) {
	...
})

  • 附加了析构器的对象不得从析构器函数内部可达（例如，通过被捕获的局部变量）。否则将阻止对象被回收，析构器也永远不会执行。 

    • f := new(myFile)
f.fd = syscall.Open(...)
runtime.SetFinalizer(f, func(_ *myFile) {
	syscall.Close(f.fd) // 错误：我们引用了外部的 f，所以这个清理函数不会执行！
})

  • 附加了析构器的对象的引用链（例如，在链表中）至少需要与链中对象数量相同的垃圾回收周期才能全部清理。保持析构器层级浅！ 

    • // 错误：回收这个链表至少需要 10 个垃圾回收周期。
node := new(linkedListNode)
for range 10 {
	tmp := new(linkedListNode)
	tmp.next = node
	node = tmp
	runtime.SetFinalizer(node, func(node *linkedListNode) {
		...
	})
}

  • 避免对在包边界返回的对象设置析构器。这使得你的包的用户有可能调用 runtime.SetFinalizer 来修改你返回的对象上的析构器，这可能成为用户可能依赖的一个意外行为。

  • 耗时较长的析构器应创建一个新的 goroutine，以避免阻塞其他析构器的执行。

  • runtime.GC 不会等待不可达对象的析构器执行完毕，只等待它们全部入队。

测试对象回收

在使用这些特性时，为使用它们的代码编写测试有时可能很棘手。以下是一些针对使用这些特性的代码编写健壮测试的技巧。

  • 避免与其他测试并行运行此类测试。尽可能增加确定性并随时掌握当前状态会很有帮助。
  • 在进入测试时使用 runtime.GC 建立基准状态。使用 runtime.GC 强制弱指针变为 nil，并将清理函数和析构器排入队列等待运行。

  • runtime.GC 不会等待清理函数和析构器运行，它只是将它们排入队列。

    为了编写最健壮的测试，请从你的测试中注入一种阻塞清理函数或析构器执行的方式（例如，从测试中向清理函数和/或析构器传递一个可选通道，并在其执行完毕后写入该通道）。如果这太难或不可能，另一种方法是循环检查某个清理后的状态是否为真。例如，os 包的测试会在一个检查文件是否已关闭的循环中调用 runtime.Gosched，一旦文件变得不可达。

  • 如果为使用析构器的代码编写测试，并且你有一个使用了析构器的对象链，那么你将需要至少与测试所能创建的最深链长度相同的 runtime.GC 调用次数，以确保所有析构器都运行。

  • 在竞争检测模式下进行测试，以发现并发清理函数之间、清理函数和析构器代码与其余代码库之间的竞态条件。

关于虚拟内存的说明

本指南主要关注垃圾回收器的物理内存使用，但一个经常出现的问题是这具体指什么，以及它与虚拟内存（通常在top等程序中显示为“VSS”）的比较。

物理内存是存在于大多数计算机实际物理RAM芯片中的内存。 虚拟内存是操作系统提供的一种对物理内存的抽象，用于隔离不同程序。 通常，程序保留不映射到任何物理地址的虚拟地址空间也是可以接受的。

由于虚拟内存只是操作系统维护的一个映射，通常进行不映射到物理内存的大型虚拟内存预留操作成本非常低。

Go运行时在几个方面普遍依赖于这种虚拟内存成本视图：

  • Go运行时从不删除它映射的虚拟内存。 相反，它使用大多数操作系统提供的特殊操作来显式释放与某个虚拟内存范围相关联的物理内存资源。

    该技术被明确用于管理内存限制，并将Go运行时不再需要的内存归还给操作系统。 Go运行时也会在后台持续释放它不再需要的内存。 更多信息请参见补充资源。

  • 在32位平台上，Go运行时会预先为堆保留128 MiB到512 MiB的地址空间，以限制碎片化问题。

  • Go运行时在多个内部数据结构的实现中使用大型虚拟内存地址空间预留。 在64位平台上，这些通常至少有大约700 MiB的虚拟内存占用。 在32位平台上，其占用可以忽略不计。

因此，诸如top中的“VSS”之类的虚拟内存指标，对于理解Go程序的内存占用通常用处不大。 相反，应关注“RSS”及类似的度量指标，它们更直接地反映了物理内存的使用情况。

优化指南

识别成本

在尝试优化Go应用程序与垃圾回收器的交互方式之前，首先需要确定垃圾回收器是否是一个主要的成本来源。

Go生态系统提供了许多用于识别成本和优化Go应用程序的工具。 有关这些工具的简要概述，请参阅诊断指南。 这里，我们将重点关注其中一部分工具，以及为理解垃圾回收器的影响和行为而应用它们的合理顺序。

  1. CPU 分析

    一个好的起点是进行CPU分析。 CPU分析提供了CPU时间消耗位置的概览，但对于未经训练的人来说，可能很难识别垃圾回收器在特定应用中扮演角色的大小。 幸运的是，理解垃圾回收器在其中的作用主要归结为知道`runtime`包中不同函数的meam。 以下是解释CPU分析时有用的函数子集。

    请注意，下面列出的函数不是叶子函数，因此它们可能不会出现在pprof工具通过top命令提供的默认输出中。 相反，请使用top -cum命令，或者直接在这些函数上使用list命令，并重点关注累计百分比列。

  • runtime.gcBgMarkWorker：后台标记工作协程的入口点。 在此处花费的时间随垃圾回收频率、对象图的复杂度和大小而增加。 它代表了应用程序用于标记和扫描操作的基准时间。

    请注意，在这些协程中，你会发现对 runtime.gcDrainMarkWorkerDedicated、 runtime.gcDrainMarkWorkerFractional 和 runtime.gcDrainMarkWorkerIdle 的调用，这些调用表明了工作者的类型。 在大部分空闲的 Go 应用程序中，Go 垃圾回收器会利用额外的（空闲）CPU 资源来更快地完成其工作， 这通过 runtime.gcDrainMarkWorkerIdle 符号来表示。 因此，此处的时间可能占据了 CPU 采样中一个很大的比例， 而 Go 垃圾回收器认为这些 CPU 资源是空闲的。 如果应用程序变得更活跃，空闲工作者中的 CPU 时间将会减少。 发生这种情况的一个常见原因是应用程序完全在一个协程中运行，但 GOMAXPROCS 设置为 >1。

  • runtime.mallocgc：堆内存内存分配器的入口点。 在此处花费的累计时间较多（>15%）通常表明分配了大量的内存。

  • runtime.gcAssistAlloc：函数协程进入此函数，以让出部分时间来协助垃圾回收器进行扫描和标记。 在此处花费的累计时间较多（>5%）表明应用程序的内存分配速度可能快于垃圾回收器的处理速度。 这表明应用程序受到了垃圾回收器的高度影响，同时也代表了应用程序用于标记和扫描的时间。 请注意，这部分时间包含在 runtime.mallocgc 的调用树中，因此也会使其数值膨胀。

  • 执行跟踪

    虽然 CPU 分析非常适合识别时间在总体上花费在哪里，但对于指示更微妙、罕见或与特定延迟相关的性能开销，它的用处较小。 另一方面，执行跟踪 提供了对 Go 程序执行的一个短窗口的丰富而深入的观察。 它们包含与 Go 垃圾回收相关的各种事件，并且可以直接观察到特定的执行路径，以及应用程序可能如何与 Go 垃圾回收器交互。 跟踪查看器中方便地将所有追踪到的垃圾回收事件标记了出来。

    关于如何开始使用执行跟踪，请参阅 runtime/trace 包的文档。

  • 垃圾回收跟踪

    当其他方法都无效时，Go 垃圾回收器提供了几种不同的特定跟踪，这些跟踪能更深入地洞察垃圾回收行为。 这些跟踪总是直接打印到标准错误输出（STDERR），每个垃圾回收周期一行，并通过所有 Go 程序都识别的 GODEBUG 环境变量进行配置。 它们主要用于调试 Go 垃圾回收器本身，因为需要熟悉垃圾回收器实现的细节，但偶尔也能有助于更好地理解垃圾回收行为。

    核心的垃圾回收跟踪通过设置 GODEBUG=gctrace=1 来启用。 此跟踪产生的输出文档记录在 runtime 包文档的环境变量部分。

    一个名为"协调器跟踪"的补充垃圾回收跟踪提供了更深入的见解，通过设置 GODEBUG=gcpacertrace=1 来启用。 解释此输出需要理解垃圾回收器的"协调器"（参见附加资源），这超出了本指南的范围。

    • 消除堆分配

    减少垃圾回收开销的一种方法是从一开始就让垃圾回收器管理更少的值。 下面描述的技术可以产生一些最大的性能改进，因为正如 GOGC 部分 所演示的，Go 程序的分配速率是垃圾回收频率的一个主要因素，而垃圾回收频率是本指南使用的关键开销指标。

    堆分析

    在确定垃圾回收是一个显著的开销来源之后，消除堆分配的下一步是找出大部分分配来自哪里。 为此，内存分析（实际上是堆内存分析）非常有用。 查看文档了解如何开始使用它们。

    内存分析描述了程序中堆分配的来源，通过分配时的堆栈跟踪来识别它们。 每个内存分析可以以四种方式分解内存。

    • inuse_objects—分解活跃对象的数量。
    • inuse_space—按字节数分解活跃对象使用的内存量。
    • alloc_objects—分解自 Go 程序开始执行以来已分配的对象数量。
    • alloc_space—分解自 Go 程序开始执行以来分配的内存总量。

    切换这些不同的堆内存视图可以通过 pprof 工具的 -sample_index 标志，或者在工具交互使用时通过 sample_index 选项来完成。

    注意：内存分析默认仅对堆对象子集进行采样，因此不会包含每次堆分配的详细信息。但这足以识别性能热点。若需调整采样率，请参见runtime.MemProfileRate。

    为降低GC成本，alloc_space通常是最有用的视图，因为它直接对应分配速率。此视图能指出可带来最大优化收益的分配热点。

    逃逸分析

    通过堆分析确定候选堆分配点后，应如何消除它们？关键在于利用Go编译器的逃逸分析机制，让编译器为这些内存寻找更高效的存储方案（例如存储在goroutine栈中）。值得庆幸的是，Go编译器能够说明将Go值逃逸到堆的原因。掌握这些信息后，重组源代码以改变分析结果就成为关键（这通常是最大的挑战，但已超出本指南范围）。

    访问Go编译器逃逸分析信息最简单的方式是使用编译器支持的调试标志——它会以文本格式描述编译器对特定包应用或未应用的所有优化，包括值是否逃逸。尝试以下命令（其中[package]为Go包路径）： 

    $ go build -gcflags=-m=3 [package]

    此信息也可在支持LSP的编辑器中作为可视化覆盖层呈现，并可通过代码操作触发。 例如在VS Code中，执行"Source Action... > Show compiler optimization details"命令即可为当前包启用诊断提示（也可运行"Go: Toggle compiler optimization details"命令）。 通过以下配置项控制显示哪些注解：

    1. 通过将ui.diagnostic.annotations设置为包含escape来启用逃逸分析覆盖层。

    此外，Go编译器还提供机器可读（JSON）格式的信息，可用于构建自定义工具。详细信息请参阅Go源代码中的相关文档。

    特定实现的优化

    Go GC对活跃内存的分布情况敏感，因为复杂的对象和指针图不仅会限制并行性，还会增加GC的工作量。因此GC针对某些常见结构进行了优化。以下列出对性能优化最直接有效的几项：

    注意：应用以下优化可能会降低代码可读性（掩盖设计意图），且可能无法跨Go版本持续有效。建议仅在关键位置应用此类优化。可通过识别成本章节中的工具定位这些关键位置。

    • 无指针值与其他值被分区存储。

      因此，从不严格需要指针的数据结构中移除指针可能具有优势，因为这能减少GC对程序造成的缓存压力。采用基于索引而非指针的数据结构（尽管类型安全性较低）可能带来性能提升。仅当对象图明显复杂且GC在标记扫描上耗费大量时间时才值得采用此策略。

    • GC将在值中最后一个指针处停止扫描。

      因此，将结构体类型的值中的指针字段集中在开头可能具有优势。仅当应用程序明显将大量时间用于标记扫描时才值得采用此策略。（理论上编译器可自动执行此优化，但目前尚未实现，结构体字段按源代码中声明的顺序排列。）

    此外，GC几乎必须与它遇到的每个指针交互，因此例如使用切片的索引代替指针，有助于降低GC成本。

    Linux透明大页（THP）

    当程序访问内存时，CPU需要将所使用的虚拟内存地址转换为物理内存地址。为此CPU会查阅页表——这是由操作系统管理的、表示虚拟内存到物理内存映射的数据结构。页表中的每个条目代表一个不可分割的物理内存块（称为页），页表因此得名。

    透明大页（THP）是Linux的一项功能，它透明地用称为大页的更大内存块替换支撑连续虚拟内存区域的物理内存页。通过使用更大的块，表示相同内存区域所需的页表条目更少，从而改善页表查找时间。但更大的块也意味着若系统仅使用大页的一小部分，则会造成更多浪费。

    在生产环境中运行Go程序时，在Linux上启用透明大页可以提升吞吐量并降低延迟，但代价是增加额外内存使用。堆内存较小的应用程序通常无法从THP中获益，反而可能因额外内存占用高达50%而产生负面影响。然而，对于堆内存较大（1 GiB或以上）的应用程序，THP通常能带来显著收益（吞吐量提升可达10%），而额外内存开销却相对较低（仅1-2%或更少）。无论何种情况，了解您的THP配置都至关重要，建议通过实验确定最佳设置。

    在Linux环境中，您可以通过修改/sys/kernel/mm/transparent_hugepage/enabled来启用或禁用透明大页。更多详情请参阅官方Linux管理指南。若您选择在Linux生产环境中启用透明大页，针对Go程序我们建议采用以下附加设置：

    • 将/sys/kernel/mm/transparent_hugepage/defrag设置为defer或defer+madvise。

      该设置控制Linux内核将常规页面合并为大页的积极程度。defer指示内核延迟并在后台合并大页。更积极的合并策略可能在内存受限系统中引发停顿，反而损害应用程序延迟性能。defer+madvise类似于defer，但对系统中其他显式请求大页且需要大页提升性能的应用程序更为友好。

    • 将/sys/kernel/mm/transparent_hugepage/khugepaged/max_ptes_none设置为0。

      该设置控制Linux内核守护进程在尝试分配大页时可额外分配的页面数量。默认设置处于最大积极模式，常常会抵消Go运行时将内存归还操作系统的工作。在Go 1.21之前，Go运行时曾试图缓解默认设置的负面影响，但这会带来CPU开销。从Go 1.21+和Linux 6.2+开始，Go运行时不再修改大页状态。

      如果您在升级到Go 1.21.1或更高版本后遇到内存使用量增加的情况，请尝试应用此设置，这很可能解决您的问题。作为额外解决方案，您可以调用Prctl函数并传入PR_SET_THP_DISABLE以在进程级别禁用大页，或者设置GODEBUG=disablethp=1（将在Go 1.21.6和Go 1.22中新增）以禁用堆内存的大页支持。请注意，GODEBUG设置可能在后续版本中被移除。

    关于GOGC的补充说明

    GOGC章节提到，将GOGC值加倍会使堆内存开销翻倍，同时将GC的CPU开销减半。下面通过数学分解来说明原因。

    首先，堆目标设定了总堆内存大小的目标值。然而，这个目标主要影响新分配的堆内存，因为存活堆内存是应用程序固有的基础占用。

    目标堆内存 = 存活堆内存 + (存活堆内存 + GC根集) × GOGC / 100

    总堆内存 = 存活堆内存 + 新分配堆内存

    ⇒

    新分配堆内存 = (存活堆内存 + GC根集) × GOGC / 100

    由此可知，将GOGC值加倍也会使应用程序每个GC周期分配的新堆内存量翻倍，这正是堆内存开销的体现。注意存活堆内存 + GC根集近似表示GC需要扫描的内存量。

    接下来分析GC的CPU开销。总开销可分解为：每个周期的开销 × 一定时间段T内的GC频率。

    GC总CPU开销 = (每周期GC CPU开销) × (GC频率) × T

    每周期GC CPU开销可通过GC模型推导：

    每周期GC CPU开销 = (存活堆内存 + GC根集) × (每字节开销) + 固定开销

    注意此处忽略了清扫阶段开销，因为标记和扫描阶段开销占主导地位。

    稳态由恒定分配速率和恒定每字节开销定义，因此在稳态下可从新分配堆内存推导出GC频率：

    GC频率 = (分配速率) / (新分配堆内存) = (分配速率) / ((存活堆内存 + GC根集) × GOGC / 100)

    综合以上公式，得到总开销的完整表达式：

    GC总CPU开销 = (分配速率) / ((存活堆内存 + GC根集) × GOGC / 100) × ((存活堆内存 + GC根集) × (每字节开销) + 固定开销) × T

    对于足够大的堆（覆盖大多数情况），GC周期的边际开销远大于固定开销。这使得GC总CPU开销公式可大幅简化为：

    GC总CPU开销 = (分配速率) / (GOGC / 100) × (每字节开销) × T

    从这个简化公式可以看出，若将GOGC值翻倍，GC的总CPU开销将减半。 （请注意，本指南中的可视化模拟包含了固定开销，因此当GOGC翻倍时，GC报告的CPU开销不会精确减半。） 此外，GC的CPU开销主要取决于分配速率和扫描内存的字节成本。 关于如何具体降低这些开销的详细信息，请参阅优化指南。

    注意：存活堆内存的大小与GC实际需要扫描的内存数量之间存在差异：相同大小的存活堆但结构不同时，会导致不同的CPU开销，但内存成本相同，从而形成不同的权衡。 这就是为什么堆结构是稳态定义的一部分。 堆目标或许应只包含可扫描的存活堆，以更贴近GC需要扫描的内存量，但这会在可扫描存活堆极少而存活堆总体积较大的情况下导致退化行为。
