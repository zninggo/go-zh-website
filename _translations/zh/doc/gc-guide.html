引言

本指南旨在通过深入解析 Go 垃圾回收器，帮助高级 Go 用户更好地理解其应用的资源消耗。 它还为 Go 用户如何利用这些见解来改善其应用的资源利用率提供了指导。 本指南不假设读者具备任何垃圾回收方面的知识，但假定其熟悉 Go 编程语言。

Go 语言负责安排 Go 值的存储；在大多数情况下，Go 开发者无需关心这些值存储在哪里，或者为什么需要存储。 然而，在实践中，这些值通常需要存储在计算机的 物理内存 中，而物理内存是一种有限的资源。 由于其有限性，必须谨慎管理和回收内存，以避免在执行 Go 程序时耗尽内存。 根据需要分配和回收内存是 Go 实现的工作。

自动回收内存的另一个术语是 垃圾回收。 从宏观角度看，垃圾 回收器（或简称 GC）是一个系统，它通过识别内存中哪些部分不再需要，代表应用程序回收内存。 Go 标准工具链提供了一个随每个应用程序一起提供的运行时库，而这个运行时库包含了一个垃圾回收器。

请注意，Go 规范 并不保证本指南中描述的垃圾回收器的存在，它仅保证 Go 值的基础存储由语言自身管理。 这种省略是有意为之的，以便能够采用截然不同的内存管理技术。

因此，本指南是关于 Go 编程语言的一个特定实现，可能不适用于其他实现。 具体来说，以下指南适用于标准工具链（即 gc Go 编译器及相关工具）。 Gccgo 和 Gollvm 都使用非常相似的 GC 实现，因此许多相同的概念适用，但细节可能有所不同。

此外，本文档是动态更新的，将随着时间的推移而变化，以最佳地反映 Go 的最新发布版本。 本文档当前描述的是 Go 1.19 版本的垃圾回收器。

Go 值存储在哪里

在深入探讨 GC 之前，我们先讨论一下那些无需 GC 管理的内存。

例如，存储在局部变量中的非指针 Go 值很可能完全不会被 Go GC 管理，Go 会转而安排分配一块内存，该内存与其创建所在的词法作用域绑定。 通常，这比依赖 GC 更高效，因为 Go 编译器能够预先确定该内存何时可以被释放，并发出清理它的机器指令。 通常，我们把这种方式为 Go 值分配内存称为"栈分配"，因为空间存储在 goroutine 的栈上。

那些无法以这种方式分配内存的 Go 值，因为 Go 编译器无法确定其生命周期，被称为逃逸到堆上。 "堆"可以看作是内存分配的一个集散地，用于当 Go 值需要被放置在某个地方时。 在堆上分配内存的行为通常被称为"动态内存分配"，因为编译器和运行时都无法对该内存的使用方式和清理时机做出太多假设。 这正是 GC 的用武之地：它是一个专门识别并清理动态内存分配的系统。

Go 值需要逃逸到堆上的原因有很多。 一个原因可能是其大小是动态确定的。 例如，考虑一个切片的后备数组，其初始大小由一个变量而非常量决定。 请注意，逃逸到堆上也必须是传递性的：如果一个对 Go 值的引用被写入另一个已经被确定要逃逸的 Go 值中，那么该值也必须逃逸。

一个 Go 值是否逃逸取决于其使用的上下文以及 Go 编译器的逃逸分析算法。 试图精确列举何时发生逃逸是脆弱且困难的：算法本身相当复杂，并且在不同 Go 版本之间会发生变化。 关于如何识别哪些值逃逸、哪些不逃逸的更多细节，请参阅消除堆分配一节。

垃圾回收可能指多种不同的内存自动回收方法；例如引用计数。 在本文档的语境中，垃圾回收特指追踪式垃圾回收，它通过传递性地追踪指针来识别正在使用的（即活跃）对象。

让我们更严格地定义这些术语。

  • 对象——对象是一块动态分配的内存，包含一个或多个Go值。

  • 指针——引用对象中任意值的内存地址。 这自然包括形如*T的Go值，但也包含内置Go值的组成部分。 字符串、切片、通道、映射和接口值都包含垃圾回收器必须追踪的内存地址。

对象以及指向其他对象的指针共同构成了对象图。 为了识别活跃内存，垃圾回收器从程序的根（即确定正在被程序使用的对象指针）开始遍历对象图。 根的两个例子是局部变量和全局变量。 遍历对象图的过程称为扫描。 在Go文档中可能看到的另一个表述是对象是否“可达”，这只是意味着该对象可以通过扫描过程被发现。 另请注意，除一种特殊情况外，一旦内存变得不可达，它将始终保持不可达状态。

这种基本算法是所有追踪式垃圾回收器共有的。 追踪式垃圾回收器的区别在于它们发现内存是活跃后采取的行动。 Go的垃圾回收器使用标记-清除技术，这意味着为了跟踪进度，它还会将遇到的值标记为活跃。 一旦追踪完成，垃圾回收器将遍历堆中的所有内存，并将所有未标记的内存释放以供分配。 这个过程称为清除。

你可能熟悉的另一种技术是实际将对象移动到新的内存区域，并留下一个转发指针，该指针后续用于更新应用程序的所有指针。 我们将以这种方式移动对象的垃圾回收器称为移动式垃圾回收器；Go使用的是非移动式垃圾回收器。

垃圾回收周期

由于Go的垃圾回收器是标记-清除式垃圾回收器，它大致分为两个阶段运作：标记阶段和清除阶段。 虽然这个说法看似同义反复，但它包含了一个重要见解：在所有内存都被追踪之前，不可能将内存释放以供分配，因为可能仍有未扫描的指针使对象保持活跃。 因此，清除行为必须与标记行为完全分离。 此外，当没有垃圾回收相关工作时，垃圾回收器也可能根本不活动。 垃圾回收器在清除、关闭和标记这三个阶段之间不断循环，这被称为垃圾回收周期。 在本文档中，我们视垃圾回收周期从清除开始，然后关闭，接着标记。

接下来的几节将重点建立对垃圾回收成本的直觉，以帮助用户根据自身利益调整垃圾回收参数。

理解成本

垃圾回收器本质上是一个复杂的软件，它构建在更复杂的系统之上。 在尝试理解垃圾回收器和调整其行为时，很容易陷入细节。 本节旨在提供一个推理Go垃圾回收器成本及其调优参数的框架。

首先，考虑这个基于三个简单公理的垃圾回收成本模型。

  1. 垃圾回收只涉及两种资源：物理内存和CPU时间。

  2. 垃圾回收器的内存成本由活跃堆内存、标记阶段前分配的新堆内存以及元数据空间组成，尽管元数据空间可能与前两者成比例，但相对较小。

    第N个周期的GC内存成本 = 第N-1个周期的活跃堆内存 + 新堆内存

    活跃堆内存是由前一个垃圾回收周期确定为活跃的内存，而新堆内存是当前周期中分配的任何内存，这些内存在周期结束时可能活跃也可能不活跃。 在任何给定时间点有多少内存活跃是程序的属性，而非垃圾回收器能直接控制的。

  3. 垃圾回收器的CPU成本被建模为每个周期的固定成本，以及一个与活跃堆大小成正比的边际成本。

    第N个周期的GC CPU时间 = 每个周期的固定CPU时间成本 + 每字节的平均CPU时间成本 * 第N个周期中发现的活跃堆内存

    每个周期的固定CPU时间成本包括每周期固定发生次数的操作，例如为下一个垃圾回收周期初始化数据结构。 此成本通常很小，仅为完整性而包含。

    垃圾回收器的大部分CPU成本是标记和扫描，这由边际成本体现。 标记和扫描的平均成本取决于垃圾回收器的实现，也取决于程序的行为。 例如，更多的指针意味着更多的垃圾回收工作，因为至少垃圾回收器需要访问程序中的所有指针。 像链表和树这样的结构也更难以被垃圾回收器并行遍历，从而增加了每字节的平均成本。

    该模型忽略了清除成本，清除成本与总堆内存（包括已死亡的内存，这些内存必须被释放以供分配）成正比。 对于Go当前的垃圾回收器实现，清除速度远快于标记和扫描，因此其成本相比之下可以忽略不计。

这个模型简单而有效：它准确地归纳了垃圾回收器的主要成本构成。 它也告诉我们，垃圾回收器的总CPU成本取决于在给定时间范围内垃圾回收周期的总数。 最后，这个模型中嵌入了一个关于垃圾回收器的根本性时间/空间权衡。

为了理解其中的原因，让我们探讨一个受限但实用的场景：稳态。 从垃圾回收器的角度来看，应用程序的稳态由以下属性定义：

  • 应用程序分配新内存的速率（以每秒字节计）是恒定的。

    这意味着，从垃圾回收器的角度来看，应用程序的工作负载随时间推移大致保持不变。 例如，对于一个Web服务来说，这意味着恒定的请求速率，平均而言是同类型的请求，并且每个请求的平均生命周期大致保持不变。

  • 垃圾回收器的边际成本是恒定的。

    这意味着对象图的统计数据，如对象大小分布、指针数量和数据结构的平均深度，在每个周期之间保持不变。

让我们通过一个例子来说明。 假设某个应用程序在稳态下运行，分配速率为 10 MiB/s，而垃圾回收器可以以 100 MiB/CPU秒的速率扫描内存（此为假设数据）。 稳态对活跃堆的大小不做任何假设，但为简单起见，我们假设该应用程序的活跃堆始终为 10 MiB。 同样为了简单起见，我们假设固定的垃圾回收成本为零。 让我们来调整一下垃圾回收周期。

假设每个垃圾回收周期恰好在 1 个 CPU秒后发生。 那么，在每个垃圾回收周期结束时，我们的示例应用程序将分配了 10 MiB 的额外内存，导致总堆大小为 20 MiB。 并且在每个垃圾回收周期中，垃圾回收器将花费 0.1 CPU秒来扫描 10 MiB 的活跃堆，产生 10% 的 CPU 开销。 请记住，垃圾回收器只需要遍历活跃堆，而不是整个堆。 （注意：活跃堆恒定并不意味着所有新分配的内存都是已死亡的。 它意味着，在垃圾回收器运行后，某种组合的旧堆和新堆内存被回收，并且最终结果是每个周期发现 10 MiB 的活跃内存。）

现在假设每个垃圾回收周期的发生频率降低，每 2 个 CPU秒一次。 那么，在稳态下，我们的示例应用程序在每个垃圾回收周期时将拥有 30 MiB 的总堆大小，因为在这段时间内它将分配 20 MiB。 但在每个垃圾回收周期中，垃圾回收器仍然只需要 0.1 CPU秒来扫描 10 MiB 的活跃内存。 同样，我们假设活跃堆大小保持不变，无论分配了多少内存。 因此，这意味着我们的垃圾回收开销刚刚从 10% 降到了 5%，代价是多使用了 50% 的内存。

开销的这种变化就是前面提到的根本性时间/空间权衡。 而垃圾回收频率是这一权衡的核心： 如果我们更频繁地执行垃圾回收，那么我们使用的内存就更少，反之亦然。 但垃圾回收实际执行的频率是多少呢？ 在 Go 中，决定何时启动垃圾回收是用户可以控制的主要参数。

GOGC

在宏观层面上，GOGC 决定了垃圾回收 CPU 开销和内存使用之间的权衡。

它的运作方式是在每个垃圾回收周期结束后确定目标堆大小，即下一周期总堆大小的目标值。 垃圾回收器的目标是在总堆大小超过目标堆大小之前完成一次回收周期。 总堆大小定义为上一周期结束时的活跃堆大小，加上应用程序自上一周期以来分配的任何新堆内存。 同时，目标堆内存定义为：

目标堆内存 = 活跃堆 + (活跃堆 + GC根) * GOGC / 100

例如，考虑一个 Go 程序，其活跃堆大小为 8 MiB，goroutine 栈占用 1 MiB，全局变量中的指针占用 1 MiB。 那么，当 GOGC 值为 100 时，在下次垃圾回收运行之前将分配的新内存量为 10 MiB，即 10 MiB 工作量的 100%，总堆占用空间为 18 MiB。 当 GOGC 值为 50 时，将是 50%，即 5 MiB。 当 GOGC 值为 200 时，将是 200%，即 20 MiB。

注意：从 Go 1.18 开始，GOGC 才将根集（GC roots）包含在内。 此前，它只计算活跃堆。 通常，goroutine 栈中的内存量非常小，活跃堆大小主导了所有其他垃圾回收工作的来源，但在程序拥有数十万个 goroutine 的情况下，垃圾回收器的判断就会变得不准确。

堆目标控制着垃圾回收频率：目标越大，垃圾回收器可以等待越久才开始下一个标记阶段，反之亦然。 虽然精确的公式对于估算很有用，但最好从 GOGC 的根本目的来理解它：一个在垃圾回收 CPU 开销和内存使用之间进行权衡的选择点。 关键要点是，将 GOGC 加倍会使堆内存开销加倍，并大致将垃圾回收 CPU 成本减半，反之亦然。 （要查看完整解释，请参阅附录。）

注意：目标堆大小只是一个目标，有几个原因可能导致垃圾回收周期并非恰好在该目标处完成。 一方面，一次足够大的堆分配可能直接超出目标。 然而，其他原因出现在本指南迄今使用的垃圾回收模型之外的垃圾回收器实现中。 更多细节请参阅延迟部分，但完整细节可在附加资源中找到。

可以通过两种方式配置 GOGC：使用所有 Go 程序都识别的 GOGC 环境变量，或者使用 runtime/debug 包中的 SetGCPercent API。

请注意，GOGC 也可用于完全关闭垃圾回收（前提是内存限制不适用），方法是设置 GOGC=off 或调用 SetGCPercent(-1)。 从概念上讲，此设置等同于将 GOGC 设置为无穷大，因为在触发垃圾回收之前可以分配的新内存量是无限制的。

请注意，对于某些 GOGC 和内存限制的值，峰值内存使用量会停在设定的内存限制处，但程序其余部分的执行仍然遵循 GOGC 设定的总堆大小规则。

这个观察引出了另一个有趣的细节：即使 GOGC 设置为 off（关闭），内存限制仍然会被遵守！事实上，这种特定配置代表了对资源经济性的最大化，因为它设置了维持某个内存限制所需的最低垃圾回收频率。在这种情况下，程序执行的整个过程中，堆大小都会上升直至触及内存限制。

现在，虽然内存限制显然是一个强大的工具，但使用内存限制并非没有代价，当然也并不会使 GOGC 失去作用。

考虑一下当活跃堆增长到足以使总内存使用量接近内存限制时会发生什么。 在上面的稳态可视化中，尝试关闭 GOGC，然后逐渐调低内存限制，观察会发生什么。 注意，随着 GC 为维持一个不可能达到的内存限制而不断执行，应用程序所花费的总时间将开始以无界的方式增长。

这种情况，即程序因持续的 GC 周期而未能取得合理进展，被称为抖动。 这尤其危险，因为它实际上会使程序停滞。 更糟糕的是，这可能恰好发生在我们试图通过 GOGC 避免的同一种情况下：足够大的瞬时堆尖峰可能导致程序无限期停滞！ 尝试在瞬时堆尖峰可视化中降低内存限制（约 30 MiB 或更低），注意最糟糕的行为是如何从堆尖峰开始的。

在许多情况下，无限期停滞比内存溢出（OOM）状况更糟，后者通常会导致更快的故障。

因此，内存限制被定义为软限制。 Go 运行时不保证在所有情况下都会维持此内存限制；它只承诺会做出一些合理的努力。 对内存限制的这种放宽对于避免抖动行为至关重要，因为它给了 GC 一条出路：允许内存使用量超过限制，以避免在 GC 上花费过多时间。

其内部工作原理是，GC 对其在某个时间窗口内可使用的 CPU 时间量设置了上限（对于非常短暂的 CPU 使用尖峰有一些滞后处理）。 此限制目前设定为大约 50%，CPU 秒窗口为 2 * GOMAXPROCS。 限制 GC CPU 时间的结果是，GC 的工作被延迟，同时 Go 程序可能继续分配新的堆内存，甚至超过内存限制。

50% GC CPU 限制背后的直觉基于对具有充足可用内存的程序的最坏情况影响。 在内存限制配置错误（被错误地设置得过低）的情况下，程序最多只会慢 2 倍，因为 GC 不能占用超过其 50% 的 CPU 时间。

注意：本页的可视化未模拟 GC CPU 限制。

建议用法

虽然内存限制是一个强大的工具，而且 Go 运行时采取措施来减轻误用导致的最坏行为，但谨慎使用它仍然很重要。 以下是一系列关于内存限制在哪些场景最有用、最适用，以及在哪些场景可能弊大于利的建议。

  • 应该在你的 Go 程序的执行环境完全在你控制之下，并且该 Go 程序是唯一有权访问某组资源（例如某种内存预留，如容器内存限制）的程序时，利用内存限制。

    一个很好的例子是将 Web 服务部署到具有固定可用内存的容器中。

    在这种情况下，一个好的经验法则是额外预留 5-10% 的空间，以应对 Go 运行时未意识到的内存来源。

  • 应该随时调整内存限制以适应不断变化的条件。

    一个很好的例子是 cgo 程序，其中 C 库临时需要使用大量内存。

  • 不应该在 Go 程序可能与其他程序共享其部分有限内存，并且这些程序通常与 Go 程序解耦的情况下，将 GOGC 设置为关闭并同时设置内存限制。 相反，应保留内存限制，因为它可能有助于抑制不良的瞬态行为，但将 GOGC 设置为针对平均情况的一个更小、合理的值。

    虽然可能很想尝试为共租户程序“预留”内存，但除非程序是完全同步的（例如，Go 程序调用某个子进程并在其被调用者执行时阻塞），否则结果将不太可靠，因为不可避免地两个程序都需要更多内存。 当 Go 程序不需要时让它使用更少的内存，整体上会产生更可靠的结果。 此建议也适用于超量分配的情况，即在一台机器上运行的容器内存限制总和可能超过该机器实际可用的物理内存。

  • 不应该在部署到你无法控制的执行环境时使用内存限制，尤其是当你的程序的内存使用量与其输入成比例时。

    一个很好的例子是 CLI 工具或桌面应用程序。 当不清楚程序可能接收何种输入，或者系统上有多少可用内存时，在程序中固化一个内存限制，可能导致令人困惑的崩溃和性能下降。 此外，高级终端用户如果愿意，总是可以自行设置内存限制。

  • 不应该在程序已经接近其环境内存限制时设置内存限制以避免内存溢出条件。

    这实际上将内存溢出风险替换为应用程序严重减速的风险，即使 Go 为减轻抖动做出了努力，这通常也不是一个有利的权衡。 在这种情况下，增加环境的内存限制（然后可能再设置内存限制）或减小 GOGC（它比抖动缓解提供了更清晰的权衡）会有效得多。

延迟

本文档中的可视化模型将应用程序建模为在GC执行期间暂停的状态。 确实存在这样行为的GC实现，它们被称为"全暂停"（stop-the-world）GC。

然而，Go的GC并非完全暂停式的，它大部分工作是与应用程序并发执行的。 这主要是为了减少应用程序的延迟。 具体来说，是指单个计算单元（例如一个Web请求）的端到端持续时间。 到目前为止，本文档主要考虑了应用程序的吞吐量（例如每秒处理的Web请求数）。 请注意，GC周期一节中的每个示例都关注执行程序的总CPU持续时间。 然而，这种持续时间对于例如Web服务来说意义要小得多。 虽然吞吐量对Web服务仍然很重要（即每秒查询数），但通常每个单独请求的延迟甚至更为重要。

就延迟而言，全暂停GC可能需要相当长的时间来执行其标记（mark）和清扫（sweep）阶段，在此期间应用程序——在Web服务的上下文中，即任何正在处理的请求——无法取得进一步进展。 相反，Go的GC避免使任何全局应用程序暂停的长度与堆的大小成比例，并且核心追踪算法是在应用程序积极执行时完成的。 （在算法上，暂停与GOMAXPROCS有更强的比例关系，但最常见的影响因素是停止运行中的goroutine所需的时间。） 并发收集并非没有代价：在实践中，它通常导致设计出比等效的全暂停垃圾回收器吞吐量更低的系统。 然而，重要的是要注意，较低的延迟并不固有地意味着较低的吞吐量，并且Go垃圾回收器的性能随着时间的推移在延迟和吞吐量方面都得到了稳步改善。

Go当前GC的并发特性并不使本文档迄今讨论的任何内容失效：没有任何陈述依赖于这一设计选择。 GC频率仍然是GC在吞吐量方面权衡CPU时间和内存的主要方式，实际上，它在延迟方面也扮演着这一角色。 这是因为GC的大部分成本是在标记阶段活动期间产生的。

那么，关键要点是，降低GC频率也可能带来延迟改善。 这不仅适用于通过调整参数（如增加GOGC和/或内存限制）来降低GC频率，也适用于优化指南中描述的优化。

然而，延迟通常比吞吐量更难理解，因为它是程序每时每刻执行的产物，而不仅仅是成本的聚合。 因此，延迟与GC频率之间的联系不那么直接。 以下是为那些希望深入探究的人列出的一些可能的延迟来源。

  1. GC在标记和清扫阶段之间切换时的短暂全暂停，
  2. 当GC处于标记阶段时，由于其占用25%的CPU资源而导致的调度延迟，
  3. 用户goroutine为应对高分配率而辅助GC，
  4. 当GC处于标记阶段时，指针写入需要额外工作，以及
  5. 正在运行的goroutine必须被暂停以扫描其根。

这些延迟来源可以在执行跟踪中看到，除了需要额外工作的指针写入。

终接器、清理器与弱指针

垃圾回收利用有限的内存提供无限内存的假象。 内存被分配但从未被显式释放，与基础的手动内存管理相比，这支持了更简单的API和并发算法。 （一些具有手动内存管理的语言使用替代方法，如"智能指针"和编译时所有权跟踪来确保对象被释放，但这些特性深深嵌入到这些语言的API设计惯例中。）

只有活跃对象——即可以从全局变量或某个goroutine中的计算访问到的对象——才能影响程序的行为。 在对象变得不可达（"死亡"）之后的任何时间，它都可以被GC安全地回收。 这允许多种多样的GC设计，例如Go今天使用的追踪设计。 对象的死亡在语言层面不是一个可观察的事件。

然而，Go的运行时库提供了三个打破这种假象的特性： 清理器、 弱指针和 终接器。 这些特性中的每一个都提供了某种观察和响应对象死亡的方式，对于终接器，甚至可以逆转死亡。 这当然使Go程序复杂化，并给GC实现增加了额外负担。 尽管如此，这些特性存在是因为它们在各种情况下很有用，Go程序一直在使用它们并从中受益。

有关每个特性的详细信息，请参考其包文档 （runtime.AddCleanup、 weak.Pointer、 runtime.SetFinalizer）。 以下是关于使用这些特性的一些通用建议，概述了使用每个特性时可能遇到的常见问题，以及测试这些特性用法的建议。

通用建议

  • 编写单元测试。

    清理函数、弱指针和终结器的确切执行时机可能难以预测，而且很容易让人在连续执行多次后误以为一切正常。但同样容易犯下细微的错误。为它们编写测试可能比较棘手，但鉴于它们的用法如此微妙，测试就显得比平常更加重要。

  • 避免在典型的Go代码中直接使用这些特性。

    这些是低层级的特性，附带微妙的限制和行为。例如，不能保证清理函数或终结器会在程序退出时运行，或者根本就运行不了。它们API文档中的长篇注释应被视为警告。绝大多数Go代码并不会直接从使用这些特性中受益，只有间接使用才会受益。

  • 将这些机制的使用封装在一个包内。

    在可能的情况下，不要让这些机制的使用泄露到包的公共API中；提供接口，使用户难以或不可能误用它们。例如，与其要求用户对某个C分配的内存设置清理函数来释放它，不如编写一个包装包，并将该细节隐藏在其中。

  • 将具有终结器、清理函数和弱指针的对象的访问权限限制在创建并应用它们的包内。

    这与前一点相关，但值得单独强调，因为这是使用这些特性并以更不易出错的方式的一种非常强大的模式。例如，unique 包在底层使用了弱指针，但完全封装了被弱引用指向的对象。这些值不能被应用程序的其他部分修改，只能通过Value 方法复制，从而为包用户营造出内存无限的假象。

  • 尽可能优先确定性地清理非内存资源，将终结器和清理函数作为后备手段。

    清理函数和终结器非常适合内存资源，例如外部（如C语言）分配的内存，或对 mmap 映射的引用。C的 malloc 分配的内存最终必须由C的 free 释放。一个调用 free 的终结器，附加到C内存的包装对象上，是确保C内存在垃圾回收时最终被回收的合理方式。

    然而，非内存资源，如文件描述符，往往受到系统限制，而Go运行时通常对此一无所知。此外，在给定Go程序中，垃圾回收器的执行时机通常是包作者无法控制的（例如，GOGC 控制了GC的运行频率，而运维人员在实践中可以将其设置为多种不同的值）。这两个事实共同导致清理函数和终结器不适合作为释放非内存资源的唯一机制。

    如果你是包作者，提供了一个包装了某种非内存资源的API，请考虑提供一个显式的API来确定性地释放该资源（通过 Close 方法或类似方式），而不是依赖于通过清理函数或终结器来进行垃圾回收。相反，应优先将清理函数和终结器用作对程序员错误的最佳努力处理程序，要么像 os.File 那样无论如何都清理资源，要么将未能确定性清理的失败报告给用户。

  • 优先使用清理函数而非终结器。

    历史上，添加终结器是为了简化Go代码与C代码之间的接口，并清理非内存资源。其预期用途是将其应用于拥有C内存或其他非内存资源的包装对象上，以便在Go代码使用完毕后释放该资源。这些原因至少部分解释了为什么终结器的作用域很窄，为什么任何给定对象只能有一个终结器，以及为什么该终结器只能附加到对象的第一个字节。这种限制已经扼杀了一些用例。例如，任何希望在内部缓存传入对象信息的包，一旦对象消失，就无法清理该信息。

    但更糟糕的是，终结器效率低下且容易出错，因为它们会使所附加的对象复活，以便将其传递给终结器函数（甚至在那之后也能继续存活）。这个简单的事实意味着，如果对象是引用循环的一部分，它将永远不会被释放，并且支持该对象的内存至少要等到下一个垃圾回收周期才能被重用。

    但也正因为终结器会使对象复活，它们的执行顺序比清理函数定义得更好。因此，对于具有复杂销毁顺序要求的结构体，终结器仍然可能（但很少）有用。

    但在Go 1.24及更高版本中，对于所有其他用途，我们建议使用清理函数，因为它们比终结器更灵活、不易出错且更高效。

常见的清理问题

  • 附有清理函数的对象不得被清理函数本身引用（例如，通过捕获的局部变量）。这将阻止对象被回收，且清理函数永远无法运行。 

    • f := new(myFile)
f.fd = syscall.Open(...)
runtime.AddCleanup(f, func(fd int) {
	syscall.Close(f.fd) // 错误：我们引用了 f，所以此清理函数不会运行！
}, f.fd)

  • 附有清理函数的对象不得被清理函数的参数引用。这将阻止对象被回收，且清理函数永远无法运行。 

    • f := new(myFile)
f.fd = syscall.Open(...)
runtime.AddCleanup(f, func(f *myFile) {
	syscall.Close(f.fd)
}, f) // 错误：我们引用了 f，所以此清理函数永远不会运行。此特定情况还会引发 panic。

  • 终结器有明确的执行顺序，但清理函数没有。清理函数也可以彼此并发运行。

  • 耗时长的清理函数应创建一个新的 goroutine，以避免阻塞其他清理函数的执行。

  • runtime.GC 不会等待不可达对象的清理函数执行完毕，只会等待它们全部进入队列。

常见的弱指针问题

  • 弱指针可能会在意外的时间开始从其 Value 方法返回 nil。务必在调用 Value 时检查是否为 nil，并准备后备方案。

  • 当弱指针用作 map 的键时，它们不会影响 map 值的可达性。因此，如果一个弱指针 map 键指向的对象也能通过 map 值被访问到，那么该对象仍将被视为可达。

常见的终结器问题

  • 附有终结器的对象不得通过任何路径可达自身（换句话说，它们不能存在于引用循环中）。这将阻止对象被回收，且终结器永远无法运行。 

    • f := new(myCycle)
f.self = f // 错误：f 可从 f 访问，所以此终结器永远不会运行。
runtime.SetFinalizer(f, func(f *myCycle) {
	...
})

  • 附有终结器的对象不得被终结器函数本身引用（例如，通过捕获的局部变量）。这将阻止对象被回收，且终结器永远无法运行。 

    • f := new(myFile)
f.fd = syscall.Open(...)
runtime.SetFinalizer(f, func(_ *myFile) {
	syscall.Close(f.fd) // 错误：我们引用了外部的 f，所以此清理函数不会运行！
})

  • 附有终结器的对象的引用链（例如，在链表中）需要至少与链中对象数量一样多的 GC 周期才能全部清理完毕。请让终结器保持浅层次！ 

    • // 错误：回收此链表至少需要 10 个 GC 周期。
node := new(linkedListNode)
for range 10 {
	tmp := new(linkedListNode)
	tmp.next = node
	node = tmp
	runtime.SetFinalizer(node, func(node *linkedListNode) {
		...
	})
}

  • 避免为在包边界返回的对象附加终结器。这使得你的包的用户可以调用 runtime.SetFinalizer 来更改你返回对象上的终结器，这可能成为你包的用户最终依赖的意外行为。

  • 耗时长的终结器应创建一个新的 goroutine，以避免阻塞其他终结器的执行。

  • runtime.GC 不会等待不可达对象的终结器执行完毕，只会等待它们全部进入队列。

测试对象生命周期终结

使用这些特性时，为使用它们的代码编写测试有时会很棘手。以下是为使用这些特性的代码编写稳健测试的一些技巧。

  • 避免与其他测试并行运行此类测试。尽可能增加确定性并随时掌握状态的全局情况会大有帮助。
  • 使用 runtime.GC 在进入测试时建立基线。使用 runtime.GC 强制将弱指针置为 nil，并将清理函数和终结器排入队列等待运行。

  • runtime.GC 不会等待清理函数和终结器运行，它只是将它们排入队列。

    为了编写尽可能稳健的测试，可以从你的测试中注入一种方法来阻塞清理函数或终结器的执行（例如，从测试中传递一个可选的 channel 给清理函数和/或终结器，并在其执行完成后向 channel 写入数据）。如果这太困难或不可能，另一种方法是循环等待某个清理后的特定状态变为真。例如，os 包的测试在循环中调用 runtime.Gosched 来检查文件是否已被关闭，一旦它变为不可达。

  • 如果编写使用终结器的测试，并且你有一个使用终结器的对象链，你将需要至少与测试所能创建的最深链长度相等的 runtime.GC 调用次数，以确保所有终结器都运行。

  • 在竞争检测模式下进行测试，以发现并发清理函数之间，以及清理函数、终结器代码与其余代码库之间的竞争条件。

关于虚拟内存的说明

本指南主要关注 GC 的物理内存使用量，但经常出现的一个问题是：这具体是什么意思，以及它与虚拟内存（通常在 top 等程序中显示为 "VSS"）相比如何。

物理内存是指大多数计算机中实际物理 RAM 芯片上的内存。 虚拟内存是操作系统提供的一种对物理内存的抽象，用于隔离程序。 程序保留不映射到任何物理地址的虚拟地址空间通常也是被允许的。

由于虚拟内存只是操作系统维护的一种映射，通常只需很小的代价就能进行大量不映射物理内存的虚拟内存保留。

Go 运行时在某种程度上依赖于这种对虚拟内存成本的看法，具体体现在以下几个方面：

  • Go 运行时从不删除它映射的虚拟内存。 相反，它使用大多数操作系统提供的特殊操作来显式释放与某个虚拟内存范围相关联的任何物理内存资源。

    此技术被明确用于管理内存限制，并将 Go 运行时不再需要的内存归还给操作系统。 Go 运行时也会在后台持续释放它不再需要的内存。 更多信息请参见附加资源。

  • 在 32 位平台上，Go 运行时会预先为堆保留 128 MiB 到 512 MiB 的地址空间，以限制内存碎片问题。

  • Go 运行时在实现若干内部数据结构时使用了大的虚拟地址空间保留。 在 64 位平台上，这些通常具有约 700 MiB 的最小虚拟内存占用。 在 32 位平台上，其占用可忽略不计。

因此，像 top 中的 "VSS" 这样的虚拟内存指标对于理解 Go 程序的内存占用通常不是很有用。 相反，应关注 "RSS" 及类似的测量指标，它们更能直接反映物理内存的使用情况。

优化指南

识别成本

在尝试优化你的 Go 应用程序与 GC 的交互方式之前，首先要确认 GC 确实是一个主要的成本来源，这一点很重要。

Go 生态系统提供了多种工具来识别成本和优化 Go 应用程序。 关于这些工具的简要概述，请参阅诊断指南。 这里，我们将专注于这些工具的一个子集，以及应用它们的合理顺序，以理解 GC 的影响和行为。

  1. CPU 分析

    一个好的起点是进行 CPU 分析。 CPU 分析提供了 CPU 时间花在何处的概览，尽管对于未经训练的人来说，可能很难识别 GC 在特定应用程序中扮演角色的份量。 幸运的是，理解 GC 的角色主要归结为了解 `runtime` 包中不同函数的含义。 以下是用于解读 CPU 分析的一些实用函数子集。

    请注意，下面列出的函数不是叶函数，因此在 pprof 工具使用 top 命令提供的默认视图中可能不会显示。 相反，请使用 top -cum 命令，或者直接对这些函数使用 list 命令，并关注累积百分比列。

  • runtime.gcBgMarkWorker：后台标记工作者 goroutine 的入口点。 在此处花费的时间与 GC 频率以及对象图的复杂性和大小成正比。 它代表了应用程序在标记和扫描方面所花费时间的基础线。

    请注意，在这些 goroutine 内部，您会发现对 runtime.gcDrainMarkWorkerDedicated、 runtime.gcDrainMarkWorkerFractional 和 runtime.gcDrainMarkWorkerIdle 的调用， 这些调用指示了工作者的类型。 在大部分时间空闲的 Go 应用程序中，Go 垃圾回收器（GC）会利用额外的（空闲）CPU 资源来更快地完成其工作， 这由 runtime.gcDrainMarkWorkerIdle 符号表示。 因此，此处的时间可能占据了很大比例的 CPU 采样，Go GC 认为这些采样是空闲的。 如果应用程序变得更加活跃，空闲工作者的 CPU 时间将会减少。 这种情况发生的一个常见原因是应用程序完全在一个 goroutine 中运行，但 GOMAXPROCS 设置大于 1。

  • runtime.mallocgc：堆内存的内存分配器入口点。 在此处累计花费大量时间（>15%）通常表明有大量的内存正在被分配。

  • runtime.gcAssistAlloc：goroutine 进入此函数，将其部分时间让出以协助 GC 进行扫描和标记。 在此处累计花费大量时间（>5%）表明应用程序的内存分配速度可能超过了 GC 的处理速度。 这表明 GC 的影响特别高，同时也代表了应用程序在标记和扫描方面花费的时间。 请注意，这包含在 runtime.mallocgc 的调用树中，因此也会增加其累计时间。

  • 执行追踪

    虽然 CPU 分析非常适合识别总体时间花在哪里，但对于指示更细微、罕见或特别与延迟相关的性能成本，它们的用处较小。 另一方面，执行追踪提供了对 Go 程序执行的一个短时间窗口的丰富而深入的视角。 它们包含与 Go GC 相关的各种事件，并且可以直接观察到特定的执行路径，以及应用程序可能如何与 Go GC 交互。 所有被跟踪的 GC 事件在追踪查看器中都被方便地标注出来。

    有关如何开始使用执行追踪，请参阅 runtime/trace 包的文档。

  • GC 追踪

    当其他方法都失败时，Go GC 提供了一些不同的特定追踪，可以更深入地了解 GC 行为。 这些追踪总是直接打印到标准错误输出（STDERR），每个 GC 周期一行，并通过所有 Go 程序都识别的 GODEBUG 环境变量进行配置。 它们主要用于调试 Go GC 本身，因为需要熟悉 GC 实现的具体细节，但偶尔也可以用于更好地理解 GC 行为。

    核心 GC 追踪通过设置 GODEBUG=gctrace=1 来启用。 此追踪产生的输出文档记录在 runtime 包文档的环境变量部分。

    一个称为“节奏器追踪”（pacer trace）的补充 GC 追踪可以提供更深入的见解，通过设置 GODEBUG=gcpacertrace=1 来启用。 解释此输出需要理解 GC 的“节奏器”（参见额外资源），这超出了本指南的范围。

    • 消除堆分配

    减少 GC 开销的一种方法是让 GC 管理更少的值。 下面描述的技术可以带来一些最大的性能提升，因为正如GOGC 章节所展示的，Go 程序的分配速率是 GC 频率（本指南使用的关键成本指标）的一个主要因素。

    堆分析

    在确认 GC 是显著开销来源之后，消除堆分配的下一步是找出大部分堆分配来自何处。 为此，内存分析（实际上是堆内存分析）非常有用。 请查看文档了解如何开始使用它们。

    内存分析描述了程序中堆分配发生的位置，通过分配时的堆栈跟踪来识别它们。 每个内存分析可以从四个方面分解内存。

    • inuse_objects — 分解存活对象的数量。
    • inuse_space — 按字节数分解存活对象所使用的内存。
    • alloc_objects — 分解自 Go 程序开始执行以来已分配的对象数量。
    • alloc_space — 分解自 Go 程序开始执行以来已分配的内存总量。

    可以通过 pprof 工具的 -sample_index 标志，或在交互式使用该工具时通过 sample_index 选项来切换堆内存的这些不同视图。

    注意：内存剖析默认只对堆对象的一部分进行采样，因此不会包含每个堆分配的信息。但这足以发现热点。 要更改采样率，请参阅 runtime.MemProfileRate。

    就降低垃圾回收成本而言，alloc_space 通常是最有用的视图，因为它直接对应分配速率。这个视图将指出那些能带来最大收益的分配热点。

    逃逸分析

    一旦通过堆剖析识别出候选的堆分配点，该如何消除它们呢？关键在于利用 Go 编译器的逃逸分析，让 Go 编译器为这些内存寻找替代且更高效的存储位置，例如存储在 goroutine 栈上。 幸运的是，Go 编译器能够描述它决定将某个 Go 值逃逸到堆上的原因。了解了这一点，剩下的工作就是重新组织你的源代码以改变分析结果（这通常是最难的部分，但超出了本指南的范围）。

    至于如何获取 Go 编译器逃逸分析的信息，最简单的方法是通过 Go 编译器支持的一个调试标志，它以文本格式描述编译器对某个包应用或未应用的所有优化，这包括值是否逃逸。 尝试以下命令，其中 [package] 是某个 Go 包路径。 

    $ go build -gcflags=-m=3 [package]

    此信息也可以在支持 LSP 的编辑器中以覆盖层的形式可视化；它作为代码操作公开。 例如，在 VS Code 中，调用 "Source Action... > Show compiler optimization details" 命令可以为当前包启用诊断。（你也可以运行 "Go: Toggle compiler optimization details" 命令。） 使用以下配置设置来控制显示哪些注解：

    1. 通过设置 ui.diagnostic.annotations 以包含 escape 来启用逃逸分析的覆盖层。

    最后，Go 编译器以机器可读的（JSON）格式提供此信息，可用于构建额外的自定义工具。 有关此方面的更多信息，请参阅 Go 源代码中的文档。

    特定于实现的优化

    Go 垃圾回收器对存活内存的构成很敏感，因为复杂的对象和指针图既限制了并行性，又给垃圾回收器产生了更多工作。因此，垃圾回收器包含了一些针对特定常见结构的优化。下面列出了对性能优化最直接有用的几种。

    注意：应用下面的优化可能会通过模糊意图而降低代码的可读性，并且可能在不同的 Go 版本之间无法保持。建议仅在最重要的地方应用这些优化。可以使用识别成本部分中列出的工具来确定这些地方。

    • 无指针的值与其他值是分开存储的。

      因此，消除那些并非严格需要指针的数据结构中的指针可能是有利的，因为这减少了垃圾回收器对程序施加的缓存压力。结果，依赖索引而非指针值的数据结构，虽然类型化程度较低，但可能性能更好。只有当对象图复杂且垃圾回收器花费大量时间进行标记和扫描时，才值得这样做。

    • 垃圾回收器将在值中的最后一个指针处停止扫描。

      因此，将结构体类型值中的指针字段分组到值的开头可能是有利的。只有当应用程序确实花费大量时间在标记和扫描上时，才值得这样做。（理论上编译器可以自动完成此操作，但目前尚未实现，结构体字段按照源代码中的书写顺序排列。）

    此外，垃圾回收器必须与它看到的几乎每个指针交互，因此使用索引进入切片（slice），而不是使用指针，有助于降低垃圾回收成本。

    Linux 透明大页（THP）

    当程序访问内存时，CPU 需要将其使用的虚拟内存地址转换为物理内存地址，以指向它试图访问的数据。为此，CPU 会查询“页表”，这是一个表示虚拟内存到物理内存映射的数据结构，由操作系统管理。页表中的每个条目代表一个称为“页”的不可分割的物理内存块，因此得名。

    透明大页（THP）是 Linux 的一项功能，它透明地将支撑连续虚拟内存区域的物理内存页替换为称为“大页”的更大内存块。通过使用更大的块，表示相同的内存区域需要的页表条目更少，从而提高了页表查找时间。然而，更大的块意味着如果系统只使用了大页的一小部分，则会产生更多的浪费。

    在生产环境中运行 Go 程序时，在 Linux 上启用透明大页可以在增加内存使用的前提下，提高吞吐量并降低延迟。小堆内存的应用程序往往无法从 THP 中受益，并且最终可能会额外占用大量内存（高达 50%）。然而，大堆内存（1 GiB 或更多）的应用程序通常会获益良多（吞吐量提升高达 10%），而额外的内存开销却很小（1-2% 或更少）。无论哪种情况，了解您的 THP 设置都会有所帮助，并且总是建议进行实验验证。

    您可以通过修改 /sys/kernel/mm/transparent_hugepage/enabled 在 Linux 环境中启用或禁用透明大页。有关更多详情，请参阅官方 Linux 管理员指南。如果您选择让您的 Linux 生产环境启用透明大页，我们建议为 Go 程序采用以下附加设置。

    • 将 /sys/kernel/mm/transparent_hugepage/defrag 设置为 defer 或 defer+madvise。

      此设置控制 Linux 内核将普通页合并为大页的积极程度。defer 告诉内核惰性地并在后台合并大页。更积极的设置可能导致在内存受限的系统中出现停顿，并且常常会损害应用程序的延迟性能。defer+madvise 与 defer 类似，但对系统中其他显式请求大页并需要大页来保证性能的应用更为友好。

    • 将 /sys/kernel/mm/transparent_hugepage/khugepaged/max_ptes_none 设置为 0。

      此设置控制在尝试分配一个大页时，Linux 内核守护进程可以分配多少额外的页面。默认设置是最大化积极策略，并且常常会抵消 Go 运行时为将内存归还给操作系统所做的工作。在 Go 1.21 之前，Go 运行时会尝试缓解默认设置的负面影响，但这会带来 CPU 开销。在 Go 1.21+ 和 Linux 6.2+ 中，Go 运行时不再修改大页状态。

      如果您在升级到 Go 1.21.1 或更高版本时遇到内存使用量增加的情况，请尝试应用此设置；它很可能会解决您的问题。作为额外的变通方法，您可以调用 Prctl 函数 并使用 PR_SET_THP_DISABLE 在进程级别禁用大页，或者您可以设置 GODEBUG=disablethp=1（将在 Go 1.21.6 和 Go 1.22 中添加）来为堆内存禁用大页。请注意，GODEBUG 设置可能在未来版本中移除。

    关于 GOGC 的补充说明

    GOGC 章节声称将 GOGC 翻倍会使堆内存开销翻倍，并使垃圾回收的 CPU 开销减半。为了理解其原因，让我们从数学上进行分解。

    首先，堆目标为总堆大小设定了一个目标。然而，这个目标主要影响新分配的堆内存，因为存活堆是应用程序的基础。

    目标堆内存 = 存活堆 + (存活堆 + GC 根) * GOGC / 100

    总堆内存 = 存活堆 + 新堆内存

    ⇒

    新堆内存 = (存活堆 + GC 根) * GOGC / 100

    由此我们可以看出，将 GOGC 翻倍也会使应用程序每个周期分配的新堆内存量翻倍，这反映了堆内存的开销。请注意，存活堆 + GC 根 是对垃圾回收器需要扫描的内存量的近似值。

    接下来，让我们看看垃圾回收的 CPU 开销。总开销可以分解为每个周期的开销乘以在某个时间段 T 内的垃圾回收频率。

    总垃圾回收 CPU 开销 = (每个周期的垃圾回收 CPU 开销) * (垃圾回收频率) * T

    每个周期的垃圾回收 CPU 开销可以从垃圾回收模型推导得出：

    每个周期的垃圾回收 CPU 开销 = (存活堆 + GC 根) * (每字节开销) + 固定开销

    请注意，这里忽略了清扫阶段的开销，因为标记和扫描的开销占主导地位。

    稳态由恒定的分配速率和恒定的每字节开销定义，因此在稳态下，我们可以从这个新堆内存推导出垃圾回收频率：

    垃圾回收频率 = (分配速率) / (新堆内存) = (分配速率) / ((存活堆 + GC 根) * GOGC / 100)

    综合起来，我们得到总开销的完整公式：

    总垃圾回收 CPU 开销 = (分配速率) / ((存活堆 + GC 根) * GOGC / 100) * ((存活堆 + GC 根) * (每字节开销) + 固定开销) * T

    对于足够大的堆（这代表了大多数情况），垃圾回收周期的边际成本主导着固定成本。这使得总垃圾回收 CPU 开销的公式可以大大简化。

    总垃圾回收 CPU 开销 = (分配速率) / (GOGC / 100) * (每字节开销) * T

    从这个简化公式中我们可以看出，如果将 GOGC 的值加倍，GC 的总 CPU 开销就会减半。（请注意，本指南中的可视化模拟了固定开销，因此当 GOGC 翻倍时，它们报告的 GC CPU 开销并不会精确减半。）此外，GC 的 CPU 开销主要取决于分配速率和扫描内存的每字节开销。有关如何具体降低这些开销的更多信息，请参阅优化指南。

    注意：活跃堆的大小与 GC 实际需要扫描的内存量之间存在差异：相同大小的活跃堆，若其结构不同，会导致不同的 CPU 开销，但内存开销相同，从而产生不同的权衡。这就是为什么堆的结构是稳定状态定义的一部分。堆目标理论上应该只包含可扫描的活跃堆，作为 GC 需要扫描的内存的更近似值，但当可扫描的活跃堆量非常少而活跃堆本身很大时，这会导致退化行为。
